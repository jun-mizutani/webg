# モデルアセットとランタイム

本章では、`ModelAsset` がどのような役割を持ち、どの段階で検証され、最終的にランタイムの `Node`、`Shape`、`Skeleton`、`Animation` へと展開されるのかを解説します。

3Dモデルの読み込みにおいては、glTFやCollada、あるいは独自形式のJSONなど、入力形式によってデータの保持方法に大きな差が生じがちです。これらの形式をそのままランタイムに直結させると、利用方法まで形式ごとに依存してしまい、実装の共通化が困難になります。そこで `webg` では、形式ごとの差異を一度 `ModelAsset` という共通形式で吸収する設計を採用しています。これにより、その後の `build()`、`instantiate()`、およびアニメーション再生といった処理を、入力形式に関わらず同一の手順で扱うことが可能になります。利用者は「現在扱っているのがインポート済みのデータなのか」、あるいは「描画可能なランタイム状態なのか」を明確に区別して管理できます。

ここで最も重要な点は、`ModelAsset` は「描画オブジェクト」ではなく、CPU側で正規化された「共通データ」であるということです。`ModelAsset` 自体はGPUバッファや描画状態を保持しません。保持しているのは、`materials`、`meshes`、`skeletons`、`animations`、`nodes` といったJSON互換のデータ構造です。実際の `Shape`、`Skeleton`、`Animation`、`Node` といったランタイムオブジェクトは、後述する `build()` と `instantiate()` の段階で初めて生成されます。

また、`build()` と `instantiate()` は明確に役割が異なります。`build()` はアセット全体をランタイム化するための準備段階であり、メッシュから共有の Shape リソースを構築し、Node 定義やアニメーション定義をランタイム用に整理します。対して `instantiate()` は、その build 済みランタイムから新しい `Node`、`Shape インスタンス`、`Skeleton`、`Animation` を生成し、`Space` へ配置する段階を指します。同じモデルをシーン内に複数配置する場合、`build()` を1回だけ行い、`instantiate()` を必要な回数だけ呼び出すのが基本となります。

さらに、`validate()` メソッドと `ModelValidator` クラスは、データの整合性を早期に確認するために提供されています。id 参照の不整合、配列長の不一致、行列データの不足などを `build()` 前に検出することで、「モデルが画面に表示されない」「アニメーションが動作しない」といった問題をランタイム側まで持ち込まずに済みます。読み込み直後に `assertValid()` を実行するフローを組み込むことが推奨されます。

加えて、`scaleUniform()` はインポート済みデータ全体の寸法を統一するための重要なヘルパーです。`ModelAsset.scaleUniform(scale)` を使用すると、メッシュの頂点だけでなく、`Node` の translation、`Skeleton` のジョイント行列、`Animation` のポーズの translation も一括して拡大縮小されます。特にスキニングメッシュの場合、ジオメトリのみを縮小するとポーズとの整合性が崩れるため、このヘルパーを用いてアセット全体を同一の規則で処理することが安全です。

このように、インポーターごとの差異は `ModelLoader` で吸収され、利用者は `ModelAsset` とランタイム側の共通インターフェースのみを扱うことになります。glTF、Collada、ModelAsset JSON と入口が異なっても、`ModelLoader.load()` や `WebgApp.loadModel()` を通じて、`asset`、`runtime`、`instantiate()`、`getClipNames()` といった共通の操作体系に統合されます。

本章ではまず `ModelAsset` の必要性を深く掘り下げ、その後に `validate()` $\rightarrow$ `build()` $\rightarrow$ `instantiate()` という一連の流れを詳説します。続いて、高レイヤー（ハイレベル）経路と低レイヤー（ローレベル）経路という2つのエントリーポイントを整理し、具体的なコード例を通して、複数配置、検証、スケール調整、クリップ確認、JSON保存の実装方法を確認します。最後に、`ModelAsset` の内部構造、`build()` の実行結果、および変更時の注意点について解説します。

## なぜ `ModelAsset` を挟むのか

3Dモデルの入力形式は、外見上の類似性に反して内部構造が大きく異なります。メッシュの保持方法、マテリアルの参照形式、スキンとスケルトンの結び付け方、アニメーションの表現手法などは、glTF と Collada の間でも相違があります。これらの差異をそのままアプリケーション側へ持ち込むと、モデルを1つ表示させるだけでもインポーターごとに異なる理解が必要となり、コードの複雑化を招きます。

`webg` がランタイムの手前に `ModelAsset` を配置している理由は、こうした形式間の差異を吸収し、「ファイルをどう読み込むか」というインポート処理と、「読み込んだモデルをシーンでどう活用するか」という利用処理を完全に切り離すためです。これにより、開発者がコードを解析する場合や、生成AIを含む支援ツールがサンプルとコアを横断して調査する場合でも、`ModelAsset` を確認すれば「モデルがどのような構造に正規化されたか」を一貫して追跡できます。

具体的に `ModelAsset` は以下の4つの役割を担っています。
1. ファイル形式ごとの差異を単一のJSON互換構造へ正規化する。
2. バリデーターが整合性を確認しやすい形式でデータを保持する。
3. ビルダーが共有リソースとランタイム定義を効率的に組み立てられる形式でデータを渡す。
4. 利用者がクリップ一覧の取得、JSON保存、スケール調整を共通のインターフェースで操作できるようにする。

ここで重要なのは、`ModelAsset` は「見た目そのもの」ではなく、「見た目を構築するための共通設計図」であるという点です。実際の描画は `Shape` や `Node` が担当しますが、その基となるメッシュ、マテリアル、スケルトン、アニメーション、Node 階層の定義を統合的に管理しているのが `ModelAsset` です。

また、この構造を採用することで、`build()` 前に検証を行えるメリットが得られます。id の参照切れや配列長の不整合を JSON 段階で検出できれば、ランタイム生成後のデバッグよりも遥かに迅速に原因を特定できます。

さらに、build 済みのランタイムと `instantiate()` 後のシーンインスタンスを分離できる点も大きな利点です。現在の `webg` の実装では、同一のメッシュを複数の `Node` で使用する場合に共有リソースを使い回し、`instantiate()` ごとに個別の `Shape インスタンス`、`Skeleton`、`Animation` を生成します。この構造により、同一モデルを多数配置してもGPUバッファの消費を抑えつつ、個々のモデルが独立したアニメーション状態を持つことが可能になります。

加えて、インポート後のデータを JSON として保存し、後から比較・検証できる点も実用的です。`downloadJSON()` を使用して glTF インポーターの出力を `modelasset.json` として保存すれば、インポーターの差分確認やドキュメント作成に活用できます。

`ModelAsset` の有用性が特に顕著に現れるのは、以下のようなケースです。
- glTF や Collada を `webg` ランタイムへ読み込む際。
- インポーターの出力を JSON で保存し、内部構造を確認したい際。
- 手書きの `modelasset.json` を用いて最小構成の 3D モデルを構築したい際。
- 同一モデルを複数配置し、共有リソースの活用と独立したアニメーション制御を両立させたい際（例：`janken` サンプルにおいて、プレイヤーとCPUで同一のハンドモデルを使用しつつ、アニメーション状態のみを分ける場合）。
- インポート済みモデル全体に一定の倍率でスケールを適用したい際（例：`tile_sim` サンプルにおいて、インポートしたアクターのサイズを論理的に統一する場合）。

なお、利用にあたっては以下の点に注意してください。
- `ModelAsset` 自体はシーンに配置されるオブジェクトではありません。描画するには `build()` と `instantiate()` が必須です。
- `build()` の戻り値は「インスタンス化可能な状態のランタイム」であり、まだ `Space` の Node ツリーには組み込まれていません。
- `runtime.instantiate()` は呼び出しごとに新しいインスタンスを返すため、インスタンス間でアニメーション状態は共有されません。
- `scaleUniform()` はジオメトリだけでなく、スケルトンとアニメーションの translation にも影響を与えます。
- `downloadJSON()` はブラウザ環境向けのヘルパーであり、Node.js スクリプト等の環境での保存には適していません。
- `node.matrix` が定義されている場合は `node.transform` より優先されます。matrix と TRS（Translation, Rotation, Scale）を併記する場合は、どちらが実際に適用されるかを意識する必要があります。

## 標準フローと 2 つの入口

![ModelAsset build instantiate](fig10_01_modelasset_build_instantiate.jpg)
*ModelAsset は異なる入力形式を共通の build() / instantiate() フローへ統合する入口であり、通常 build() は 1 回、instantiate() は必要な数だけ行います。*

`ModelAsset` を利用するフローには、大きく分けて高レイヤー（ハイレベル）経路と低レイヤー（ローレベル）経路の 2 つが存在します。

通常のサンプル実装では、`WebgApp.loadModel()` または `ModelLoader.load()` を使用する高レイヤー経路が推奨されます。この経路では、形式判定、インポート、整合性検証、`build()`、そして必要に応じた `instantiate()` までを一括して実行できます。

```js
const result = await app.loadModel("./human.glb", {
  instantiate: false,
  validate: true
});

const runtime = result.runtime;
const actorA = runtime.instantiate(app.space);
const actorB = runtime.instantiate(app.space);
```

この例では、`human.glb` のインポートと `build()` を 1 回だけ行い、得られた `runtime` から 2 体のインスタンスを生成しています。`actorA` と `actorB` はそれぞれ独立した `Node`、`Shape インスタンス`、`Skeleton`、`Animation` を持ちますが、メッシュ由来の共有リソースは build 結果側で共有されます。

一方で、アセットを手動で編集する場合や、整合性検証および `build()` の各段階を詳細に制御したい場合は、`ModelAsset` を直接操作する低レイヤー経路が適しています。

```js
import ModelAsset from "./webg/ModelAsset.js";

const asset = await ModelAsset.load("./modelasset.json");
asset.assertValid();

const runtime = asset.build(app.getGL());
const instantiated = runtime.instantiate(app.space);
runtime.startAllAnimations();
```

この経路では `ModelAsset` オブジェクトを直接保持するため、`build()` を実行する前に `getClipNames()`、`downloadJSON()`、`scaleUniform()` などのヘルパーメソッドを利用できます。「まず JSON 構造を確認したい」「含まれているクリップ名を把握したい」といった用途に最適です。

`ModelAsset` を利用する際は、以下の順序で処理を構成すると効率的です。
1. インポーターまたは JSON ファイルから `ModelAsset` を取得する。
2. `validate()` または `assertValid()` でデータの整合性を確認する。
3. `build()` を実行し、共有リソースを含むランタイムを構築する。
4. `instantiate()` を呼び出し、新しいシーンインスタンスを `Space` へ配置する。
5. 必要に応じて `startAnimation()` や `startAllAnimations()` でクリップを再生する。

このように `ModelAsset` を「正しい形式に整えられたモデルデータ」として定義しておくことで、glTF や手書き JSON など、どのような入力源であっても一貫した手法でモデルを扱うことが可能になります。

## 使い方のコード例

### glTF を読み込み、同じモデルを複数配置する

同一の glTF モデルを複数配置する場合、`loadModel()` を複数回呼び出すのではなく、一度読み込んで `runtime.instantiate()` を複数回呼び出す手法が効率的です。

```js
const result = await app.loadModel("./hand.glb", {
  instantiate: false,
  validate: true
});

const player = result.runtime.instantiate(app.space);
const cpu = result.runtime.instantiate(app.space);

const playerNode = player.nodeMap.get("HandRoot");
const cpuNode = cpu.nodeMap.get("HandRoot");

playerNode.setPosition(-1.2, 0.0, 0.0);
cpuNode.setPosition( 1.2, 0.0, 0.0);

player.startAllAnimations();
cpu.startAllAnimations();
```

この実装では、`hand.glb` のジオメトリや GPU バッファは build 側で共有されます。一方で、`player` と `cpu` は別々のインスタンスであるため、アニメーションの再生位置や一時停止状態などの状態管理は互いに干渉しません。配置するモデル数が増えるほど、この手法によるメモリ効率とパフォーマンスの向上が顕著になります。

### インポート済み JSON を検証してから build する

手書きの JSON やエクスポーターの出力を利用する場合、描画前に `validate()` を実行することで、不整合によるランタイムエラーを未然に防ぐことができます。

```js
const asset = await ModelAsset.load("./modelasset.json");
const report = asset.validate();

if (!report.ok) {
  console.error(report.errors);
  throw new Error("Invalid ModelAsset");
}

const runtime = asset.build(app.getGL());
runtime.instantiate(app.space);
```

`validate()` は `{ ok, errors, warnings }` という形式のレポートを返します。エラーが検出された場合に `build()` を中断させることで、id 参照ミスや行列データの不足を早期に発見できます。インポーターの開発やアセット調整の段階では、このレポートを確認しながら修正を行うフローが推奨されます。

### インポート済みモデル全体へ一定倍率を焼き込む

アセット全体のサイズをインポート後に調整したい場合は、ジオメトリだけでなくスキンとアニメーションも含めて同一の規則でスケールさせる必要があります。`scaleUniform()` はそのための専用ヘルパーです。

```js
const asset = await ModelAsset.load("./human.json");
asset.assertValid();

const scaledAsset = ModelAsset.fromData(
  asset.cloneJSONValue(asset.getData())
);

scaledAsset.scaleUniform(0.70);

const runtime = scaledAsset.build(app.getGL());
runtime.instantiate(app.space);
```

上記の例でデータの複製（`cloneJSONValue`）を行っているのは、元のアセットを保持したまま比較検証を行うためです。`scaleUniform(0.70)` は、メッシュ頂点、`Node` の translation、`Skeleton` のジョイント行列、`Animation` のポーズの translation を一括して 0.70 倍します。スキニングモデルにおいてジオメトリのみを縮小させるとポーズが崩れるため、アセット全体に同一倍率を適用する手法が最も安全です。

### クリップ一覧を確認し、要約情報を取得する

複数のアニメーションを含むモデルを扱う際は、事前にクリップ名や長さを把握しておくことで、サンプル実装をスムーズに進めることができます。

```js
const result = await app.loadModel("./robot.glb", {
  instantiate: false
});

for (const clipName of result.getClipNames()) {
  const info = result.getClipInfo(clipName);
  console.log(clipName, info.durationMs, info.trackCount);
}
```

`getClipInfo()` は `id`、`targetSkeleton`、`keyCount`、`trackCount`、`durationMs` を返します。ランタイムをシーンに配置する前でもこれらの情報を取得できるため、「どのクリップを UI に表示させるか」や「ドキュメントに記載するクリップ名の特定」に活用できます。

### インポーターの出力を JSON として保存する

glTF や Collada インポーターが内部的にどのようにデータを処理したかを確認したい場合は、`downloadJSON()` を使用して `ModelAsset` をファイルとして保存できます。

```js
const result = await app.loadModel("./vehicle.glb", {
  instantiate: false
});

result.downloadJSON("vehicle.modelasset.json");
```

保存された JSON を解析することで、インポート後のメッシュ、スケルトン、アニメーション、`Node` がどのように正規化されたかを詳細に追跡でき、インポーターのデバッグや仕様策定に役立ちます。

### `Shape.createInstance()` による共有ジオメトリの再利用

`ModelAsset.build() -> instantiate()` で採用されている「共有リソース」の考え方は、低レイヤー（ローレベル）で `Shape` を直接組み立てる際にも適用可能です。`ModelAsset` を介さない実装であっても、「ジオメトリは一度だけ構築し、色や位置、スケールのみを変更したインスタンスを複数配置する」構成にすることで、GPU バッファの重複生成を回避できます。

`webg/samples/dof` はこの実例です。被写界深度の比較に使用される球体はすべてセグメント数が同一であるため、半径 1.0 のユニットスフィアを一度だけ作成し、各 `Node` で `Shape.createInstance()` と `node.setScale(radius)` を組み合わせて半径の差を表現しています。マテリアルはインスタンスごとに個別設定が可能なため、ジオメトリを共有したまま色のみを変更できます。

```js
const sphereSource = new Shape(app.getGL());
sphereSource.applyPrimitiveAsset(
  Primitive.sphere(1.0, 28, 20, sphereSource.getPrimitiveOptions())
);
sphereSource.endShape();

function addSphere(name, options) {
  const shape = sphereSource.createInstance();
  shape.setMaterial("smooth-shader", {
    use_texture: 0,
    color: options.color,
    ambient: 0.70,
    specular: 1.10,
    power: 58.0
  });

  const node = app.space.addNode(null, name);
  node.setPosition(options.x, options.y, options.z);
  node.setScale(options.radius);
  node.addShape(shape);
  return node;
}
```

この実装の要点は、半径の差をジオメトリの再生成で解決せず、`Node` の一様スケールで調整している点にあります。`Shape` インスタンスはマテリアルや表示/非表示状態を個別に保持できますが、ジオメトリ、GPU バッファ、静的バウンディングボックスは共有リソースを参照します。つまり、「見た目の差分のみをインスタンス側に持たせ、負荷の高いメッシュ構築は共有側に集約する」という役割分担になります。

`dof` サンプルでは、この共有ジオメトリ化により、診断情報の集計結果が劇的に改善されています。インスタンス化前は `vertexCount=12252`、`triangleCount=22436`、`meshCount=23` であったものが、インスタンス化後は `vertexCount=681`、`triangleCount=1156`、`meshCount=4` まで削減されました。`shapeCount` が不変なのはシーン上のインスタンス数自体は変わらないためですが、`meshCount` や `vertexCount` は一意な共有リソースのみをカウントするため、最適化の効果が明確に現れています。同一のプリミティブを多数配置する際は、まずこの手法を検討することが推奨されます。

## `build()` の結果と `ModelAsset` の中身を見る

`ModelAsset.build(gpu)` の戻り値は、単なるデータの集合ではなく、以下の 3 層構造で構成されています。

1. アセット由来の定義: `runtime.nodes` および `runtime.nodeMap` に格納される、アセットから変換された Node 定義。
2. build 済みの共有リソース: `runtime.shapeResources` に格納される、複数の Node で共有される描画リソース群。
3. instantiate 後のランタイムインスタンス: `runtime.instantiate(space)` 呼び出しによって生成される、個別の `Node`、`Shape インスタンス`、`Skeleton`、`Animation`。

この構造により、`build()` の結果をキャッシュし、必要なタイミングで `instantiate()` を呼び出す効率的な運用が可能になります。逆に、モデルを配置するたびに `loadModel()` を呼び出すと、インポートと `build()` が繰り返されるため、同一メッシュを大量に使用する場面ではパフォーマンス上の不利となります。

`ModelAsset` のトップレベル構造（JSON スキーマ）は、主に以下の項目で構成されています。

```json
{
  "version": "1.0",
  "type": "webg-model-asset",
  "meta": {},
  "materials": [],
  "meshes": [],
  "skeletons": [],
  "animations": [],
  "nodes": []
}
```

各項目の役割は以下の通りです。
- `materials`: メッシュが参照する材質定義。`shaderParams` に `color`、`ambient`、`specular`、`power` などを指定し、`build()` 時に `Shape` 側へ反映されます。
- `meshes`: ジオメトリ、マテリアル参照、および必要に応じたスキン情報を保持します。`positions` と `indices` が必須であり、`normals` と `uvs` は任意です。ここから共有の `Shape` リソースが構築されます。
- `skeletons`: ジョイント階層、レストポーズ、Inverse Bind Matrix を定義します。スキニングメッシュを動作させるための骨格定義であり、アニメーションの操作対象となります。
- `animations`: クリップ定義。`targetSkeleton`、`times`、`tracks` を持ち、特定のスケルトンをどのように動作させるかを定義します。なお、状態遷移や再生順序の制御はランタイム側で管理します。
- `nodes`: モデル内の配置単位。使用するメッシュ、結び付けるスケルトン、およびローカル変換を定義します。`node.parent` は Node id を参照し、シーングラフの親子関係を構築します。

`ModelAsset` の理解を深めるには、スキーマだけでなく `validate()` と `build()` の動作を併せて把握することが重要です。`validate()` は、トップレベルの `type` や `version`、id の重複、参照整合性、ジオメトリ配列長、行列長、アニメーショントラックの整合性を検証し、以下の形式で結果を返します。

```js
{
  ok: true,
  errors: [],
  warnings: []
}
```

`errors` は `build()` の実行を妨げる致命的な不整合であり、`warnings` は実行は可能であるものの確認が推奨される項目です。

`build()` は、検証を通過したアセットからランタイムを構築します。具体的には、メッシュから共有 `Shape` リソースを作成し、Node ごとに `Shape` テンプレートを紐付け、スケルトンとアニメーションの構築情報を保持します。この段階ではまだシーンへの配置は行われません。

最終的に `instantiate()` が呼ばれた時点で、初めて `Space.addNode()` が実行され、`Shape` インスタンスが Node に付加され、`Skeleton` と `Animation` がインスタンスごとに個別に生成されます。これにより、同一の build 結果から生成された複数のインスタンスであっても、それぞれが独立したアニメーション再生状態を保持できます。

## 例を通して確かめる

`ModelAsset` の役割を具体的に確認するには、以下のサンプルコードおよびテストケースの参照が有効です。

- `webg/samples/gltf_loader`: glTF のインポート、クリップ情報の取得、および JSON 保存の動作を確認できます。
- `webg/samples/collada_loader`: Collada 形式から `ModelAsset` へ正規化されるフローを確認できます。
- `webg/samples/json_loader`: 手書きの JSON を `ModelAsset` として直接読み込む例です。
- `webg/samples/janken`: 同一の build 済みランタイムから複数の `instantiate()` を行い、アニメーション状態を独立して制御する実例です。
- `webg/samples/tile_sim`: インポート済みアクターのサイズ調整および複数配置の実装例です。
- `webg/unittest/primitive_modelasset`: `Primitive` $\rightarrow$ `ModelAsset` $\rightarrow$ `validate` $\rightarrow$ `build` という最小構成のパイプラインを確認できるため、構造の理解に最適です。

## 変更時の注意点

`ModelAsset` 関連の機能を変更・拡張する際は、影響範囲が広いため、以下の要素をセットで確認してください。

- スキーマ変更時: `ModelValidator` および `ModelBuilder` の修正が必要です。
- メッシュ・スキン変更時: ジオメトリ処理およびスキニング反映ロジックを確認してください。
- スケルトン変更時: レストポーズ、Inverse Bind Matrix、およびアニメーションの結び付け処理を確認してください。
- アニメーション変更時: トラックの整合性検証およびランタイムのクリップ生成ロジックを確認してください。
- 高レイヤー API 変更時: `ModelLoader` および各サンプルの実装への影響を確認してください。

特に、「`build()` 結果で共有すべきリソース」と「`instantiate()` ごとに独立させるべき状態」の分離を厳格に維持することが重要です。GPU バッファ、ジオメトリ、静的バウンディングボックスは共有側に集約し、アニメーション再生状態、スケルトンの現在ポーズ、マテリアルの個別差分、および表示/非表示状態などのランタイム状態は必ずインスタンス側で独立して保持させる必要があります。

## 参考用の最小例

最小構成の `ModelAsset` は、`type`、`version`、`meshes`、`nodes` を必須項目として持ち、`mesh.geometry.positions` と `mesh.geometry.indices` が正しく定義されていれば `build()` が可能です。スケルトンやアニメーションは任意項目であり、必須ではありません。

```json
{
  "version": "1.0",
  "type": "webg-model-asset",
  "meta": {
    "name": "triangle"
  },
  "materials": [
    {
      "id": "mat0",
      "shaderParams": {
        "color": [0.4, 0.8, 1.0, 1.0],
        "ambient": 0.3,
        "specular": 0.4,
        "power": 24.0
      }
    }
  ],
  "meshes": [
    {
      "id": "mesh0",
      "name": "triangle",
      "material": "mat0",
      "geometry": {
        "vertexCount": 3,
        "polygonCount": 1,
        "positions": [-1, -1, 0, 1, -1, 0, 0, 1, 0],
        "normals": [0, 0, 1, 0, 0, 1, 0, 0, 1],
        "uvs": [0, 0, 1, 0, 0.5, 1],
        "indices": [0, 1, 2]
      }
    }
  ],
  "nodes": [
    {
      "id": "node0",
      "name": "triangleNode",
      "parent": null,
      "mesh": "mesh0",
      "transform": {
        "translation": [0, 0, 0],
        "rotation": [0, 0, 0, 1],
        "scale": [1, 1, 1]
      }
    }
  ]
}
```

この最小例は、手書き JSON によるテストの出発点として適しています。まずはアニメーションやスケルトンを含まないメッシュと Node の基本経路を確認し、その後段階的にスキンやクリップを追加することで、問題の切り分けを容易に行うことができます。
