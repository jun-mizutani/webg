# モデルアセットとランタイム

この章では、`ModelAsset` が何を表し、どの段階で検証され、どの段階でランタイムの `Node`、`Shape`、`Skeleton`、`Animation` へつながるのかを見ていきます。3D モデルの読み込みでは、glTF、Collada、手書き JSON など、入口となる形式ごとの差が大きくなりがちです。それらをそのままランタイムへ直結すると、使い方まで形式ごとにばらばらになりやすくなります。`webg` では、その差を 1 回 `ModelAsset` で吸収し、その後の `build()`、`instantiate()`、アニメーション再生を同じ見方で扱えるようにしています。これにより、「今見ているのはインポート済みデータなのか」「もう描画できるランタイムなのか」を読み分けやすくなります。 

この章で最初に押さえたいのは、`ModelAsset` は「描画オブジェクト」ではなく、CPU 側で正規化された共通データだということです。`ModelAsset` 自体は GPU バッファや描画状態を持ちません。保持しているのは、`materials`、`meshes`、`skeletons`、`animations`、`nodes` などの JSON 互換データです。実際の `Shape`、`Skeleton`、`Animation`、`Node` は `build()` と `instantiate()` の段階で初めて作られます。 

もうひとつ重要なのは、`build()` と `instantiate()` は役割が違うという点です。`build()` はアセット全体をランタイム化する準備段階であり、ここでメッシュから共有の Shape リソースを作り、Node 定義やアニメーション定義をランタイム用に整理します。`instantiate()` は、その build 済みランタイムから新しい `Node`、`ShapeInstance`、`Skeleton`、`Animation` を生成して `Space` へ配置する段階です。同じモデルを複数置くときは、`build()` を 1 回だけ行い、`instantiate()` を必要な回数だけ呼ぶのが基本になります。さらに、`validate()` は「壊れていないか」を早い段階で確認するためにあり、`ModelValidator` は id 参照の不整合、配列長の不一致、行列長の不足などを build 前に見つけるための層です。読み込み直後に `assertValid()` を通しておくと、「画面に出ない」「アニメーションが動かない」といった問題をランタイム側まで持ち込まずに済みます。 

また、`scaleUniform()` はインポート済みデータ全体の寸法をそろえるための入口です。`ModelAsset.scaleUniform(scale)` は、メッシュの頂点だけでなく、`Node` の translation、`Skeleton` のジョイント行列、`Animation` のポーズの translation もまとめて拡大縮小します。スキニングメッシュではジオメトリだけを縮めてもポーズが合わなくなるため、このヘルパーを使ってアセット全体を同じ規則で扱うのが安全です。利用者の視点では、インポーターの違いは `ModelLoader` で吸収し、`ModelAsset` とランタイム側の共通入口を見る、という考え方が自然です。glTF、Collada、ModelAsset JSON は入口が違っても、`ModelLoader.load()` や `WebgApp.loadModel()` を通すと、`asset`、`runtime`、`instantiate()`、`getClipNames()` といった共通の見え方へそろいます。サンプル側ではインポーターの細部を追うより、この共通の見え方を基準に考えるほうが読みやすくなります。 

この章では、まず `ModelAsset` がなぜ必要なのかを確認し、そのあとで `validate()`、`build()`、`instantiate()` の流れを見ます。続いて、高レベル経路とローレベル経路の 2 つの入口を整理し、具体的なコード例を通して複数配置、検証、スケール調整、クリップ確認、JSON 保存の使い方を見ていきます。最後に、`ModelAsset` の中身、`build()` の結果、変更時の注意点、最小例を確認します。 

## なぜ `ModelAsset` を挟むのか

3D モデルの入力形式は、見た目は似ていても中身の持ち方がかなり違います。メッシュがどこにあり、マテリアルがどう参照され、スキンとスケルトンがどう結び付き、アニメーションがどう表現されるかは、glTF と Collada でも差があります。その差をサンプル側へそのまま持ち込むと、モデルを 1 つ表示するだけでもインポーターごとに別の理解が必要になります。`webg` では、そうした差をランタイムの手前で吸収するために `ModelAsset` を置いています。これにより、「ファイルをどう読むか」と「読んだモデルをシーンでどう使うか」を切り分けられます。人間がコードを読むときも、生成 AI を含む支援ツールがサンプルとコアを横断して調べるときも、まず `ModelAsset` を見れば「このモデルがどのような構造へ正規化されたか」を追えるようになります。 

`ModelAsset` の役割は大きく 4 つあります。ファイル形式ごとの差を 1 つの JSON 互換構造へ正規化すること、バリデーターが整合性を確認しやすい形でデータを保持すること、ビルダーが共有リソースとランタイム定義を組み立てやすい形で渡すこと、サンプルや利用者がクリップ一覧、JSON 保存、スケール調整を同じ入口で扱えるようにすることです。ここで大事なのは、`ModelAsset` が「見た目そのもの」ではなく、「見た目を作るための共通データ」だという点です。実際の描画は `Shape` や `Node` が行いますが、その元になるメッシュ、マテリアル、スケルトン、アニメーション、Node 階層の定義をまとめているのが `ModelAsset` です。 

`ModelAsset` を間に挟む理由は、単にインポートを楽にするためだけではありません。まず、検証を `build()` より前に行えることが重要です。id の参照切れや配列長の不整合は、ランタイムを作ってから見つけるより、JSON 段階で見つけたほうが原因を追いやすくなります。次に、build 済みのランタイムと `instantiate()` 後のシーンインスタンスを分けられることが重要です。現在の `webg` 実装では、同じメッシュを複数の `Node` で使うときに共有リソースを共有し、`instantiate()` ごとに新しい `ShapeInstance`、`Skeleton`、`Animation` を作れるようになっています。この構造により、同じモデルを多数配置しても GPU バッファを使い回しながら、アニメーション状態は独立して持てます。さらに、インポート後のデータを JSON として保存し、後から比較できることも大きな利点です。`downloadJSON()` を使えば、glTF インポーターの出力を `modelasset.json` として残せるため、インポーターの差分確認やサンプルの説明にも使いやすくなります。 

`ModelAsset` は、次のような場面で使うと効果が分かりやすくなります。glTF や Collada を `webg` のランタイムへ読み込むとき、インポーターの出力を JSON として保存し内容を確認したいとき、手書きの `modelasset.json` を使って最小の 3D モデルを作りたいとき、同じモデルを複数配置し共有リソースと独立したアニメーションを両立したいとき、インポート済みモデル全体へ一定倍率のスケールを焼き込みたいときです。特に、`janken` のように同じ hand model を player と cpu で使いながらアニメーション状態だけを分けたい場面では、`build()` を 1 回、`instantiate()` を複数回という流れが自然です。`tile_sim` のようにインポート済み actor のサイズを論理的にそろえたい場面では、`ModelAsset.scaleUniform()` とランタイム側のスケールを組み合わせる考え方が役立ちます。 

先に意識しておきたい注意点もあります。`ModelAsset` 自体はシーンに配置されません。画面へ出したいときは `build()` と `instantiate()` が必要です。`build()` の戻り値は「インスタンス化可能な状態のランタイム」であり、まだ `Space` の `Node` ツリーではありません。`runtime.instantiate()` は毎回新しいインスタンスを返すため、別インスタンス間でアニメーション状態は共有されません。`scaleUniform()` はジオメトリだけでなくスケルトンとアニメーションの translation も変えるため、インポート後データ全体に影響します。`downloadJSON()` はブラウザ側で使うヘルパーであり、Node.js のスクリプトから同じ方法で保存する用途には向いていません。また、`node.matrix` がある場合は `node.transform` より優先されます。matrix と TRS を併記するときは、どちらが実際に使われるかを意識する必要があります。 

## 標準フローと 2 つの入口
<!-- 図候補: 図10-1 ModelAsset -> build -> instantiate 図 -->

![図10-1 ModelAsset build instantiate](fig10_01_modelasset_build_instantiate.jpg)

*図10-1 `ModelAsset` は異なる入力形式を共通の `build()` / `instantiate()` フローへそろえる入口であり、通常 `build()` は 1 回、`instantiate()` は必要な数だけ行います。*

`ModelAsset` を使う流れは、大きく分けて高レベル経路とローレベル経路の 2 つがあります。普段のサンプルでは、`WebgApp.loadModel()` または `ModelLoader.load()` を使うのが素直です。この経路では、形式判定、インポート、整合性検証、`build()`、必要なら `instantiate()` までをまとめて行えます。 

```js
const result = await app.loadModel("./human.glb", {
  instantiate: false,
  validate: true
});

const runtime = result.runtime;
const actorA = runtime.instantiate(app.space);
const actorB = runtime.instantiate(app.space);
```

この例では、`human.glb` のインポートと `build()` は 1 回だけ行い、その結果の `runtime` から 2 体ぶんの新しいインスタンスを生成しています。`actorA` と `actorB` は別々の `Node`、`ShapeInstance`、`Skeleton`、`Animation` を持ちますが、メッシュ由来の共有リソースは build 結果側で共有されます。 

アセットを手で編集したい場合や、整合性検証や `build()` を段階ごとに確認したい場合は、`ModelAsset` を直接扱うほうが分かりやすくなります。 

```js
import ModelAsset from "./webg/ModelAsset.js";

const asset = await ModelAsset.load("./modelasset.json");
asset.assertValid();

const runtime = asset.build(app.getGL());
const instantiated = runtime.instantiate(app.space);
runtime.startAllAnimations();
```

この経路では、`ModelAsset` を自分で保持しているため、`getClipNames()`、`downloadJSON()`、`scaleUniform()` などのヘルパーを `build()` 前に使えます。サンプルを作るときに「まず JSON を確認したい」「どのクリップが入っているかを知りたい」といった用途に向いています。`ModelAsset` を使うときは、インポーターまたは JSON から `ModelAsset` を得る、`validate()` または `assertValid()` で整合性を確認する、`build()` で共有リソースを含むランタイムを作る、`instantiate()` で新しいシーンインスタンスを `Space` へ置く、必要なら `startAnimation()` や `startAllAnimations()` でクリップを再生する、という順で考えると迷いにくくなります。つまり、`ModelAsset` は「今すぐ画面に出るもの」ではなく、「正しい形にそろえたモデルデータ」です。ここを最初にそろえておくと、glTF を読んだときも、手書き JSON を扱うときも、サンプルとコアの説明がぶれにくくなります。 

## 使い方のコード例

### glTF を読み込み、同じモデルを複数配置する

同じ glTF を複数配置したいときは、`loadModel()` を 2 回呼ぶより、1 回読み込んで `runtime.instantiate()` を複数回呼ぶほうが自然です。 

```js
const result = await app.loadModel("./hand.glb", {
  instantiate: false,
  validate: true
});

const player = result.runtime.instantiate(app.space);
const cpu = result.runtime.instantiate(app.space);

const playerNode = player.nodeMap.get("HandRoot");
const cpuNode = cpu.nodeMap.get("HandRoot");

playerNode.setPosition(-1.2, 0.0, 0.0);
cpuNode.setPosition( 1.2, 0.0, 0.0);

player.startAllAnimations();
cpu.startAllAnimations();
```

この例では、`hand.glb` のジオメトリや GPU バッファは build 側で共有されます。一方で、`player` と `cpu` は別のインスタンスなので、アニメーションの再生位置や一時停止状態は互いに干渉しません。大量配置したいモデルほど、この使い方の効果が大きくなります。 

### インポート済み JSON を検証してから build する

手書き JSON やエクスポーターの出力をそのまま使うときは、画面に出す前に `validate()` を通しておくと原因を追いやすくなります。 

```js
const asset = await ModelAsset.load("./modelasset.json");
const report = asset.validate();

if (!report.ok) {
  console.error(report.errors);
  throw new Error("Invalid ModelAsset");
}

const runtime = asset.build(app.getGL());
runtime.instantiate(app.space);
```

`validate()` は `{ ok, errors, warnings }` を返します。エラーがあると `build()` 前に止められるため、id 参照のミスや行列長の不足を早い段階で見つけられます。サンプルやインポーターを作っている途中は、まずこの形でレポートを見たほうが状況を整理しやすくなります。 

### インポート済みモデル全体へ一定倍率を焼き込む

アセット全体のサイズをインポート後にそろえたいときは、ジオメトリだけでなくスキンとアニメーションも含めて同じ規則で扱う必要があります。`scaleUniform()` はそのためのヘルパーです。 

```js
const asset = await ModelAsset.load("./human.json");
asset.assertValid();

const scaledAsset = ModelAsset.fromData(
  asset.cloneJSONValue(asset.getData())
);

scaledAsset.scaleUniform(0.70);

const runtime = scaledAsset.build(app.getGL());
runtime.instantiate(app.space);
```

ここで複製を挟んでいるのは、元のアセットを残したまま比較できるようにするためです。`scaleUniform(0.70)` は、メッシュ頂点、`Node` の translation、`Skeleton` のジョイント行列、`Animation` のポーズの translation をまとめて 0.70 倍します。スキニングモデルをジオメトリだけ縮めるとポーズが崩れやすいため、サイズを変えるときはアセット全体へ同じ倍率を掛けるほうが安全です。 

### クリップ一覧を確認し、要約情報を見る

アニメーションが複数入ったモデルでは、「クリップ名は何か」「長さはどれくらいか」を先に見たほうがサンプルを組みやすくなります。 

```js
const result = await app.loadModel("./robot.glb", {
  instantiate: false
});

for (const clipName of result.getClipNames()) {
  const info = result.getClipInfo(clipName);
  console.log(clipName, info.durationMs, info.trackCount);
}
```

`getClipInfo()` は `id`、`targetSkeleton`、`keyCount`、`trackCount`、`durationMs` を返します。ランタイムをシーンへ置く前でもクリップ情報は調べられるため、「どのクリップを UI に並べるか」「サンプルの説明文にどのクリップ名を書くか」を決めるときに便利です。 

### インポーターの出力を JSON として保存する

glTF インポーターや Collada インポーターの出力を後から確認したいときは、`downloadJSON()` を使って `ModelAsset` を保存できます。 

```js
const result = await app.loadModel("./vehicle.glb", {
  instantiate: false
});

result.downloadJSON("vehicle.modelasset.json");
```

この保存結果を見ると、インポート後にメッシュ、スケルトン、アニメーション、`Node` がどのような `ModelAsset` へ正規化されたのかを追えます。インポーターの差分確認や文書作成のときに役立ちます。 

### `Shape.createInstance()` で共有ジオメトリを手で再利用する

`ModelAsset.build() -> instantiate()` の共有リソースという考え方は、ローレベルに `Shape` を直接組み立てる場面でもそのまま使えます。`ModelAsset` を通さないサンプルでも、「ジオメトリは 1 回だけ build し、色や位置やスケールだけを変えたインスタンスを複数置く」という構成にすると、GPU バッファと頂点生成の重複を避けやすくなります。`webg/samples/dof` はこの実例です。被写界深度の比較に使う球は segment 数がすべて同じなので、半径 1.0 の unit sphere を 1 回だけ作り、各 `Node` は `Shape.createInstance()` と `node.setScale(radius)` で半径差を表しています。マテリアルはインスタンスごとに個別設定できるため、色だけを変えつつジオメトリは共有できます。 

```js
const sphereSource = new Shape(app.getGL());
sphereSource.applyPrimitiveAsset(
  Primitive.sphere(1.0, 28, 20, sphereSource.getPrimitiveOptions())
);
sphereSource.endShape();

function addSphere(name, options) {
  const shape = sphereSource.createInstance();
  shape.setMaterial("smooth-shader", {
    use_texture: 0,
    color: options.color,
    ambient: 0.70,
    specular: 1.10,
    power: 58.0
  });

  const node = app.space.addNode(null, name);
  node.setPosition(options.x, options.y, options.z);
  node.setScale(options.radius);
  node.addShape(shape);
  return node;
}
```

このコードで大事なのは、半径差をジオメトリで作り分けず、`Node` の一様スケールで調整している点です。`Shape` のインスタンスはマテリアルや hide 状態を個別に持てる一方、ジオメトリ、GPU バッファ、静的バウンディングボックスは共有リソースを参照します。つまり、「見た目の差分だけをインスタンス側へ置き、重いメッシュ build は共有側へまとめる」という分担になります。`dof` サンプルでは、diagnostics 集計を共有リソース単位に見直した結果、インスタンス化前は `vertexCount=12252`、`triangleCount=22436`、`meshCount=23` だったものが、インスタンス化後は `vertexCount=681`、`triangleCount=1156`、`meshCount=4` まで減っています。ここで `shapeCount` が変わらないのは、シーン上のインスタンス数自体は同じだからです。一方で `meshCount`、`vertexCount`、`triangleCount` は一意な共有リソースだけを数えるため、共有ジオメトリ化の効果がそのまま見えるようになります。通常サンプルで同じプリミティブを多数置くときは、このやり方を最初に検討するのが自然です。 

## `build()` の結果と `ModelAsset` の中身を見る

`ModelAsset.build(gpu)` の戻り値は、単なる配列の寄せ集めではありません。現在の実装では、次の 3 層を意識すると理解しやすくなります。アセット由来の定義、build 済みの共有リソース、`instantiate()` 後の新しいランタイムインスタンスです。`runtime.nodes` と `runtime.nodeMap` は、アセットから作られた `Node` 定義です。`runtime.shapeResources` は、同じメッシュを複数の `Node` で使うときに共有される描画リソース群です。`runtime.instantiate(space)` を呼ぶと、この build 済みデータから新しい `Node`、`ShapeInstance`、`Skeleton`、`Animation` が作られ、`Space` へ配置されます。この構造があるため、`build()` の結果をキャッシュしておき、必要なときだけ `instantiate()` する使い方がしやすくなっています。反対に、同じモデルを毎回 `loadModel()` し直すと、インポートと `build()` を繰り返すことになるため、同じメッシュを大量に使う場面では不利になります。 

`ModelAsset` のトップレベルは、現行実装では主に次の項目で構成されます。 

```json
{
  "version": "1.0",
  "type": "webg-model-asset",
  "meta": {},
  "materials": [],
  "meshes": [],
  "skeletons": [],
  "animations": [],
  "nodes": []
}
```

ここではスキーマを丸暗記するより、「どの役割のデータか」で分けて読むほうが分かりやすくなります。`materials` は、メッシュが参照する材質定義です。`shaderParams` に `color`、`ambient`、`specular`、`power` などを入れ、`build()` 時にマテリアルとして `Shape` 側へ反映します。`meshes` は、ジオメトリとマテリアル参照、必要に応じてスキンを持つ部分です。`positions` と `indices` が中心で、`normals` と `uvs` は任意です。`build()` 時にはここから共有の `Shape` リソースが作られます。`skeletons` は、ジョイント階層、レストポーズ、inverse bind matrix を表します。スキニングメッシュを正しく動かすための骨格定義であり、アニメーションの対象になるのもこの層です。`animations` は、クリップ定義です。`targetSkeleton`、`times`、`tracks` を持ち、1 本のクリップがどのスケルトンをどのように動かすかを表します。状態遷移や再生順はこの層ではなくランタイム側で扱います。`nodes` は、モデル内の配置単位です。どのメッシュを使い、どのスケルトンを結び、どのローカル変換を持つかを定義します。`node.parent` は `Node` id を参照し、シーングラフの親子関係を表します。 

`ModelAsset` を理解するときは、スキーマだけでなく `validate()` と `build()` の見え方も一緒に押さえておくと役立ちます。`validate()` は、トップレベルの `type` や `version`、各 id の重複、参照の整合、ジオメトリ配列長、行列長、アニメーショントラックの整合などを確認します。戻り値は次の形です。 

```js
{
  ok: true,
  errors: [],
  warnings: []
}
```

`errors` は `build()` を止める不整合です。`warnings` は `build()` 自体は進められるが、後で見直したい項目です。サンプルの初期確認では `assertValid()` でも十分ですが、インポーターやエクスポーターを作る場面では `validate()` でレポートを直接見るほうが状況を追いやすくなります。`build()` は、整合性検証を通ったアセットからランタイムを組み立てます。現在の実装では、メッシュから共有の `Shape` リソースを作り、`Node` ごとに `Shape` テンプレートを結び、必要ならスケルトンとアニメーションの構築情報も保持します。ここではまだシーンに配置されません。`instantiate()` は、build 済みランタイムから新しいインスタンスを作る段階です。ここで初めて `Space.addNode()` が呼ばれ、`Shape` インスタンスが `Node` に付き、`Skeleton` と `Animation` もインスタンスごとに新しく作られます。同じ build 結果から複数回 `instantiate()` しても、各インスタンスのアニメーション再生状態は独立します。 

## 例を通して確かめる

`ModelAsset` の役割を具体的に確かめたいときは、次の例が役立ちます。`webg/samples/gltf_loader` は glTF をインポートし、クリップ情報と JSON 保存を確かめる例です。`webg/samples/collada_loader` は Collada を `ModelAsset` に正規化する流れを確かめる例です。`webg/samples/json_loader` は、手書き JSON をそのまま `ModelAsset` として読む例です。`webg/samples/janken` は、同じ build 済みランタイムから複数 `instantiate()` し、アニメーション状態を独立させる実例です。`webg/samples/tile_sim` は、インポート済み actor のサイズ調整や複数 actor 配置を考える実例です。`webg/unittest/primitive_modelasset` は、`Primitive -> ModelAsset -> validate -> build` の最小例として読みやすくなっています。 

## 変更時の注意点

`ModelAsset` 周りを変更するときは、1 か所だけを見るとずれやすいため、少なくとも次の組を一緒に確認してください。スキーマを変えるなら `ModelValidator` と `ModelBuilder`、メッシュやスキンを変えるならジオメトリ処理とスキニング反映、スケルトンを変えるならレストポーズ、inverse bind matrix、アニメーション結び付け、アニメーションを変えるならトラックの整合性検証とランタイムのクリップ生成、高レベル API を変えるなら `ModelLoader` とサンプルの使い方です。特に、共有リソースとインスタンス側の状態の分離が入ってからは、「`build()` 結果で共有してよいもの」と「`instantiate()` ごとに独立すべきもの」を混ぜないことが重要です。GPU バッファ、ジオメトリ、静的バウンディングボックスは共有側に寄せられますが、アニメーション再生状態、スケルトンの現在ポーズ、マテリアル差分や hide 状態のようなランタイム状態はインスタンス側で独立している必要があります。 

## 参考用の最小例

最小構成の `ModelAsset` は、少なくとも `type`、`version`、`meshes`、`nodes` を持ち、`mesh.geometry.positions` と `mesh.geometry.indices` が整っていれば `build()` できます。スケルトンとアニメーションは必須ではありません。 

```json
{
  "version": "1.0",
  "type": "webg-model-asset",
  "meta": {
    "name": "triangle"
  },
  "materials": [
    {
      "id": "mat0",
      "shaderParams": {
        "color": [0.4, 0.8, 1.0, 1.0],
        "ambient": 0.3,
        "specular": 0.4,
        "power": 24.0
      }
    }
  ],
  "meshes": [
    {
      "id": "mesh0",
      "name": "triangle",
      "material": "mat0",
      "geometry": {
        "vertexCount": 3,
        "polygonCount": 1,
        "positions": [-1, -1, 0, 1, -1, 0, 0, 1, 0],
        "normals": [0, 0, 1, 0, 0, 1, 0, 0, 1],
        "uvs": [0, 0, 1, 0, 0.5, 1],
        "indices": [0, 1, 2]
      }
    }
  ],
  "nodes": [
    {
      "id": "node0",
      "name": "triangleNode",
      "parent": null,
      "mesh": "mesh0",
      "transform": {
        "translation": [0, 0, 0],
        "rotation": [0, 0, 0, 1],
        "scale": [1, 1, 1]
      }
    }
  ]
}
```

この最小例は、手書き JSON を試すときの出発点として向いています。最初はアニメーションやスケルトンを入れず、メッシュと `Node` の経路だけを通し、その後でスキンやクリップを追加していくほうが原因を切り分けやすくなります。
