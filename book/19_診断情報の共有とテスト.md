# 診断情報の共有とテスト

デバッグ用の文字列を画面の隅に表示させるだけでは、現在の 3D アプリケーションの状態を他者に正確に伝えることは困難です。真に必要とされるのは、その瞬間の状態を「レポート」としてまとめ、画面上で確認し、必要に応じてそのまま共有できる仕組みです。

本章では、`DebugDock`、`Diagnostics`、`DebugProbe`、および `WebgApp` を通じて、診断情報を単に「見るもの」ではなく「共有するもの」として扱う設計思想について解説します。webg の診断システムは、アプリケーションの現在状態を画面で読み取り、そのままクリップボード経由で共有することを目的として構成されています。

例えば、描画された画面の状態を AI（大規模言語モデル）に伝えて解析を依頼したい場合、`DebugDock` の `Copy Summary` または `Copy JSON` を利用します。これにより、その瞬間のカメラ位置、視野角、シーン規模、警告、およびサンプル固有の状態がクリップボードにコピーされます。利用者はそれを貼り付けるだけで、AI に対して「現在見えているはずの状態」を数値やテキストとして正確に渡すことができます。つまり、診断情報は単なる補助的な表示ではなく、アプリケーションの内部状態を外部へ出力するための正式なインターフェースなのです。

ここで重要なのは、webg の診断機能（diagnostics）が単一の巨大な UI ではなく、役割の異なるコンポーネントの組み合わせで構成されている点です。
- `DebugConfig.js`：デバッグモードとリリースモードの切り替え、および診断機能の有効/無効を管理します。
- `DebugDock.js`：現在状態の表示と、レポートのコピー操作を担当します。
- `Diagnostics.js`：レポートオブジェクトを生成し、要約テキストや JSON 形式に整形します。
- `DebugProbe.js`：特定の瞬間における状態採取を支援します。

このように、「診断情報の正本（マスターデータ）を生成する層」と、「それを表示またはコピーする層」が明確に分離されています。本章では、まず `DebugDock` とクリップボード共有を中心とした情報の流れを整理し、レポートの内容について詳しく見ていきます。続いて、`WebgApp` が自動で収集する情報と、開発者が意図的に追加すべき情報の使い分け、そして特定の瞬間を切り出すワンショットプローブや起動失敗時のレポート処理について解説します。

## 診断情報を共有するという設計思想

webg の診断システムは、ブラウザのコンソール（console）を主役にした仕組みではありません。もちろん開発者ツールを用いてコンソールを確認することは可能ですが、標準的な共有経路として `Copy Summary` と `Copy JSON` を提供しています。`navigator.clipboard.writeText()` を用いてレポートを直接クリップボードに書き込むことで、利用者がチャットツールやバグ報告フォームへ即座に貼り付けられる形式を実現しています。

この設計を採用している理由は、画面を見ている本人にしか分からない情報を、再現性のある形で外部へ伝達するためです。`Diagnostics` 自体は表示部品ではなく「レポート生成層」であり、第14章で整理した表示経路においても、`DebugDock` や `FixedFormatPanel` に情報を流し込む前の「正本」を作成する役割を担っています。表示と共有の両方が、常に同一のレポートを基準に動作することがこの設計の肝となります。

## `DebugDock` による状態の可視化と操作

`DebugDock` は、診断情報の中心となる開発者向けドックです。ここには `Current State` の表示領域と、`Copy Summary`、`Copy JSON` の 2 つのボタンが配置されています。画面上で現在状態を確認し、そのまま同じ内容をコピーできるため、確認した内容と共有した内容に齟齬が生じません。

`WebgApp` の標準設定では、デバッグ用のキー入力にプレフィックス方式を採用しています。最も重要な操作は `F9` キーを押した後に `M` キーを押すことで、これによりデバッグモードとリリースモードを切り替えることができます。同様に、`F9` の後に `C` で要約を、`F9` の後に `V` で JSON をコピーできます。本稿では便宜上「F9 + M」と表記することがありますが、これは同時押しではなく、「`F9` を押した後に `M` を押す」という順次入力であることを留意してください。

リリースモードでは `enableDiagnostics` と `enableProbe` が無効になり、デバッグ用のドックや補助表示も非表示となります。そのため、診断機能を利用する際は、まずデバッグモードへ切り替えることが基本となります。

## レポートの構成：Summary と JSON

### 要約（Summary）に含まれる情報

`Copy Summary` は `Diagnostics.toSummaryText()` によって生成される要約テキストです。構成は `[Overview]`、`[Latest Issue]`、`[Key Stats]`、`[Warnings]`、`[Recent Details]` の 5 つのセクションで成り立っており、人間と AI が効率的に状況を把握できるよう最適化されています。

1. Overview: `source`（ソースファイル）、`system`（システム名）、`stage`（現在の段階）、`ok`（正常判定）、`timestamp`（時刻）が含まれます。
2. Latest Issue: 最新のエラー情報を表示します。エラーがない場合は最新の警告（warning）を、それもない場合は `(none)` と表示されます。
3. Key Stats: カメラの位置・姿勢・ターゲット、距離、視野角、アプリケーション全体のシェーダークラス、シーン規模、キャンバスおよびディスプレイサイズ、フレームカウント、稼働時間（uptimeSec）などが並びます。
4. Warnings: 警告配列の末尾から最大 4 件を表示します。
5. Recent Details: 詳細配列の末尾から最大 8 件を表示します。

要約は、「今、何が起きているか」を最小限のコストで共有するための入り口としての役割を果たします。

### JSON 形式で拡張される情報

`Copy JSON` も要約と同じレポートを基にしていますが、`detailLevel: "json"` として組み立て直されるため、より詳細な構造情報が含まれます。特に重要なのが `context.webgAuto` セクションです。ここには以下の情報が格納されます。

- カメラのターゲット、視点位置（eye position）、視点姿勢（eye attitude）
- アプリケーション全体で使用しているシェーダークラス
- シーン内で使用されているシェーダークラスの一覧とその使用数
- マテリアル ID の一覧と各マテリアルの使用数

AI に詳細な解析を依頼する場合や、レンダリング構成まで含めて精査したい場合は、この JSON 形式が適しています。

### `Current State` とコピー結果の更新タイミング

`Current State` の表示と `Copy Summary` の内容は同一の要約テキストに基づいています。異なるのは更新のタイミングです。`Current State` はパフォーマンスへの影響を避けるため、`WebgApp` 側で 1 秒のキャッシュを保持しています。一方、`Copy Summary` と `Copy JSON` は、ボタンが押された瞬間に `forceRefresh: true` でレポートを再生成します。したがって、通常は `Current State` で概況を把握し、正確な瞬間の値を外部に渡したいときだけコピーボタンを利用するという運用になります。

## `WebgApp` による自動統計情報の収集

診断機能の導入は非常にシンプルです。以下のような基本構成で動作します。

```js
const app = new WebgApp({
  document,
  debugTools: {
    mode: "debug",
    system: "scene",
    source: "./scene.json"
  }
});

await app.init();
app.setDiagnosticsStage("runtime");
app.configureDebugKeyInput();
```

この設定により、デスクトップ環境であれば `DebugDock` が表示され、`Copy Summary` / `Copy JSON` および `F9` プレフィックスによる操作が可能になります。導入の手間は最小限に抑えられており、ここに必要な情報を随時追加していく構成になっています。

`WebgApp` は、開発者が明示的に指定しなくても、起動直後の環境情報と現在のシーン規模を自動的に収集します。`app.init()` の最終段階で `checkEnvironment({ stage: "ready" })` が呼び出され、`webgpuSupported`、`hasScreen`、`hasShader`、`hasProjection`、`hasSpace`、`hasEye`、`hasInput`、`hasMessage` といった基本整合性がレポートに記録されます。

さらに、`collectCurrentSceneStats()` によって、以下の統計情報が自動的に収集されます。
- シーン規模: `nodeCount`、`shapeCount`、`meshCount`、`vertexCount`、`triangleCount`、`boneCount`、`visibleShapeCount`、`hiddenShapeCount`、`skinnedShapeCount`
- カメラ・描画条件: `eyeX/Y/Z`、`eyeYaw/Pitch/Bank`、`cameraTargetX/Y/Z`、`eyeDistance`、`fovX/Y`、`canvasWidth/Height`、`displayWidth/Height`、`frameCount`、`uptimeSec`、`appShaderClass`

この自動収集機能のおかげで、サンプルごとにカメラ位置やシーン規模を確認するための文字列を手書きする必要はありません。これは、第4章や第5章で説明した「`WebgApp` が入力、HUD、デバッグ、診断情報の土台をまとめて初期化する」という設計思想に基づいています。診断機能は後付けのオプションではなく、`WebgApp` の標準構成の一部として組み込まれています。

## アプリケーション側で追加する診断情報

自動収集される情報だけでは、そのアプリケーション固有の文脈や意図を十分に伝えることはできません。そこで `WebgApp` では、アプリケーション固有の `stage`、`stats`、`details`、`warnings` を追加するためのインターフェースが用意されています。

ここでは、「どのサンプルでも共通する情報は自動統計」に、「そのサンプルで意味を持つ情報は追加情報」に分担させることが重要です。

### 実行段階（stage）の記録

まず、現在アプリケーションがどの処理段階にあるかを記録します。

```js
app.setDiagnosticsStage("fetch");
sceneAsset = await SceneAsset.load("./scene.json");

app.setDiagnosticsStage("build");
sceneRuntime = await app.loadScene(sceneAsset.getData());

app.setDiagnosticsStage("runtime");
```

これにより、レポートを読んだ人が「この値がリソース取得中（fetch）のものか、構築中（build）のものか、あるいは実行中（runtime）のものか」を明確に判別できるようになります。

### 統計情報（stats）の追加

サンプル固有の重要な数値は `mergeDiagnosticsStats()` を用いて追加します。

```js
app.mergeDiagnosticsStats({
  mode: state.mode,
  focusDistance: dof.focusDistance.toFixed(1),
  focusRange: dof.focusRange.toFixed(1),
  blurScale: dof.getBlurScale().toFixed(2),
  clipCount,
  score
});
```

ここには「そのアプリケーションの利用者が最初に知りたい値」を配置します。例えば、被写界深度（DoF）のサンプルであれば `focusDistance`、ローダーのサンプルであれば `clipCount`、ゲームであれば `score` のように、自動統計と同階層に配置したい重要な数値をここに集約します。

### 詳細情報（details）の追加

補足的な情報は `addDiagnosticsDetail()` で追加します。

```js
app.addDiagnosticsDetail(`clip=${selectedClipName}`);
app.addDiagnosticsDetail(`targetPos=(${x.toFixed(2)}, ${y.toFixed(2)}, ${z.toFixed(2)})`);
app.addDiagnosticsDetail(`bgmPreset=${selectedBgmPreset}`);
```

詳細情報は `Recent Details` セクションに表示されるため、長文の説明よりも `key=value` 形式の短い記述が適しています。統計情報に載せるほどではないが、解析に必要な補足情報をここに配置することで、レポートが整理されます。

### 警告情報（warnings）の追加

注意を促したい内容は `addDiagnosticsWarning()` で追加します。

```js
if (envReport.warnings?.length) {
  app.addDiagnosticsWarning(envReport.warnings[0]);
}
```

警告情報は `Latest Issue` と `Warnings` の両方に表示されるため、単なる補足よりも優先的に確認してほしい内容に適しています。要約を最初に読むという設計であるため、警告情報の適切な運用は非常に重要です。

### `checkEnvironment()` との併用

`checkEnvironment()` は、プロジェクション、シェイプ、ウェイト、メッセージなどの基本整合性を確認し、そのレポートを返します。

```js
const envReport = app.checkEnvironment({
  stage: "runtime-check",
  shapes: [activeShape]
});

app.mergeDiagnosticsStats({
  envOk: envReport.ok ? "yes" : "no",
  envWarning: envReport.warnings?.[0] ?? "-"
});
```

この手法を用いることで、利用者は要約を見ただけで「そのシーンが基本条件を満たしているか」を即座に判断できます。`checkEnvironment()` の結果を診断情報に統合することで、別画面を確認することなく状況を共有でき、効率的なデバッグが可能になります。

## ワンショットプローブ（One-shot Probe）の活用

連続的に更新される状態ではなく、「設定を切り替えた直後の 1 回だけ」の状態を固定して採取したい場合があります。その際は `configureDiagnosticsCapture()` と `createProbeReport()` を使用します。これは常時ログを記録する代わりではなく、「欲しい瞬間を 1 枚のレポートとして切り出す」ための補助機能です。

```js
function makeProbeReport(frameCount) {
  const report = app.createProbeReport("runtime-probe");
  Diagnostics.addDetail(report, `mode=${state.mode}`);
  Diagnostics.addDetail(report, `targetPaused=${state.targetPaused ? "yes" : "no"}`);
  Diagnostics.mergeStats(report, {
    frameCount,
    targetAngleDeg: state.targetAngleDeg.toFixed(2)
  });
  return report;
}

app.configureDiagnosticsCapture({
  labelPrefix: "camera_controller",
  collect: () => makeProbeReport(app.screen.getFrameCount())
});
```

これを登録しておけば、`captureDiagnosticsSummary()` や `captureDiagnosticsReportJSON()` を呼び出すことで、指定したフレーム後などの状態を採取できます。

```js
app.captureDiagnosticsSummary({ afterFrames: 1 });
app.captureDiagnosticsReportJSON({ afterFrames: 1 });
```

プローブは、モード切り替え直後、アニメーション再開直後、あるいはユーザー入力直後など、一瞬の状態を確定させたい場合に非常に有効です。通常の要約が「現在の状態」を示すのに対し、プローブは「特定の瞬間の状態」を固定して記録します。

## 起動失敗時のレポート処理

アプリケーションの起動に失敗した場合、画面にエラーパネルを表示するだけでなく、診断レポートをエラー版に差し替えておくことで、問題の共有が容易になります。

```js
document.addEventListener("DOMContentLoaded", () => {
  start().catch((error) => {
    app?.setDiagnosticsReport?.(Diagnostics.createErrorReport(error, {
      system: "camera_controller",
      source: "samples/camera_controller/main.js",
      stage: app?.getDiagnosticsReport?.()?.stage ?? "start"
    }));
    app?.showErrorPanel?.(error, {
      title: "camera_controller failed",
      id: "start-error"
    });
  });
});
```

この実装により、起動失敗時であっても `Copy Summary` や `Copy JSON` を通じて状況を外部へ渡すことができます。成功時だけでなく、失敗時にも共有経路を維持することが、webg の診断設計における大きな利点です。これは第15章で解説した `FixedFormatPanel` や `showErrorPanel()` の役割とも密接に関連しています。

## `WebgApp` を使用しない個別実装による経路

通常は `WebgApp` を利用するのが自然ですが、`samples/sound` のように独自の UI を構築し、`Diagnostics.js` を直接利用する方法もあります。これは個別実装による経路ですが、音響サンプルのように 3D シーンの統計情報よりも UI の調整値の方が重要な場合、このアプローチの方が適しています。つまり、診断機能は `WebgApp` 専用ではなく、レポート生成層として単独で利用可能です。

```js
const report = Diagnostics.createSuccessReport({
  system: "sound",
  source: "samples/sound/main.js",
  stage: "runtime"
});

Diagnostics.addDetail(report, `soundEffect=${selectedSoundEffect}`);
Diagnostics.mergeStats(report, {
  masterVol: ui.masterVol.value,
  seVol: ui.seVol.value,
  bgmVol: ui.bgmVol.value
});
```

この経路では `WebgApp` による自動統計は行われませんが、その分、独自 UI の状態をダイレクトにレポートへまとめることができます。これは第18章の sound サンプルが `WebgApp` を使わずに音響 UI に特化していた構成と整合しています。標準経路では `WebgApp` がカメラやシーン規模を自動収集し、個別実装経路では開発者が必要な値を手動で組み立てるという使い分けになります。

基本形から応用、そして個別実装まで幅広く例示されているため、目的に応じて参照先を選択してください。

## まとめ

本章で最も重要な点は、診断情報を単なる「画面上の補助表示」ではなく、「現在の状態を共有するための正式な出力口」として扱うことです。システムの中心にあるのは「現在のレポート」であり、それを `Copy Summary` と `Copy JSON` によって外部へ持ち出せる点に価値があります。利用者は `DebugDock` で状態を確認し、そのまま AI や開発者に情報を共有できます。

`WebgApp` は共通情報を自動的に収集しますが、アプリケーション固有の意味を伝えるには、`stage`、`stats`、`details`、`warnings` を適切に使い分ける必要があります。さらに、必要に応じてワンショットプローブやエラーレポートへの差し替えを組み合わせることで、診断情報は極めて実用的なツールとなります。

第14章と第15章で整理した表示経路において、診断機能は「レポートを生成する層」という特別な位置にあり、その上に `DebugDock` や `FixedFormatPanel` が構築されていると理解することで、全体の設計思想がより明確に見えてくるはずです。
