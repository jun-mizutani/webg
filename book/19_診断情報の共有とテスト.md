# 診断情報の共有とテスト

デバッグ用の文字列を画面の隅へ出すだけでは、いまの 3D アプリの状態を他者へ正確に伝えることはできません。必要なのは、その瞬間の状態をレポートとしてまとめ、画面で読み、必要ならそのまま共有できることです。本章では、`DebugDock`、`Diagnostics`、`DebugProbe`、`WebgApp` を通して、診断情報を「見るもの」ではなく「共有するもの」として扱う考え方を整理します。 `webg` の診断情報は、アプリの現在状態を画面で読み、そのままクリップボード経由で共有するための仕組みとして構成されています。

いまの `webg` では、描画された画面の状態を AI に伝えたいとき、まず `DebugDock` の `Copy Summary` または `Copy JSON` を使います。すると、その瞬間のカメラ位置、視野角、シーン規模、警告、サンプル固有の状態がクリップボードへコピーされます。利用者はそれをそのまま貼り付けるだけで、AI に「見えているはずの状態」を数値やテキストとして渡せます。つまり、診断情報は単なる補助表示ではなく、アプリの現在状態を共有するための正式な出口です。

ここで最初に押さえたいのは、`webg` の diagnostics が 1 つの巨大な UI ではなく、役割の違う部品の組み合わせだという点です。`DebugConfig.js` は debug / release の切り替えと診断系機能の可否を管理し、`DebugDock.js` は現在状態の表示とコピーを担当し、`Diagnostics.js` はレポート object を作って summary / text / JSON に整形し、`DebugProbe.js` は一瞬だけの採取を助けます。つまり、「診断情報の正本を作る層」と「それを見せたりコピーしたりする層」が分かれています。

この章では、まず `DebugDock` とクリップボード共有を中心に診断情報の流れを整理し、そのあとで summary と JSON に何が入るかを見ます。続いて、`WebgApp` が自動で入れる情報と、アプリ側が追加すべき情報を分けて確認します。そのうえで one-shot probe、起動失敗時のレポート、`WebgApp` を使わない場合の経路を見ていきます。つまり本章の中心は、「どう表示するか」よりも「どうレポートを作り、どう共有するか」です。

## 診断情報を共有するという考え方

`webg` の diagnostics は、ブラウザーの console を主役にした仕組みではありません。もちろん必要なら開発者ツールを開いて console を見られますが、標準の共有経路は `Copy Summary` と `Copy JSON` です。`navigator.clipboard.writeText()` を使ってレポートをそのままクリップボードへ書き込み、利用者が会話欄やバグ報告へ貼り付けられる形を最初から用意しています。 

この設計にしている理由は、画面を見ている本人だけが分かる情報を、再現性のある形で外へ渡したいためです。Diagnostics 自体は表示部品ではなくレポート生成層であり、第14章で整理した表示経路の中でも、`DebugDock` や `FixedFormatPanel` に流し込む前の「正本」を作る役割にあります。つまり、表示と共有の両方が同じレポートを基準に動くようにしている点が重要です。

## `DebugDock` と現在状態の表示

`DebugDock` は、診断情報の中心になる開発者向けドックです。ここには `Current State` の表示欄と、`Copy Summary`、`Copy JSON` の 2 つのボタンがあります。画面で現在状態を読み、そのまま同じ内容をコピーできるため、読んだ内容と共有した内容がずれにくくなっています。

`WebgApp` の既定設定では、デバッグ用のキー入力は prefix 方式です。もっとも重要な操作は `F9` のあとに `M` で、これが debug / release の切り替えです。同じ仕組みで `F9` のあとに `C` で summary、`F9` のあとに `V` で JSON をコピーできます。文書ではしばしば「F9 + M」と略記できますが、実際には同時押しではなく「`F9` のあとに `M`」です。release では `enableDiagnostics` と `enableProbe` が無効になり、debug 用の dock や補助表示も隠れるため、まず debug へ戻すのが基本になります。

## `Copy Summary` と `Copy JSON`

### Summary に入る情報

`Copy Summary` は `Diagnostics.toSummaryText()` が作る要約テキストです。構成は `[Overview]`、`[Latest Issue]`、`[Key Stats]`、`[Warnings]`、`[Recent Details]` の 5 つで、人と AI が最初に読むことを前提に並んでいます。まず何についてのレポートかを把握し、次に最新の問題を見て、そのあと主要な数値と補足へ進む流れになっています。

`Overview` には `source`、`system`、`stage`、`ok`、`timestamp` が入り、`Latest Issue` には最新の error、なければ最新の warning、何もなければ `(none)` が入ります。`Key Stats` にはカメラ位置、姿勢、target、距離、視野角、アプリ全体の shader クラス、シーン規模、canvas / display サイズ、frameCount、uptimeSec などが並びます。さらに、`Warnings` は warning 配列の末尾側から最大 4 件、`Recent Details` は detail 配列の末尾側から最大 8 件を表示します。つまり summary は、「まず何が起きているか」を短く共有するための入口です。

### JSON で増える情報

`Copy JSON` は summary と同じ current report を基にしていますが、`detailLevel: "json"` で組み立て直すため、追加の構造情報が入ります。特に重要なのは `context.webgAuto` で、ここへ camera target、eye position、eye attitude、アプリ全体の shader クラス、シーンで使っている shader クラス一覧、shader ごとの使用数、material ID 一覧、material ごとの使用数などが入ります。AI に詳しい解析を頼みたいときや、レンダリング構成まで含めて見せたいときはこちらが向いています。 

### `Current State` とコピー結果の関係

`Current State` も `Copy Summary` も、同じ summary テキストを基にしています。違うのは更新タイミングです。`Current State` は毎フレーム再計算されるのではなく、`WebgApp` 側で 1 秒のキャッシュを持ちます。一方で `Copy Summary` と `Copy JSON` は、ボタンを押した時点で `forceRefresh: true` で report を作り直します。したがって、通常は `Current State` を目で読み、正確にその瞬間の値を外へ渡したいときだけボタンで fresh なレポートを切り出す、と考えると分かりやすくなります。 

## `WebgApp` が自動で入れる情報

もっとも基本的な構成は次のとおりです。

```js
const app = new WebgApp({
  document,
  debugTools: {
    mode: "debug",
    system: "scene",
    source: "./scene.json"
  }
});

await app.init();
app.setDiagnosticsStage("runtime");
app.configureDebugKeyInput();
```

この時点で、desktop 条件を満たせば `DebugDock` が表示され、`Copy Summary` と `Copy JSON` が使え、`F9` のあとに `M`、`C`、`V` が動き、環境の基本情報と自動統計が summary / JSON へ入ります。つまり diagnostics を使い始める入口そのものはかなり小さく、そこへアプリ固有情報を足していく構成です。 

`WebgApp` は、利用者が何も追加しなくても、起動直後の環境情報と現在のシーン規模を自動で入れます。`app.init()` の最後では `checkEnvironment({ stage: "ready" })` が呼ばれ、`webgpuSupported`、`hasScreen`、`hasShader`、`hasProjection`、`hasSpace`、`hasEye`、`hasInput`、`hasMessage` のような基本整合をレポートへ残します。さらに `collectCurrentSceneStats()` により、`nodeCount`、`shapeCount`、`meshCount`、`vertexCount`、`triangleCount`、`boneCount`、`visibleShapeCount`、`hiddenShapeCount`、`skinnedShapeCount` が入ります。カメラ位置や描画条件としては `eyeX`、`eyeY`、`eyeZ`、`eyeYaw`、`eyePitch`、`eyeBank`、`cameraTargetX`、`cameraTargetY`、`cameraTargetZ`、`eyeDistance`、`fovX`、`fovY`、`canvasWidth`、`canvasHeight`、`displayWidth`、`displayHeight`、`frameCount`、`uptimeSec`、`appShaderClass` も自動収集されます。

この自動統計の利点は、サンプルごとに毎回カメラやシーン規模の確認文字列を手書きしなくてよいことです。第5章や第4章で `WebgApp` が入力、HUD、デバッグ、診断情報の土台までまとめて初期化する、と説明した内容とも一致しています。つまり diagnostics は、後から貼り足す機能ではなく `WebgApp` の標準構成の一部として読むと整理しやすくなります。

## アプリ側で追加する情報

自動で入る情報だけでは、そのアプリの意味が十分に伝わらないことがあります。そこで `WebgApp` 側には、アプリ固有の `stage`、`stats`、`details`、`warnings` を追加する入口が用意されています。ここで重要なのは、自動統計は「どのサンプルにも共通する情報」、追加情報は「そのサンプルで意味のある情報」という分担にすることです。

### `stage` を残す

まず、いまどの段階の情報かをレポートへ残します。

```js
app.setDiagnosticsStage("fetch");
sceneAsset = await SceneAsset.load("./scene.json");

app.setDiagnosticsStage("build");
sceneRuntime = await app.loadScene(sceneAsset.getData());

app.setDiagnosticsStage("runtime");
```

これにより、レポートを読んだ人や AI が「今の値が fetch 中なのか、build 中なのか、runtime なのか」を見分けやすくなります。diagnostics は値そのものだけでなく、その値がどの段階のものかも重要です。 

### `stats` を追加する

サンプル固有の重要な数値は `mergeDiagnosticsStats()` で追加します。

```js
app.mergeDiagnosticsStats({
  mode: state.mode,
  focusDistance: dof.focusDistance.toFixed(1),
  focusRange: dof.focusRange.toFixed(1),
  blurScale: dof.getBlurScale().toFixed(2),
  clipCount,
  score
});
```

ここへ入れるべきなのは、「そのアプリの利用者が最初に知りたい値」です。被写界深度の sample なら `focusDistance`、ローダー sample なら `clipCount`、ゲームなら `score` のように、自動統計と同じ階層へ載せたい重要な値をここへ寄せます。

### `details` を追加する

短い補足は `addDiagnosticsDetail()` で足します。

```js
app.addDiagnosticsDetail(`clip=${selectedClipName}`);
app.addDiagnosticsDetail(`targetPos=(${x.toFixed(2)}, ${y.toFixed(2)}, ${z.toFixed(2)})`);
app.addDiagnosticsDetail(`bgmPreset=${selectedBgmPreset}`);
```

details は `Recent Details` に出るため、長文説明より `key=value` の形で短く読める値に向いています。stats に載せるほどではないが、レポートを読む人へ補足したい情報をここへ置くと整理しやすくなります。 

### `warnings` を追加する

注意を促したい内容は `addDiagnosticsWarning()` で追加します。

```js
if (envReport.warnings?.length) {
  app.addDiagnosticsWarning(envReport.warnings[0]);
}
```

warning は `Latest Issue` と `Warnings` の両方で目に入りやすくなるため、単なる補足より優先して読ませたい内容に向いています。summary を最初に読む設計だからこそ、warnings の使い方はかなり重要です。 

### `checkEnvironment()` を併用する

`checkEnvironment()` は、projection、shape、weight、message などの基本整合を確認するレポートを返します。現在のサンプルでもよく使われています。

```js
const envReport = app.checkEnvironment({
  stage: "runtime-check",
  shapes: [activeShape]
});

app.mergeDiagnosticsStats({
  envOk: envReport.ok ? "yes" : "no",
  envWarning: envReport.warnings?.[0] ?? "-"
});
```

この形にしておくと、利用者は summary を見ただけで「そのシーンが基本条件を満たしているか」を判断できます。`checkEnvironment()` の結果を diagnostics へ統合することで、別画面を見ずに状況を共有しやすくなります。 

## one-shot probe を使う

連続更新中の状態ではなく、「切り替えた直後の 1 回だけ」を固定したいことがあります。そのときは `configureDiagnosticsCapture()` と `createProbeReport()` を使います。ここでの考え方は、常時ログの代わりではなく「欲しい瞬間を 1 枚だけ切り出す」ことです。つまり probe は連続レポートの代替ではなく、瞬間採取のための補助です。 

```js
function makeProbeReport(frameCount) {
  const report = app.createProbeReport("runtime-probe");
  Diagnostics.addDetail(report, `mode=${state.mode}`);
  Diagnostics.addDetail(report, `targetPaused=${state.targetPaused ? "yes" : "no"}`);
  Diagnostics.mergeStats(report, {
    frameCount,
    targetAngleDeg: state.targetAngleDeg.toFixed(2)
  });
  return report;
}

app.configureDiagnosticsCapture({
  labelPrefix: "camera_controller",
  collect: () => makeProbeReport(app.screen.getFrameCount())
});
```

これを登録しておくと、`captureDiagnosticsSummary()` や `captureDiagnosticsReportJSON()` で 1 フレーム後などの状態を採取できます。

```js
app.captureDiagnosticsSummary({ afterFrames: 1 });
app.captureDiagnosticsReportJSON({ afterFrames: 1 });
```

probe は、モード切り替え直後、アニメーション再開直後、入力直後など、一瞬の状態を確定させたいときに向いています。通常の summary が「いま」のレポートだとすると、probe は「この瞬間だけ」を固定するレポートです。 

## 起動失敗を report に残す

起動に失敗した場合は、画面へエラーパネルを出すだけでなく、diagnostics レポートもエラー版へ差し替えておくと、あとで共有しやすくなります。

```js
document.addEventListener("DOMContentLoaded", () => {
  start().catch((error) => {
    app?.setDiagnosticsReport?.(Diagnostics.createErrorReport(error, {
      system: "camera_controller",
      source: "samples/camera_controller/main.js",
      stage: app?.getDiagnosticsReport?.()?.stage ?? "start"
    }));
    app?.showErrorPanel?.(error, {
      title: "camera_controller failed",
      id: "start-error"
    });
  });
});
```

この形にしておけば、失敗時でも `Copy Summary` や `Copy JSON` で状況をそのまま外へ渡せます。成功時だけでなく失敗時にも共有経路を保つことが、現在の diagnostics 設計の大きな利点です。第15章で整理した `FixedFormatPanel` や `showErrorPanel()` の役割ともつながります。

## `WebgApp` を使わない場合

通常は `WebgApp` を使うほうが自然ですが、`samples/sound` のように独自 UI を組み、`Diagnostics.js` を直接使う方法もあります。これは例外経路ですが、音のように 3D シーン統計より UI の調整値が重要なサンプルでは、この形のほうが分かりやすいことがあります。つまり diagnostics は `WebgApp` 専用ではなく、レポート生成層として単独でも使える、ということです。

```js
const report = Diagnostics.createSuccessReport({
  system: "sound",
  source: "samples/sound/main.js",
  stage: "runtime"
});

Diagnostics.addDetail(report, `soundEffect=${selectedSoundEffect}`);
Diagnostics.mergeStats(report, {
  masterVol: ui.masterVol.value,
  seVol: ui.seVol.value,
  bgmVol: ui.bgmVol.value
});
```

この経路では、`WebgApp` が持つ自動統計は使いません。その代わり、独自 UI の状態をそのままレポートへまとめられます。これは第18章の sound sample が `WebgApp` を使わずに音の UI に集中していた構成とも整合します。つまり、標準経路では `WebgApp` が camera や scene 規模を自動収集し、例外経路では利用者が必要な値を自分で組み立てる、という違いになります。

## 参照すべき sample

利用者が実例を読みたいときは、`samples/scene` が `Copy Summary` と `Copy JSON` を使う基本例として分かりやすく、`samples/camera_controller` はカメラ関連の `mode` や target 状態を追加する例として向いています。`samples/dof` は `focusDistance` などのパラメータを summary に載せる例、`samples/model_shape` は build 経路と validator 状態をレポートへ載せる例、`samples/collada_loader` は loader と runtime の接続状態を diagnostics へ残す例として読めます。さらに、`samples/sound` は `WebgApp` を使わずに `Diagnostics.js` を直接使う例外経路として役立ちます。つまり、基本形から応用形、例外経路まで揃っているため、必要な場面に応じて参照先を選べます。

## まとめ

この章で最も大事なのは、診断情報を「画面の隅の補助表示」としてではなく、「現在状態を共有するための正式な出口」として扱うことです。主役は current report であり、そのレポートを `Copy Summary` と `Copy JSON` で外へ持ち出せることにあります。利用者は `DebugDock` を見て、そのまま AI や他者へ貼り付けられます。つまり diagnostics の中心は「表示」ではなく「共有可能なレポートを作ること」です。

また、`WebgApp` は共通情報を自動で集めますが、それだけでは各サンプルの意味までは伝わりません。`stage`、`stats`、`details`、`warnings` をアプリ側で足し、必要なら one-shot probe やエラーレポート差し替えも使うことで、診断情報はかなり実用的になります。第14章と第15章で整理した表示経路の中でも、diagnostics は「レポートを作る層」として特別な位置にあり、その上に `DebugDock` や `FixedFormatPanel` が乗っていると考えると、全体の設計がさらに見えやすくなります。
