# アニメーションの実践とアセット

この章は、`webg/samples/animation_state`、`webg/samples/janken`、`webg/samples/tile_sim` を中心に、`Animation`、`Action`、`AnimationState` を実際のアプリへどのように載せるかを整理した実践ガイドです。第12章が各層の役割を説明する章だとすると、この章は「実際のサンプルではどこに何を書くのか」「アセットのキー区間やサイズをどう判断するのか」を確認する章です。ここで重視するのは、現行サンプルと現行 core 実装に沿って、利用者がそのままアプリ開発へ持ち込める読み方を示すことです。そのため、以前のアセットデバッグ用サンプルではなく、現在の代表サンプルである `animation_state`、`janken`、`tile_sim` を主な参照先として扱います。 

この章で最初に押さえておきたいのは、現在のサンプルでは `animation_state`、`janken`、`tile_sim` の 3 本が主な参照先だということです。`animation_state` は条件評価型の `AnimationState`、`janken` は入力駆動の `AnimationState`、`tile_sim` はスキニング付き glb を TileMap actor として扱う実例です。この 3 本を押さえると、今の `webg` でアニメーションをどのように使うかを現行サンプルだけで追えます。また、実装の分かれ目は「何を再生するか」と「どう進めるか」です。サンプル側が決めるのは、どのステートやアクションを選ぶかです。実際の補間や bone への反映は、`Animation -> Schedule -> Task -> CoordinateSystem` が担当します。この役割分担を意識すると、入力処理とアニメーション処理を混ぜにくくなります。 

さらに、`fromKey / toKey` と `entryDurationMs` は、アセット側の意味とランタイム側の意味を分けて読む必要があります。`fromKey / toKey` はどのキー区間を使うか、`entryDurationMs` は現在姿勢からその区間へ何 ms で入るかです。サンプルの見え方が崩れたときは、この 2 つを別々に確認する必要があります。また、hand 系の現行アセットは `samples/gltf_loader/hand.glb` です。現行サンプルの `animation_state` と `janken` は、ともにこの `hand.glb` を使います。`N0` は `12-13`、`N2` は `4-5`、`N5` は `10-11` といった区間対応も、この現行アセットを基準に読む必要があります。さらに `tile_sim` のように glb actor をワールドへ載せる場合は、バウンディングボックス、glTF root のローカルスケール、アプリ側で欲しい表示倍率を分けて扱うほうが判断しやすくなります。 

この章では、まず `AnimationState` から `Task` までの実フローを現行サンプルに沿って追い、そのあとで `animation_state`、`janken`、`tile_sim` の役割差を見ます。後半では、`hand.glb` のキー区間の見方と、glb actor のサイズ決定を扱います。つまりこの章は、アニメーションの概念説明ではなく、「現行サンプルを読むときにどの層をどう見ればよいか」を整理する章です。 

## `AnimationState` から `Task` までの実フロー

`AnimationState` から実際の移動・回転までを追うと、役割はかなり明確に分かれています。まず `AnimationState` は「何を再生するか」を決めるだけで、bone の座標値や補間式には触りません。そこから下の層で、再生対象の選択、区間の切り出し、時間補間、そして個々の bone への命令実行が順番に行われます。流れを順に書くと、次のようになります。 

1. `AnimationState.update(context, deltaMs)` が今のステートを更新し、必要なら遷移を選びます。`resolveTransition()` が真を返した場合は `setState()` が呼ばれ、`onExit`、`currentState` の更新、`onEnter`、`playStateTarget()` まで進みます。
2. `playStateTarget()` はコントローラーの型に応じて `start()`、`startAction()`、`startAnimation()`、`restartAnimation()` のどれかを呼びます。`Action` をコントローラーに渡している場合は、ここで `action` 名が再生開始されます。
3. `Action.start(actionId, options)` は `action` から最初の `pattern` を取り出し、`currentPattern` を決めたうえで `transitionToKey()` を呼びます。`Action.play()` は再生中の `pattern` が終わったら次の `pattern` へ進み、必要ならループします。
4. `transitionToKey(entryDurationMs, fromKey, toKey)` は `Animation.js` 側へ `startTimeFromTo()` を投げます。ここで `pattern` の区間と entry duration が、実際の補間対象になります。
5. `Animation.startTimeFromTo(time, keyFrom, keyTo)` は `transitionTo()` を呼び、`keyFrom` と `keyTo` のポーズを使って `Schedule` へコマンドを流します。`putRotTransByMatrix` で開始姿勢を置き、その後 `doRotTrans` で時間補間を進めます。
6. `Animation.setData(skeleton, bind_shape_matrix)` では、bone ごとに `Task` を作り、各キー区間について `[0, putRotTransByMatrix, ...]` と `[time, doRotTrans, [1.0]]` のコマンドペアを登録します。つまり、補間の元になるコマンド列はこの段階で組み立てられます。
7. `Schedule.directExecution()` は各 bone 用 `Task` に同じコマンドを配り、`time > 0` の場合は `insertCurrentCommand()` で実行中のコマンド列へ差し込みます。`Schedule.doCommand()` は `delta_msec` を計算して、各 `Task.execute(delta_msec)` を回します。
8. `Task.execute(delta_msec)` が `remaining_time` を見ながらコマンドを実行し、`execCommand()` を通して `CoordinateSystem.prototype.putRotTransByMatrix` や `CoordinateSystem.prototype.doRotTrans` を対象 bone に呼び出します。ここが実際の移動・回転の適用地点です。 

この分離がある理由は、ステートマシンと補間実行を混ぜないためです。`AnimationState` は「今どのアニメーションを見せたいか」を決めるだけにしておくと、入力条件やゲームプレイ条件の変更で補間コードを壊しにくくなります。逆に `Task` は「命令を時間でどう刻むか」だけに集中できるので、同じ仕組みをアニメーション以外のコマンド列にも流用しやすくなります。`webg/samples/janken` のようなコマンド型では、入力を受けた瞬間に `setState()` で `action` を出し直し、その `action` が `Animation` と `Task` のコマンド列へ流れます。`webg/samples/animation_state` のような条件評価型では、毎フレームのコンテキストからステートを選び、そのステートが下層の `action` を始めます。どちらの場合も、実際に bone を動かしているのは `Task` です。デバッグしたいときは、`AnimationState.getDebugInfo()` でステート遷移を見て、`Action.getActionInfo()` で現在の `action` / `pattern` を見て、`Animation` の `getClipInfo()` や `Task` のコマンド列を追うと、どの層で見え方が変わったかを切り分けやすくなります。 

## `webg/samples/animation_state` の `N0` を使った具体例

`webg/samples/animation_state` は、`Action` と `AnimationState` の両方を現行サンプルとして追える最短の参照先です。`main.js` では `hand.glb` のポーズ区間を次の `pattern` と `action` として登録し、そのあと `AnimationState` から `pose0` を開始しています。 

```js
const pattern = { id: "N0", fromKey: 12, toKey: 13, entryDurationMs: 250 };
action.addPattern(pattern);
action.addAction(pattern.id, [pattern.id]);
animationState.setState("pose0", {
  context: {
    desiredPoseId,
    nowMs: app.space.now()
  },
  force: true
});
```

ここでの呼び出し順は、コード上では次のようになります。 

1. `AnimationState.setState("pose0")` が `pose0` ステートを `currentState` にする
2. `AnimationState.playStateTarget()` が `state.action === "N0"` を見て `Action.start("N0")` を呼ぶ
3. `Action.start("N0")` が先頭 `pattern` として `N0` を選び、`transitionToKey(250, 12, 13)` を呼ぶ
4. `transitionToKey(250, 12, 13)` は `anim.startTimeFromTo(250, 12, 13)` を呼ぶ
5. `Animation.startTimeFromTo(250, 12, 13)` は `transitionTo(250, 12, 13)` を呼ぶ
6. `transitionTo()` は、各 bone について `this.poses[i][12]` を `args` に入れ、`ones` に `1.0` を入れる
7. `Schedule.directExecution(0, CoordinateSystem.prototype.putRotTransByMatrix, args)` が、各 `Task` に開始姿勢を渡す
8. `Schedule.directExecution(250, CoordinateSystem.prototype.doRotTrans, ones, 24, 25)` が、各 `Task` に `doRotTrans(1.0)` を 250 ms かけて流す
9. `Task.execute(delta_msec)` が毎フレーム呼ばれ、`remaining_time` を減らしながら `doRotTrans(1.0)` を進める
10. `CoordinateSystem.doRotTrans(1.0)` が `execRotation()` と `execTranslation()` を呼び、回転は `slerp`、移動は直線補間で反映される。 

この例で重要なのは、`12 -> 13` は **pattern のキー範囲** であって、`250 ms` は **その pattern を再生するために与えた時間** だということです。`fromKey / toKey` は `clip` のキーを選ぶためのインデックスで、`entryDurationMs` はランタイム側の補間時間です。`animation_state` では `AnimationState.update()` が内部で `action.play()` まで進めるため、サンプル側は desired state を更新するだけで済みます。`webg/samples/janken` と `webg/samples/animation_state` も同じ `pattern` 定義を使いますが、こちらは `AnimationState` が「どの `action` を始めるか」を選ぶだけです。実際の移動・回転の補間は、同じく `Action -> Animation -> Schedule -> Task -> CoordinateSystem` の順で流れます。 

## `startFromTo()` 系と `startTimeFromTo()` 系の違い

`webg` では、見た目は似ていても `startFromTo()` と `startTimeFromTo()` で役割が違います。違いを一言で言うと、`startFromTo()` は **clip のキー区間をそのまま再生する入口** で、`startTimeFromTo()` は **現在姿勢から `fromKey` へ入るための時間を別指定する入口** です。`Animation.startFromTo(keyFrom, keyTo)` の実装は次のようになっています。 

```js
// キー区間を直接再生する
startFromTo(keyFrom, keyTo) {
  this.setPose(keyFrom);
  this.schedule.startFromTo(keyFrom * 2, keyTo * 2 - 1);
}
```

ここでの動きはかなり素直です。`setPose(keyFrom)` が `fromKey` の姿勢をその場で適用し、`schedule.startFromTo(keyFrom * 2, keyTo * 2 - 1)` が命令列のその区間を `clip` 側の時間差で進めます。たとえば `startFromTo(12, 13)` なら、`setPose(12)` のあとに `schedule.startFromTo(24, 25)` が呼ばれます。つまり、**12 の姿勢にはすでに着いた状態から、12 -> 13 の区間を `clip` 側の時間差で再生する** 形です。`fromKey` へ入るための追加補間はありません。 

これに対して `Animation.startTimeFromTo(time, keyFrom, keyTo)` は、`entryDurationMs` を別に与えて、現在姿勢から `fromKey` へ入るための補間を先に作ります。 

```js
// 指定した時間でキーへ入り込む
startTimeFromTo(time, keyFrom, keyTo) {
  this.transitionTo(time, keyFrom, keyTo);
}
```

`transitionTo(250, 12, 13)` では、各 bone について `this.poses[i][12]` を集め、`schedule.directExecution(0, CoordinateSystem.prototype.putRotTransByMatrix, args)` と `schedule.directExecution(250, CoordinateSystem.prototype.doRotTrans, ones, 24, 25)` を順に呼びます。その結果、**現在姿勢から 12 の姿勢へ 250 ms で入る** 処理と、**そのあと 12 -> 13 の `clip` 区間へ入る** 処理が 2 段構成で行われます。`N0` のような `pattern` がこの形を使うのは、欲しいのが「`clip` 区間をそのまま見ること」ではなく、「今の姿勢から次の状態へ自然に入る entry」だからです。したがって、`startFromTo()` は `clip` そのものを確認する用途に向き、`startTimeFromTo()` は `idle -> walk` や `rock -> scissors` のような状態切り替えに向いています。 

## サンプルから見た入口の整理

実際のサンプルから触れる入口を整理すると、次のようになります。`webg/samples/animation_state/main.js` では、`Action` の `pattern` / `action` を現行サンプル上で組み立て、サンプル側は `desiredPoseId` を変えるだけにし、`AnimationState` へ制御を委譲しています。`AnimationState.update()` により、内部では `Action.start()` と `action.play()` が進みます。`webg/samples/janken/main.js` では、入力に応じて `AnimationState.setState(..., { force: true })` を呼び、同じく `Action` へ流します。`webg/samples/tile_sim/alpha_actor.js` では、`human.glb` の `clip` 情報を現行ランタイム上で読み、TileMap actor の idle / move 表示へつなぎます。こちらは `Action` の hand pose 切り替えではなく、スキニング付き glb をワールド上へどう置くかを見る入口になります。 

この整理から分かるとおり、`startFromTo()` をサンプルから直接呼ぶ場面はほぼありません。`startFromTo()` は `Animation` の内部で、`clip` 全体の再生や `Action` が作ったコマンド列を処理するための下位関数です。サンプル側が意識するのは主に `Action.startPattern()`、`Action.start()`、`AnimationState.setState()` であり、その先で `startTimeFromTo()` 系へ落ちていく、と考えると追いやすくなります。 

## `webg/samples/animation_state` と `webg/samples/janken` の読み方

`webg/samples/animation_state` は、glTF 版の `hand.glb` を使って `AnimationState` の条件評価型を確認する現行サンプルです。サンプル側は desired state を更新するだけにし、どの `action` を始めるかの判断は `AnimationState` へ委譲しています。サンプル側で `Action.start()` を直接呼ぶ構成では、「どの `pattern` / `action` を始めるか」をサンプルコード自身が明示的に持ちます。一方 `webg/samples/animation_state` はサンプル側が `desiredPoseId` を更新し、`AnimationState` が `pose0` から `pose5` のステートを管理し、それぞれに対応する `N0` から `N5` のアクションを開始します。この違いは、役割の分かれ方の違いです。前者は「再生対象の選択」までサンプル側が持ち、後者は「どの状態を望むか」だけをサンプル側が持ちます。そのぶん後者は、状態遷移の条件を増やしたいときに整理しやすくなります。さらにこのサンプルでは、`1` から `6` で desired state を変えたり、`/` で次の desired state へ進めたり、`P` で自動巡回したり、`@` で current state を再始動したりできます。HUD では current state、target state、現在の action、現在の pattern、最後の transition が出るので、「望んだ state」と「実際に動いている action」の関係を追えます。 

これに対して `webg/samples/janken` は、`AnimationState` のコマンド型利用を確認するサンプルです。条件評価型と違い、毎フレーム次のステートを計算するより、「入力があった瞬間にその手を出す」ことが自然な場面ではこちらの形が合います。`webg/samples/animation_state` は `desiredPoseId` を毎フレーム更新し、`update(context)` が `transitions` を評価して遷移先を決めます。つまり条件評価型です。これに対して `webg/samples/janken` は、入力イベントごとに `setState(choiceId, { force: true })` を呼び、`update()` は action 進行とステート情報の維持だけを行います。ここでの違いは、どこで判断するかです。前者は「毎フレームの条件」が判断材料で、後者は「入力イベントそのもの」が判断材料です。`janken` では、毎フレーム条件を見て `rock / scissors / paper` を選ぶより、`G`、`C`、`P` の入力を受けた瞬間にその手を出したいです。また、同じ手を続けて出し直すこともあるので、同じステートを再始動できることが重要です。これは `transition` の真偽を毎フレーム判定するより、「同じステートを `force` 付きで出し直すコマンド」と考えるほうが自然です。そのため `janken` は、`AnimationState` が条件評価型でしか使えないわけではない、ということを示す重要なサンプルでもあります。 

## 補間問題を切り分けるときの結論

過去の検証を通して共有しておきたい結論は、補間品質の問題を `AnimationState` の有無だけで説明しないことです。見え方の問題は、もっと下の層で起きていることが多いです。まず、`AnimationState` は補間品質の改善機構ではありません。`AnimationState` は「何を再生するか」を決めるヘルパーです。そこにステートマシンを導入しても、中間ポーズが自動生成されたり、cross-fade が自動追加されたり、sparse なキーフレームが自然なモーションに変換されたりはしません。こうした話は、`Animation` や `Action`、あるいはアセット側の設計の話です。さらに、ポーズの見え方はキー配置と `entryDurationMs` に強く依存します。hand pose のようなサンプルでは、`2 -> 3` や `12 -> 13` のような短い保持区間を多用します。このとき見え方を大きく左右するのは、現在姿勢から次の key pose までの距離と、`entryDurationMs` をどれだけ取るかです。1 本の `clip` の中に確認用ポーズだけが疎に並んでいるアセットを、そのままステート遷移へ使うと、ポーズによって滑らかに見える場合と急に切り替わる場合が混在します。これはステートマシンの問題ではなく、アセット / `pattern` 設計の性質です。したがって、見え方が良くないときはステートではなく、まずキー配置と `pattern` を疑うのが正しい順番です。 

また、停止後再開では `Schedule` の基準時刻も見え方へ効きます。過去の検証では、停止後に action を再始動したとき、`Schedule` の基準時刻更新が不足すると、最初の補間が一気に進んで見えることが確認されました。つまり、同じ `fromKey / toKey` と `entryDurationMs` を使っていても、時間基準がずれると見え方が変わります。そのため、見え方を切り分けるときは `pattern` だけでなく、再始動直後の最初のフレームで大きな delta time が入っていないか、pause / resume / restart のあとで時間基準が適切に更新されているか、ステートを切り替えたフレームで再生をすぐ進めていないかも見る必要があります。最終的な切り分けの順番としては、まずアセット側を見る、次に `clip` の直接再生で確認する、次に `Action` を見る、その次に `AnimationState` を見る、最後に `Schedule` を見る、という順が整理しやすくなります。この順で見ると、「アセットのキーフレーム問題」と「ステートマシンの使い方問題」と「時間管理問題」を混同しにくくなります。 

## 今後の拡張余地

現時点の `AnimationState` は、できるだけ薄いステートマシンとして設計されています。今後の拡張としては、`clip blend / cross-fade`、ランタイムイベントフック、ステートごとの entry / exit 診断情報ヘルパー、複数レイヤー、上半身 / 下半身マスク、blend tree などが考えられます。ただし、これらは何でも追加すればよいわけではありません。現在の `webg` の規模では、まず `Action` をコントローラーとする薄いステートマシンを保ったほうが、人間にも読みやすく、切り分けもしやすいです。今後の拡張は、「便利だから入れる」のではなく、「役割の分離を保ったまま必要か」で判断するのがよいです。 

## 付録 1: `hand.glb` のキー区間と姿勢の対応

hand 系サンプルを読むときに迷いやすいのは、`fromKey / toKey` がどの姿勢を指しているかと、同じ姿勢が複数フレーム続いている理由です。この付録は、その 2 点をまとめて確認するためのものです。`2026-04-07 JST` 時点の `webg/samples/gltf_loader/hand.glb` では、ポーズの保持区間を次のように読むと分かりやすくなります。 

| key 区間 | 姿勢       | `animation_state` / `janken` の pattern |
| ------ | -------- | -------------------------------------- |
| 0      | レストポジション | なし                                     |
| 1      | グー       | なし                                     |
| 2-3    | 1本伸ばす    | `N1: 2 -> 3`                           |
| 4-5    | チョキ      | `N2: 4 -> 5`                           |
| 6-7    | 3本伸ばす    | `N3: 6 -> 7`                           |
| 8-9    | 4本伸ばす    | `N4: 8 -> 9`                           |
| 10-11  | パー       | `N5: 10 -> 11`                         |
| 12-13  | グー       | `N0: 12 -> 13`                         |

ここでの `fromKey / toKey` は、`Action` がそのポーズを見せるための再生区間です。`entryDurationMs: 250` は、その区間へ入るときにランタイム側で 0.25 秒かけて姿勢を近づける指定です。同じ姿勢が連続している区間は、単なる重複ではありません。その姿勢を一定時間見せるためであり、次の姿勢へ入る直前に保持区間を作るためであり、再生時に急に跳ねないように遷移の見え方を落ち着かせるためでもあります。たとえば `4-5: チョキ` なら、`4` から `5` のあいだを使ってチョキの姿勢を保持し、そのポーズをサンプル側のステートから再利用できます。ここを「同じ絵が 2 回入っている」と読むより、「その姿勢の滞在時間が表現されている」と読むほうが実態に合います。入力や desired state の変化に応じて姿勢を切り替えるサンプルでも、この保持区間には意味があります。入力直後の切り替えを自然に見せやすくなり、あとで `Action` の別再生や `AnimationState` の別の使い方へ流用しやすくなり、アセット側のキーフレーム構成としても「どの姿勢をどのくらい見せるか」を表しやすくなるからです。 

Blender から glTF / GLB を出すときの注意も、ここで押さえておくと役立ちます。`webg/samples/gltf_loader/hand.glb` のように glTF / GLB を使う場合も、アニメーションをキーフレームベースで扱いたいなら、`CUBICSPLINE` ではなく `LINEAR` で出すことが基本です。ただし current importer は `CUBICSPLINE` を受け取ったときに前後の tangent を読み飛ばし、中央の値だけを取り出して `LINEAR` として再生します。つまり、`CUBICSPLINE` はそのまま厳密再生するのではなく、`LINEAR` へ近似変換したうえで扱う前提です。Blender から glTF / GLB を出すときは、`Sampling` を無効にし、全フレームのベイクではなくキーフレームのみを出力する方向へ寄せ、補間も `LINEAR` にしておくのが安全です。また、アーマチュアでは必要な bone / action がきちんと key を持っていることも確認する必要があります。この条件がそろうと、glTF / GLB も「毎フレームの近似データ」ではなく「元のキーフレーム列」を保ちやすくなります。逆に `Sampling` が有効なまま、あるいは補間が `CUBICSPLINE` のままだと、`gltf_loader` 側では tangent を落として `LINEAR` に近似するので、厳密な曲線再現は期待しないほうがよいです。現時点の `webg` では、hand 系の glTF animation は `LINEAR` 前提で考えるのが安全です。 

## インポートしたアニメーション付きモデルのサイズをどう決めるか

スキニングメッシュやアニメーション付きモデルのサイズ調整で迷いやすいのは、「glTF 側のルートノードが持つスケール」と「サンプル側で見た目のために掛けたいスケール」が混ざりやすいことです。ここを曖昧にしたまま `0.8` や `0.7` のような値を試すと、一見合っても、別アセットを入れたときやインポーターの条件が変わったときに再び判断がぶれます。`webg` では、アニメーション付きモデルの大きさを次の 3 段階に分けて考えると整理しやすくなります。

1. インポート済み mesh 自体の大きさ
2. glTF root node が持つスケール
3. サンプル / アプリ側で「このワールドに対してどのくらいで見せたいか」という表示倍率

大事なのは、1 と 2 はアセットから読める値であり、3 はアプリケーション側の設計値だという点です。1 と 2 を読まずに 3 だけを調整すると、「なぜその値なのか」が残りません。逆に 1 と 2 を先に測れば、3 は「このワールドで欲しい高さは何か」という設計判断だけに切り分けられます。 

アニメーション付きモデルのサイズを論理的に決めるには、最低でも 2 つを測ります。1 つはインポート済み mesh のバウンディングボックス高さ、もう 1 つは glTF root node が持つインポート済み一様スケールです。`tile_sim` の `human.glb` では、これを `webg/samples/tile_sim/alpha_actor.js` で次の 2 つに分けて扱っています。`measureRuntimeShapeBounds()` はインポート後の shape 群から `bounds.height`、`bounds.minY`、`bounds.centerX`、`bounds.centerZ` を読み、`resolveImportedRootUniformScale()` は root node の `localMatrix` からインポート済み root scale を読みます。ここで `bounds.height` は「shape geometry そのものの高さ」です。まだ root のローカルスケールは入っていません。一方で `importedRootScale` は「root node の local transform があとから何倍するか」を表します。つまり、この 2 つは別物です。glTF / GLB では、見た目の大きさが geometry だけで決まるとは限りません。DCC ツール側で armature root や mesh root にスケールが入っていることがあり、そのスケールはアニメーションや node 階層の意味の一部でもあります。これを「大きすぎるから無視する」としてしまうと、そのアセットが本来持っていた transform 情報をサンプル側で落としてしまいます。そこで `webg` では、root のローカルスケールは復元したまま使い、最後にアプリ側の node scale を逆算して全体の大きさを合わせる考え方を取っています。こうすると、インポート済みデータの意味を残したまま、ワールドに対する表示サイズだけを整えられます。 

考え方を式にすると、アニメーション付きモデルの最終高さは次のように読めます。 

```text
finalHeight = bboxHeight * importedRootScale * runtimeNodeScale
```

ここで `bboxHeight` はインポート済み mesh のバウンディングボックス高さ、`importedRootScale` は glTF root のローカル transform が持つ一様スケール、`runtimeNodeScale` はサンプル / アプリ側で最後に掛ける node scale です。もし「このアセットをワールド上で最終的に高さ `targetHeight` で見せたい」と決めるなら、逆算すべき値は次の 1 つです。 

```text
runtimeNodeScale = targetHeight / (bboxHeight * importedRootScale)
```

この式の利点は、試行回数ではなく測定値だけで決まることです。`bboxHeight` と `importedRootScale` はインポート済みデータから取れます。残る `targetHeight` だけがアプリケーション設計の値です。つまり、「アセットをどう読むか」と「ワールドでどの大きさにしたいか」をきれいに分けられます。`tile_sim` では、式をそのまま高さではなく比率で持っています。`webg/samples/tile_sim/alpha_actor.js` の `desiredDisplayScale` がそれです。 

```js
const importedRootScale = resolveImportedRootUniformScale(roots);
const runtimeNodeScale = desiredDisplayScale / importedRootScale;
```

この書き方は、`bboxHeight` を共通基準として扱うときに読みやすくなります。意味としては次と同じです。 

```text
targetHeight = bboxHeight * desiredDisplayScale
runtimeNodeScale = targetHeight / (bboxHeight * importedRootScale)
                 = desiredDisplayScale / importedRootScale
```

つまり `desiredDisplayScale` は、「`bboxHeight` に対して最終的に何倍で見せたいか」という設計値です。`runtimeNodeScale` 自体を直接いじるのではなく、`desiredDisplayScale` を先に決め、`runtimeNodeScale` はインポート済みデータから逆算する、という順番にすると判断がぶれにくくなります。 

`2026-04-02 JST` 時点で `webg/samples/tile_sim/human.glb` を確認すると、root node は 1 つで、`sx = sy = sz = 1.4680135250091553`、すなわち `importedRootScale = 1.4680135250091553` を持っていました。これは root scale が一様であることを意味します。したがって `tile_sim` では、この値をそのまま `importedRootScale` として使えます。そのうえで、現在の `desiredDisplayScale` は Alpha / Warden が `0.70`、Bravo / Mule が `0.64` です。したがって、実際に `offsetNode` へ掛かる `runtimeNodeScale` は、Alpha / Warden で `0.70 / 1.4680135250091553 = 0.47683484387218733`、Bravo / Mule で `0.64 / 1.4680135250091553 = 0.4359632858259999` になります。これを掛け戻すと、それぞれほぼ `0.70`、`0.64` となり、glTF 側の root scale を捨てずに、最終表示倍率だけをアプリ側の設計値へそろえられていることが分かります。 

`tile_sim` の `human.glb` actor では、バウンディングボックスは単にカメラ目安に使うだけではありません。`webg/samples/tile_sim/alpha_actor.js` では、`bounds.height` を cell 内の前後左右オフセットの基準量、`bounds.minY` を足元を tile 上面へ合わせるための baseY 補正、`bounds.centerX / bounds.centerZ` を cell 中央へ寄せるためのオフセット補正にも使っています。ここでも重要なのは、「これらは最終的に見せたい倍率で計算する」ということです。`tile_sim` では `desiredDisplayScale` を使っているので、インポート済み root scale が変わっても、cell 上でのまとまり方や足元位置は最終表示サイズに対して一貫しやすくなっています。 

最後に迷いやすいのは `targetHeight` をどう決めるかですが、ここはインポーターの問題ではなくサンプル設計の問題です。判断基準としては、代理ボディと同じくらいに見せたい、`BALL_RADIUS` やマーカー高さに対して何倍にしたいかで決めたい、TileMap の cell 上面に対してどのくらいの占有感を持つべきかで決めたい、のどれかを先に決めると整理しやすくなります。たとえば `tile_sim` のように論理移動の土台が `BALL_RADIUS` を持っている場合は、次のように規格値を持つ考え方が自然です。 

```text
targetHeight = BALL_RADIUS * K
```

この形なら、「この model は 0.7 倍だった」ではなく、「このサンプルの前線ユニットは ball 半径の何倍の高さにする」と説明できます。サイズが期待とずれるときは、`bounds.height` を確認する、`importedRootScale` を確認する、`targetHeight` または `desiredDisplayScale` を確認する、最後に `runtimeNodeScale` を確認する、という順で見ると、「インポーターの問題」「アセットの問題」「サンプルの設計値の問題」を混同しにくくなります。結局のところ、アニメーション付きモデルのサイズ決定で大事なのは、`runtimeNodeScale` を直接いじることではありません。まずインポート済みデータから `bboxHeight` と `importedRootScale` を測り、そのうえでアプリケーション側が欲しい `targetHeight` を決め、最後に `runtimeNodeScale` を逆算することです。この流れにしておくと、glTF 側のローカルスケールを捨てない、インポート済みデータの意味を保てる、サンプルのワールド内の規格を文書へ書ける、別アセットへ差し替えても再計算しやすい、という利点があります。`tile_sim` の `human.glb` で行った調整は、この考え方を具体的なコードと実測値へ落とした例です。 

## まとめ

この章で最も大事なのは、現行サンプルを「アニメーション一般論」としてではなく、「どの層をどこで使っているか」という視点で読むことです。`animation_state` は条件評価型の `AnimationState`、`janken` は入力駆動の `AnimationState`、`tile_sim` はスキニング付き glb を TileMap actor として扱う実例です。どのサンプルでも、サンプル側が決めるのは「何を再生するか」であり、実際の補間や bone への反映は `Animation -> Schedule -> Task -> CoordinateSystem` が担当しています。ここが見えると、入力処理、状態選択、補間実行を混ぜにくくなります。 

また、この章では、`startFromTo()` と `startTimeFromTo()` の違い、`hand.glb` のキー区間の読み方、保持区間の意味、glTF / GLB のエクスポート条件、そして `tile_sim` の actor サイズを `bboxHeight`、`importedRootScale`、`runtimeNodeScale` に分けて決める考え方を整理しました。ここまで押さえておくと、アニメーション付きモデルの見え方が崩れたときも、「キー区間の問題なのか」「入り込み時間の問題なのか」「時間基準の問題なのか」「サイズ設計の問題なのか」を順に切り分けやすくなります。次章では、こうして扱ってきた HUD、dialogue、panel といった表示まわりを、UI 設計という観点で整理していきます。
