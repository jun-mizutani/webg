# WGSLの読み方

本章では、`webg` のシェーダーを解読するために必要な WGSL の基礎を、体系的に解説します。単に構文を列挙するのではなく、「CPU 側の setter で設定した値が WGSL のどこに到達するのか」「サンプルを読む際にどこから着手すべきか」という視点を含めて整理します。

`webg` の利用者がすぐに WGSL を書き始める必要はありません。しかし、ノーマルマップが機能する仕組みや、フラットシェーディングによる面単位の陰影生成、スキニングの処理工程、そして binding が増加した際に整合性をどう取るべきかを理解するためには、WGSL の構造を把握することが不可欠です。

本章では、`book/examples/Phong.js`、`book/examples/NormPhong.js`、`book/examples/BonePhong.js`、`book/examples/BoneNormPhong.js`、および `webg/SmoothShader.js` の読解を前提として説明します。なお、`Phong` 系 4 本のファイルは `webg` のコア機能ではなく、比較検証のために `book/examples` に配置された学習用サンプルファイルです。一方、`webg/SmoothShader.js` は `Phong` 系 4 本の機能に加え、フラットシェーディングの切り替え機能も含んでいます。

まず押さえていただきたいのは、WGSL を単なる「GPU に渡す文字列」としてではなく、CPU 側の実装と対になったものとして捉えることです。たとえば `setAmbientLight(0.3)` を実行したとき、その値は単に JavaScript の変数に格納されて終わるわけではありません。値はユニフォームバッファの特定のオフセットへ書き込まれ、`queue.writeBuffer()` を介して GPU へ転送され、最終的に WGSL の `uniforms.params0.x` のようなフィールドから参照されます。したがって、WGSL 単体ではなく、CPU 側の setter と対比させて理解することが重要です。

また、シェーダーを読解する際に、最初から複雑な数式を追う必要はありません。まずは `@vertex` と `@fragment` のエントリーポイント、`@group` / `@binding` によるリソース参照、そして `@location` と `@builtin` の受け取り口を確認してください。これにより、そのシェーダーが「何を読み込み、何を返すのか」という全体像が見えてきます。`webg` では `Shape.setMaterial()` や `shader.setDefaultParam()` を通じて多くのパラメータが注入されるため、WGSL だけを読んでも意図を完全に把握することは困難です。本章の目的は、サンプルコード、JavaScript クラス、そして WGSL の 3 点をつなぐための解読手順を提示することにあります。

## WGSL を読む目的

WebGPU において、CPU 側の JavaScript はバッファやテクスチャの準備を担い、WGSL は頂点ごとの座標変換とピクセルごとの色決定を担います。WGSL を読む目的は、大きく分けて次の 3 点に整理できます。

第一に、利用者として、現在使用しているマテリアルがどのような前提条件で動作しているかを把握すること。第二に、開発者として、新しいパラメータの追加やデバッグ表示の実装を行う際に、どこを変更すべきかを特定すること。そして第三に、サンプルを拡張する際に、安全に踏襲すべき構造を見極めることです。

`webg` の文書では、シェーダーを単なる「見た目を決める部品」としてではなく、リソースの紐付けと描画パイプラインの一部として扱います。そのため、WGSL の読解においても「数式の意味」だけでなく、「どのリソースがどこから供給されているか」というデータフローを重視します。

## エントリーポイントの確認

WGSL の読解を始める際は、まず頂点シェーダーとフラグメントシェーダーのエントリーポイントを確認してください。最小構成の例を以下に示します。

```wgsl
struct VSIn {
  @location(0) position : vec3f,
};

struct VSOut {
  @builtin(position) position : vec4f,
};

@vertex
fn vsMain(input : VSIn) -> VSOut {
  var out : VSOut;
  out.position = vec4f(input.position, 1.0);
  return out;
}

@fragment
fn fsMain() -> @location(0) vec4f {
  return vec4f(1.0, 0.0, 0.0, 1.0);
}
```

構造は至ってシンプルです。`@vertex` が頂点シェーダーの入口であり、`@fragment` がフラグメントシェーダーの入口となります。`VSIn` は頂点バッファから受け取る入力データを、`VSOut` は頂点シェーダーから後段（ラスタライザおよびフラグメントシェーダー）へ渡す出力を定義しています。`@builtin(position)` はクリップ空間上の位置を示し、`@location(0)` はカラーアタッチメント 0 へ返される色を指定します。この基本構造を把握することで、「頂点側で位置を決定し、フラグメント側で色を決定する」というフローが明確になります。

WGSL では、`f32`、`i32`、`u32`、`vec2f`、`vec3f`、`vec4f`、`mat4x4f` といった型が使用されます。`webg` のシェーダーで頻繁に登場するのは、以下の型です。

| 型              | 意味             | `webg` での典型例              |
| -------------- | -------------- | ------------------------- |
| `vec2f`        | 2 要素の浮動小数点ベクトル | UV座標、画面座標                   |
| `vec3f`        | 3 要素の浮動小数点ベクトル | position、normal           |
| `vec4f`        | 4 要素の浮動小数点ベクトル | color、light、パラメータ束        |
| `mat4x4f`      | 4x4 行列         | projection、model-view     |
| `array<vec4f>` | `vec4` の配列     | bone palette、詰め込み済みユニフォーム |

`webg` のシェーダーでは、複数のパラメータを `params0` や `params1` といった `vec4f` 型にまとめて格納する手法が採られています。これは、フィールド数を抑制してユニフォームレイアウトを整理するためです。読解の際は、`params0.x` のような記述を見かけたら、「JavaScript 側にこれに対応する setter が存在するはずだ」と想定することで、実装間のつながりを把握しやすくなります。

## リソースの受け渡し（Location と Builtin）

`@location(n)` は、頂点バッファの属性や、頂点シェーダーからフラグメントシェーダーへ補間されて渡される値のインデックスを指定します。

```wgsl
struct VSIn {
  @location(0) position : vec3f,
  @location(1) normal : vec3f,
  @location(2) texCoord : vec2f,
};
```

ここで重要なのは、WGSL 側の `@location` 番号が、CPU 側の `shaderLocation` 設定と完全に一致している必要がある点です。位置と法線の番号が不整合を起こすと、ライティング計算が不正な結果となります。WGSL 側だけを確認して原因が特定できない場合は、パイプライン作成時の属性設定を確認してください。また、頂点シェーダーの出力における `@location` は、「フラグメントシェーダーへ何を渡しているか」を示しており、ここにはワールド法線、UV、視線方向などの中間値が流れます。

`@builtin` は、GPU が内部的に管理する特殊な値を参照するために使用されます。`@builtin(position)` はクリップ空間上の最終的な位置であり、頂点シェーダーは必ずこの値を返さなければなりません。また、`@builtin(front_facing)` はフラグメントシェーダーでポリゴンの表裏判定に使用でき、`webg` の `backface_debug` 機能はこの仕組みを利用しています。

```wgsl
struct FSIn {
  @location(0) uv : vec2f,
  @builtin(front_facing) frontFacing : bool,
};
```

`front_facing` を利用することで、裏面のみを別色で描画するといったデバッグ表示が可能です。ただし、条件分岐の記述方法によってはテクスチャサンプリングの制約に抵触する場合があるため、単なる `bool` 値ではなく、「シェーダー全体の構造に影響を与える制御値」として扱うのが安全です。

WebGPU では、ユニフォームバッファ、テクスチャ、サンプラーを bind group と binding 番号で管理します。WGSL 側では以下のように参照されます。

```wgsl
@group(0) @binding(0) var<uniform> uniforms : Uniforms;
@group(0) @binding(1) var uTexture : texture_2d<f32>;
@group(0) @binding(2) var uSampler : sampler;
```

これらの記述を確認した際は、まず JavaScript 側で同一の binding 番号を持つ bind group レイアウトが定義されているか、`getBindGroup()` 等の処理でリソースが正しい順序で格納されているか、また `Shape.setMaterial()` で設定したテクスチャがどの binding に割り当てられているかを確認してください。`Phong` シェーダーではユニフォーム、カラーテクスチャ、サンプラーがひとまとめに構成されています。`NormPhong` ではこれにノーマルマップ用テクスチャが加わり、`BonePhong` や `BoneNormPhong` ではボーン関連のユニフォームサイズが増大するため、グループの分割構成が重要な読解ポイントとなります。

## ユニフォームバッファと CPU 側の対応付け

`webg` のシェーダーでは、ユニフォーム構造体の中に、行列、光源情報、フォグ設定、パラメータ束が順に定義されています。

```wgsl
struct Uniforms {
  projection : mat4x4f,
  modelView  : mat4x4f,
  normalMat  : mat4x4f,
  light      : vec4f,
  params0    : vec4f,
  params1    : vec4f,
};
```

読解のコツは、「フィールド名から CPU 側の setter を逆引きすること」です。たとえば `projection` であれば `setProjectionMatrix()`、`modelView` であれば `setModelViewMatrix()`、`normalMat` であれば `setNormalMatrix()`、`light` であれば `setLightPosition()`、そして `params0` / `params1` はマテリアルパラメータを格納する setter 群に対応しています。この対応関係を把握することで、WGSL は断片的な数式の集まりではなく、「JavaScript 側で準備した値を最終的に消費する場所」として理解できるようになります。

`webg` のシェーダーを解析する際は、「setter $\rightarrow$ ユニフォームへの格納 $\rightarrow$ WGSL のフィールド」という 3 段階のフローで追跡することが有効です。例として `Phong` シェーダーの場合、`shape.setMaterial("phong", { ambient: 0.3, specular: 0.9, power: 40.0 })` を呼び出すと、内部で `doParameter()` が各 setter を実行し、setter が `uniformData` の所定オフセットを更新し、`writeBuffer()` によって GPU へ転送され、最終的に WGSL の `params0` から値が読み出されます。この流れに慣れることで、パラメータを追加する際に修正すべき箇所が明確になります。

## テクスチャサンプリングの読解

WGSL でテクスチャを参照する典型的な記述は以下の通りです。

```wgsl
let baseColor = textureSample(uTexture, uSampler, input.uv);
```

ここで確認すべき点は、`uTexture` と `uSampler` がそれぞれどの binding に割り当てられているか、そして `input.uv` が頂点シェーダーから正しく渡されているか、の 3 点です。`webg` ではテクスチャ未設定時にデフォルトテクスチャを適用する仕様となっており、見た目上は正常に動作しているように見えることがあります。したがって、サンプリング式が存在していても、利用者が意図したテクスチャが正しく渡されているかは、`shape.setMaterial()` の設定側とあわせて確認する必要があります。

通常のカラーテクスチャには `textureSample()` を使用しますが、非一様な制御や微分計算が絡む場面では、WGSL の制約により `textureSampleLevel()` を使用する実装が見られます。`webg` の `NormPhong` および `BoneNormPhong` において、ノーマルマップの参照に `textureSampleLevel(..., 0.0)` が使用されているのは、ノーマルマップ固有の理由というよりも、「WGSL の言語仕様上の制約に基づいた安全な実装」を選択しているためです。

ここで重要なのは、「なぜレベルを固定してサンプリングしているのか」をデバッグの観点から理解することです。WGSL は GLSL と異なり、テクスチャサンプリングを記述できる場所に厳格な制約があります。WebGPU ではこの種のバリデーションが非常に厳密であるため、サンプルコードが正常に動作している理由を言語仕様の側面から押さえておくことは非常に有益です。

## ノーマルマップとフラットシェーディングの仕組み

`NormPhong` および `BoneNormPhong` では、接線ベクトル（Tangent）属性を必須としていません。その代わりに、フラグメントシェーダー内で `dpdx` / `dpdy` を用いた UV の微分から接線ベクトルおよび従法線ベクトルを再構成し、TBN 行列を構築しています。この微分演算は、`SmoothShader` の `flat_shading` において「現在描画している三角形の面法線」を算出する用途にも利用されています。

`NormPhong` におけるノーマルマップの処理を追う際は、以下の順序で確認すると理解しやすくなります。
1. `normal_texture` をサンプリングしている行を確認する。
2. サンプリングした法線値を、どの範囲（例: [0, 1] から [-1, 1]）へ変換しているかを確認する。
3. TBN 行列をどのように構築しているかを確認する。
4. `normal_strength` によって、どのように法線を補間しているかを確認する。

この順序で追うことで、ノーマルマップの効果が強すぎる、あるいは弱すぎるといった調整がどこで行われているかを特定できます。

一方、`SmoothShader` で `flat_shading` を有効にした場合、同一の bind group 構成（テクスチャ、ノーマルマップ、フォグ、スキニングパラメータ）を維持したまま、法線の算出方法のみを切り替えることができます。通常の `SmoothShader` は頂点法線を頂点シェーダーからフラグメントシェーダーへ補間して使用しますが、`flat_shading` 有効時の経路では、ビュー空間位置 `vPosition` の微分から、そのフラグメントが属する三角形の面法線を直接算出します。

```wgsl
let faceNormal = normalize(cross(dpdy(input.vPosition), dpdx(input.vPosition)));
let facing = select(-1.0, 1.0, input.frontFacing);
let nnormal = faceNormal * facing;
```

読解のポイントは以下の通りです。
- `input.vPosition` は、頂点シェーダーで `modelView * position` 計算が行われたビュー空間上の位置であること。
- `dpdx` と `dpdy` は、その位置が画面上でどちらにどれだけ変化しているか（微分値）を表していること。
- 2 つの変化ベクトルの外積を取ることで、その三角形の面法線が得られること。
- `frontFacing` を用いて向きを揃えることで、両面描画や `backface_debug` 時でも整合性が保たれること。

この手法では頂点法線を補間しないため、同一のメッシュであっても面ごとの陰影がはっきりと現れます。WGSL を読む際は、「最終的な法線 `nnormal` をどの計算で導出しているか」に注目することで、通常の `SmoothShader`、`flat_shading` 有効時の `SmoothShader`、および `BoneNormPhong` の違いを整理して理解できます。

## スキニングの読解

`BonePhong` および `BoneNormPhong` を読む際は、まず頂点シェーダーを確認してください。ここではボーンインデックス（bone index）とウェイト（weight）を用いて、複数のボーン行列を合成しています。

```wgsl
let i0 = i32(input.index.x) * 3;
let i1 = i32(input.index.y) * 3;
```

`webg` では 1 本のボーンを `vec4 x 3` の形式で保持しているため、4x4 行列をそのまま配置する設計とは記述が異なります。ここで「なぜ 3 なのか」という疑問が生じますが、これは位置変換に必要な成分のみを抽出して格納したパレット表現を採用しているためです。スキニング処理を追う際は、「頂点入力に bone index / weight が存在するか」「ユニフォーム側に bone palette が定義されているか」「複数本のボーン行列をどのように合成しているか」「変形後の `position` と `normal` がどこへ出力されているか」という順序で確認してください。

## ダイナミックオフセットと大量描画の設計

WGSL の構文自体にダイナミックオフセット（dynamic offset）の記述は現れませんが、`webg` のシェーダーを理解する上では極めて重要な概念です。`Phong` 系、`Font`、`Wireframe` などのシェーダーでは、1 つのユニフォームバッファに複数オブジェクト分のデータを連続して配置し、描画ごとにオフセットをずらして参照する設計を採用しています。

この設計の利点は、同一のシェーダーを使用するオブジェクトを大量に描画する際、描画ごとにバッファを再作成したり再バインドしたりする必要がないことです。WGSL 側でユニフォーム構造体が一定のサイズに厳密に整理されているのは、この運用を前提としているためです。つまり、WGSL のレイアウトを読み解くことは、描画パフォーマンスの最適化手法を理解することに直結します。

## 混同しやすい概念

`binding` と `location` は全く異なる概念です。`@binding(1)` と `@location(1)` は、同じ番号であっても意味が異なります。`binding` はテクスチャやユニフォームバッファなどの外部リソースの接続口であり、`location` は頂点属性やシェーダー間での補間値の受け渡し口です。番号のみで同一視しないよう注意してください。

また、`vec4f` にまとめられたパラメータは、意味を省略しているのではなく、メモリレイアウトを最適化するための実装上の表現です。`params0.x` といった記述は直感的ではありませんが、JavaScript 側の setter 名と対応付けることで、その意味を明確に把握できます。

さらに、GLSL の仕様をそのまま適用しないことも重要です。WGSL は GLSL と類似した部分が多いものの、binding の明示的な指定、型の厳密な判定、テクスチャサンプリングの制約など、細かな仕様が異なります。過去の経験で補完するのではなく、「このシェーダーが WebGPU の仕様に基づきどのように記述されているか」を都度確認することを推奨します。

## `webg` シェーダーの推奨読解順序

`webg` のシェーダーを解析する際は、以下の順序で追跡することをお勧めします。

1. `Shape.setMaterial()` の実装例を確認する。
2. シェーダークラス側でどのパラメータを受け取っているかを確認する。
3. `@group` / `@binding` のリソース構成を確認する。
4. 頂点シェーダーの入力（`VSIn`）と出力（`VSOut`）を確認する。
5. フラグメントシェーダーにおける色の決定ロジックを確認する。
6. 必要に応じて、CPU 側のユニフォーム更新処理を確認する。

この順序で追うことで、WGSL の数式が「何のために存在しているのか」という目的を見失わずに読解できます。

実際に `Shader.js` や `Phong`、`NormPhong`、`BonePhong`、`BoneNormPhong` を CPU 側の実装から追跡したい場合は、第 9 章「シェーダーの実装」を参照してください。

どのシェーダーをどのように使い分けるかをアプリケーション構築の視点から確認したい場合は、第 7 章「シェーダーとマテリアル」が対応しています。また、`WebgApp` を用いたアプリケーション全体の構成について再確認したい場合は、第 5 章「WebgApp によるアプリ構成」に戻ることで、知識を整理できるはずです。

本章の目的は、WGSL を単なる構文としてではなく、`webg` のサンプル、JavaScript クラス、そして描画結果を相互に結びつけるための「読解術」として身に付けることです。この視点を持つことで、シェーダーのカスタマイズやデバッグの効率が飛躍的に向上します。
