# UI表示の設計

この章では、`webg` の文字表示と情報表示をどう使い分けるかを考えます。`Message`、`DialogueOverlay`、`Diagnostics`、`DebugDock`、`FixedFormatPanel`、`UIPanel`、`Touch`、`WebgApp` の関係を、表示の役割ごとに見ていきます。

`webg` の表示系が分かりにくくなりやすい理由は、表示経路が 1 本ではないからです。canvas HUD、シーンの上に重ねる DOM UI、画面端へ固定する DOM、画面下へ固定する touch button DOM、デバッグドック、診断情報レポートという複数の経路があり、それぞれは必要があって分かれています。しかし文書やサンプルを個別に読み始めると、「短い案内はどこへ出すか」「会話は何を使うか」「調査用の情報はどこへ出すか」「入力代わりのボタンはどこへ出すか」が混在しやすくなります。この章の役割は、その判断基準を最初にそろえることです。

最初に押さえたいのは、API 名から選ぶのではなく、「誰に何を見せるか」で選ぶほうが迷いにくい、ということです。短い案内なら canvas HUD、会話や選択肢ならシーンの上に重ねる DOM オーバーレイ、キー入力の代わりの仮想ボタンなら画面下固定の `Touch`、調査用の詳細は診断情報とデバッグドック、長文の読み物や失敗理由は `FixedFormatPanel` が自然です。特に重要なのは、`Message` は短い ASCII 用であり、日本語や一般的な UTF-8 文章を表示する用途には使えない、という点です。会話、チュートリアル本文、選択肢は `DialogueOverlay` と `UIPanel` を使うのが標準になります。また `Diagnostics` は会話やタイトル表示とは別の役割を持ちます。これは表示物ではありますが、役割は「状態の記録」であり、ゲーム内の会話やタイトル UI とは分けて考える必要があります。さらに `Touch` はボタンを出す点では `UIPanel` と似ていますが、役割はまったく違います。`Touch` はキー入力の代わりに押す仮想ボタン群で、画面下に固定表示され、`InputController` やキー状態へ入力を流すための部品です。シーンアクションパネルやステータスパネルを組むための `UIPanel` とは分けて考えるほうが自然です。そして `WebgApp` は `setGuideLines()`、`setStatusLines()`、`setControlRows()`、`createHelpPanel()`、`startDialogue()`、`showFixedFormatPanel()` を通じて、表示の置き場所をまとめて扱いやすくします。新しいサンプルでは、まず `WebgApp` のヘルパーから始めると流れをそろえやすくなります。

なお、`WebgApp` は既定ではビューポート全体を使う構成ですが、`layoutMode: "embedded"` を使うと固定サイズ canvas を本文中へ埋め込み、HelpPanel、dialogue、`FixedFormatPanel`、touch controls も canvas host 基準で一緒に動かせます。教材ページでは、この違いを意識して「ビューポート固定の UI なのか、本文と一緒にスクロールする UI なのか」を決める必要があります。

`webg` の情報表示は、まず次の 7 分割で考えると整理しやすくなります。ここでは「どこに表示されるか」も一緒に押さえておきます。`Message` / `setGuideLines()` / `setStatusLines()` は canvas 上でシーンに重ねる領域に表示され、用途は利用者向けの短い ASCII HUD です。`DialogueOverlay` は canvas 外の DOM ですが、見た目としてはシーンの上に重ねて見せ、用途は会話、チュートリアル本文、選択肢のような UTF-8 の文章です。`UIPanel` も canvas 外の DOM で、シーンの上に重ねる UI を作る用途です。`FixedFormatPanel` は canvas 外の DOM で、既定では画面端へ固定表示し、用途はエラー、長文の説明、全文 diagnostics のような「読むためのテキスト」です。`Touch` は canvas 外の DOM で、既定では画面下に固定表示され、用途はキー入力の代わりに押す保持入力 / 単発入力ボタンです。`Diagnostics` は自体は表示部品ではなくレポートの生成層で、用途は状態をレポートとして作り、テキスト / JSON / コピー / 保存の出力へ流すことです。`DebugDock` は canvas 外で、デスクトップでのデバッグ時の右側固定 DOM です。用途は開発者向け情報の集約表示です。この 7 分割を先に決めておくと、「同じ内容を 3 か所に重ねる」状態を避けやすくなります。たとえば score と lives は `Message`、会話は `DialogueOverlay`、pause menu のボタンは `UIPanel`、入力代わりの仮想ボタンは `Touch`、読み込み失敗は `FixedFormatPanel`、probe やレポートは `Diagnostics`、PC の詳細操作は `DebugDock` という分担にすると、役割がぶれません。

## `Message` と canvas HUD

`Message` は、canvas 上でシーンに重ねる短い ASCII 文字列のための高レベル HUD です。`Text` より上位で、`id`、`anchor`、`block` の単位で置けるので、タイトルや短いステータス表示を扱いやすくなっています。ここに向いているのは、START、PAUSE、HIGH SCORES、SCORE、TIME、LIVES のような短くて更新頻度の高いものです。ゲーム内のタイトルやスコアボードは、利用者が一瞬で読めることが大事なので、DOM オーバーレイより canvas HUD のほうが自然なことが多いです。ここで重要なのは、`Message` では日本語や一般的な UTF-8 文章を出せないことです。長いか短いかに関係なく、日本語の説明を出したい時点で `DialogueOverlay` か `UIPanel` を使うべきです。

```js
app.setGuideLines([
  "[Enter] START",
  "[ArrowLeft/Right] MOVE"
], {
  anchor: "bottom-left",
  x: 0,
  y: -2,
  color: [0.92, 0.96, 1.0]
});

app.setStatusLines([
  `SCORE ${score}`,
  `LIVES ${lives}`,
  `TIME ${timeLeft}`
], {
  anchor: "top-left",
  x: 0,
  y: 0,
  color: [1.0, 0.88, 0.72]
});
```

より直接 `Message` を触りたい場合は、次のように `id` を持たせます。

```js
app.message.setLine("title", "START", {
  anchor: "center",
  y: -2,
  color: [1.0, 0.96, 0.72]
});

app.message.setBlock("scoreboard", [
  `HIGH SCORES`,
  `1. ${scores[0]}`,
  `2. ${scores[1]}`,
  `3. ${scores[2]}`
], {
  anchor: "top-right",
  x: -1,
  y: 0,
  color: [0.92, 0.96, 1.0]
});
```

この使い方の代表サンプルは `webg/samples/breakout` です。タイトル、スコア、コンボ、時間、残機のようなゲーム寄りの短い表示を `Message` 系 HUD にまとめています。最小確認は `webg/unittest/message` が向いています。

## `DialogueOverlay` と `UIPanel`

`DialogueOverlay` は、会話とチュートリアル本文のための標準経路です。表示先は canvas 外の DOM ですが、見た目としてはシーンの上に重なって見えます。speaker、本文、選択肢、次へ、再表示のような構造をそのまま DOM オーバーレイに出せるので、日本語や長い説明文をそのまま扱えます。`DialogueOverlay` は DOM オーバーレイ専用の会話ヘルパーと考えるのが自然です。会話 UI を作るときは、`WebgApp.startDialogue()` でエントリを渡し、`nextDialogue()` と `chooseDialogue()` で進行だけを制御します。

```js
const intro = [
  {
    speaker: "guide",
    title: "Tutorial",
    lines: [
      "ようこそ",
      "ここでは移動と攻撃を順番に確認します"
    ]
  },
  {
    speaker: "guide",
    lines: [
      "先に進む道を選んでください"
    ],
    choices: [
      { label: "左へ行く", next: [{ speaker: "guide", lines: ["左の説明へ進みます"] }] },
      { label: "右へ行く", next: [{ speaker: "guide", lines: ["右の説明へ進みます"] }] }
    ]
  }
];

app.startDialogue(intro, {
  title: "Tutorial",
  footer: "Enter: next  1/2: choose  R: restart"
});
```

毎フレームは短い入力確認だけで十分です。

```js
if (app.wasActionPressed("next")) {
  app.nextDialogue();
}
if (app.wasActionPressed("choice1")) {
  app.chooseDialogue(0);
}
if (app.wasActionPressed("choice2")) {
  app.chooseDialogue(1);
}
```

この役割の標準確認は `webg/unittest/dialogue` です。UTF-8 の本文、分岐の差し込み、再開の流れを最小構成で追えます。

一方 `UIPanel` は、ボタン、パネル、ヒント、pill のような操作 UI を DOM オーバーレイに出すための部品です。表示先は canvas 外の DOM ですが、シーンの上に重ねる UI を作るための部品だと考えると分かりやすいです。これは会話専用ではなく、シーンアクションパネル、ツールパレット、プレビューカード、簡易メニューのような UI を組むための土台です。`DialogueOverlay` も内部では `UIPanel` を使っていますが、利用者が直接触るときは「会話」ではなく「操作パネル」を作りたい場面が中心です。Scene JSON のアクションボタンや、テーマプレビューの操作パネルが代表例です。

```js
const uiPanels = new UIPanel({
  document,
  theme: app.uiTheme.uiPanel
});

const layout = uiPanels.createLayout({
  id: "scene-tools",
  leftWidth: "minmax(300px, 420px)",
  rightWidth: "minmax(260px, 340px)",
  scrollColumns: true
});

const panel = uiPanels.createPanel(layout.left, { stack: true });
uiPanels.createEyebrow(panel, "Scene Actions");
uiPanels.createTitle(panel, "Runtime Controls", 1);
uiPanels.createHint(panel, "scene action と同じ処理を button から呼ぶ例");

const row = uiPanels.createButtonRow(panel);
const pauseButton = uiPanels.createButton(row, { text: "Pause Scene" });
const resetButton = uiPanels.createButton(row, { text: "Reset Camera" });

pauseButton.addEventListener("click", () => togglePause());
resetButton.addEventListener("click", () => resetCamera());
```

この役割を見るなら `webg/samples/scene` が最初の参照先です。左のアクションボタンと右のステータスパネルがあり、scene file と action handler の関係をまとめて見られます。テーマ適用まで含めて確認したいなら `webg/unittest/theme` が向いています。

### 通常サンプルの操作説明は `createHelpPanel()` を標準にする

`bloom` のような「教材寄りのサンプルで、操作説明をシーンの上へ出したいが、邪魔なときは畳みたい」という場面では、`WebgApp.createHelpPanel()` を標準の入口として使うのが自然です。サンプル側は 1 行 1 操作の line 配列を渡すだけで、左上に `Hide Help` / `Show Help` を持つパネルを置けます。デバッグドックの幅も `WebgApp` 側が追従するため、サンプルごとにドックのオフセットを計算し直す必要はありません。

```js
const helpLines = [
  "Drag or Arrow keys: orbit camera",
  "[ / ] or wheel: zoom",
  "[1]/[2] threshold -/+",
  "[3]/[4] strength -/+",
  "[space] pause"
];

app.createHelpPanel({
  id: "sampleHelpOverlay",
  lines: helpLines
});
```

この表示方法は、今後の通常サンプルで最初に選ぶ既定形として扱うのがよいです。例外は、`janken` や `circular_breaker` のようにゲーム HUD 自体が演出の一部になっているサンプルと、`tile_sim` のように説明量が多く独自レイアウトを持つサンプルです。そうした例外を除けば、「操作説明は畳める help panel、現在値は HUD、調査用の詳細は diagnostics」という分担が最も整理しやすくなります。

## `FixedFormatPanel` は長文とエラー用

`FixedFormatPanel` は、「短い HUD に入れるべきではない長文」を読む場所です。表示先は canvas 外の DOM で、通常は画面端へ固定表示されます。内容は `<pre>` と同じように整形済みテキストをそのまま読む前提です。エラー全文、長い diagnostics、説明メモ、読み込み結果の一覧など、落ち着いて読む必要があるテキストを扱います。短いステータス表示と長文を同じ場所へ混ぜると、利用者も開発者も読みづらくなります。ローダーのエラーやバリデーション失敗は `showErrorPanel()` へ、長い説明は `showFixedFormatPanel()` へ分けるのが安全です。

```js
try {
  await loadSomething();
} catch (err) {
  app.showErrorPanel(err, {
    id: "load-error",
    title: "scene load failed"
  });
}

app.showFixedFormatPanel([
  "Overview",
  `clipCount=${clipCount}`,
  `warningCount=${warningCount}`,
  `triangleCount=${triangleCount}`
].join("\n"), {
  id: "scene-overview"
});
```

この役割は `webg/samples/gltf_loader`、`webg/samples/collada_loader`、`webg/samples/json_loader` のようなローダーサンプルでよく出ます。`webg/unittest/theme` でも `FixedFormatPanel` のテーマ更新を確認できます。教材ページの途中へ canvas を置くときは、`WebgApp` の `layoutMode: "embedded"` と `showFixedFormatPanel()` を組み合わせると、長文パネルも canvas host 基準で動きます。スクロールする本文の中に 3D 実行例と補助説明を同居させたいときは、`webg/unittest/embedded` が確認用の最短例になります。

## `Touch` はキー入力代わりのボタン表示

`Touch` は、`UIPanel` のようにシーンアクションパネルを組む部品ではありません。表示先は canvas 外の DOM ですが、既定では画面下に固定され、役割は「キー入力の代わりに押すボタン」を出すことです。保持入力と単発入力を分けて扱い、`InputController` とつなぐとキーボードと同じキー状態へ流せます。モバイルや coarse pointer の端末では、見た目のカード UI よりも「どのキーの代わりに押すか」が大事です。そのため `Touch` は `UIPanel` よりレイアウトも単純で、画面下に横並びまたは複数行で収める前提になっています。シーン上のステータスや選択肢を見せたいときは `UIPanel`、移動や confirm を押させたいときは `Touch` と分けると、役割が混ざりません。比較名を迷ったときは [16_タッチ機能と入力.md](./16_タッチ機能と入力.md) の一覧を見ると、`space` や function key も含めてそろえやすくなります。

```js
import Touch from "./webg/Touch.js";

const touch = new Touch(document, {
  touchDeviceOnly: false
});

touch.create({
  groups: [
    {
      id: "move",
      buttons: [
        { key: "arrowleft", label: "←", kind: "hold" },
        { key: "arrowright", label: "→", kind: "hold" }
      ]
    },
    {
      id: "action",
      buttons: [
        { key: "enter", label: "Start", kind: "action" },
        { key: "r", label: "R", kind: "action" }
      ]
    }
  ],
  onPress: ({ key }) => input.press(key),
  onRelease: ({ key }) => input.release(key),
  onAction: ({ key }) => {
    if (key === "enter") startGame();
    if (key === "r") resetGame();
  }
});
```

`InputController.installTouchControls()` を使うと、`Touch` のボタン群をキーボードと同じ action map へ流せます。`webg/samples/breakout`、`webg/samples/tile_sim`、`webg/unittest/touch`、`webg/unittest/input_controller` などが、この役割を読む入口です。表示系の観点では、`Touch` は「入力 UI」であり、「情報表示 UI」ではない点が大事です。さらに `WebgApp` を `layoutMode: "embedded"` で使う場合は、touch controls もビューポート下端ではなく canvas host 下端へそろいます。本文全体をスクロールする教材ページで「3D 実行例の直下にだけ入力ボタンを置きたい」ときは、この挙動を前提にすると構成を決めやすくなります。

## `Diagnostics` と `DebugDock`

`Diagnostics` は表示部品ではなく、状態をレポートオブジェクトとして残すためのヘルパーです。まずレポートを作り、そのあとテキスト / JSON / コピー / 保存 / パネルへ流すので、console 出力だけに依存しません。ここが `Message` や `DialogueOverlay` といちばん違う点です。

```js
app.setDiagnosticsStage("runtime");
app.mergeDiagnosticsStats({
  score,
  lives,
  brickCount,
  envOk: app.checkEnvironment().ok
});
app.addDiagnosticsDetail(`phase=${app.getScenePhase()}`);
app.addDiagnosticsWarning("ball speed reached clamp");
```

必要なら 1 回だけ probe を取ります。

```js
app.requestProbe({
  label: "breakout_runtime_probe",
  format: "text",
  afterFrames: 1,
  collect: () => app.createProbeReport("runtime-probe")
});
```

`Diagnostics` の代表サンプルは `webg/samples/bloom`、`webg/samples/dof`、`webg/samples/scene`、`webg/samples/gltf_loader` です。状態の記録が必要なサンプルは、この層を通すと後から比較しやすくなります。

これに対して `DebugDock` は、デスクトップでのデバッグ用の補助表示です。右側の固定領域にパラメータ、probe 状態、コピー / 保存 / ログ出力操作をまとめて出せるので、canvas HUD を情報だらけにしなくて済みます。これは最終 UI ではなく、開発中の見やすさを支える場所です。`setControlRows()` を使うと、canvas HUD とデバッグドックの両方へ同じ row を流せます。パラメータ調整サンプルでは、まずこの形を使うのがいちばん扱いやすくなります。

```js
const rows = [
  { label: "Threshold", decKey: "1", incKey: "2", value: bloom.threshold.toFixed(2) },
  { label: "Strength", decKey: "3", incKey: "4", value: bloom.strength.toFixed(2) },
  { label: "View", cycleKey: "V", value: state.view }
];

app.setControlRows(rows, {
  anchor: "top-left",
  minScale: 0.80
});

app.setDebugDockRows(rows);
```

さらに diagnostics 用のキー入力をそろえると、調査が標準化しやすくなります。

```js
app.configureDebugKeyInput({
  prefixKey: "f9",
  files: {
    text: "bloom_diagnostics.txt",
    json: "bloom_diagnostics.json"
  }
});
```

この役割の代表サンプルは `webg/samples/bloom` と `webg/samples/dof` です。パラメータの row 表示、デバッグ / 通常表示の切替、probe の操作を一緒に見られます。

### Dock とオーバーレイの表示位置で起こりやすい問題

レイアウト調整では、`DebugDock` を右側へ表示したときに、`DialogueOverlay` 由来のパネルが canvas 外へ出ることがあります。症状は単純に「右へずれる」だけではなく、window 幅によっては正常に見えたり、window resize を一度行うと直ったり、Dock の表示切替直後だけ古い位置へ残ったりすることがあります。そのため、サンプル側の幅指定ミスに見えても、実際には core 側のレイアウト再計算の順序や CSS クラスの切替漏れが原因になっている場合があります。

この問題を理解するときに大事なのは、`webg` の表示系には「canvas 自体の描画領域」と「canvas の上に重ねる DOM オーバーレイ」が別経路で存在することです。`DebugDock` はビューポート右端に固定表示される一方、通常の viewport 構成では `DialogueOverlay`、`FixedFormatPanel`、`Touch`、サンプル独自の minimap オーバーレイも DOM 側で固定配置されます。ここで Dock を出したときに canvas だけを狭めると、DOM オーバーレイはまだビューポート全幅を使って右端へ置かれるため、「canvas 外にはみ出したように見える」状態が起きます。逆に DOM オーバーレイだけにオフセットを入れて canvas を狭めないと、Dock と canvas が重なって見えます。再発しやすい条件や対策も整理しておきます。まず `WebgApp.applyViewportLayout()` が canvas の `right` を狭め、同時に `--webg-canvas-right-inset` を document へ配布する必要があります。そのうえで、固定 DOM 側は `DialogueOverlay` なら `dockOffsetProvider()`、`Touch` や minimap のような独自 DOM なら CSS 変数経由で、同じ inset を参照しなければなりません。さらに `DialogueOverlay` のようにレスポンシブ用クラスを持つ UI では、通常時の root だけでなく `.is-stacked` や `.is-spread-columns` といったクラス側の `inset` 定義でも dock offset を保持する必要があります。通常 root が持っていた `--webg-overlay-dock-offset` がクラス側で上書きされて消えると、「特定の幅では正しく、特定の幅では外へ出る」という問題が起きやすくなります。

対策としては、Dock と同居する固定 DOM を追加するとき、次の順で考えると壊れにくくなります。まず canvas を狭める役割は `WebgApp.applyViewportLayout()` に集約し、サンプルごとに canvas style を個別補正しないこと。次に DOM オーバーレイ側はビューポート基準ではなく、「canvas 右端までの有効表示領域」を基準にしたオフセットを受け取ること。`DialogueOverlay` や `FixedFormatPanel` は `getDockOffset()` を使い、独自 DOM は `--webg-canvas-right-inset` を参照します。最後にレスポンシブ用クラスを持つ component は、通常時と派生クラスの両方で同じオフセットが残るかを確認します。ここを忘れると、resize やクラス切り替えの境目でだけ位置が変わります。見分け方としては、`DebugDock` を表示した状態で、minimap や touch button のような単純な固定 DOM が正しいのに特定のパネルだけが外へ出るなら、`DialogueOverlay` / `UIPanel` のクラス切り替え側を疑います。逆に固定 DOM が全部外へ出るなら、`--webg-canvas-right-inset` の配布か canvas 縮小そのものが疑わしいです。Dock の表示切替直後だけ崩れて resize すると直るなら、`toggleDebugMode()` や `syncDebugDockVisibility()` から `applyViewportLayout()` / `dialogue.syncLayout()` が即時に呼ばれているかを確認します。サンプル側で `vw` 幅を多用している場合は、Dock で canvas が狭くなってもパネル幅だけがビューポート基準で太いまま残ることがあるので、`tile_sim` のように固定上限付き width にそろえるほうが安全です。結局のところ、この一連の修正方針は、`DebugDock` を右に出すなら canvas を狭める、固定 DOM は同じ inset を共有する、レスポンシブ用クラス側でも dock offset を落とさない、という 3 点へ整理できます。今後別サンプルや別オーバーレイを追加するときも、この 3 点を外さなければ「Dock を出すと一部 UI だけ canvas 外へ出る」問題をかなり避けやすくなります。

## `WebgApp` helper をどう組み合わせるか

表示を組み立てるときは、最初から `Message`、`DialogueOverlay`、`Diagnostics` を全部直接触るより、`WebgApp` のヘルパーから選ぶほうが流れをつかみやすくなります。考え方は次の順番が自然です。画面ですぐ読ませたい短い案内には `setGuideLines()` と `setStatusLines()` を使います。パラメータと操作の一覧には `setControlRows()` を使います。会話とチュートリアルには `startDialogue()` を使います。長文とエラーには `showFixedFormatPanel()` と `showErrorPanel()` を使います。調査のために残す状態記録には `Diagnostics` と `configureDebugKeyInput()` を使います。この順番で組むと、画面で見せる情報と調査のために残す情報が混ざりにくくなります。

### `WebgApp.setUiTheme()` でテーマを切り替える

`WebgApp.setUiTheme()` は、DOM ベースの補助 UI を runtime 中にまとめて切り替えるための入口です。`Message` のような canvas HUD の色や 3D object のマテリアルは対象ではなく、`DebugDock`、`FixedFormatPanel`、`createHelpPanel()` が作る help panel、`startDialogue()` が作る dialogue overlay の見た目を同じ方向へそろえる役割を持ちます。教材ページの背景色を dark と light で切り替えたいときや、同じサンプルを tool 画面と記事ページの両方で見せたいときは、このメソッドを基準にすると整理しやすくなります。引数には `WebgUiTheme` と同じ 3 group 構造の object を渡します。`debugDock` はデスクトップ用 dock、`fixedFormatPanel` は長文やエラーの固定パネル、`uiPanel` はシーン上に重ねる軽量パネル群の色と余白を持ちます。`webg/WebgUiTheme.js` には既定 dark の `DEFAULT_UI_THEME`、明るい tool 向けの `DEFAULT_UI_LIGHT_THEME`、暖色 accent を強めた `DEFAULT_UI_SUNSET_THEME`、緑とシアン寄りの `DEFAULT_UI_FOREST_THEME` が用意されており、`UI_THEME_PRESETS` にはこれらが `dark`、`light`、`sunset`、`forest` の名前でまとまっています。preset 一覧から選ばせる UI を作りたいときは、`UI_THEME_PRESETS` を入口にすると列挙漏れを防ぎやすくなります。ここで大事なのは、`WebgApp` が自分で管理している UI と、サンプル側が直接生成した `UIPanel` は更新経路が違うことです。`app.setUiTheme(...)` を呼ぶと、`WebgApp` が持つ dock、fixed panel、help panel、dialogue は自動で更新されます。一方で、`new UIPanel(...)` で作った sample 固有 overlay は `WebgApp` が参照を持っていないため、`uiPanels.setTheme(app.uiTheme.uiPanel)` を追加で呼んで同期します。ここを忘れると、dock と fixed panel だけが切り替わり、独自 overlay だけ古い色のまま残ります。

次の例は、`book/examples/14_02.html` にかなり近い構成で、`UI_THEME_PRESETS` から 4 種類の preset を選び、`WebgApp` 管理の UI と sample 固有の `UIPanel` をそろえて更新するものです。`FixedFormatPanel` を一緒に開いているのは、「dock と overlay だけでなく、長文パネルも同じ preset へ変わる」ことを視覚的に確認しやすくするためです。本文では毎フレームの HUD 更新までは省き、テーマ切替そのものに関係する部分へ絞っています。

```js
import WebgApp from "./webg/WebgApp.js";
import UIPanel from "./webg/UIPanel.js";
import { UI_THEME_PRESETS } from "./webg/WebgUiTheme.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  layoutMode: "embedded",
  fixedCanvasSize: {
    width: 720,
    height: 540,
    useDevicePixelRatio: false
  }
});
await app.init();

const host = app.ensureCanvasHost();
const uiPanels = new UIPanel({
  document,
  theme: app.uiTheme.uiPanel
});
const layout = uiPanels.createLayout({
  id: "theme-switcher",
  containerElement: host,
  positioningMode: "absolute",
  leftWidth: "minmax(0, 320px)",
  rightWidth: "minmax(0, 260px)",
  top: 12,
  left: 12,
  right: 12,
  compactWidth: 700,
  collapseWidth: 880
});

const panel = uiPanels.createPanel(layout.left, { stack: true });
uiPanels.createEyebrow(panel, "Theme");
uiPanels.createTitle(panel, "WebgApp.setUiTheme()", 1);
uiPanels.createHint(panel, "Buttons or [1]-[4] to switch presets");
const buttonGrid = uiPanels.createButtonGrid(panel, { columns: 2 });

const infoPanel = uiPanels.createPanel(layout.right, { stack: true });
uiPanels.createEyebrow(infoPanel, "Preview");
uiPanels.createTitle(infoPanel, "Current preset", 2);
const overview = uiPanels.createTextBlock(infoPanel, {
  text: "theme=dark\nbutton / border / panel colors"
});
const status = uiPanels.createStatusBlock(layout, {
  id: "theme-status",
  text: "theme: dark"
});

const presetEntries = [
  { key: "1", id: "dark" },
  { key: "2", id: "light" },
  { key: "3", id: "sunset" },
  { key: "4", id: "forest" }
];

function applyThemePreset(themeId) {
  const nextTheme = UI_THEME_PRESETS[themeId];
  if (!nextTheme) {
    throw new Error(`Unknown UI theme preset: ${themeId}`);
  }

  app.setUiTheme(nextTheme);
  uiPanels.setTheme(app.uiTheme.uiPanel);
  overview.textContent = [
    `theme=${themeId}`,
    `panel=${app.uiTheme.uiPanel.panelBackground}`,
    `button=${app.uiTheme.uiPanel.buttonBackground}`
  ].join("\n");
  status.textContent = `theme: ${themeId}`;

  app.showFixedFormatPanel([
    `theme=${themeId}`,
    `dockBg=${app.uiTheme.debugDock.rootBackground}`,
    `fixedBg=${app.uiTheme.fixedFormatPanel.errorBackground}`,
    `panelBg=${app.uiTheme.uiPanel.panelBackground}`
  ].join("\n"), {
    id: "theme-preview",
    title: "Theme Preview"
  });
}

for (const entry of presetEntries) {
  const button = uiPanels.createButton(buttonGrid, {
    text: `[${entry.key}] ${entry.id}`
  });
  button.addEventListener("click", () => applyThemePreset(entry.id));
}

app.createHelpPanel({
  id: "themeExampleHelp",
  lines: [
    "[1]-[4]: switch theme preset",
    "help / fixed panel / overlay update together"
  ]
});

app.attachInput({
  onKeyDown: (key, ev) => {
    if (ev.repeat) return;
    const entry = presetEntries.find((item) => item.key === key);
    if (entry) {
      applyThemePreset(entry.id);
    }
  }
});

applyThemePreset("dark");
```

この例では `presetEntries` を手書きして、button 表示とキー割り当てを同時に固定しています。preset 名をそのまま列挙したいだけなら `Object.keys(UI_THEME_PRESETS)` を使っても構いませんが、読者へ見せる順番やキー表示を安定させたいなら、この例のように配列を明示したほうが整理しやすくなります。`WebgApp` の help panel や dialogue を使っている場合は、このコードを加えるだけでそれらも同時に更新されます。実行可能な完全版は `book/examples/14_02.html` で、そこでは guide/status lines の更新と HTML 枠も加えた状態で確認できます。確認用としては `webg/unittest/theme` も分かりやすく、dark / light / sunset / forest を runtime 中に切り替えながら、dock、fixed panel、overlay の透明度や文字色が崩れないかを見比べられます。`samples/scene` のように sample 側で `UIPanel` を直接組んでいる例では、「`app.setUiTheme()` の直後に `uiPanels.setTheme(app.uiTheme.uiPanel)` を呼ぶ」という流れを守ることが重要です。

## 例を通して役割を確かめる

情報表示の役割は、実際のサンプルをいくつか通すとつかみやすくなります。短い canvas HUD の標準を見るなら `webg/samples/breakout` が代表です。タイトル、スコア、コンボ、時間、残機のようなゲーム寄りの短い表示を読む代表サンプルです。`Message` や `WebgApp` の guide/status ヘルパーをどこまで使うかを判断したいときは、ここを基準にするとそろえやすくなります。`webg/samples/axis` も canvas HUD の教育用サンプルとして有効で、白背景の上でガイドと control row をどう読ませるかを確認できます。ゲーム寄りの HUD は `breakout`、教材寄りの HUD は `axis` という 2 本立てで読むと役割差を追いやすくなります。パラメータ HUD とデバッグドックの標準を見るなら `webg/samples/bloom` が最初の基準です。ここでは通常サンプル用の `createHelpPanel()` とパラメータ HUD の標準例が見られます。操作説明を左上の畳めるパネルへ置き、現在値は canvas HUD、調査用の詳細は `DebugDock` と `Copy Summary` / `Copy JSON` へ分けています。`webg/samples/dof` は、パラメータ調整と view 切替を canvas HUD 中心で見せる標準例です。help panel までは要らないが、パラメータ row と diagnostics は欲しい、という場面の比較対象として読むと役割差が分かりやすくなります。`webg/samples/proctex`、`webg/samples/skinning`、`webg/samples/billboard`、`../discontinued/demo1`、`webg/samples/camera_controller`、`webg/samples/json_loader`、`webg/samples/collada_loader`、`webg/samples/gltf_loader` も `setControlRows()` 系の延長として読めますが、表示系の基準サンプルとしては `bloom` / `dof` を先に見るほうが迷いません。

シーンの上に重ねる操作 UI の標準を見るなら `webg/samples/scene` が代表です。`UIPanel` の代表サンプルで、ボタンとステータスパネルをシーンの上に重ねながら、scene action と diagnostics 状態を同時に扱っています。`UIPanel` の実運用を見るなら、まずここを参照するのが自然です。会話とチュートリアルの標準を見るなら `webg/unittest/dialogue` が最短です。UTF-8 の本文、分岐、再表示、ステータスの流れだけに絞っているので、会話 UI の標準を確認するにはここが最も分かりやすくなります。テーマとパネル群の見え方確認には `webg/unittest/theme` が向いています。`DebugDock`、`FixedFormatPanel`、`UIPanel` をまとめて比較するテストで、見た目の統一、テーマの切替、パネル間の一貫性を確認したいときに役立ちます。`webg/unittest/embedded` は、固定サイズ canvas を本文へ埋め込んだ状態で HelpPanel、dialogue、fixed panel、touch が一緒に動くかを確認する test です。専用の操作画面を持つ例としては `webg/samples/sound` があります。音調整用の専用 HTML UI を直接組んでいる例で、音パラメータを一覧で触りたいときには分かりやすい一方、通常のゲーム HUD や会話 UI と同じ見方で読む必要はありません。用途の違いを見比べるつもりで読むと整理しやすくなります。`Text` と `Message` を小さく確かめる例としては `webg/unittest/message` が役立ちます。`webg/unittest/touch` も touch 入力の状態表示を素直に読む小さな例で、ここでは `Message` の直接操作がよく合います。`webg/unittest/tilemap` では、tile の選択結果や表示域の追従を小さく確かめるために `Message` を直接使っています。ここでは表示そのものより、TileMap ランタイムと短い状態表示の結び付き方を見ると読みやすくなります。直接書いた HUD を読む例としては `webg/samples/detouch`、`webg/samples/bone_creature`、`webg/samples/janken`、`webg/samples/circular_breaker`、`webg/samples/shapes` などがあります。ここでは、`WebgApp` の共通ヘルパーを使った整った表示というより、場面ごとの演出や用途に合わせて HUD を組む発想を見ると読みやすくなります。とくに `webg/samples/janken` の中央結果表示や `webg/samples/circular_breaker` のゲーム HUD は、短い ASCII 表示として見ると分かりやすい例です。一方で、最初に表示構成を学ぶ段階では、まず `WebgApp` の `createHelpPanel()`、guide/status/control row ヘルパー、または `UIPanel` / `DialogueOverlay` を使う例と見比べるほうが整理しやすくなります。`Touch` の使い方を見る例としては、`webg/unittest/touch` が `Touch` をそのまま確かめる小さな例で、`webg/unittest/input_controller` は `Touch` と `InputController` を一緒に見る例です。`webg/samples/breakout` や `webg/samples/tile_sim` はゲーム内へ touch button を組み込んだ応用例として読みやすく、密度やレイアウト変化だけを見たいときは `webg/unittest/touch` から入ると整理しやすくなります。`Touch` は入力代替 UI なので、操作 UI を中心に見る `scene` とは別の役割として考えると分かりやすくなります。

## 新しいサンプルを作るときの判断表
<!-- 図候補: 図14-1 UI 部品選択表 -->

![図14-1 UI 部品選択表](fig14_01_ui_component_selector.jpg)

*図14-1 表示部品は API 名から選ぶより、「短い情報」「長文」「操作」「診断」のどれを主役にするかで選ぶほうが整理しやすくなります。*

新しいサンプルを追加するときは、まずどの表示経路を主役にするかを 1 つ決めると混ざりにくくなります。タイトル、START、HIGH SCORES、score、lives のような短い ASCII HUD なら `Message` または `setGuideLines()` / `setStatusLines()` を使います。通常サンプルの操作説明を畳めるパネルにしたいなら `createHelpPanel()` を使います。option 一覧、パラメータ一覧、操作表なら `setControlRows()`、必要なら `DebugDock` を組み合わせます。会話、チュートリアル本文、選択肢、日本語の長文なら `startDialogue()` と `DialogueOverlay` を使います。pause menu やアクションボタンなら `UIPanel`、キー入力代わりの保持入力 / 単発入力ボタンなら `Touch`、読み込み失敗、バリデーションエラー、全文レポートなら `showErrorPanel()` / `showFixedFormatPanel()`、copy / save / probe / JSON 化したいランタイム状態なら `Diagnostics` を使う、という考え方にすると迷いにくくなります。

1 つのサンプルの中で複数の経路を使うこと自体は問題ありませんが、同じ情報を canvas HUD、シーン上の DOM、固定パネルに重複して出し始めると読みにくくなります。現行サンプルの整理でも、この「主表示経路を 1 つ先に決める」方針を基準にすると判断しやすくなります。この章の結論を一言でまとめると、短い ASCII HUD は `Message`、会話と本文は `DialogueOverlay`、操作パネルは `UIPanel`、長文とエラーは `FixedFormatPanel`、入力代替は `Touch`、状態記録は `Diagnostics`、PC デバッグ表示は `DebugDock`、そしてそれらの入口をそろえるのが `WebgApp` です。ここが分かると、以後のサンプルで「どの表示をどこへ置くべきか」がかなり決めやすくなります。
