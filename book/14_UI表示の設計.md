# UI表示の設計

この章では、`webg` における文字表示と情報表示の使い分けについて解説します。`Message`、`DialogueOverlay`、`Diagnostics`、`DebugDock`、`FixedFormatPanel`、`UIPanel`、`Touch`、そしてこれらを統合的に管理する `WebgApp` の関係性を、それぞれの表示役割ごとに整理して見ていきましょう。

`webg` の表示系が複雑に見える理由は、情報の出力経路が一本ではないためです。canvas HUD、シーン上に重ねる DOM UI、画面端に固定する DOM、画面下部に固定するタッチボタン DOM、デバッグドック、そして診断情報レポートという複数の経路が存在し、それぞれが異なる目的で設計されています。しかし、個別のサンプルや文書だけを読み始めると、「短い案内はどこに出すべきか」「会話や選択肢には何を使うべきか」「調査用の詳細情報はどこに表示させるか」といった判断に迷うことがあります。そこで本章では、実装前にまず揃えておきたい判断基準を明確にします。

設計のポイントは、API名から機能を選ぶのではなく、「誰に、何を見せたいか」という目的から選択することです。例えば、短い案内であれば canvas HUD、会話や選択肢であればシーン上の DOM オーバーレイ、キー入力の代わりとなる仮想ボタンであれば画面下固定の `Touch`、詳細な調査情報であれば診断情報（Diagnostics）とデバッグドック、そして長文の読み物やエラー理由の提示には `FixedFormatPanel` が適しています。

特に注意が必要なのは、`Message` は短い ASCII 文字専用であり、日本語や一般的な UTF-8 文章の表示には対応していない点です。会話やチュートリアルの本文、選択肢などを表示させる場合は、`DialogueOverlay` や `UIPanel` を使用するのが標準的な構成となります。

また、`Diagnostics` は表示部品ではなく「状態の記録」を担う層であり、ゲーム内の会話やタイトル UI とは明確に役割が異なります。同様に `Touch` もボタンを表示させる点では `UIPanel` と似ていますが、その役割は「キー入力の代わりとなる仮想ボタン群」であり、`InputController` を通じてキー状態へ入力を流すための部品です。シーンアクションパネルやステータスパネルを構築するための `UIPanel` とは切り分けて考えることで、設計がシンプルになります。

これらの表示経路を効率的に扱うため、`WebgApp` は `setGuideLines()`、`setStatusLines()`、`setControlRows()`、`createHelpPanel()`、`startDialogue()`、`showFixedFormatPanel()` といったヘルパーメソッドを提供しています。新しいサンプルを構築する際は、まずこれらの `WebgApp` ヘルパーから検討することで、全体の構成を見通しよく整えることができます。

なお、`WebgApp` は通常ビューポート全体を使用する構成ですが、`layoutMode: "embedded"` を指定すると、固定サイズの canvas を本文中に埋め込むことができます。このモードでは HelpPanel、Dialogue、`FixedFormatPanel`、Touch Controls も canvas ホスト基準で動作します。教材ページなどを制作する場合は、UIが「ビューポートに固定されるべきか」あるいは「本文と一緒にスクロールすべきか」を意識して選択してください。

`webg` の情報表示は、以下の 7 つの役割に分割して考えると整理しやすくなります。

1. `Message` / `setGuideLines()` / `setStatusLines()`: canvas 上のシーンに重ねて表示される、利用者向けの短い ASCII HUD。
2. `DialogueOverlay`: canvas 外の DOM で構成され、シーン上に重ねて表示される。会話、チュートリアル本文、選択肢などの UTF-8 文章に使用。
3. `UIPanel`: canvas 外の DOM で構成され、シーン上に重ねる操作 UI（ボタンやパネル）を構築するために使用。
4. `FixedFormatPanel`: canvas 外の DOM で、通常は画面端に固定表示される。エラー全文や長文の説明など、「読むためのテキスト」に使用。
5. `Touch`: canvas 外の DOM で、通常は画面下部に固定表示される。キー入力の代わりとなる保持入力・単発入力ボタンに使用。
6. `Diagnostics`: 表示部品ではなくレポート生成層。状態をレポートとして作成し、テキスト、JSON、コピー、保存などの出力へ流す。
7. `DebugDock`: canvas 外のデスクトップ用デバッグ領域（右側固定 DOM）。開発者向け情報の集約表示に使用。

例えば、スコアや残機は `Message`、会話は `DialogueOverlay`、ポーズメニューのボタンは `UIPanel`、仮想ボタンは `Touch`、読み込み失敗の詳細は `FixedFormatPanel`、内部状態のレポートは `Diagnostics`、PCでの詳細操作は `DebugDock` と分担させることで、役割が混在することを避けられます。

## `Message` と canvas HUD

`Message` は、canvas 上でシーンに重ねる短い ASCII 文字列を表示するためのハイレベル（高レイヤー）な HUD です。`Text` クラスよりも上位の概念であり、`id`、`anchor`、`block` という単位で配置できるため、タイトルやステータス表示を効率的に管理できます。

ここでの活用例としては、"START"、"PAUSE"、"HIGH SCORES"、"SCORE"、"TIME"、"LIVES" といった、短く更新頻度の高い情報の表示が最適です。ゲーム内のタイトルやスコアボードは、利用者が一瞬で認識できる必要があるため、DOM オーバーレイよりも canvas HUD の方が自然な表現となります。

前述の通り、`Message` では日本語や UTF-8 文章を表示できません。文字数に関わらず、日本語の説明を表示させたい場合は、必ず `DialogueOverlay` または `UIPanel` を使用してください。

```js
app.setGuideLines([
  "[Enter] START",
  "[ArrowLeft/Right] MOVE"
], {
  anchor: "bottom-left",
  x: 0,
  y: -2,
  color: [0.92, 0.96, 1.0]
});

app.setStatusLines([
  `SCORE ${score}`,
  `LIVES ${lives}`,
  `TIME ${timeLeft}`
], {
  anchor: "top-left",
  x: 0,
  y: 0,
  color: [1.0, 0.88, 0.72]
});
```

より直接的に `Message` を制御したい場合は、次のように `id` を指定して操作します。

```js
app.message.setLine("title", "START", {
  anchor: "center",
  y: -2,
  color: [1.0, 0.96, 0.72]
});

app.message.setBlock("scoreboard", [
  `HIGH SCORES`,
  `1. ${scores[0]}`,
  `2. ${scores[1]}`,
  `3. ${scores[2]}`
], {
  anchor: "top-right",
  x: -1,
  y: 0,
  color: [0.92, 0.96, 1.0]
});
```

この実装の代表的な例は `webg/samples/breakout` です。タイトル、スコア、コンボ、時間、残機といったゲーム的な短い表示を `Message` 系の HUD にまとめています。最小構成での動作確認には `webg/unittest/message` が適しています。

## `DialogueOverlay` と `UIPanel`

`DialogueOverlay` は、会話やチュートリアルの本文を表示するための標準的な経路です。表示先は canvas 外の DOM ですが、視覚的にはシーンの上に重なって表示されます。話者（speaker）、本文、選択肢、次へボタン、再表示などの構造をそのまま DOM オーバーレイとして出力できるため、日本語を含む長い説明文を適切に扱うことができます。

`DialogueOverlay` は、DOM オーバーレイ専用の会話ヘルパーとして利用するのが自然です。会話 UI を構築する際は、`WebgApp.startDialogue()` に対話データ（エントリ）を渡し、`nextDialogue()` や `chooseDialogue()` で進行を制御します。

```js
const intro = [
  {
    speaker: "guide",
    title: "Tutorial",
    lines: [
      "ようこそ",
      "ここでは移動と攻撃を順番に確認します"
    ]
  },
  {
    speaker: "guide",
    lines: [
      "先に進む道を選んでください"
    ],
    choices: [
      { label: "左へ行く", next: [{ speaker: "guide", lines: ["左の説明へ進みます"] }] },
      { label: "右へ行く", next: [{ speaker: "guide", lines: ["右の説明へ進みます"] }] }
    ]
  }
];

app.startDialogue(intro, {
  title: "Tutorial",
  footer: "Enter: next  1/2: choose  R: restart"
});
```

メインループ内では、以下のように短い入力確認を行うだけで制御が可能です。

```js
if (app.wasActionPressed("next")) {
  app.nextDialogue();
}
if (app.wasActionPressed("choice1")) {
  app.chooseDialogue(0);
}
if (app.wasActionPressed("choice2")) {
  app.chooseDialogue(1);
}
```

この機能の標準的な動作については `webg/unittest/dialogue` で確認できます。UTF-8 本文の表示、分岐の差し込み、再開の流れを最小構成で追うことができます。

一方、`UIPanel` は、ボタン、パネル、ヒント、pill（ピル状ラベル）などの操作 UI を DOM オーバーレイに配置するための部品です。会話専用ではなく、シーンアクションパネル、ツールパレット、プレビューカード、簡易メニューといった UI を構築するための土台となります。`DialogueOverlay` も内部的に `UIPanel` を使用していますが、利用者が直接 `UIPanel` を使う場面は、「会話」ではなく「操作パネル」を作りたい場合が中心となります。Scene JSON のアクションボタンや、テーマプレビューの操作パネルなどがその代表例です。

```js
const uiPanels = new UIPanel({
  document,
  theme: app.uiTheme.uiPanel
});

const layout = uiPanels.createLayout({
  id: "scene-tools",
  leftWidth: "minmax(300px, 420px)",
  rightWidth: "minmax(260px, 340px)",
  scrollColumns: true
});

const panel = uiPanels.createPanel(layout.left, { stack: true });
uiPanels.createEyebrow(panel, "Scene Actions");
uiPanels.createTitle(panel, "Runtime Controls", 1);
uiPanels.createHint(panel, "scene action と同じ処理を button から呼ぶ例");

const row = uiPanels.createButtonRow(panel);
const pauseButton = uiPanels.createButton(row, { text: "Pause Scene" });
const resetButton = uiPanels.createButton(row, { text: "Reset Camera" });

pauseButton.addEventListener("click", () => togglePause());
resetButton.addEventListener("click", () => resetCamera());
```

この役割を確認するには `webg/samples/scene` が最適です。左側のアクションボタンと右側のステータスパネルがあり、scene file と action handler の関係性をまとめて確認できます。テーマ適用まで含めて確認したい場合は `webg/unittest/theme` を参照してください。

### 操作説明には `createHelpPanel()` を活用する

教材寄りのサンプルなどで、「操作説明をシーン上に表示させたいが、不要なときは畳んでおきたい」という場面では、`WebgApp.createHelpPanel()` を利用するのが標準的な手法です。サンプル側で 1 行 1 操作の line 配列を渡すだけで、左上に `Hide Help` / `Show Help` ボタンを持つパネルを配置できます。また、デバッグドックの幅についても `WebgApp` 側が自動的に追従するため、サンプルごとにドックのオフセットを再計算する必要はありません。

```js
const helpLines = [
  "Drag or Arrow keys: orbit camera",
  "[ / ] or wheel: zoom",
  "[1]/[2] threshold -/+",
  "[3]/[4] strength -/+",
  "[space] pause"
];

app.createHelpPanel({
  id: "sampleHelpOverlay",
  lines: helpLines
});
```

今後の通常サンプルでは、この方法を標準形として採用することを推奨します。ただし、`janken` や `circular_breaker` のように HUD 自体が演出の一部となっている場合や、`tile_sim` のように説明量が多く独自のレイアウトが必要な場合は例外として扱います。それ以外は、「操作説明は畳める Help Panel、現在値は HUD、調査用の詳細は Diagnostics」という分担にすることで、UIが最も整理されます。

## `FixedFormatPanel` は長文とエラー用

`FixedFormatPanel` は、「短い HUD には収まりきらない長文」を提示するための場所です。表示先は canvas 外の DOM で、通常は画面端に固定表示されます。内容は `<pre>` タグと同様に整形済みテキストとして表示されるため、エラー全文、詳細な診断レポート、説明メモ、リソース読み込み結果の一覧など、利用者が落ち着いて読む必要があるテキストに適しています。

短いステータス表示と長文を同じ場所に混在させると、視認性が低下します。そのため、リソース読み込みのエラーやバリデーション失敗は `showErrorPanel()` へ、詳細な説明は `showFixedFormatPanel()` へと分けるのが安全です。

```js
try {
  await loadSomething();
} catch (err) {
  app.showErrorPanel(err, {
    id: "load-error",
    title: "scene load failed"
  });
}

app.showFixedFormatPanel([
  "Overview",
  `clipCount=${clipCount}`,
  `warningCount=${warningCount}`,
  `triangleCount=${triangleCount}`
].join("\n"), {
  id: "scene-overview"
});
```

この役割は `webg/samples/gltf_loader`、`webg/samples/collada_loader`、`webg/samples/json_loader` などのローダー系サンプルで頻繁に利用されています。また、`webg/unittest/theme` でも `FixedFormatPanel` のテーマ更新を確認できます。

教材ページの中に canvas を埋め込む場合は、`WebgApp` の `layoutMode: "embedded"` と `showFixedFormatPanel()` を組み合わせることで、長文パネルも canvas ホスト基準で動作させることが可能です。スクロールする本文の中に 3D 実行例と補助説明を同居させたい場合の最短例は `webg/unittest/embedded` で確認できます。

## `Touch` はキー入力代わりのボタン表示

`Touch` は、`UIPanel` のように複雑なシーンアクションパネルを構築するための部品ではありません。表示先は canvas 外の DOM ですが、通常は画面下部に固定され、「キー入力の代わりに押すボタン」を提供することを目的としています。

保持入力（hold）と単発入力（action）を分けて扱い、`InputController` と連携させることで、キーボードと同じキー状態をエミュレートできます。タッチ操作などの精度の低いポインタを使用する端末では、見た目のカード UI よりも「どのキーの代わりとして機能するか」が重要です。そのため `Touch` は `UIPanel` よりもシンプルなレイアウトを採用し、画面下部に横並び、あるいは複数行で収まる設計になっています。

シーン上のステータスや選択肢を表示させたい場合は `UIPanel` を、移動や決定などの入力を促したい場合は `Touch` を使用することで、役割を明確に分離できます。ボタンの割り当てに迷った際は、[16_タッチ機能と入力.md](./16_タッチ機能と入力.md) の一覧を参照し、`space` やファンクションキーを含めて検討してください。

```js
import Touch from "./webg/Touch.js";

const touch = new Touch(document, {
  touchDeviceOnly: false
});

touch.create({
  groups: [
    {
      id: "move",
      buttons: [
        { key: "arrowleft", label: "←", kind: "hold" },
        { key: "arrowright", label: "→", kind: "hold" }
      ]
    },
    {
      id: "action",
      buttons: [
        { key: "enter", label: "Start", kind: "action" },
        { key: "r", label: "R", kind: "action" }
      ]
    }
  ],
  onPress: ({ key }) => input.press(key),
  onRelease: ({ key }) => input.release(key),
  onAction: ({ key }) => {
    if (key === "enter") startGame();
    if (key === "r") resetGame();
  }
});
```

`InputController.installTouchControls()` を使用すると、`Touch` のボタン群をキーボードと同じアクションマップへ流し込むことができます。`webg/samples/breakout`、`webg/samples/tile_sim`、`webg/unittest/touch`、`webg/unittest/input_controller` などが、この機能の実装例です。

表示系の観点から見ると、`Touch` は「入力 UI」であり、「情報表示 UI」ではないという点が重要です。また、`WebgApp` を `layoutMode: "embedded"` で運用する場合、タッチコントロールはビューポート下端ではなく canvas ホストの下端に揃います。これにより、「3D 実行例の直下にのみ入力ボタンを配置したい」という構成を実現しやすくなります。

## `Diagnostics` と `DebugDock`

`Diagnostics` は表示部品ではなく、システムの状態をレポートオブジェクトとして記録するためのヘルパーです。まずレポートを生成し、その後でテキスト、JSON、コピー、保存、あるいは専用パネルへと出力するため、単なる console 出力に依存しない柔軟な運用が可能です。ここが `Message` や `DialogueOverlay` と決定的に異なる点です。

```js
app.setDiagnosticsStage("runtime");
app.mergeDiagnosticsStats({
  score,
  lives,
  brickCount,
  envOk: app.checkEnvironment().ok
});
app.addDiagnosticsDetail(`phase=${app.getScenePhase()}`);
app.addDiagnosticsWarning("ball speed reached clamp");
```

必要に応じて、以下のようにプローブ（probe）を一度だけ取得することも可能です。

```js
app.requestProbe({
  label: "breakout_runtime_probe",
  format: "text",
  afterFrames: 1,
  collect: () => app.createProbeReport("runtime-probe")
});
```

`Diagnostics` の活用例は `webg/samples/bloom`、`webg/samples/dof`、`webg/samples/scene`、`webg/samples/gltf_loader` などで確認できます。状態の記録が必要なサンプルでは、この層を経由させることで、後の比較検証が容易になります。

対して `DebugDock` は、デスクトップ環境でのデバッグを補助するための表示領域です。画面右側の固定領域にパラメータ、プローブ状態、コピー/保存操作、ログ出力をまとめて表示できるため、canvas HUD を情報で埋め尽くすことなく開発を進められます。これは最終的なユーザー UI ではなく、開発効率を支えるためのツールです。`setControlRows()` を使用すると、canvas HUD とデバッグドックの両方に同じ行情報を流せるため、パラメータ調整を行うサンプルでは非常に効率的な構成となります。

```js
const rows = [
  { label: "Threshold", decKey: "1", incKey: "2", value: bloom.threshold.toFixed(2) },
  { label: "Strength", decKey: "3", incKey: "4", value: bloom.strength.toFixed(2) },
  { label: "View", cycleKey: "V", value: state.view }
];

app.setControlRows(rows, {
  anchor: "top-left",
  minScale: 0.80
});

app.setDebugDockRows(rows);
```

さらに、診断用のキー入力を設定することで、調査フローを標準化できます。

```js
app.configureDebugKeyInput({
  prefixKey: "f9",
  files: {
    text: "bloom_diagnostics.txt",
    json: "bloom_diagnostics.json"
  }
});
```

この構成の代表例は `webg/samples/bloom` と `webg/samples/dof` です。パラメータの行表示、デバッグ/通常表示の切り替え、プローブ操作をまとめて確認できます。

### レイアウトの仕組みと注意点

レイアウト調整において、`DebugDock` を表示した際に `DialogueOverlay` 由来のパネルが canvas 外へはみ出してしまう現象が発生することがあります。これは単なる幅指定のミスではなく、`webg` の表示系における「canvas 描画領域」と「DOM オーバーレイ」の管理経路が異なることに起因します。

`DebugDock` はビューポートの右端に固定されますが、通常のビューポート構成では `DialogueOverlay`、`FixedFormatPanel`、`Touch` などの DOM 要素も同様に固定配置されます。ここで Dock を表示して canvas 領域だけを狭めた場合、DOM オーバーレイ側が依然としてビューポート全幅を基準に配置されていると、結果として canvas 外にはみ出したように見えてしまいます。

この問題を回避し、安定したレイアウトを実現するためのポイントは以下の 3 点です。

1. canvas 領域の制御を集約する: canvas を狭める処理は `WebgApp.applyViewportLayout()` に集約し、サンプル側で個別に canvas style を補正しないようにします。これにより、`--webg-canvas-right-inset` という CSS 変数が document に適切に配布されます。
2. 有効表示領域を基準にする: DOM オーバーレイ側はビューポート基準ではなく、「canvas 右端までの有効表示領域」を基準にしたオフセットを参照させます。`DialogueOverlay` や `FixedFormatPanel` は内部で `getDockOffset()` を使用しており、独自 DOM を構築する場合は `--webg-canvas-right-inset` を参照させる必要があります。
3. レスポンシブクラスでのオフセット保持: `DialogueOverlay` のように `.is-stacked` などのレスポンシブ用クラスを持つ UI では、ルート要素だけでなく、各クラス側の定義においても dock offset が保持されている必要があります。ここが漏れていると、ウィンドウ幅の変更（リサイズ）やクラス切り替えのタイミングで位置がずれる原因となります。

トラブルシューティングの際は、まず「特定のパネルだけが外に出るのか（クラス切り替え側の問題）」、あるいは「すべての固定 DOM が外に出るのか（CSS 変数の配布や canvas 縮小自体の問題）」を確認してください。また、リサイズ後に直る場合は、`toggleDebugMode()` 等から `applyViewportLayout()` や `dialogue.syncLayout()` が即時に呼ばれているかを確認してください。

## `WebgApp` ヘルパーをどう組み合わせるか

表示構成を組み立てる際は、個別のクラスを直接操作するよりも、`WebgApp` のヘルパーメソッドから選択することで、設計の流れをスムーズに構築できます。推奨される検討順序は以下の通りです。

1. 短い案内: `setGuideLines()` および `setStatusLines()`
2. パラメータ・操作一覧: `setControlRows()` （必要に応じて `DebugDock` を併用）
3. 会話・チュートリアル: `startDialogue()`
4. 長文・エラー提示: `showFixedFormatPanel()` および `showErrorPanel()`
5. 状態記録・調査: `Diagnostics` および `configureDebugKeyInput()`

この順序で検討することで、「ユーザーに見せる情報」と「開発者が調査するための情報」を明確に分離でき、UI の混雑を防ぐことができます。

### `WebgApp.setUiTheme()` によるテーマ切り替え

`WebgApp.setUiTheme()` は、DOM ベースの補助 UI の見た目をランタイム中に一括して切り替えるための入口です。これは `Message` などの canvas HUD や 3D オブジェクトのマテリアルではなく、`DebugDock`、`FixedFormatPanel`、ヘルプパネル、対話オーバーレイの配色や余白を統一するために使用します。

引数には `WebgUiTheme` と同じ 3 グループ構造（`debugDock`, `fixedFormatPanel`, `uiPanel`）を持つオブジェクトを渡します。`webg/WebgUiTheme.js` には、既定の `DEFAULT_UI_THEME`（dark）のほか、`DEFAULT_UI_LIGHT_THEME`（light）、`DEFAULT_UI_SUNSET_THEME`（sunset）、`DEFAULT_UI_FOREST_THEME`（forest）が用意されており、これらは `UI_THEME_PRESETS` にまとめられています。

ここで重要な注意点として、`WebgApp` が管理している UI と、サンプル側で直接生成した `UIPanel` では更新経路が異なることが挙げられます。`app.setUiTheme()` を呼ぶと、`WebgApp` 管理下の UI は自動的に更新されますが、独自に `new UIPanel(...)` で作成したパネルは自動更新されません。そのため、独自パネルを併用している場合は、以下のように明示的にテーマを同期させる必要があります。

```js
app.setUiTheme(nextTheme);
uiPanels.setTheme(app.uiTheme.uiPanel);
```

以下に、`UI_THEME_PRESETS` からプリセットを選択し、`WebgApp` 管理 UI と独自 `UIPanel` を同期して更新する実装例を示します。

```js
import WebgApp from "./webg/WebgApp.js";
import UIPanel from "./webg/UIPanel.js";
import { UI_THEME_PRESETS } from "./webg/WebgUiTheme.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  layoutMode: "embedded",
  fixedCanvasSize: {
    width: 720,
    height: 540,
    useDevicePixelRatio: false
  }
});
await app.init();

const host = app.ensureCanvasHost();
const uiPanels = new UIPanel({
  document,
  theme: app.uiTheme.uiPanel
});
const layout = uiPanels.createLayout({
  id: "theme-switcher",
  containerElement: host,
  positioningMode: "absolute",
  leftWidth: "minmax(0, 320px)",
  rightWidth: "minmax(0, 260px)",
  top: 12,
  left: 12,
  right: 12,
  compactWidth: 700,
  collapseWidth: 880
});

const panel = uiPanels.createPanel(layout.left, { stack: true });
uiPanels.createEyebrow(panel, "Theme");
uiPanels.createTitle(panel, "WebgApp.setUiTheme()", 1);
uiPanels.createHint(panel, "Buttons or [1]-[4] to switch presets");
const buttonGrid = uiPanels.createButtonGrid(panel, { columns: 2 });

const infoPanel = uiPanels.createPanel(layout.right, { stack: true });
uiPanels.createEyebrow(infoPanel, "Preview");
uiPanels.createTitle(infoPanel, "Current preset", 2);
const overview = uiPanels.createTextBlock(infoPanel, {
  text: "theme=dark\nbutton / border / panel colors"
});
const status = uiPanels.createStatusBlock(layout, {
  id: "theme-status",
  text: "theme: dark"
});

const presetEntries = [
  { key: "1", id: "dark" },
  { key: "2", id: "light" },
  { key: "3", id: "sunset" },
  { key: "4", id: "forest" }
];

function applyThemePreset(themeId) {
  const nextTheme = UI_THEME_PRESETS[themeId];
  if (!nextTheme) {
    throw new Error(`Unknown UI theme preset: ${themeId}`);
  }

  app.setUiTheme(nextTheme);
  uiPanels.setTheme(app.uiTheme.uiPanel);
  overview.textContent = [
    `theme=${themeId}`,
    `panel=${app.uiTheme.uiPanel.panelBackground}`,
    `button=${app.uiTheme.uiPanel.buttonBackground}`
  ].join("\n");
  status.textContent = `theme: ${themeId}`;

  app.showFixedFormatPanel([
    `theme=${themeId}`,
    `dockBg=${app.uiTheme.debugDock.rootBackground}`,
    `fixedBg=${app.uiTheme.fixedFormatPanel.errorBackground}`,
    `panelBg=${app.uiTheme.uiPanel.panelBackground}`
  ].join("\n"), {
    id: "theme-preview",
    title: "Theme Preview"
  });
}

for (const entry of presetEntries) {
  const button = uiPanels.createButton(buttonGrid, {
    text: `[${entry.key}] ${entry.id}`
  });
  button.addEventListener("click", () => applyThemePreset(entry.id));
}

app.createHelpPanel({
  id: "themeExampleHelp",
  lines: [
    "[1]-[4]: switch theme preset",
    "help / fixed panel / overlay update together"
  ]
});

app.attachInput({
  onKeyDown: (key, ev) => {
    if (ev.repeat) return;
    const entry = presetEntries.find((item) => item.key === key);
    if (entry) {
      applyThemePreset(entry.id);
    }
  }
});

applyThemePreset("dark");
```

この例では、ボタン表示とキー割り当てを安定させるため、`presetEntries` 配列を用いて明示的に管理しています。実行可能な完全版は `book/examples/14_02.html` で、ガイド/ステータスラインの更新や HTML 枠を含めた状態で確認できます。また、`webg/unittest/theme` では、テーマ切り替え時にドックやパネルの透明度、文字色が崩れないかを検証できます。

## 例を通して役割を確かめる

情報表示の役割をより深く理解するために、目的別の代表的なサンプルを紹介します。

### 短い HUD とパラメータ表示
ゲーム的な HUD 構成を確認したい場合は、`webg/samples/breakout` が最適です。タイトル、スコア、時間、残機などの短い表示を `Message` や `WebgApp` ヘルパーでどのようにまとめているかを確認できます。また、`webg/samples/axis` は、白背景上でガイドと control row をどのように提示させるかを示す教育的な例として有効です。

パラメータ調整とデバッグドックの標準構成については、`webg/samples/bloom` を参照してください。操作説明を畳める Help Panel に置き、現在値は canvas HUD、詳細情報は `DebugDock` とレポート出力に分けるという、理想的な分担がなされています。`webg/samples/dof` は、Help Panel を使わずにパラメータ row と diagnostics を中心に据えた構成となっており、比較対象として有用です。

### 操作 UI と会話
シーン上に重ねる操作 UI の実運用については、`webg/samples/scene` が代表例です。`UIPanel` を用いてボタンとステータスパネルを構築し、scene action と diagnostics 状態を同時に扱う構成となっています。

会話やチュートリアルの実装については、`webg/unittest/dialogue` が最短の確認経路です。UTF-8 本文の表示、分岐、再表示といった会話 UI の基本フローに絞って実装されています。

### テーマと埋め込み構成
テーマの一貫性やパネル間の見え方を確認するには、`webg/unittest/theme` が適しています。`DebugDock`、`FixedFormatPanel`、`UIPanel` を同時に比較し、テーマ切り替えによる影響を確認できます。また、固定サイズ canvas を本文に埋め込んだ状態での UI 挙動（HelpPanel, Dialogue, FixedFormatPanel, Touch）を確認したい場合は、`webg/unittest/embedded` を参照してください。

### 特殊な HUD 演出と入力
演出に合わせて HUD を構築している例として、`webg/samples/janken`（中央の結果表示）や `webg/samples/circular_breaker`（ゲーム専用 HUD）があります。これらは共通ヘルパーによる整った表示ではなく、場面に応じた演出としての HUD 構築の参考になります。

`Touch` の利用例については、`webg/unittest/touch` で基本機能を、`webg/unittest/input_controller` で `InputController` との連携を確認できます。`webg/samples/breakout` や `webg/samples/tile_sim` では、ゲーム内へのタッチボタン組み込みという応用例として実装されています。

## 新しいサンプルを作るときの判断表

![UI 部品選択表](fig14_01_ui_component_selector.jpg)

*表示部品は API 名から選ぶのではなく、「短い情報」「長文」「操作」「診断」のどれを主役にするかで選択してください。*

新しいサンプルを追加する際は、まず主となる表示経路を一つ決めることで、UI の混在を防ぐことができます。

* 短い ASCII HUD $\rightarrow$ `Message` または `setGuideLines()` / `setStatusLines()`
* 畳める操作説明パネル $\rightarrow$ `createHelpPanel()`
* オプション・パラメータ一覧 $\rightarrow$ `setControlRows()` （必要に応じて `DebugDock` を併用）
* 会話・チュートリアル・日本語長文 $\rightarrow$ `startDialogue()` と `DialogueOverlay`
* ポーズメニュー・アクションボタン $\rightarrow$ `UIPanel`
* キー入力代わりの仮想ボタン $\rightarrow$ `Touch`
* 読み込み失敗・バリデーションエラー・全文レポート $\rightarrow$ `showErrorPanel()` / `showFixedFormatPanel()`
* ランタイム状態の記録（コピー/保存/JSON化） $\rightarrow$ `Diagnostics`

一つのサンプルで複数の経路を併用することは問題ありませんが、同じ情報を複数の経路で重複して表示させないよう注意してください。

本章の結論として、短い ASCII HUD は `Message`、会話と本文は `DialogueOverlay`、操作パネルは `UIPanel`、長文とエラーは `FixedFormatPanel`、入力代替は `Touch`、状態記録は `Diagnostics`、PC デバッグ表示は `DebugDock`、そしてこれらすべての入口を統合するのが `WebgApp` です。この役割分担を理解することで、今後の開発において「どの情報をどこに配置すべきか」を迷わず決定できるようになります。
