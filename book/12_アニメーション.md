# アニメーション

この章では、`webg/Animation.js`、`webg/Action.js`、`webg/AnimationState.js`、`webg/Schedule.js`、`webg/Task.js` の役割と使い方を整理します。ここでいうアニメーションは、単に「動く」という意味ではありません。`webg` では、アニメーションを `clip -> pattern -> action -> state` という層に分けて考えます。こうして層を分けておくと、何をアセット側で持ち、何をランタイム側で組み立て、何をサンプルやゲームコード側で制御しているのかが見えやすくなります。

この章で最初に押さえたいのは、アニメーションの見え方を 1 つの仕組みでまとめて考えないことです。`clip` はインポーターが持ち込んだ生のキーフレーム列であり、`pattern` はその内部区間、`action` はその区間の束、`state` はどの `action` または `clip` を今使うかを決める上位制御です。たとえば `AnimationState` を入れてもキーフレーム自体が増えるわけではなく、滑らかさそのものは `Action` が選ぶ `fromKey / toKey`、`entryDurationMs`、そして下層の時間管理に強く依存します。ここを最初に分けておくと、「アセット側の問題」と「ランタイム側の制御」が混ざりにくくなります。

また、サンプルの読み方もこの区別にそのまま対応します。`collada_loader` や `gltf_loader` は、インポーターが `clip` 全体を正しく復元できているかを確認するサンプルです。ここでは読み込んだ `clip` を最初から最後まで再生して、キーフレーム列そのものの見え方を確認します。一方で `hand`、`janken`、`animation_state` は、`Action` を使って `clip` の一部を切り出して使うサンプルです。このため、後者は `fromKey / toKey` に強く依存し、Blender のエクスポート設定が変わってキー数や配置が変わると、同じアセット名でも見え方が変わります。ここが、ローダーサンプルとポーズ切替サンプルのもっとも大きな違いです。

さらに重要なのは、`fromKey / toKey` と `entryDurationMs` の意味を混同しないことです。`fromKey / toKey` は 0 始まりのキーインデックスであり、現在の `Action.js` では `fromKey < toKey` が必要です。したがって、「1 つのキーだけを選ぶ」というより、「短い保持区間を切り出す」と考えるほうが実装に合っています。これに対して `entryDurationMs` は、ソースファイルのキーフレーム間隔そのものではありません。これはランタイム側で、「現在姿勢から、その `pattern` の開始姿勢へ何 ms で入るか」を指定する値です。アセット側のキー時刻と似た見え方になることはあっても、自動で一致するわけではありません。

アセット側の前提も、ここでそろえておく必要があります。Blender から出すアセットは `Y-up` を前提にし、`webg` 側で軸補正を足す前提では扱いません。Collada をキーフレームベースで使うときは、全フレーム出力ではなく、`Sampling` 間隔を大きくし、`Keep Keyframes` を有効にしたエクスポートが向いています。glTF / GLB をキーフレームベースで使うときは、`Sampling` を無効にし、補間も `LINEAR` にして出すのが安全です。現在の `webg/GltfShape.js` は `CUBICSPLINE` を受け取っても、前後 tangent を落として中央値だけを抜き出し、`LINEAR` のキー列として近似します。そのため、`CUBICSPLINE` をそのまま厳密再生する前提では読まないほうが安全です。hand 系アセットは、現状では `0: rest`, `1: goo`, `2-3: N1`, `4-5: N2`, `6-7: N3`, `8-9: N4`, `10-11: N5`, `12-14: N0` と読むと整理しやすくなります。同じ姿勢が続いている区間は単なる重複ではなく、その姿勢を保持して見せる時間や、次の状態へ入るための余白を表しています。

この章では、まず各層の役割をはっきり分け、そのあとで `Animation.js`、`Action.js`、`AnimationState.js`、`Schedule.js`、`Task.js` の順に見ていきます。最後に、どの層まで使えばよいか、そして誤解しやすい点を整理します。つまり、本章の中心は「実装をどう読むか」であり、「サンプルで何が起きているか」を見失わないための地図を作ることです。

## 各層の役割を分けて考える

まず `clip` は、1 本の連続アニメーションです。ランタイムでは `Animation.js` のオブジェクトがこれに相当します。ある skeleton に対して、「どの時刻に、どの bone が、どの姿勢を取るか」という列をまとめて持っています。DCC ツール側で `Idle`、`Walk`、`Jump` を別々に出したなら、普通はそれぞれが別 `clip` になります。`clip` は最下層なので、「再生する」「停止する」「特定キー区間へ入る」といったローレベルな役割を持ちますが、「このゲーム状態では walk を使う」といった判断はしません。あくまで再生対象です。 

`pattern` は、その `clip` の内部区間です。`Action.js` では、特定のキー範囲を次のような形で登録します。 

```js
// 特定のキー区間を定義し、ランタイムでの入り込み時間を指定
{
  id: "step_left",
  fromKey: 2,
  toKey: 5,
  entryDurationMs: 120
}
```

ここで大切なのは、`pattern` が新しいアニメーションを作る仕組みではなく、すでに存在する `clip` のどこを使うかを宣言する仕組みだということです。`fromKey` は入り口のキー、`toKey` は終端のキーであり、`entryDurationMs` は現在姿勢からその入り口へ何 ms で入るかをランタイム側で決める値です。したがって、`pattern` は「区間の選択」と「入り込み時間」の定義であって、キーフレーム列そのものではありません。

`action` は、利用者やサンプル側が扱いやすい単位へ `pattern` を束ねたものです。たとえば左足区間と右足区間を `walk_loop` としてまとめておけば、サンプル側は細かな区間名ではなく `walk_loop` という動作名だけで扱えます。 

```js
// 複数の pattern を束ねて、ループ設定を含めて定義
{
  id: "walk_loop",
  patterns: ["step_left", "step_right"],
  loop: true
}
```

`Action.js` は、この `action` を再生しながら現在の `pattern` を管理し、終端へ達したら次へ進むか、先頭へ戻るか、そのまま止まるかを決めます。つまり `action` は、「どの区間を、どの順番で見せるか」をまとめる層です。 

`state` は、「今どの `action` または `clip` を使うか」を決める上位制御です。`AnimationState.js` では、状態名と遷移条件を持つオブジェクトとして扱います。

```js
// 状態名、使用する action、遷移条件を定義
{
  id: "walk",
  action: "walk_loop",
  transitions: [
    { to: "idle", test: (ctx) => ctx.moveSpeed <= 0.1 },
    { to: "jump", test: (ctx) => ctx.jumpPressed }
  ]
}
```

ここでの `state` はアニメーションデータそのものを持ちません。持っているのは、「この状態では何を再生するか」と「どの条件で次へ移るか」です。したがって、`action` は再生対象のまとまりであり、`state` はその再生対象を選ぶ判断単位です。この 2 つは似て見えても役割が違います。

## `Animation.js` は clip をそのまま再生する層

`Animation.js` は最下層の `clip` 再生オブジェクトです。名前だけを見ると汎用アニメーションマネージャーのように見えますが、実際には「1 本の `clip` を skeleton に適用して進める」ことに役割を絞っています。`setTimes(times)` と `setBonePoses(bonePoses)` でキー情報を持ち、`setData(skeleton, bindShapeMatrix)` でランタイム再生に必要な内部タスクを作り、`start()` や `play()` で進行します。つまり `Animation` は、「そのアニメーション列をどう時間で進めるか」を担当する層です。 

ローダーサンプルで `clip` 全体を確認するときは、この層を直接見るのがいちばん素直です。たとえば次のような形です。 

```js
// ローダーサンプル：clip 全体を確認するための基本的な使い方
const model = await app.loadModel("model.dae", {
  format: "collada",
  instantiate: true,
  startAnimations: false
});

const clipId = model.asset.getClipNames()[0];
const runtime = model.runtime;

runtime.restartAnimation(clipId);

function update() {
  runtime.playAnimation(clipId);
}
```

この使い方が向いているのは、アセット側のアニメーションが正しいか、`clip` 全体の見え方そのものを確認したいか、まだ `Action` や `AnimationState` を重ねずにインポーターの結果を確かめたい場面です。つまり、「まず `clip` が正しいか」を見るときの入口が `Animation.js` です。 

## `Action.js` は clip の中を切り出して使いやすくする層

`Action.js` は、1 本の `Animation` を参照し、その内部区間を `pattern` / `action` として再利用しやすくするランタイムコントローラーです。存在理由は単純で、`clip` 全体ではなく、その一部分だけを分かりやすい名前で使いたい場面があるからです。hand pose のように、1 本の `clip` の中へ複数ポーズが入っているアセットでは、この層が特に重要になります。 

```js
// Action を初期化し、pattern と action を定義
const action = new Action(animation);

action.addPattern({
  id: "step_left",
  fromKey: 2,
  toKey: 5,
  entryDurationMs: 120
});

action.addPattern({
  id: "step_right",
  fromKey: 6,
  toKey: 9,
  entryDurationMs: 120
});

// action を定義して再生
action.addAction("walk_loop", ["step_left", "step_right"]);
action.start("walk_loop");
```

毎フレームは `play()` を呼びます。 

```js
// 毎フレーム呼び出して進行させる
action.play();
```

ここで大切なのは、`Action` が `clip` を別形式へ変換しているわけではない、という点です。すでにある `clip` の中から「左足」「右足」「グー」「チョキ」のような区間を切り出し、それを利用者向けの名前へまとめているだけです。`Action` は、サンプル側が毎回キーフレーム番号を意識しなくても済むようにする層だと考えると分かりやすくなります。 

## `AnimationState.js` は再生対象の選択を分離する層

`AnimationState.js` は、今どの `action` または `clip` を再生するべきかを決める上位制御です。役割を一言で言えば、「再生対象の選択をサンプル側の分岐から切り離す」ことです。条件分岐が増えてくると、サンプル側のコードは「どの状態で何を始めるか」と「実際に再生を進める処理」が混ざりやすくなります。そこで `AnimationState` を 1 層入れると、「何を再生するか」の判断だけを別にできます。

最小の state 定義は、次のような形です。 

```js
// state の基本定義：id、action、遷移条件
{
  id: "idle",
  action: "idle",
  transitions: [
    { to: "walk", test: (ctx) => ctx.moveSpeed > 0.1 }
  ]
}
```

遷移定義は、さらに次のような形になります。 

```js
// transition の詳細設定：優先度、クールダウン時間
{
  to: "jump",
  test: (ctx) => ctx.jumpPressed,
  priority: 10,
  cooldownMs: 80
}
```

ここでの `state` はアニメーションデータではなく、再生対象の選択規則です。同じ `action` を複数の `state` から参照することもできますし、`state` が直接 `clip` を指すこともできます。したがって、`state` と `action` は似て見えても同じものではありません。

標準的な導入手順は、まず `Action` または clip 再生側を用意し、その上に `AnimationState` を載せる形です。たとえば次のように書けます。

```js
// Action で区間を切り、それを state で参照
const clipId = model.asset.getClipNames()[0];
const animation = model.runtime.getAnimation(clipId);
const action = new Action(animation);

action.addPattern({ id: "pose0", fromKey: 2, toKey: 3, entryDurationMs: 220 });
action.addPattern({ id: "pose1", fromKey: 4, toKey: 5, entryDurationMs: 220 });
action.addAction("pose0", ["pose0"]);
action.addAction("pose1", ["pose1"]);

const sm = new AnimationState(action, {
  initialState: "pose0"
});

sm.addState({ id: "pose0", action: "pose0" });
sm.addState({ id: "pose1", action: "pose1" });
```

毎フレームは `update(context)` を呼びます。 

```js
// 毎フレームコンテキストを渡して更新
sm.update({
  moveSpeed,
  jumpPressed,
  grounded,
  nowMs
});
```

`AnimationState` には大きく 2 つの使い方があります。1 つはこのように毎フレーム `update(context)` を呼び、条件から遷移先を選ぶ条件評価型です。もう 1 つは、入力イベントやゲームイベントの瞬間に `setState()` を直接呼ぶコマンド型です。`webg/samples/animation_state` は前者、`webg/samples/janken` は後者の代表例です。つまり `AnimationState` は、遷移評価型でしか使えないわけではありません。

たとえば `janken` のように、入力があった瞬間にその手を出したい場面では、次のような使い方が自然です。

```js
// 入力イベントに応じて状態を切り替える
function onJankenInput(choiceId) {
  handState.setState(choiceId, {
    startOptions: {
      entryDurationMs: 250
    },
    force: true
  });
}
```

この形では、毎フレーム状態を計算する必要がなく、入力イベントそのものを高レベルコマンドとして扱えます。`force: true` を使うと、同じ `state` を再始動できます。ここが条件評価型との大きな違いです。

## `Schedule.js` と `Task.js` は補間実行を担当する層

`AnimationState` や `Action` が再生対象を決めたあと、実際の移動と回転はさらに下の層へ流れます。ここを担当するのが `Schedule.js` と `Task.js` です。章全体の見方としては、`AnimationState` は「何を再生するか」を決めるだけで、補間式や bone の座標値には直接触れません。下層では、選ばれた区間が時間付きのコマンド列になり、それが各 bone へ実行されます。

流れを要約すると、`AnimationState.update()` が必要なら遷移を決め、`Action.start()` が現在の `pattern` を選び、`transitionToKey(entryDurationMs, fromKey, toKey)` が `Animation.startTimeFromTo()` を呼びます。その後 `Animation.setData()` が bone ごとの `Task` を組み立て、`Schedule.directExecution()` がコマンド列を流し、`Task.execute(delta_msec)` が `CoordinateSystem.prototype.putRotTransByMatrix` や `CoordinateSystem.prototype.doRotTrans` を対象 bone に適用します。つまり、実際に bone を動かしているのは最下層の `Task` です。

この分離があるため、ステートマシンと補間実行を混ぜずに済みます。`AnimationState` は「今どの動きを見せたいか」だけを持ち、`Task` は「その命令を時間でどう刻むか」に集中できます。入力条件やゲームプレイ条件が変わっても、補間コードそのものを壊しにくいのはこのためです。

## どの層を使うべきか

ここまで見てくると、実際にはどの層まで使えばよいかが問題になります。`Animation` だけで足りるのは、`clip` 全体を 1 本再生できれば十分なときです。ローダーサンプルのように、インポーターが復元した `clip` の見え方そのものを確認したい場面がこれに当たります。`Action` まで使うべきなのは、`clip` の内部キー区間を明示的に再利用したいときです。指ポーズや step のように、短い区間を分かりやすい名前でまとめたい場合や、サンプル側が `action` 再生を直接持っても複雑になりすぎない場合に向いています。`AnimationState` を重ねるべきなのは、状態名を持ちたいとき、入力や gameplay 条件をアニメーション再生コードから分離したいとき、HUD や診断情報に `state` 情報を共通形式で出したいとき、または入力イベントを高レベルコマンドとして扱いたいときです。判断基準は、条件分岐がどれだけ増えたかと、再生対象の選択を別層へ出したほうが読みやすいかどうかです。

## よくある誤解

最初に押さえておきたい誤解は、「`AnimationState` を入れると遷移が必ず滑らかになる」というものです。これは違います。`AnimationState` は再生先を決めるヘルパーであり、補間品質そのものを改善する仕組みではありません。滑らかさは、アセット側のキー配置、`Action` の `fromKey / toKey`、`entryDurationMs`、そして `Animation.transitionTo()` の結果で決まります。

次に、「`Action` があればステートマシンは不要」という理解も、小規模サンプルでは当てはまっても、一般化はできません。条件分岐が増えると、サンプル側が「どの `action` を始めるか」を持ちすぎるようになります。そのときステートマシンを別層にすると、再生対象の選択規則をまとめて読めるようになります。不要かどうかは機能数ではなく、役割の整理で判断するほうが実態に合います。

また、「`state` と `action` は同じもの」というのも誤解です。`action` は再生対象の並びであり、`state` はどの `action` を使うかを決める判断単位です。同じ `action` を複数の `state` から参照することもできますし、`state` が直接 `clip` を指すこともできます。さらに、「`AnimationState` は遷移評価型でしか使えない」というのも誤解です。`janken` のように入力イベントごとに `setState(..., { force: true })` を呼ぶコマンド型も自然な使い方です。

## まとめ

この章で最も大事なのは、`webg` のアニメーションを 1 枚の仕組みとしてではなく、`clip -> pattern -> action -> state` という層で分けて理解することです。`clip` はインポーターが持ち込んだキーフレーム列、`Action` はその内部区間を切り出して再利用しやすくする層、`AnimationState` は今どの `action` または `clip` を使うかを決める上位制御です。そしてその下では、`Animation -> Schedule -> Task` が、実際の補間と bone への反映を担当します。ここが見えると、アセット側の問題、アクション区間の問題、状態遷移の問題を切り分けやすくなります。

また、この章では、ローダーサンプルとポーズ切替サンプルの違い、`fromKey / toKey` と `entryDurationMs` の違い、Collada と glTF / GLB のエクスポート条件、hand 系アセットのキー配置、そして条件評価型とコマンド型という `AnimationState` の 2 つの使い方を整理しました。次章では、こうした層を現行サンプルの上でどう使うか、つまり `animation_state`、`janken`、`tile_sim` をどの順で読み、アセットのキー区間やサイズをどう判断するかを実践的に見ていきます。
