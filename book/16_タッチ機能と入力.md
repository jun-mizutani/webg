# タッチ機能と入力

## Touch と入力の設計思想

本章では、`webg/Touch.js` と `webg/InputController.js` を中心に、webg におけるタッチ入力の構成方法を解説します。文字表示や DOM UI 全体の使い分けについて先に把握したい場合は、[14_UI表示の設計.md](./14_UI表示の設計.md) を参照してください。`Touch` と `UIPanel` の役割の違いを明確にすることで、設計の意図をより深く理解できます。また、入力が 3D 空間のどの地点に到達したかを判定する仕組みについては、次章の [17_衝突判定・raycast・picking.md](./17_衝突判定・raycast・picking.md) で詳しく解説します。

webg における「Touch」とは、単に画面上にボタンを配置することを指すのではありません。仮想ボタンの表示から、そこから得られた入力をキー入力のステートやアクションへと流す「入力経路」までを包括的に設計することを意味します。視覚的な UI を構築するだけでなく、それが適切に入力システムへ伝達される経路を設計しなければ、実用的な操作性は得られません。特にモバイル端末や粗いポインター（coarse pointer：タッチデバイスなど）を利用する環境において、押しやすく、かつ直感的な操作感を提供するための構成を検討します。

## 入力設計における重要事項

実装に入る前に、`Touch`、`UIPanel`、そして `InputController` の関係性を整理します。これらの役割を正しく理解することが、効率的な実装への近道となります。

### Touch は「表示部品」であり「入力の入口」である
`Touch` は Canvas 外の DOM に仮想ボタンを表示しますが、その本質的な役割は見た目だけではありません。`onPress`、`onRelease`、`onAction` といったコールバックを通じて、アプリケーション側のキー入力ステートやアクションへ入力を橋渡しすることにあります。つまり、見た目はボタン群ですが、目的は「キー入力の代替手段」を提供することにあります。

### UIPanel と Touch の使い分け
`UIPanel` もボタンを配置できますが、その用途はシーンアクション、メニュー、ステータスカード、会話の選択肢といった「操作 UI」です。対して `Touch` は、`ArrowLeft` や `A`、`R` といったキーボード入力を代替する「固定ボタン群」を指します。`UIPanel` はシーンの上に重ねるカードやパネルとして、`Touch` は画面下部に固定される仮想キー列として使い分けることで、UI の役割が明確になります。

### InputController による入力の統合
標準的な実装では、`InputController.installTouchControls()` を利用して、キーボードとタッチ入力の両方を一括して管理することを推奨します。`Touch` を直接制御することも可能ですが、`installTouchControls()` を介してホールド入力を `keyState` へ、アクション入力を `pulseAction()` へ流すことで、アプリケーション側は入力ソースを意識せず、単一の判定ロジックで処理を行うことができます。

### ホールド入力とアクション入力の分離
入力設計の基本として、「押している間だけ有効なホールド入力」と「1 回の操作で 1 回だけ反応するアクション入力」を明確に分離します。
- ホールド入力: 移動、回転、視点操作など。
- アクション入力: リセット、決定、ポーズ、モード切替など。
この整理を最初に行うことで、ボタン数が増加しても処理フローをシンプルに保つことができます。

### キー名称の統一
キーボード入力では `event.key.toLowerCase()` を使用して小文字で管理します。タッチボタン側でも同様に小文字のキー名を使用することで、`InputController` によるアクションマップを共通して利用でき、実装の重複を避けることができます。

## 役割の定義

### Touch.js
`Touch` は仮想ボタン UI を構築するためのクラスです。表示先は Canvas 外の DOM であり、通常は `position: fixed` によって画面下部に配置されます。`groups` 配列に基づいたボタン群を生成し、`onPress`、`onRelease`、`onAction`、`onAnyPress` を通じてアプリケーション側へ通知します。

`Touch` は汎用的なパネル管理を目的としたものではなく、「押下中の入力」と「単発の入力」を画面下部の固定 UI として提供することに特化した設計となっています。そのため、レイアウトもシーン上の自由配置ではなく、下端のボタン列としての最適化が行われています。

### InputController.js
`InputController` は、キーボードとタッチ入力を共通の `keyState` およびアクションステートに統合するクラスです。`attach()` メソッドでキーボードの `keydown` / `keyup` イベントを捕捉し、`installTouchControls()` メソッドで `Touch` 機能を組み込みます。これにより、サンプル側では `input.has("a")`、`input.getAction("jump")`、`input.wasActionPressed("pause")` のように、入力元を意識せずにステートを参照することが可能になります。

## Touch と UIPanel の比較

両者はともに DOM を利用しますが、その目的は明確に異なります。

- Touch
  - 画面下部に固定配置される。
  - キーボード入力の代替として機能する。
  - ホールド入力とアクション入力を明確に区別する。
  - キーボードと共通のキー名（小文字）で管理する。
- UIPanel
  - シーンの上にオーバーレイとして重ねて表示される。
  - アクションボタン、ステータス表示、メニュー、選択肢などを構成する。
  - 情報提示やシーン全体の操作 UI として機能する。
  - 必ずしもキー入力の代替である必要はない。

例えば、「ポーズメニュー」や「カメラリセット」などのシーン操作ボタンは `UIPanel` が適しています。一方で、「← / → / A / D / R」のようなゲームプレイに直結する入力ボタンは `Touch` が適しています。これらを適切に使い分けることで、操作 UI と入力代替 UI の干渉を防ぐことができます。

## 標準的な導入手順

### キーボード入力の接続
まず `InputController.attach()` を使用してキーボード入力を有効にします。キー名は内部で `normalizeKey()` を通り小文字に統一されるため、比較時は小文字で記述してください。

ここで注意すべき点は、受け取るキー名が raw の `event.key` とは異なる場合があることです。例えば `Space` は `" "` ではなく `space`、`Esc` は `escape` として処理されます。タッチボタンと共通の条件で扱いたい場合は、以下の名称を使用してください。

#### 名称で指定する必要がある特殊キー一覧

| キー | 比較に使用する名称 |
| :--- | :--- |
| Space | `space` |
| Escape | `escape` |
| Enter | `enter` |
| Tab | `tab` |
| Backspace | `backspace` |
| Delete | `delete` |
| Shift | `shift` |
| Control | `control` |
| Ctrl | `ctrl` |
| Alt | `alt` |
| Option | `option` |
| Meta | `meta` |
| Command | `command` |
| Cmd | `cmd` |
| Home | `home` |
| End | `end` |
| PageUp | `pageup` |
| PageDown | `pagedown` |
| ArrowLeft | `arrowleft` |
| ArrowRight | `arrowright` |
| ArrowUp | `arrowup` |
| ArrowDown | `arrowdown` |
| F1 〜 F12 | `f1` 〜 `f12` |
| Numpad0 〜 Numpad9 | `numpad0` 〜 `numpad9` |

ここで挙げた modifier key 名は、単に `InputController.has()` で比較するときだけでなく、`EyeRig` や `WebgApp.createOrbitEyeRig()` の `panModifierKey` に指定するときにも使えます。特に `panModifierKey` では、pointer drag 側でも同じ名称を参照するため、実用上は `shift`、`control` / `ctrl`、`alt` / `option`、`meta` / `command` / `cmd` のような modifier key 名を使うと覚えておくと整理しやすくなります。

### Touch コントロールの追加
次に `installTouchControls()` を呼び出し、`Touch` 機能を組み込みます。ホールド入力用ボタンは `press()` / `release()` を通じてステートに反映され、アクション入力用ボタンは `pulseAction()` を通じて 1 フレーム分のアクションステートとして反映されます。

```js
import InputController from "./webg/InputController.js";

const input = new InputController(document);

input.attach({
  onKeyDown: (key) => {
    if (key === "escape") pauseGame();
  }
});

input.installTouchControls({
  touchDeviceOnly: false,
  autoSpread: true,
  groups: [
    {
      id: "move",
      buttons: [
        { key: "arrowleft", label: "←", kind: "hold", ariaLabel: "move left" },
        { key: "arrowright", label: "→", kind: "hold", ariaLabel: "move right" }
      ]
    },
    {
      id: "turn",
      buttons: [
        { key: "a", label: "A", kind: "hold", ariaLabel: "turn left" },
        { key: "d", label: "D", kind: "hold", ariaLabel: "turn right" }
      ]
    },
    {
      id: "action",
      buttons: [
        { key: "r", label: "R", kind: "action", ariaLabel: "reset" }
      ]
    }
  ],
  onAction: ({ key }) => {
    if (key === "r") resetPlayer();
  }
});

function update() {
  if (input.has("arrowleft")) moveLeft();
  if (input.has("arrowright")) moveRight();
  if (input.has("a")) rotateLeft();
  if (input.has("d")) rotateRight();
}
```

この構成の最大の利点は、キーボードとタッチ入力のどちらが操作されていても、アプリケーション側は同じステートを参照すればよい点にあります。

### アクションマップの活用
`InputController.registerActionMap()` を使用している場合は、タッチボタンもアクション名に紐付けます。`installTouchControls()` のアクションボタンは内部的に `pulseAction()` を呼び出すため、`wasActionPressed()` や `getAction()` と自然に連携します。これは `webg/unittest/input_controller` や `webg/samples/breakout` のように、操作体系を完全に統一したい場合に非常に有効です。

### Canvas ジェスチャーによるカメラ操作
仮想ボタンによる入力とは別に、モバイル環境でマウスの代わりにシーンを見回したい場合は、Canvas 上のポインターイベントを利用したジェスチャー操作を実装します。

仮想ボタンは「特定のコマンド（キー）の代替」であり、Canvas ジェスチャーは「空間的な操作」であるため、これらは役割を分けて実装することを推奨します。`webg/samples/tile_sim` ではこの方針を採用しており、ボタンによる移動操作と、Canvas 上の直接操作を併用しています。

実装のステップは以下の通りです。

1. `canvas.style.touchAction = "none"` を設定し、ブラウザ標準のスクロールやピンチ操作による干渉を防ぎます。
2. `pointerdown` / `pointermove` / `pointerup` / `pointercancel` イベントでポインターを追跡し、単指操作と多指操作を区別します。
3. 単指ドラッグでは、移動量（`dx / dy`）をカメラの回転（yaw / pitch）に反映させます。
4. 二本指ドラッグでは、中心位置の移動量をカメラターゲットの平行移動へ、指間距離の変化をズームへ変換します。
5. タップとドラッグを区別する場合、移動量のしきい値（threshold）を設け、それを超えなかった `pointerup` のみをタップ（選択処理）として判定します。

この手法により、ボタン配置に依存せず、Canvas 上を直接触ってシーンを操作することが可能になります。

## Touch クラスの直接利用

機能の個別検証や、最小限の構成で動作を確認したい場合は、`InputController` を介さず `Touch` クラスを直接利用します。`webg/unittest/touch` ではこの形式を用いて、ボタンの密度やレイアウトの挙動を検証しています。

```js
import Touch from "./webg/Touch.js";

const keyState = new Set();
const press = (key) => keyState.add(String(key).toLowerCase());
const release = (key) => keyState.delete(String(key).toLowerCase());

const touch = new Touch(document, {
  touchDeviceOnly: false
});

touch.create({
  autoSpread: true,
  groups: [
    {
      id: "move",
      buttons: [
        { key: "arrowleft", label: "←", kind: "hold" },
        { key: "arrowright", label: "→", kind: "hold" }
      ]
    },
    {
      id: "action",
      buttons: [
        { key: "r", label: "R", kind: "action" }
      ]
    }
  ],
  onPress: ({ key }) => press(key),
  onRelease: ({ key }) => release(key),
  onAction: ({ key }) => {
    if (key === "r") resetPlayer();
  }
});
```

この実装では、ホールド入力とアクション入力がどのように通知されるかを直接制御できます。ただし、キーボードとの統合やアクションマップの管理は自前で実装する必要があります。

### createActionButtons() の利用
アクションボタンのみを配置したい場合は、`createActionButtons()` メソッドが利用可能です。これはすべてのボタンを `kind: "action"` として生成する簡易的なメソッドであり、単発のコマンドを並べるメニューなどの実装に便利です。

## レイアウトと表示仕様

`Touch` のルート要素は画面下部に固定されます。標準 CSS では `left: 0`、`right: 0`、`bottom: 0` が設定され、デバイスの `safe-area-inset` も考慮されています。

### ボタン密度（Density）
ボタンの総数に応じて、視認性と押しやすさを維持するためにボタンサイズが自動的に調整されます（`applyDensitySize()` が担当）。

- 5 個以下: `52px`
- 6 〜 8 個: `48px`（高密度表示）
- 9 個以上: `44px`（コンパクト表示）

### レイアウトモード
`applyLayoutMode()` は、画面幅とボタン数に基づいて「単一行」「複数行」「展開配置」を自動的に切り替えます。`autoSpread: true` が設定されている場合は、利用可能な余白に応じて左右に広がった配置が選択されます。`webg/unittest/touch` では、クエリパラメータ `?touchButtons=<数>` を指定することで、これらのレイアウト変化を確認できます。

### デバッグ用データ属性
デバッグを容易にするため、ルート要素には以下のデータ属性が付与されます。
- `data-touch-count`
- `data-touch-density`
- `data-touch-layout`
- `data-touch-spread`

## オプション設定

### touchDeviceOnly
デフォルトでは `true` です。粗いポインター（coarse pointer）を持つ端末でのみ表示させたい場合に利用します。PC 環境でも常時表示させて動作確認を行いたい場合は `false` に設定してください。

### force
`force: true` を指定すると、デバイスの判定に関わらず強制的にタッチデバイスとして扱い、UI を表示します。

### className
ルート要素に独自のクラス名を付与できます。サンプルごとに CSS を調整し、標準レイアウトを微修正したい場合に有効です。

### onAnyPress
いずれかのボタンが押された瞬間に実行したい処理がある場合に利用します。初回入力の検知や、入力ログの採取などに適しています。

## 実践的な参照例

実装の際は、以下のサンプルやテストコードを参照してください。

- webg/unittest/touch
  `Touch` クラスの最小構成テストです。ホールド/アクション入力の挙動や、ボタン数による密度・レイアウトの変化を確認するのに最適です。
- webg/unittest/input_controller
  キーボードとタッチ入力を `InputController` へ統合する標準的なフローを確認できます。アクションマップやパルスアクションの実装例が含まれています。
- webg/samples/breakout
  ゲームプレイ操作にタッチボタンを組み込んだ代表的な例です。左右移動、開始、ポーズ、リスタートをキーボードと共通の経路で処理しています。
- webg/samples/tile_sim
  移動用タッチボタンと Canvas ジェスチャーを併用した高度な例です。TileMap の操作、カメラのドラッグ/ピンチズームなどの実装を確認できます。

## まとめ：使い分けの指針

- キーボードとタッチ入力を共通のロジックで扱いたい $\rightarrow$ `InputController` を使用。
- タッチボタンの最小挙動を個別に検証したい $\rightarrow$ `Touch` を直接使用。
- シーン上の操作パネルやメニューを構築したい $\rightarrow$ `UIPanel` を使用。
- 会話やチュートリアルを提示したい $\rightarrow$ `DialogueOverlay` を使用。
- ボタンの意味を画面上で説明したい $\rightarrow$ `Message` や `WebgApp` のガイド/ステータスヘルパーを併用。

この切り分けを徹底することで、入力代替 UI と情報表示 UI が混在することを防ぎ、保守性の高いコードを維持できます。

## 注意点とよくあるミス

- キー名の比較: `event.key` をそのまま比較せず、必ず `toLowerCase()` を通した小文字の名称で比較してください。
- 入力種類の混同: 移動などの連続的な操作は「ホールド入力」、リセットや決定などの単発操作は「アクション入力」に明確に分けてください。
- UI の目的外利用: 入力代替は `Touch`、シーン操作 UI は `UIPanel` と使い分けてください。
- 開発環境での表示: PC での動作確認時は `touchDeviceOnly: false` を設定してください。
- ユーザーへのガイド: タッチボタンを配置する際は、その機能が何であるかを `Message` 等で提示し、ユーザーが迷わないように配慮してください。
