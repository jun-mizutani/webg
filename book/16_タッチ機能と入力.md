# 16 タッチ機能と入力

最終更新：2026-04-17 JST

## Touch と入力の詳細

この章では、`webg/Touch.js` と `webg/InputController.js` を中心に、webg における Touch 入力の構成方法を見ていきます。文字表示や DOM UI 全体の使い分けから入りたい場合は、先に [14_UI表示の設計.md](./14_UI表示の設計.md) を読むと、`Touch` と `UIPanel` の役割の違いを見失いにくくなります。入力した先が 3D 空間のどこに当たるかを続けて整理したい場合は、次章の [17_衝突判定・raycast・picking.md](./17_衝突判定・raycast・picking.md) を合わせて読むと流れがつながります。

ここでいう Touch は、単にボタンを画面へ出すことだけを指しません。webg では、仮想ボタンの表示と、そこからキー入力のステートやアクションへ入力を流す経路までをひとまとめに考えます。見た目だけを整えても入力に流れなければ意味がありませんし、入力だけ通っても、モバイルや coarse pointer 端末で押しやすく見えていなければ使いにくいからです。

## この章で押さえること

この章で最初に迷いやすいのは、「`Touch` はただの UI 部品なのか」「`UIPanel` と何が違うのか」「`InputController` を通すべきか」を同時に考えてしまう点です。先に次の背景を押さえておくと、あとでコードと文書を行き来するときに判断しやすくなります。

### A. `Touch` は表示部品であると同時に入力の入口

`Touch` は Canvas 外の DOM に仮想ボタンを出しますが、役割は見た目だけではありません。`onPress`、`onRelease`、`onAction` を通じて、アプリ側のキー入力のステートやアクションへ入力を橋渡しすることが本体です。見た目はボタン群でも、目的は「押せる操作パネル」ではなく「キー入力の代替」です。

### B. `UIPanel` は操作 UI、`Touch` は入力代替 UI

`UIPanel` もボタンを出せますが、用途はシーンアクション、メニュー、ステータスカード、会話選択肢のような操作 UI です。これに対して `Touch` は、`ArrowLeft`、`A`、`R` のようなキーの代わりに押す固定ボタン群です。どちらも DOM を使いますが、`UIPanel` はシーンの上に重ねるカード / パネル、`Touch` は画面下に固定する仮想キー列と考えると整理しやすくなります。

### C. 標準経路は `InputController.installTouchControls()`

新しいサンプルでは、まず `InputController` にキーボードと Touch の両方を集めるのが分かりやすくなります。`Touch` を直接使うこともできますが、通常は `installTouchControls()` を通して、ホールド入力を `keyState`、アクション入力を `pulseAction()` へ流すほうが、サンプル側の判定を 1 系統に保ちやすくなります。

### D. ホールド入力とアクション入力は最初に分ける

押している間だけ効く入力はホールド入力、1 回押したら 1 回だけ反応したい入力はアクション入力に分けます。移動、回転、視点操作はホールド入力、reset、confirm、pause、モード切替はアクション入力にするのが基本です。この整理を最初にしておくと、Touch ボタンが増えても処理フローが崩れにくくなります。

### E. キー名はキーボードと同じ lower-case にそろえる

キーボード側は `event.key.toLowerCase()`、Touch 側も同じ lower-case のキー名を使うのが基本です。`ArrowLeft`、`arrowleft`、`A`、`a` の差をサンプル側で意識しなくて済み、`InputController` によるアクションマップもそのまま使いやすくなります。

## この章の進め方

webg の Touch 入力は、まず `Touch` で仮想ボタンを出し、その入力を `InputController` でキーボードと共通のステートへ流す、という 2 段構成で考えると整理しやすくなります。`Touch` は画面下に固定表示される仮想キー列、`InputController` はその入力を `keyState` やアクション名へ束ねる部品です。

通常のサンプルでは、`InputController.installTouchControls()` を使うのが標準です。これでホールド入力は `press()` / `release()`、アクション入力は `pulseAction()` に入り、サンプル側はキーボードと Touch を同じ `has()`、`getAction()`、`wasActionPressed()` で読めます。`Touch` を直接使うのは、内部テストや、`InputController` を通さずに最小構成を確かめたい場合です。

実例としては、最小確認なら `webg/unittest/touch`、アクションマップと合わせて見るなら `webg/unittest/input_controller`、ゲームへの組み込み例なら `webg/samples/breakout` と `webg/samples/tile_sim` が分かりやすくなります。`webg/samples/circular_breaker`、`webg/samples/janken`、`webg/samples/billboard`、`webg/samples/shapes` なども Touch ボタンを使っていますが、まずは上記の 4 つを先に読むと流れをつかみやすくなります。

## 1. 役割の分け方

### 1.1 `Touch.js`

`Touch` は、仮想ボタン UI を作るクラスです。表示先は Canvas 外の DOM で、通常は `position: fixed` による画面下配置になります。`groups` 配列からボタン群を生成し、`onPress`、`onRelease`、`onAction`、`onAnyPress` でアプリ側へ通知します。

ここで大事なのは、`Touch` が「汎用のパネル管理用部品」ではない点です。ボタンを自由配置するための部品ではなく、「押されている間の入力」と「1 回だけの入力」を画面下の固定 UI として出すことに役割を絞っています。そのため、レイアウトもシーン上のカード群ではなく、下端のボタン列に向いた設計になっています。

### 1.2 `InputController.js`

`InputController` は、キーボードと Touch を同じ `keyState` とアクションのステートにまとめるクラスです。`attach()` でキーボードの `keydown` / `keyup` を受け、`installTouchControls()` で `Touch` を組み込みます。サンプル側は `input.has("a")`、`input.getAction("jump")`、`input.wasActionPressed("pause")` のように、入力ソースを意識せず読むだけで済みます。

`Touch` を直接使うこともできますが、複数のサンプルで同じ操作を扱うなら、`InputController` にまとめたほうが読みやすく、保守性も高まります。

## 2. `Touch` と `UIPanel` の違い

`Touch` と `UIPanel` はどちらも DOM を使いますが、目的が異なります。

* `Touch`

  * 画面下に固定する
  * キー入力の代わりに押す
  * ホールド入力 / アクション入力に分ける
  * キーボードと同じキー名にそろえる
* `UIPanel`

  * シーンの上に重ねる
  * アクションボタン、カード、ステータス、メニュー、選択肢を組む
  * 情報表示やシーン操作の UI を構成する
  * キー入力の代替とは限らない

たとえば `Pause Scene` や `Reset Camera` のシーン操作ボタンは `UIPanel` が向いています。一方で `← / → / A / D / R` のような「キーボードの代わりに押す」ボタンは `Touch` が向いています。ここを混ぜると、操作 UI と入力代替 UI が同じ場所に重なりやすくなります。

## 3. 標準の導入手順

### 3.1 キーボードを先に接続する

まず `InputController.attach()` でキーボードを受けます。キー名は `normalizeKey()` を通って lower-case へそろうので、サンプル側はその前提で書くほうが安全です。

重要なのは、ここで受け取る `key` は raw の `event.key` ではない、という点です。たとえば `Space` は `" "` ではなく `space`、`Esc` は `escape` として渡ります。`attachInput()` や `Touch` と同じ条件で扱いたいときは、以下の比較名へそろえてください。

### 3.2 `installTouchControls()` で Touch を追加する

そのあと `installTouchControls()` で `Touch` を組み込みます。ホールド入力用のボタンは `press()` / `release()` に入り、アクション入力用のボタンは `pulseAction()` として 1 フレーム分のアクションステートに反映されます。

```js
import InputController from "./webg/InputController.js";

const input = new InputController(document);

input.attach({
  onKeyDown: (key) => {
    if (key === "escape") pauseGame();
  }
});

input.installTouchControls({
  touchDeviceOnly: false,
  autoSpread: true,
  groups: [
    {
      id: "move",
      buttons: [
        { key: "arrowleft", label: "←", kind: "hold", ariaLabel: "move left" },
        { key: "arrowright", label: "→", kind: "hold", ariaLabel: "move right" }
      ]
    },
    {
      id: "turn",
      buttons: [
        { key: "a", label: "A", kind: "hold", ariaLabel: "turn left" },
        { key: "d", label: "D", kind: "hold", ariaLabel: "turn right" }
      ]
    },
    {
      id: "action",
      buttons: [
        { key: "r", label: "R", kind: "action", ariaLabel: "reset" }
      ]
    }
  ],
  onAction: ({ key }) => {
    if (key === "r") resetPlayer();
  }
});

function update() {
  if (input.has("arrowleft")) moveLeft();
  if (input.has("arrowright")) moveRight();
  if (input.has("a")) rotateLeft();
  if (input.has("d")) rotateRight();
}
```

この形の利点は、キーボードでも Touch でも同じステートを見ればよいことです。操作の実装を 2 系統に分けずに済みます。

### 3.3 アクションマップを使う場合

`InputController.registerActionMap()` を使っているサンプルでは、Touch ボタンもそのアクション名に寄せます。`installTouchControls()` のアクションボタンは内部で `pulseAction()` を呼ぶので、`wasActionPressed()` や `getAction()` と自然につながります。

これは `webg/unittest/input_controller` や `webg/samples/breakout` のように、キーボードと Touch を完全に同じゲームプレイ操作へ束ねたい場面で特に有効です。

### 3.4 Canvas ジェスチャーでカメラを動かす

ここまでの説明は主に `Touch.js` と `InputController.js` を使った仮想ボタン入力ですが、モバイルでマウスの代わりにシーンを見回したい場合は、それだけでは足りません。カメラの回転、平行移動、ズームのような連続入力は、画面下のボタン群ではなく `canvas` 上のポインターイベントで扱うほうが自然です。

このときの考え方は、`Touch` の仮想ボタンと Canvas ジェスチャーを別の役割として分けることです。仮想ボタンは `ArrowLeft` や `GO` の代わりに押す固定入力、Canvas ジェスチャーはシーンを直接つかんで動かす入力です。両方を同じサンプルに入れても構いませんが、処理フローは別に保つほうが読みやすくなります。

`tile_sim` では、この方針でモバイル向けのカメラジェスチャーを入れています。マウスではドラッグで orbit、ホイールでズーム、クリックで tile inspection を行い、Touch では 1 本指ドラッグで orbit、2 本指ドラッグでカメラターゲットの平行移動、ピンチでズーム、タップで tile inspection を行います。ボタン群は引き続き移動や `GO`、preset 切替に使い、カメラジェスチャーは Canvas 側で別に受けます。

実装手順は次の順序で考えると整理しやすくなります。

1. `canvas.style.touchAction = "none"` を設定し、ブラウザー標準のスクロール / ピンチにカメラジェスチャーを奪われないようにします。
2. `pointerdown / pointermove / pointerup / pointercancel` で Touch ポインターを追跡し、1 本指と 2 本指を分けて扱います。
3. 1 本指では `dx / dy` を orbit の yaw / pitch へ反映します。
4. 2 本指では中心位置の移動量をカメラターゲットの平行移動へ変換し、指間距離の増減をズームへ変換します。
5. `tap` と `drag` を分けたい場合は、移動量の threshold を設け、threshold を超えなかった `pointerup` だけを選択処理に使います。

この方式の利点は、カメラジェスチャーがボタン配置に依存しないことです。移動系の仮想ボタンを画面下に置いたままでも、シーンの見回しは Canvas 上を直接触って行えます。TileMap のように `click / tap` でセルを拾うサンプルでは特に相性がよく、`tile_sim` がそのまま参照例になります。

## 4. `Touch` を直接使う場合

`Touch` を直接使うのは、内部テストや最小構成の確認に向いています。`webg/unittest/touch` はこの形で、`Touch` 自体の密度やレイアウトを見やすくしています。

```js
import Touch from "./webg/Touch.js";

const keyState = new Set();
const press = (key) => keyState.add(String(key).toLowerCase());
const release = (key) => keyState.delete(String(key).toLowerCase());

const touch = new Touch(document, {
  touchDeviceOnly: false
});

touch.create({
  autoSpread: true,
  groups: [
    {
      id: "move",
      buttons: [
        { key: "arrowleft", label: "←", kind: "hold" },
        { key: "arrowright", label: "→", kind: "hold" }
      ]
    },
    {
      id: "action",
      buttons: [
        { key: "r", label: "R", kind: "action" }
      ]
    }
  ],
  onPress: ({ key }) => press(key),
  onRelease: ({ key }) => release(key),
  onAction: ({ key }) => {
    if (key === "r") resetPlayer();
  }
});
```

この形では、`Touch` がホールド入力とアクション入力をどう通知するかを直接追えます。その代わり、キーボードとの統合やアクションマップの管理は自前になります。新規サンプルの標準経路としては、まず `InputController` を通すほうが安全です。

### 4.1 `createActionButtons()`

アクションボタンだけをまとめて置きたい場合は `createActionButtons()` を使えます。これはすべてのボタンを `kind: "action"` に統一する簡易的な入口です。メニューや単発のコマンドだけを並べるときに便利ですが、ホールド入力を含むゲームプレイ操作では通常の `create()` のほうが自然です。

## 5. レイアウトと表示位置

`Touch` の root は画面下固定です。標準 CSS では `left: 0`、`right: 0`、`bottom: 0` で、safe-area-inset も考慮します。つまり、シーンの上へカードを重ねる `UIPanel` と違い、`Touch` は画面端の固定 UI として見るべきです。

### 5.1 密度

`Touch` はボタン数に応じてサイズを自動で少し変えます。

* 5 個以下：`52px`
* 6 個から 8 個：`48px` の高密度表示
* 9 個以上：`44px` のコンパクト表示

これにより、ボタン数が増えても、まずは画面内へ収めやすくなります。`applyDensitySize()` がこの調整を担当します。

### 5.2 レイアウト

`applyLayoutMode()` は横幅とボタン数を見て、単一行 / 複数行 / 展開配置を切り替えます。`autoSpread: true` のときは、余白に応じて左右へ開いた配置も選びます。`webg/unittest/touch` は `?touchButtons=<数>` でボタン数を増やし、この挙動を確認するためのテストです。

### 5.3 data 属性

デバッグ用に root へ次の属性が付きます。

* `data-touch-count`
* `data-touch-density`
* `data-touch-layout`
* `data-touch-spread`

`webg/unittest/touch` ではこの値も HUD に出し、現在どのレイアウトになっているかを確認できるようにしています。

## 6. よく使う option

### 6.1 `touchDeviceOnly`

`touchDeviceOnly: true` が初期値です。coarse pointer 端末だけで表示したいときに向いています。`false` にすると PC でも常時表示できるので、デスクトップで確認したいサンプルや unittest に向いています。

### 6.2 `force`

`Touch` 自体の option に `force: true` を渡すと、ポインター判定にかかわらず coarse pointer 扱いにできます。特殊な確認用途や PC での強制表示に使います。

### 6.3 `className`

`className` で root に独自のクラスを足すと、サンプルごとの CSS 調整がしやすくなります。標準のレイアウトを少しだけ変えたい場合に便利です。

### 6.4 `onAnyPress`

`onAnyPress` は、どのボタンでも押された瞬間に呼びたいときに使います。初回入力の検出や簡単なログ採取に向いています。

## 7. 現在読むべき sample / unittest

### 7.1 `webg/unittest/touch`

`Touch` を直接使う最小テストです。ホールド入力 / アクション入力の違い、`?touchButtons=<数>` による密度変化、ボタンサイズとレイアウト属性を確認するにはここが最短です。

### 7.2 `webg/unittest/input_controller`

キーボードと Touch を `InputController` へ統合する標準経路を確認するテストです。アクションマップ、pulse action、manual touch controls まで含めて見たいときに向いています。

### 7.3 `webg/samples/breakout`

ゲーム内で Touch ボタンをゲームプレイ操作に使う代表サンプルです。左右移動、開始、pause、restart をキーボードと同じ経路へ流しています。

### 7.4 `webg/samples/tile_sim`

少し大きめのサンプルの中で、移動系の Touch ボタンと Canvas ジェスチャーをどう併用するかを見る入口です。TileMap ランタイム、`GO` / preset / reset の Touch ボタン、ドラッグ / ピンチによるカメラ操作を一緒に読みたいときに向いています。

## 8. どう使い分けるか

* キーボードと Touch を同じロジックで扱いたい。`InputController` を使います。
* Touch ボタンの最小挙動を直接確かめたい。`Touch` を使います。
* シーン上の操作パネルやメニューを組みたい。`UIPanel` を使います。
* 会話やチュートリアルを出したい。`DialogueOverlay` を使います。
* Touch ボタンの意味を画面上に説明したい。`Message` や `WebgApp` のガイド / ステータスヘルパーを併用します。

この切り分けを守ると、入力代替 UI と情報表示 UI が混ざりにくくなります。

## 9. よくあるミス

* `event.key` をそのまま比較してしまう。`toLowerCase()` を通してから比較します。
* ホールド入力とアクション入力を同じ扱いにしてしまう。移動はホールド入力、reset / confirm はアクション入力に分けます。
* `Touch` と `UIPanel` を同じ用途で使ってしまう。入力代替は `Touch`、シーンの操作 UI は `UIPanel` に分けます。
* PC で確認したいのに `touchDeviceOnly: true` のままにする。デスクトップ確認時は `touchDeviceOnly: false` を使います。
* Touch ボタンの意味を画面へ出さない。`Message` か `WebgApp` のガイド / ステータスを併用します。
