# 18 サウンド

最終更新：2026-04-09 JST

この章では、`AudioSynth` と `GameAudioSynth` を中心に、webg のサウンド機能を見ていきます。

背景として、webg のサウンドは単に「効果音を 1 回鳴らすヘルパー」ではありません。`AudioContext` の遅延初期化、SE と BGM のバス分離、ディレイとリバーブの独立した調整、BGM の簡易シーケンサを 1 つの流れとして扱います。音の調整を追いやすくするために、「何を鳴らすか」と「どう聞かせるか」を分けて考える設計になっています。

## この章で押さえること

### A. 音の開始は、利用者が画面を操作した後に行う

`AudioSynth` はコンストラクタで `AudioContext` を生成しません。ブラウザーには、利用者が何も操作していない状態で突然音が鳴るのを防ぐため、音声再生の開始を制限する仕組みがあるためです。`resume()` が呼ばれて初めて `ensureContext()` が `AudioContext` を生成し、マスター、SE バス、BGM バス、ディレイ、リバーブの経路を組みます。`webg/samples/sound` に `Audio Start` ボタンがあるのは、この制約をそのまま表に出しているためです。

### B. `AudioSynth` は音の基盤、`GameAudioSynth` はゲーム向けの実用プリセット

`AudioSynth` は Web Audio API の配線、SE 発音、BGM シーケンサ、エンベロープ、ディレイ、リバーブを担当します。`GameAudioSynth` はその上にメロディプリセットと SE カタログを載せます。場面ごとの BGM preset や通知音の割り当ては sample / game_support 側へ分離できるので、通常の利用では `GameAudioSynth` から始め、その上に必要な preset 層を追加するのが適切です。

### C. SE と BGM は最初から別バスで扱う

webg では SE と BGM を別バスにしているので、音量、ディレイ、リバーブの調整も分けて扱えます。ゲームでは、SE だけを強調したい場面、BGM だけを控えめにしたい場面があるため、この分離は実用上かなり重要です。

### D. リバーブの「量」と「性格」は別に調整する

`setSeReverb()` / `setBgmReverb()` はセンド / リターンの量を変える API です。`setSeReverbImpulse()` / `setBgmReverbImpulse()` は Convolver のインパルスレスポンス、つまり `room` / `hall` / `plate` の性格と長さを変える API です。量と性格を分けて考えると、音設計の説明と UI の操作が一致しやすくなります。

### E. インパルスレスポンスは外部ファイルではなく手続き的に生成している

現在の webg では、インパルスレスポンス専用クラスを別ファイルに分けておらず、`AudioSynth.js` の `createImpulseResponse()` が直接ステレオ `AudioBuffer` を作ります。`room`、`hall`、`plate` の違いは、プロファイルと初期反射の差で作っており、文書からもその仕組みを追跡できるように構成しておくことが重要です。

## この章の流れ

webg でサウンドを扱うときは、まず `GameAudioSynth` を使うのが適切です。使う流れは次の順で考えると理解しやすくなります。

1. `const synth = new GameAudioSynth()` を作る
2. 利用者がボタンを押すなどの操作を行った後に `await synth.resume()` を呼ぶ
3. `setMasterVolume()`、`setSeVolume()`、`setBgmVolume()` で音量バランスを整える
4. SE なら `playSe(name)`、BGM なら `setMelody(name)` と `startBgm()` を使う
5. ディレイとリバーブをバスごとに分けて調整する
つまり、まず `AudioContext` を利用可能にし、そのあと「SE か BGM か」「量を変えたいのか、特性を変えたいのか」を分けて触るのが標準です。

## 1. 背景

ゲームやサンプルのサウンドでは、単にオシレーターを鳴らせば十分ということは多くありません。短い効果音とループする BGM では求める性格が異なりますし、ディレイやリバーブの量と空間感も別々に調整したくなります。さらに、同じ素材でも envelope やリバーブ設定で印象が大きく変わるため、音を鳴らしながらその場で調整できることが重要になります。

webg のサウンド機能は、そのために `AudioSynth` と `GameAudioSynth` を分けています。基盤側は配線と時間制御に集中し、上位層はゲーム向けのプリセット名で扱えるようにすることで、音の構造と実際の使い方を分け、可読性を高めています。

## 2. 役割

サウンド関連の役割は、次の 3 つの層に分けると分かりやすくなります。

1. `AudioSynth`
   Web Audio API の配線、SE 発音、BGM シーケンサ、エンベロープ、ディレイ、リバーブ、インパルスレスポンス生成を担当します
2. `GameAudioSynth`
   メロディプリセット、SE カタログ、ゲーム向け発音 helper を担当します
3. `webg/samples/sound`
   実際の UI を通して、音設計と確認を同じ画面で繰り返すサンプルです

ここで重要なのは、`AudioSynth` が「どう鳴らすか」の層であり、`GameAudioSynth` が「どんな名前で何を鳴らすか」の層だという点です。

## 3. 理由

この構成を取る理由は 3 つあります。

まず、SE と BGM を同じ API で無理に扱わないためです。単発の SE と連続する BGM では、バスもエンベロープも時間の扱いも異なります。

次に、ディレイとリバーブの量と性格を分けて説明しやすくするためです。音量調整だけでなく、空間の特性そのものも扱えるようにしておくと、サンプルの比較がしやすくなります。

さらに、ゲーム側の実装では「コアが用意する基盤」と「そのゲーム固有の preset」を分けたほうが、役割の切り分けが分かりやすくなるためです。共通部分は `GameAudioSynth` に寄せ、場面ごとの BGM preset や通知音の対応は sample 側で持つと、あとから調整範囲を追いやすくなります。

## 4. 使いどころ

webg のサウンドは、次のような場面で特に役立ちます。

1. サンプルに単発の SE と簡易 BGM をすぐ組み込みたいとき
2. オシレーターベースの軽い音設計をリポジトリ内だけで完結させたいとき
3. SE と BGM の音量、ディレイ、リバーブを別々に調整したいとき
4. `room` / `hall` / `plate` の空間感を外部 IR ファイルなしで比較したいとき
5. 短い操作で複数の効果音や BGM preset を聞き比べたいとき

特に `webg/samples/sound` は、「音を鳴らす」「比較聴取する」「その場で調整する」を 1 画面で繰り返し確認できるサンプルとして読むと分かりやすくなります。

## 5. 注意点

サウンドを使うときに先に意識しておきたい注意点は、次のとおりです。

1. `resume()` 前は `AudioContext` が作成されていないので、まず利用者の操作をきっかけにして音声機能を有効化する必要があります
2. SE と BGM は別バスなので、音量や FX 設定もバスごとに変える前提です
3. `setSeReverb()` / `setBgmReverb()` は量、`setSeReverbImpulse()` / `setBgmReverbImpulse()` は IR の性格変更です
4. BGM は単発再生ではなく `scheduleBgm()` の先読みで進行するため、BPM やメロディを変えると流れ全体が変わります
5. `webg/samples/sound` は `WebgApp` を使わず、音の UI を直接組んでいるサンプルです。他のサンプルと入口がやや異なる点に注意が必要です

## 6. 標準フロー

サウンドの使い方は、高レベル経路と低レベル経路の 2 つに分けて考えると分かりやすくなります。

### 6.1 高レベル経路

通常の利用におけるアプリやサンプルでは、`GameAudioSynth` を直接使うのが適切です。

```js
import GameAudioSynth from "./webg/GameAudioSynth.js";

const synth = new GameAudioSynth();

button.addEventListener("click", async () => {
  await synth.resume();
  synth.setMasterVolume(0.25);
  synth.playSe("poyoon");
});
```

この形では、`GameAudioSynth` が内部で `AudioSynth` の基盤を利用し、プリセット名だけで SE を呼び出せます。最初の 1 音を出すまでの流れとしては、これが最も素直です。

### 6.2 低レベル経路

音色や FX の仕組みそのものを操作したいときは、`AudioSynth` を直接使います。

```js
import AudioSynth from "./webg/AudioSynth.js";

const synth = new AudioSynth();
await synth.resume();

synth.setSeDelay(0.14, 0.30, 0.24);
synth.setSeReverb(0.32, 0.55);
synth.playTone(440, 0.12, {
  type: "triangle",
  profile: "soft",
  gain: 0.12
});
```

この経路では、音色、エンベロープ、ディレイ、リバーブを細かく追えます。仕組みを理解したいときや、新しい上位のシンセサイザーを作成したいときはこちらが適しています。

## 7. 使い方のコード例

### 7.1 最初の SE を鳴らす

まずは `resume()` と `playSe()` の最小例です。

```js
const synth = new GameAudioSynth();

await synth.resume();
synth.playSe("poyoon");
```

ここで重要なのは、`resume()` が利用者の操作の後に呼ばれることです。コンストラクタだけでは、まだ音は出力されません。

### 7.2 SE と BGM の音量を分ける

SE と BGM のバランスは、最初にバスで分けておくほうが扱いやすくなります。

```js
synth.setMasterVolume(0.25);
synth.setSeVolume(0.90);
synth.setBgmVolume(0.75);
```

マスターは全体の基準、SE / BGM の音量は相対バランスという役割で見ると整理しやすくなります。

### 7.3 ディレイとリバーブの量を調整する

ディレイとリバーブの「量」を変える API です。

```js
synth.setSeDelay(0.11, 0.26, 0.22);
synth.setBgmDelay(0.18, 0.22, 0.18);

synth.setSeReverb(0.32, 0.55);
synth.setBgmReverb(0.28, 0.48);
```

`setSeDelay()` / `setBgmDelay()` はディレイ時間、フィードバック、ウェット量を変えます。`setSeReverb()` / `setBgmReverb()` はセンド / リターンの量を変えます。ここは「空間の量」の調整です。

### 7.4 リバーブの性格を変える

同じリバーブ量でも、IR の性格を変えると聞こえ方は大きく変わります。

```js
synth.setSeReverbImpulse({
  kind: "room",
  durationSec: 2.2,
  decay: 1.6
});

synth.setBgmReverbImpulse({
  kind: "hall",
  durationSec: 4.0,
  decay: 1.9
});
```

ここで変わるのはセンド / リターンの量ではなく、ConvolverNode に入るインパルスレスポンスそのものです。UI 上で言えば、`Reverb Kind`、`Reverb Length`、`Reverb Decay` がこの層に対応します。

### 7.5 エンベロープを調整する

SE と BGM はエンベロープも別に扱えます。

```js
synth.setSeEnvelopePreset("soft", {
  attack: 0.005,
  decay: 0.05,
  sustain: 0.55,
  release: 0.08
});

synth.setBgmEnvelope({
  attack: 0.03,
  decay: 0.20,
  sustain: 0.60,
  release: 0.40
});
```

SE は単発音の輪郭が重要で、BGM は持続感が重要なので、別々に持っているほうが自然です。

### 7.6 メロディプリセットと BGM preset を切り替える

BGM を melody 名と sample 側の preset で扱う例です。

```js
import { applyGameBgmPreset } from "./game_support/GameAudioPresets.js";

synth.setMelody("minor_drive");
synth.startBgm();

applyGameBgmPreset(synth, "boss");
```

`setMelody()` はメロディプリセットを直接切り替える API です。場面向けのまとまりを使いたい場合は、`game_support/GameAudioPresets.js` のような helper で melody、BPM、root、BGM envelope をまとめて適用すると、core の責務を増やさずに済みます。

### 7.7 通知音を sample 側でまとめる

ゲーム側で「開始」「停止」「クリア」などの通知音をまとめたいときは、SE 名の対応表を sample 側に置きます。

```js
import { playGameEventSound } from "./game_support/GameAudioPresets.js";

playGameEventSound(synth, "clear");
playGameEventSound(synth, "fail");
```

ここでは「どの波形で何 Hz を鳴らすか」ではなく、「どのゲーム上の出来事を鳴らすか」で選べます。通知音の命名や割り当てはゲームごとに変わりやすいので、core ではなく sample / app 側に置くほうが追跡しやすくなります。

## 8. `AudioSynth` の役割

`AudioSynth` は、音の基盤を担当します。現在の実装では、主に次の責務があります。

1. `ensureContext()` と `resume()` による `AudioContext` の遅延初期化
2. マスター / SE バス / BGM バスの構築
3. SE / BGM それぞれのディレイ / リバーブチェーンの構築
4. `playTone()` と簡易 `playSe()` による単発音の発音
5. メロディを使った BGM の先読みスケジューリング
6. エンベローププリセットの保持
7. インパルスレスポンスの手続き生成

ここで重要なのは、IR 生成も別クラスではなく `AudioSynth.js` の中に入っていることです。現在の webg では、リバーブの特性もこのクラスの責務として捉えるほうが実装に合っています。

## 9. インパルスレスポンスの作り方

この節は、仕組みの説明として残しておくべき重要な部分です。

`AudioSynth.createImpulseResponse(ctx, durationSec, decay, options)` は、録音済みのインパルスファイルを読むのではなく、手続き的にステレオ `AudioBuffer` を生成します。流れは次のとおりです。

1. `getImpulseProfile(kind)` で `room` / `hall` / `plate` ごとのプロファイルを取る
2. 左右 2 チャネルの `AudioBuffer` を作成する
3. `writeImpulseTail()` で、時間が進むほどスムージングを強めた減衰ノイズ列を各チャンネルに書き込む
4. `addEarlyReflections()` で、数本の初期反射を左右別タイミングで加算する
5. `normalizeImpulse()` でピークをそろえる

この構成にしている理由は、外部アセットなしで `room` / `hall` / `plate` の性格差を比較できるようにするためです。

### 9.1 `getImpulseProfile(kind)`

`room`、`hall`、`plate` ごとに、`alphaStart`、`alphaEnd`、`reflectionBurstMs`、`earlyReflections` を返します。

* `room`
  短く近い反射を意識したプロファイルです
* `hall`
  広く長いテールを意識したプロファイルです
* `plate`
  明るく密な印象を意識したプロファイルです

ここで決めているのは、単なる長さではなく「反射の性格」です。

### 9.2 `writeImpulseTail()`

`writeImpulseTail()` は、時間が進むほどスムージングを強めたノイズ列を作ります。単純なホワイトノイズに `(1 - t)^decay` を掛けるだけではなく、`alphaStart` から `alphaEnd` へ変化するスムージングを入れることで、テール後半の高域が少しずつ落ちやすくなります。

この処理により、単純なノイズ減衰よりも、やや自然なリバーブテールに聞こえやすくなります。

### 9.3 `addEarlyReflections()`

`addEarlyReflections()` は、冒頭に数本の初期反射を加えます。左チャンネルと右チャンネルで時間差を持たせるので、単なるモノラル的な残響よりも少し空間感が出やすくなります。

ここが `room`、`hall`、`plate` の聞こえ方の違いを作る大きな要素です。特に `room` は近く短く、`hall` は遅く長く、`plate` は早く密という傾向をここで作っています。

### 9.4 `normalizeImpulse()`

初期反射を加えるとピークが上がりやすいので、最後に `normalizeImpulse()` でチャンネル間のピークをそろえます。こうしておくと、`kind` を切り替えても極端な音量差になりにくくなります。

### 9.5 現在の既定値

現行実装の既定値は次のとおりです。

* SE リバーブインパルス

  * `kind = "hall"`
  * `durationSec = 3.2`
  * `decay = 1.8`
* BGM リバーブインパルス

  * `kind = "hall"`
  * `durationSec = 4.0`
  * `decay = 1.9`

SE より BGM を少し長くしているのは、フレーズの尾を聞き取りやすくするためです。

## 10. BGM の組み立て方

`AudioSynth` の BGM は、単に 1 曲のオーディオファイルを再生するのではなく、`scheduleBgm()` と `scheduleBgmStep()` で少し先の音を予約しながら進行します。

### 10.1 基本の流れ

1. `startBgm()` がスケジューリング処理を開始する
2. `scheduleBgm(lookAheadSec)` が少し先までのノートを予約する
3. `scheduleBgmStep(step, when)` がベースとリードを組み立てる
4. `stopBgm(fadeSec)` がフェードアウトして止める

この設計にしている理由は、BGM を「再生ファイル」ではなく「時系列で進むスコア」として扱うためです。

### 10.2 メロディプリセットの意味

メロディプリセットは、単なる音の並びではなく、小さなスコア断片として読むと分かりやすくなります。`scale`、`bassPattern`、`leadDegrees` または `leadGate`、`leadHoldSteps`、`leadDur`、`bassDur`、`leadType`、`bassType`、`leadGain`、`bassGain` などがまとまっています。

特に `leadDegrees` を長く取り、`bassPattern` を 8 ステップ以上にすると、同じ 1 小節の反復に聞こえにくくなります。現在の `GameAudioSynth` は、この方針に沿って `minor_drive`、`bright_arp`、`hero_run`、`night_drive` などを定義しています。

## 11. `GameAudioSynth` の役割

`GameAudioSynth` は `AudioSynth` の上位層として、ゲーム向けのプリセットを格納しています。現在の実装では、主に次の責務があります。

1. `installMelodyPresets()`
   BGM メロディプリセットを登録します
2. `installSePresets()`
   SE カタログと実体を登録します
3. `playGameTone()`
   ゲーム向けの Tone ヘルパーです
4. `playSe()` / `playGameSe()`
   SE 名を厳密に解決して再生します
5. `getSoundEffectList()` / `getSoundEffectInfo()`
   sound sample などで catalog と profile 情報を取り出します

通常の利用では、まずこちらから入るのが適切です。

## 12. `webg/samples/sound` の見方

`webg/samples/sound` は、音設計と確認を同じ画面で繰り返すサンプルです。ここでは `WebgApp` を使わず、`GameAudioSynth` と `AudioSynth` の調整 UI を直接組み合わせています。

画面は大きく次の 3 つのブロックに分かれます。

1. `System`
   `Audio Start` と `Master Vol` を扱います
2. `Sound Effects`
   SE の選択、連続比較、SE エンベロープ、SE ディレイ / リバーブを扱います
3. `Background Music`
   BGM の開始 / 停止、BPM、メロディ、BGM エンベロープ、BGM ディレイ / リバーブを扱います

このサンプルを使うときは、まず `Audio Start`、次に `Play SE` や `Audition All`、最後に `BGM Start` という順で見ると追跡しやすくなります。

### 12.1 `tail_probe` の役割

`tail_probe` は、リバーブとエンベロープの差を聞き分けやすくする確認用 SE です。冒頭の短い輪郭音でアタックを、その後の少し長い胴鳴りでディケイ / サステイン / リリースとリバーブテールを追跡しやすくしています。違いが分かりにくいときは、まずこれを選ぶと分かりやすくなります。

### 12.2 UI を簡潔に保つ理由

`webg/samples/sound` は 3D 表示や WebGPU diagnostics を持たないため、現在は音調整に必要な UI だけへ絞っています。ここで重視しているのは、SE / BGM の素材選択、音量、ディレイ、リバーブ、エンベロープを短い操作で往復しながら聞き比べられることです。sample 側の独自 BGM preset helper を置かず、core に含まれる `GameAudioSynth` と `AudioSynth` の確認へ寄せている点も、この簡潔さに含まれます。

音作り sample では、設定値をレポート化するよりも、実際に鳴らして差を聞き比べる操作が中心になるため、画面構成もその目的へ合わせて簡潔にしています。

## 13. 変更時の注意点

サウンド周りを変更するときは、1 か所だけを見るとずれが生じやすいので、少なくとも次の組を一緒に確認してください。

1. バス構成を変えるなら `ensureContext()` と `buildFxChain()`
2. リバーブの特性を変えるなら `getImpulseProfile()`、`writeImpulseTail()`、`addEarlyReflections()`
3. BGM の流れを変えるなら `startBgm()`、`scheduleBgm()`、`scheduleBgmStep()`
4. sample / app 側の preset helper を増やすなら `GameAudioSynth`、`game_support/GameAudioPresets.js`、その helper を使う sample / app の UI と文書
5. sound sample の UI や説明を変えるなら `webg/samples/sound/main.js`、`webg/samples/sound/sound.txt`、`webg/samples/index.html`

特に、インパルスレスポンスの仕組みを変える場合は、「量を変える API」と「特性を変える API」を混ぜないほうが安全です。
