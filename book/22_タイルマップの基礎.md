# タイルマップの基礎

TileMap は、単に床タイルを並べて表示するためのコンポーネントではありません。各セル（cell）が持つ高さ、地形（terrain）、移動可否、選択、パス探索、表示範囲の追従といった複数の役割を、同一のデータセットで管理するための基盤です。本章では、TileMap を盤面における「信頼できる唯一の情報源（source of truth）」として捉え、`tiles` や `generator`、`displayArea`、`terrainMaterials` をどのように定義すべきかを整理します。

まず理解しておくべき点は、TileMap が単なる「見た目の床」ではなく、盤面の基準となる唯一のデータであるということです。`height`（高さ）や `terrain`（地形）は描画用の装飾ではなく、`canMove()` による移動判定、`pickCell()` によるセルの選択、`findPath()` による経路探索、`followCell()` によるカメラ追従など、すべての機能が同じセルデータを参照します。したがって、TileMap を利用する際は「先にセルデータを確定させ、後からノードやカメラをそれに合わせる」という設計順序で考えることが最も安全です。

また、`displayArea` は盤面そのものではなく、現在ユーザーに見せるための「窓」であるという点に注意してください。`width` / `height` と `tiles` / `generator` が盤面全体の定義であるのに対し、`displayArea` はその上に重ねる可視範囲を定義します。広大な地形でカメラを追従させたい場合は、この `displayArea` を `followCell()` で動かすことになります。なお、`canMove()` は判定結果のみを返し、実際の移動処理や演出は別のレイヤーで実装します。また、`edgeCutMask` はビルドプロセスの中で自動的に計算されるため、通常は参照するだけで十分です。

## TileMap の役割と構成

TileMap は、3D 空間上の高さを持つ盤面を一つのランタイムとして扱うためのヘルパーです。その役割は大きく分けて以下の 5 点に集約されます。

1. `tiles` または `generator` からフルグリッドのセルを組み立てる。
2. `terrainMaterials` を用いて、地形ごとの外見を制御する。
3. `pickCell()`、`canMove()`、`findPath()` により、盤面上の判定処理を一元管理する。
4. `displayArea` と `followCell()` によって、可視範囲のスクロールを制御する。
5. `placeNodeOnCell()` により、セル上のオブジェクトを配置するための基準座標を算出する。

重要なのは、TileMap がパス探索専用や描画専用の部品ではないということです。見た目、入力、判定のすべてを同一のセルデータに統合するための基盤として機能します。この設計思想は、後続の章で扱う `placeNodeOnCell()` や `followCell()`、`pickCell()` の動作前提となっており、第 24 章の `tile_sim` においても TileMap が盤面の source of truth として扱われています。本章で定義する盤面データが、その後のすべての挙動を決定づけることになります。

## Scene JSON による盤面定義

TileMap は `scene.tileMap` として定義します。基本となる項目は `width`、`height`、`tiles` または `generator`、そして必要に応じて `displayArea` を指定します。「どのような盤面を構築するか」を先に決定し、その後に可視範囲や外見の設定を追加していくことで、定義を整理しやすくなります。

### 手動定義によるマップ作成（tiles）

パズルゲームや固定ステージなど、設計者が意図的に配置を決めたい場合は `tiles` を使用します。

```js
const scene = {
  tileMap: {
    width: 4,
    height: 3,
    displayArea: {
      x: 0,
      y: 0,
      width: 4,
      height: 3
    },
    tiles: [
      { x: 0, y: 0, height: 0, terrain: "floor" },
      { x: 1, y: 0, height: 0, terrain: "floor" },
      { x: 2, y: 0, height: 1, terrain: "slope" },
      { x: 3, y: 0, height: 2, terrain: "peak" },

      { x: 0, y: 1, height: 0, terrain: "floor" },
      { x: 1, y: 1, height: 1, terrain: "slope" },
      { x: 2, y: 1, height: 1, terrain: "slope" },
      { x: 3, y: 1, height: 2, terrain: "peak" },

      { x: 0, y: 2, height: 0, terrain: "floor" },
      { x: 1, y: 2, height: 0, terrain: "floor" },
      { x: 2, y: 2, height: 1, terrain: "slope" },
      { x: 3, y: 2, height: 1, terrain: "wall" }
    ]
  }
};
```

`tiles` を使用する場合、`width` と `height` で指定した範囲の全セルを定義する必要があります。TileMap には「未定義のセルを自動的に floor 扱いにする」という仕様はありません。定義漏れや重複がある場合、`build()` 時にエラーが発生するため、全セルが正しく定義されているかを確認してください。

### 自動生成によるマップ作成（generator）

地形の検証や広大なマップの試作には `generator` が適しています。

```js
const scene = {
  tileMap: {
    width: 20,
    height: 15,
    displayArea: {
      x: 0,
      y: 0,
      width: 10,
      height: 8
    },
    generator: {
      type: "noise",
      seed: 42,
      heightMin: 0,
      heightMax: 5
    }
  }
};
```

`generator` を使用すると、`width * height` 分のセルが自動的に生成されます。`terrain` は生成された `height` に基づいて `floor` / `slope` / `peak` へ自動的に分類されます。

### 可視範囲の制御（displayArea）

`displayArea` は、盤面全体の中から「現在どこを表示させるか」を決定する窓の役割を果たします。`x` / `y` / `width` / `height` で指定され、内部的に `minCol` / `maxCol` / `minRow` / `maxRow` の矩形領域へと正規化されます。

描画およびセルの選択（pick）処理では、この窓の範囲に加えて、周囲に一定の余白（`displayAreaPadding`）を設けた範囲を候補として処理します。これにより、画面端でタイルが急に消える現象を防ぎ、操作性を向上させています。

また、タイルの描画方式として、全高をテクスチャ付きの柱として描くのではなく、上面のみを薄いキャップ（cap）として描画し、その下部をテクスチャのない土台（foundation）で埋める構造を採用しています。初期設定では `visibleHeightSteps = 1` となっており、最上面から 1 段分だけがテクスチャ付きのキャップとして描画されます。これにより、高いセルであっても側面の情報量が過剰にならず、視認性を維持できます。

```js
displayArea: {
  x: 3,
  y: 2,
  width: 8,
  height: 6
}
```

この例では、列 3〜10、行 2〜7 の範囲が表示されます。指定した矩形が盤面サイズを超えている場合は、TileMap 側で自動的に盤面サイズ内に制限されます。

## TileMap の詳細パラメータ

`scene.tileMap` では、盤面の寸法や高さ、描画設定などの初期値を定義できます。これらの値は、セルのワールド座標や見た目の厚み、表示窓の追従位置に直接影響します。

### 平面寸法に関する設定

`cellSize` は、グリッド上の 1 セルを何ユニット間隔で配置するかを決定します。`createCellSpec()` では `col * cellSize` および `row * cellSize` を基準にセルの境界（`minx` / `maxx` / `minz` / `maxz`）を算出するため、盤面全体の広さや中心位置の計算の基礎となります。

`cellGap` は、隣接するセル間に設ける隙間の幅です。`createCellSpec()` において、この値の半分（`cellGap * 0.5`）だけセルのボックスを内側に縮小させることで、セル同士が完全に密着せず、盤面としての区切りが明確になります。実質的なセルの幅は `tileSize = cellSize - cellGap` となり、キャップ形状や土台形状の寸法はこの値を基準に計算されます。

### 高さ座標に関する設定

`floorY` は、すべてのセルの底面となる共通の Y 座標です。この値を下げると盤面全体がワールド座標の下方へ移動し、結果として土台部分の高さが増加します。

`baseTopY` は、`height = 0` のセルの上面を配置する基準高さです。実際の上面座標（`topY`）は `topY = baseTopY + height * heightStepY` で算出されます。`floorY` を固定したまま `baseTopY` を上げると、すべてのセルの上面が一様に持ち上がり、柱全体の厚みが増します。

`heightStepY` は、論理的な高さ `height` が 1 増えるごとに、ワールド座標でどれだけ上面を上げるかを決定する値です。移動判定や `edgeCutMask` は論理的な `height` を参照しますが、描画上の高さやノードの配置座標は、この `heightStepY` を介して変換されます。

### 視覚的表現に関する設定

`visibleHeightSteps` は、描画上の上面側を何段分だけテクスチャ付きのキャップとして表示させるかを指定します。`resolveVisibleCapHeight()` では `heightStepY * visibleHeightSteps` をキャップの厚みとして計算し、残りの部分は `resolveFoundationHeight()` によってテクスチャのない土台として処理されます。これは盤面データ自体を変更するのではなく、あくまで「同じ盤面をどう見せるか」を調整するための設定です。

`surfaceShading` は、キャップ部分の法線処理を決定します。
- `flat`（初期値）: 各面が独立した頂点を持つため、エッジが強調された外見になります。
- `smooth`: キャップ上部を共有頂点で再構成し、法線を平均化します。これにより、土台の区切りは維持したまま、上面付近だけを滑らかに見せることができます。

`smooth` 設定時は、上端に向けてわずかなベベル（面取り）が追加され、中央の平坦な面を残しつつエッジを丸めるため、地形としての自然な質感を表現できます。

### 表示窓の追従設定

`displayFollowCol` および `displayFollowRow` は、`followCell()` を用いて表示窓をプレイヤーなどの対象に追従させる際、対象セルを表示矩形のどの位置に配置するかを決定する初期値です。

例えば `displayFollowCol` が 0 であれば対象セルは左端に、2 であればより中央寄りに配置されます。見下ろし視点のゲームでは、プレイヤーを完全な中央に置くよりも少し手前寄りに配置したほうが、進行方向の先読みがしやすくなるため、これらの値で調整を行います。

`displayAreaPadding` は、`displayArea` の外側に設ける緩衝帯のセル数です。`getDisplayAreaCells()` ではこのパディング分だけ範囲を広げてセルを収集するため、画面端でのタイルの消失や、選択（pick）判定の不安定さを解消できます。

## 地形素材（terrainMaterials）と外見の設定

### 地形ごとのマテリアル定義

地形ごとの外見を変更するには、`scene.tileMap.terrainMaterials` を定義します。これは `TileMap.build()` 時に参照されるため、Scene JSON の構築段階で設定しておくことが推奨されます。

```js
scene.tileMap.terrainMaterials = {
  default: {
    materialId: "norm-phong",
    color: [0.70, 0.74, 0.84, 1.0],
    texture: groundTexture,
    normal_texture: groundNormalTexture,
    use_texture: 1,
    use_normal_map: 1
  },
  floor: {
    color: [0.30, 0.54, 0.96, 1.0]
  },
  slope: {
    color: [0.36, 0.70, 0.42, 1.0]
  },
  peak: {
    color: [0.92, 0.76, 0.42, 1.0]
  },
  wall: {
    color: [0.52, 0.30, 0.30, 1.0]
  }
};
```

`default` の設定をベースとし、各地形（`floor` や `peak` など）で個別の設定がある場合はそれが上書きされます。ノーマルマップを適用する場合は、`SmoothShader` の normal map 対応経路を有効にしたマテリアル設定を選択してください。見た目の制御は `terrainMaterials` が担い、盤面ロジックは `height` と `terrain` というデータで管理されるため、外見とロジックを明確に分離して管理できます。

### TileMap 特有の UV マッピング

TileMap の UV 展開は、通常の `Shape` による自動 UV とは異なる仕組みを採用しています。

キャップの上面は「セル外形を上から見た投影座標」を使用します。外形の左端を `U=0`、右端を `U=1`、上端を `V=0`、下端を `V=1` とします。例えば、左右が 0.2 ずつ削られたセルであれば、上面の U 範囲は `0.2 .. 0.8` となります。これを `0 .. 1` に再正規化せず、そのまま投影することで、広大な床に一枚のテクスチャを投影するような表現を実現しています。

一方、側面は面ごとに独立したシンプルな UV を使用します。これにより、削れた斜面や垂直面においても規則的な貼り方を維持しています。土台（foundation）の側面は、面を正面から見た際に左下が `(0,0)`、右上が `(1,1)` となる矩形 UV を基本とし、高さに応じて上端の `V` 座標を補正する手順で処理されます。

実装上の重要な注意点として、TileMap は `addVertexPosUV()` で明示的に UV を指定しています。このとき、`Shape` のテクスチャマッピングモードが初期値（`0`）のままだと、標準の球面シーム補正（sphere seam correction）が走り、指定した UV が意図せず改変されてしまいます。これを避けるため、TileMap ではシェイプ作成時にマッピングモードを `1` に切り替え、補正を無効化しています。

また、UV の座標系は `webg` 全体の規約および Blender の仕様に合わせ、「下側が 0」として定義されています。つまり、上面の `nw / ne / sw / se` を考える際、`v` は south 側が `0`、north 側が `1` となります。画像入力時に上下が反転して見える場合は、UV 座標を反転させるのではなく、画像読み込み時の `flipV` 設定で調整してください。

さらに、面の表裏については、`webg` のシェーダー初期値である `frontFace: "ccw"`（反時計回り）に従っています。`TileMap.js` の `addPrismSlice()` は各面を外側から見て反時計回りの順序で `addQuadFace()` へ渡し、それがそのまま `addPlane()` へ引き継がれます。頂点の定義順を確認することで、法線の向きを容易に追跡できる設計になっています。

## ビルドの標準フロー

新規アプリケーションでは、`WebgApp` と `loadScene()` を利用するフローが最も効率的です。盤面定義を scene 全体の一部として組み込むことで、カメラ、HUD、モデルなどの他の要素と整合性を保ったまま構築できます。

```js
import WebgApp from "./webg/WebgApp.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  camera: {
    target: [8.0, 0.0, 8.0],
    distance: 18.0,
    yaw: 0.0,
    pitch: -26.0,
    roll: 0.0
  }
});

await app.init({
  shader: "norm-phong"
});

const runtime = await app.loadScene(scene);
const tileMap = runtime.tileMap;
```

この手法の利点は、スクリーン、シェーダー、空間、カメラを個別に組み立てる手間が省けることです。`tileMap` 以外に `primitives` や `models` も同一の `scene` 定義に含められるため、プロジェクトの規模が拡大しても処理フローを変更せずに済みます。

一方、ユニットテストや小規模な動作確認などで TileMap 単体を構築したい場合は、`TileMap.fromScene()` を使用します。

```js
import TileMap from "./webg/TileMap.js";

const tileMap = TileMap.fromScene(scene, app.space, app.getGL());
tileMap.build();
```

通常は `loadScene()` を利用し、局所的なテストにのみ `TileMap.fromScene().build()` を利用するという使い分けを推奨します。

## まとめ

本章で最も重要なのは、TileMap を単なる「床を並べる部品」ではなく、「盤面の意味を一つに集約する基盤」として扱うことです。

- 盤面データ: `width`、`height`、`tiles` または `generator` で定義する。
- 表示制御: `displayArea` で「見せる窓」を管理する。
- 外見設定: `terrainMaterials` と各種初期値で調整する。
- 機能連携: ビルド後、`canMove()`、`pickCell()`、`findPath()`、`placeNodeOnCell()` を同一のセルデータに接続する。

特に、`tiles` を手動で記述する場合は全グリッドを完全に埋めること、`displayArea` は盤面そのものではなく可視範囲であること、`visibleHeightSteps` や `surfaceShading` は盤面データではなく描画上の設定であること、そして `edgeCutMask` はビルド後の参照用であること、この 4 点を明確に区別してください。

固定ステージであれば `tiles`、試作や広域地形であれば `generator` を選択し、通常は `app.loadScene(scene)` を通じて構築することで、一貫性のある実装フローを維持できます。次章では、この TileMap ランタイムを用いて、実際の移動処理、セルの選択、およびカメラの追従を実現する方法について解説します。
