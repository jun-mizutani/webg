# タイルマップの基礎

TileMap は単に床タイルを並べて表示する部品ではありません。各 cell は高さ、terrain、移動可否、選択、path 探索、表示範囲の追従といった複数の役割を同じデータから扱います。本章では、TileMap を盤面の source of truth として捉え、`tiles` や `generator`、`displayArea`、`terrainMaterials` をどのように定義するかを整理します。 

ここで最初に押さえたいのは、TileMap が「見た目の床」ではなく、盤面の基準となる唯一のデータだという点です。`height` と `terrain` は描画用の飾りではなく、`canMove()`、`pickCell()`、`findPath()`、`followCell()` も同じ cell データを見ます。つまり、TileMap を使うときは「先に cell データを決め、あとから node や camera を合わせる」と考えるほうが安全です。 

さらに重要なのは、`displayArea` が盤面そのものではなく、現在見せる窓だということです。`width` / `height` と `tiles` / `generator` が盤面そのもので、`displayArea` はその上に重ねる可視範囲です。広い地形で camera を追従させたいときは、この `displayArea` を `followCell()` で動かします。また `canMove()` は判定だけを返し、実際の移動や演出は別に行います。`edgeCutMask` も build の途中で自動計算されるため、通常は読むだけで十分です。

## TileMap は何をまとめる部品か

TileMap は、3D 空間上の高さ付き盤面を 1 つのランタイムとして扱うヘルパーです。役割は大きく 5 つあります。`tiles` または `generator` から full grid の cell を組み立てること、`terrainMaterials` を使って terrain ごとの見た目を分けること、`pickCell()`、`canMove()`、`findPath()` で盤面の判定をまとめること、`displayArea` と `followCell()` で可視範囲のスクロールを扱うこと、そして `placeNodeOnCell()` で「cell の上に object を置く」基準位置を返すことです。ここで大事なのは、TileMap が path 探索専用でも描画専用でもないことです。見た目、入力、判定を同じ cell データへ寄せるための基盤として使います。 

この考え方は、あとで TileMap ランタイムを使う章とも自然につながります。第23章で扱う `placeNodeOnCell()`、`followCell()`、`pickCell()` は、ここで定義した盤面データを前提に動きますし、第24章の `tile_sim` でも TileMap は盤面の source of truth として扱われています。つまり、本章は見た目の説明というより、盤面定義の基礎を固める章です。

## Scene JSON で盤面を定義する

TileMap は `scene.tileMap` として定義します。最初に覚えるべき項目は、`width`、`height`、`tiles` または `generator`、そして必要なら `displayArea` です。ここで重要なのは、「どの盤面を作るか」を先に決め、そのあとで可視範囲や見た目を足していくことです。盤面定義と表示窓を混ぜて考えないほうが整理しやすくなります。 

### `tiles` を使う

手作りの map を使う最小例は次のようになります。これは puzzle や固定ステージに向いています。 

```js
const scene = {
  tileMap: {
    width: 4,
    height: 3,
    displayArea: {
      x: 0,
      y: 0,
      width: 4,
      height: 3
    },
    tiles: [
      { x: 0, y: 0, height: 0, terrain: "floor" },
      { x: 1, y: 0, height: 0, terrain: "floor" },
      { x: 2, y: 0, height: 1, terrain: "slope" },
      { x: 3, y: 0, height: 2, terrain: "peak" },

      { x: 0, y: 1, height: 0, terrain: "floor" },
      { x: 1, y: 1, height: 1, terrain: "slope" },
      { x: 2, y: 1, height: 1, terrain: "slope" },
      { x: 3, y: 1, height: 2, terrain: "peak" },

      { x: 0, y: 2, height: 0, terrain: "floor" },
      { x: 1, y: 2, height: 0, terrain: "floor" },
      { x: 2, y: 2, height: 1, terrain: "slope" },
      { x: 3, y: 2, height: 1, terrain: "wall" }
    ]
  }
};
```

この例で重要なのは、`width=4`、`height=3` に対して 12 cell を全部書いていることです。TileMap は「書かれていない cell を自動で floor 扱いする」仕様ではありません。`tiles` を使う経路では、その範囲の全 cell を 1 回ずつ定義する必要があります。欠けている cell や重複した cell があると `build()` はエラーになります。手作りの map を使うときは、ここを最初に確認すると原因を追いやすくなります。

### `generator` を使う

自動生成を使う最小例は次のようになります。これは terrain の見え方確認や広いマップの試作に向いています。 

```js
const scene = {
  tileMap: {
    width: 20,
    height: 15,
    displayArea: {
      x: 0,
      y: 0,
      width: 10,
      height: 8
    },
    generator: {
      type: "noise",
      seed: 42,
      heightMin: 0,
      heightMax: 5
    }
  }
};
```

`generator` を使うと、`width * height` 分の cell は自動で埋まります。`terrain` は height から自動で `floor` / `slope` / `peak` へ分類されます。最初に形の確認だけをしたいときは、この経路のほうが早く始められます。固定盤面なら `tiles`、試作段階なら `generator` と分けて考えると、どちらを選ぶべきかが明確になります。 

### `displayArea` を持つ

`displayArea` は、盤面全体ではなく「現在見せる窓」です。`x` / `y` / `width` / `height` で書く形がいちばん分かりやすく、内部では `minCol` / `maxCol` / `minRow` / `maxRow` を持つ矩形へ正規化されます。つまり、盤面データそのものとは別に、「いまどこを見せるか」を切り分けるための値です。広い地形で camera を追従させる場合にこの分離が特に効きます。 

現在の TileMap 実装では、描画と pick はこの窓そのものではなく、その周囲に少し余白を足した範囲を候補にします。既定では `displayAreaPadding = 2` として扱い、画面端で tile が急に消えないようにしています。また、cell の見た目は full height の柱を最後まで texture 付きで描くのではなく、上面側だけを薄い cap として描きます。既定では `visibleHeightSteps = 1` なので、各 cell は「最上面から 1 step 分だけ」を texture 付き cap として描画し、その下側は必要なぶんだけ texture を使わない foundation で埋めます。移動判定や pick に使う full height box とは役割を分けています。 

```js
displayArea: {
  x: 3,
  y: 2,
  width: 8,
  height: 6
}
```

この例なら、列 3 から 10、行 2 から 7 までを見せます。盤面より大きい矩形を書いても、TileMap 側で盤面サイズへ収まるように制限されます。必要なら `displayAreaPadding` を scene 定義側で上書きして、描画余白の広さを調整できます。表示窓は build 後に `followCell()` や `setDisplayArea()` で動かしますが、その場合でも「盤面そのもの」と「いま見せる窓」は別に考えるのが大切です。

## TileMap の既定値

`scene.tileMap` には `width`、`height`、`tiles`、`generator` だけでなく、TileMap ランタイムの既定値も含められます。ここは単なる飾りではなく、cell のワールド座標、見た目の厚み、表示窓の追従位置に直接効きます。コードを読むときは、「盤面の平面寸法を決める値」「高さ座標を決める値」「描画上の cap の見せ方を決める値」「表示窓の follow を決める値」に分けて追うと理解しやすくなります。 

### 平面寸法を決める値

`cellSize` は 1 cell を grid 上で何 unit 間隔で並べるかを決めます。`createCellSpec()` では `col * cellSize`、`row * cellSize` を基準にして cell の `minx` / `maxx` / `minz` / `maxz` を組み立てるため、盤面全体の広さや cell 中心の位置計算にそのまま入ります。`build()` の最後に計算する `bounds.centerX`、`bounds.centerZ` もこの値を使います。つまり `cellSize` は、見た目の 1 マス幅であると同時に、ワールド上で隣の cell がどれだけ離れるかを決める値です。 

`cellGap` は隣り合う cell のあいだに残す隙間です。`createCellSpec()` では `cellGap * 0.5` を左右に配り、cell の box を内側へ少し縮めます。これにより、cell 同士が完全に接した一枚板ではなく、盤面として読みやすい区切りが出ます。`tileSize = cellSize - cellGap` は cell の実幅そのものなので、cap shape や foundation shape の平面寸法を読むときも、この値の感覚を基準にすると追いやすくなります。 

### 高さ座標を決める値

`floorY` は、すべての cell に共通する底面の Y 座標です。`createCellSpec()` では `box.miny` に入り、cell の中心 Y や foundation の高さ計算の基準にもなります。つまり `floorY` を下げると、盤面全体がワールドの下側へ沈み、`baseTopY` や `heightStepY` が同じでも柱の全高は大きくなります。TileMap の高さ表現を読むときは、「底がどこにあるか」を決める値として見ると分かりやすくなります。 

`baseTopY` は `height = 0` の cell でも、上面をどの高さへ置くかを決める基準値です。`createCellSpec()` では `topY = baseTopY + height * heightStepY` を使うため、`baseTopY` は「地表の 0 段目」をどこに置くかの基準になります。たとえば `floorY` を固定したまま `baseTopY` を上げると、すべての cell の上面が一様に持ち上がり、height 差はそのままでも見た目の柱は厚くなります。 

`heightStepY` は、cell の `height` が 1 増えるごとに、上面 `topY` を何 unit 上げるかを決める値です。`height` は論理的な段数で、実際のワールド高さは `heightStepY` を掛けて作ります。したがって `heightStepY` を大きくすると、同じ `height = 3` でもより背の高い柱に見えます。TileMap の移動判定や `edgeCutMask` は論理 height を見ますが、描画上の高さ差や node を置く位置はこの値を通してワールド座標へ変換されます。 

### 見せ方を決める値

`visibleHeightSteps` は、論理高さそのものではなく、描画上で上面側を何 step 分だけ texture 付きの cap として見せるかを決める値です。`resolveVisibleCapHeight()` では `heightStepY * visibleHeightSteps` を cap の厚みとして計算し、残りの高さは `resolveFoundationHeight()` により texture を使わない foundation 側へ回します。そのため `visibleHeightSteps = 1` のときは、各 cell の最上面から 1 段分だけが texture 付きの主役として見え、背の高い cell でも側面の情報量が増えすぎません。ここは「盤面データを変える値」ではなく、「同じ盤面をどう見せるかを変える値」です。 

`surfaceShading` は、cap 部分の法線を `flat` のまま保つか、`smooth` で共有するかを決める値です。既定値は `flat` で、top face と各 side face が独立した頂点を持ち、cell の edge が立った見え方になります。`smooth` を指定すると、cell 上部の cap だけを shared vertex で組み立て直し、top と斜面の法線が corner ごとに平均化されます。これにより、tile の gap や foundation は保ったまま、上面まわりだけを少し滑らかに見せられます。現在の smooth cap は、上端へ横方向比率 0.05、縦方向もほぼ同程度の bevel を入れて中央の平らな面を少し残すため、bevel 帯そのものも視認しやすくなっています。 

```js
tileMap: {
  width: 8,
  height: 8,
  visibleHeightSteps: 1,
  surfaceShading: "smooth",
  tiles
}
```

ここで大事なのは、`smooth` がマップ全体を 1 枚の地形 sheet に変えるわけではないことです。`cellGap` による隙間と、cell ごとの独立した foundation はそのまま残ります。つまり、「盤面の区切りは残しつつ、各 cell の上部だけを共有法線で試す」ための option です。まずは `flat` を比較基準にして通常の見え方を確認し、そのあとで `smooth` を試すと差が追いやすくなります。 

### 表示窓追従を決める値

`displayFollowCol` は、`followCell()` を使って表示窓を player などへ追従させるとき、対象 cell を表示矩形の左から何列目に置くかを決める既定値です。`followCell()` は `minCol = cell.col - followCol` を基準に表示矩形を組み立てるため、この値が 0 なら対象 cell は左端寄り、1 なら左から 2 列目、2 ならさらに中央寄りに置かれます。camera そのものの角度を決める値ではなく、「スクロール後に対象が窓のどこへ来るか」を決める値と考えると読みやすくなります。 

`displayFollowRow` はその行方向版です。`followCell()` では `minRow = cell.row - followRow` を使うため、対象 cell を表示矩形の上から何行目に置くかがこの値で決まります。見下ろし型のパズルや斜め見下ろし camera では、player を完全中央に置くより少し手前寄りに保つほうが先読みしやすい場合があるため、この値で表示窓の追従位置を調整します。 

`displayAreaPadding` は、実際の `displayArea` の外側へ何 cell 分余白を足して描画候補と pick 候補を集めるかを決める値です。`getDisplayAreaCells()` では `minCol - padding`、`maxCol + padding` のように範囲を少し広げて cell を集めるため、表示窓ぎりぎりで tile が急に消えたり、画面端で pick が不安定になったりするのを抑えられます。つまり `displayArea` が「今見せたい窓」だとすると、`displayAreaPadding` は「その窓の外側にどれだけ緩衝帯を持つか」を決める値です。 

## `terrainMaterials` と見た目の設定

### terrain ごとの material

terrain ごとの見た目を変えたいときは、`scene.tileMap.terrainMaterials` を構築前に定義します。ここは `TileMap.build()` の中で読まれるため、構築後に差し替えるより、Scene JSON を作る段階でそろえるほうが自然です。つまり、盤面データの準備段階で「terrain ごとにどう見せるか」まで含めて決めておく、という流れにすると整理しやすくなります。 

```js
scene.tileMap.terrainMaterials = {
  default: {
    materialId: "norm-phong",
    color: [0.70, 0.74, 0.84, 1.0],
    texture: groundTexture,
    normal_texture: groundNormalTexture,
    use_texture: 1,
    use_normal_map: 1
  },
  floor: {
    color: [0.30, 0.54, 0.96, 1.0]
  },
  slope: {
    color: [0.36, 0.70, 0.42, 1.0]
  },
  peak: {
    color: [0.92, 0.76, 0.42, 1.0]
  },
  wall: {
    color: [0.52, 0.30, 0.30, 1.0]
  }
};
```

`default` が土台になり、`floor` や `peak` の個別設定があれば上書きされます。normal map を効かせる場合は `NormPhong` のような対応シェーダーを選ぶ必要があります。`webg/samples/tile_sim` は terrain ごとの見え方をサンプル全体の中で確認しやすい参照先です。ここでのポイントは、見た目の差し替えは `terrainMaterials` が受け持ち、盤面データ自体は `height` と `terrain` に残ることです。見た目と盤面ロジックを同じ terrain 名でつないでいる、と考えると理解しやすくなります。

### TileMap 特有の UV と texture mapping

ここで注意したいのは、TileMap の surface UV は通常の `Shape` の自動 UV とは少し考え方が違うことです。TileMap では、`cap` の上面だけが「cell の外形を上から見た投影座標」を使い、外形の左端を `U=0`、右端を `U=1`、上端を `V=0`、下端を `V=1` として扱います。たとえば east / west を 0.2 ずつ削った cell なら、上面の U 範囲は `0.2 .. 0.8` になります。ここを上面だけ `0 .. 1` に再正規化しないのが重要です。つまり、TileMap の貼り方は「1 枚の床へ投影する」感覚に寄せてあり、通常の shape の単純な自動 UV と同一ではありません。 

一方で側面は、各面ごとに独立した単純な UV を使います。上面と同じ投影をそのまま共有するのではなく、面単位で `topuv` と `bottomuv` を結んだ分かりやすい貼り方を保つことで、削れた斜面と垂直面の両方が同じ規則で読めるようにしています。そのため、TileMap の見た目を調整したいときは、追加の UV 数値を増やすよりも、`edgeCutMask` 後の形状と texture 画像そのものの見え方を意識するほうが自然です。foundation の側面だけは別扱いで、面を正面から見たときに左下が `(0,0)`、右上が `(1,1)` になる矩形 UV を先に置き、そのあと高さ分だけ上端 `V` を補正する手順を使います。`surfaceShading: "smooth"` を使う場合も、外周 shoulder 側には従来の `topuv`、中央の平らな top 側にはそれを少し内側へ寄せた UV、bottom 側には `bottomuv` を持たせるため、貼り方の規則自体は大きく変わりません。 

実装上の注意として、TileMap は `addVertexPosUV()` で明示的に指定した UV を各頂点へ直接入れています。このとき `Shape` の texture mapping mode を既定の `0` のままにすると、球面 UV 用の seam 補正が `addTriangle()` 側で走り、手動で指定した `0..1` の UV まで改変されます。TileMap ではこれを避けるため、shape 作成時に texture mapping mode を `1` に切り替えて、旧来の sphere seam 補正が入らないようにしています。ここを忘れると、上面へ 1 枚画像を貼るだけのはずが、edge 側を引き延ばしたような見え方になります。 

もう 1 つ大事なのは、TileMap の論理的な UV 自体は `webg` 全体の規約に合わせて Blender と同じく「下が 0」であることです。つまり、上面の `nw / ne / sw / se` を考えるときも、`v` は south 側が `0`、north 側が `1` です。一方で WebGPU へ画像を取り込むと、画像の上下の見え方とはそのまま一致しない場合があります。そのため、`num256.png` のような確認用画像を使うときは、論理的な UV をひっくり返すのではなく、画像入力側で `flipV` の補正を入れて向きを合わせるのが正しい整理です。さらに、面の表裏も TileMap 実装を読むときの重要な前提です。`webg` のシェーダー既定値は `frontFace: "ccw"` なので、外側から見て反時計回りに並んだ頂点が表面になります。現在の `TileMap.js` では `addPrismSlice()` が各面を「外側から見て CCW」の順で `addQuadFace()` へ渡し、`addQuadFace()` はその順をそのまま `addPlane()` に渡します。面の点順を読むだけで法線の向きを追えるようになっているのは、この前提があるためです。 

## `build` の標準フロー

新規アプリでは、`WebgApp` と `loadScene()` を使う経路がいちばんそろえやすくなります。つまり、盤面定義ができたあとは、通常は scene 全体の一部として TileMap を build するのが自然です。TileMap だけを特殊扱いするより、scene 全体の構築フローへ乗せたほうが、camera、HUD、models とも接続しやすくなります。第11章の Scene JSON でも、`tileMap` はシーンの一部として build される構造で説明されています。

```js
import WebgApp from "./webg/WebgApp.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  camera: {
    target: [8.0, 0.0, 8.0],
    distance: 18.0,
    yaw: 0.0,
    pitch: -26.0,
    bank: 0.0
  }
});

await app.init({
  shader: "norm-phong"
});

const runtime = await app.loadScene(scene);
const tileMap = runtime.tileMap;
```

この形の利点は、Screen、Shader、Space、camera を別々に組み立てなくてよいことです。tileMap だけでなく primitives や models も同じ `scene` に含められるので、サンプルを大きくしていくときにも処理フローを変えずに済みます。TileMap を scene の一部として自然に扱える点が大きな利点です。

TileMap 単体を直接構築したいときは、`TileMap.fromScene()` を使います。

```js
import TileMap from "./webg/TileMap.js";

const tileMap = TileMap.fromScene(scene, app.space, app.getGL());
tileMap.build();
```

この経路は unittest や小さい確認用スクリプトに向いています。複数 entry を含む scene 全体を扱うなら、`app.loadScene(scene)` のほうが自然です。つまり、通常は `loadScene()`、最小確認や局所テストでは `TileMap.fromScene().build()` と分けて考えると、どちらを使うべきか判断しやすくなります。

## まとめ

この章で最も大事なのは、TileMap を「床を並べる見た目の部品」ではなく、「盤面の意味を 1 つに集約する基盤」として扱うことです。盤面データは `width`、`height`、`tiles` または `generator` で決め、表示窓は `displayArea` で別に持ち、見た目は `terrainMaterials` と既定値で整え、build 後に `canMove()`、`pickCell()`、`findPath()`、`placeNodeOnCell()` を同じ cell データへ接続します。ここがそろっていると、見た目だけが独立した TileMap になりにくくなります。 

特に重要なのは、`tiles` を手で書く場合は full grid を欠けなく埋めること、`displayArea` は盤面そのものではなく現在見せる窓であること、`visibleHeightSteps` や `surfaceShading` は盤面データではなく見せ方を変える値であること、`edgeCutMask` は通常は build 後に読むだけでよいこと、この 4 点です。固定ステージなら `tiles`、試作や広域地形なら `generator`、通常の build は `app.loadScene(scene)`、最小確認は `TileMap.fromScene(...).build()` という選び分けを基準にすると、実装の流れを保ちやすくなります。次章では、この TileMap ランタイムを実際にどう使って移動、選択、camera 追従へつなぐかを見ていきます。
