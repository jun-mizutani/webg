# タッチ入力の設計

キーボード中心で作った 3D アプリをモバイルへ持っていくと、最初に困るのは「何を UI として見せるか」ではなく、「その入力を既存の操作系へどう接続するか」です。ボタンを画面へ出すだけでは足りず、押している間だけ効く入力と、1 回だけ発火したい入力を分け、キーボードと Touch を同じ判定系へ流し込めなければ、サンプルごとに処理が二重化しやすくなります。本章では、`Touch` を単なるボタン表示ではなく入力の入口として捉え、`InputController` と組み合わせてキーボードと Touch を同じ経路へ統合する考え方を整理します。 

ここで重要なのは、`webg` の Touch 入力が「見た目のボタン群」と「入力経路の統合」を最初から一緒に考える設計になっていることです。`Touch` は DOM に仮想ボタンを出しますが、役割は見た目だけではありません。`onPress`、`onRelease`、`onAction` を通して、ホールド入力やアクション入力をアプリ側へ橋渡しするのが本体です。さらに新しいサンプルでは、`InputController.installTouchControls()` を使って、Touch 側もキーボード側も同じ `keyState` とアクション判定へ寄せる構成が標準になります。つまり本章の中心は、「仮想ボタンをどう描くか」よりも、「仮想ボタンをどう既存の入力系へ接続するか」です。 

最初に結論だけを言うと、通常のサンプルでは `InputController` を先に接続し、その上に `installTouchControls()` で Touch を追加するのがいちばん自然です。`Touch` を直接使うのは、内部テストや最小構成の確認に向いています。移動や回転のような「押している間だけ効く入力」はホールド入力、reset や confirm のような「1 回押したら 1 回だけ反応したい入力」はアクション入力へ分けます。さらに、Touch ボタンのキー名もキーボードと同じ lower-case へそろえておくと、サンプル側のロジックを 1 系統に保ちやすくなります。 

## タッチ入力をどう位置づけるか

Touch 入力を扱うとき、まず整理すべきなのは、`Touch` が「汎用 UI 部品」ではなく、「キー入力の代替 UI」だという点です。`UIPanel` のボタンが scene action、メニュー、会話選択肢のような操作 UI であるのに対し、`Touch` は `ArrowLeft`、`ArrowRight`、`A`、`D`、`R` のような既存キーの代わりに押す固定ボタン群です。どちらも DOM を使いますが、役割はかなり異なります。つまり、`Touch` は「シーンの上にボタンを重ねる一般 UI」ではなく、「入力系の一部」として読むほうが自然です。 

この役割が見えてくると、ホールド入力とアクション入力も分けて考えやすくなります。移動、回転、視点操作のように押している間だけ効いてほしいものはホールド入力です。reset、confirm、pause、モード切替のように 1 回の押下だけを拾いたいものはアクション入力です。Touch 入力を設計するときは、まずこの 2 種類を分け、その上で「どの入力を仮想ボタンとして画面へ出すか」を決めるほうが整理しやすくなります。 

さらに、Touch 側のキー名はキーボード側と同じ lower-case にそろえるのが基本です。キーボードは `event.key.toLowerCase()` を通し、Touch 側も `arrowleft`、`arrowright`、`a`、`d`、`r` のように同じ名前を使うと、アプリ側は入力源を意識せずに `has()` や `wasActionPressed()` だけを見れば済みます。Touch の見た目を考える前に、まず入力名をそろえることが大切です。 

## `Touch` と `UIPanel` の違い

`Touch` と `UIPanel` はどちらも DOM に要素を出しますが、目指しているものが違います。`Touch` は画面下に固定した仮想キー列で、キーボードの代わりに押されることを前提にしています。これに対して `UIPanel` は、ボタン、カード、説明文、日本語のヘルプをシーンの上に重ねるオーバーレイ基盤です。つまり `UIPanel` は「何かを読ませたり操作させたりする UI」、`Touch` は「既存キー入力を代替する UI」です。 

たとえば `Pause Scene` や `Reset Camera` のような scene action は `UIPanel` のボタンが自然です。一方で `← / → / A / D / R` のように、ゲームプレイ中に継続的に押すことを想定した入力は `Touch` が自然です。ここを混ぜると、操作 UI と入力代替 UI が同じ場所へ集まり、見た目の意味も処理の意味も曖昧になりやすくなります。第15章で整理した HUD / オーバーレイの役割分担を、この章では「入力代替」という観点から使い直す、と考えると分かりやすくなります。 

## 標準の入力経路

`webg` で Touch 入力を使うときの標準経路は、まず `InputController` にキーボードを接続し、そのあとで `installTouchControls()` を追加する形です。この順序にすると、キーボードも Touch も同じ `keyState` とアクション判定へ集約されるため、サンプル側の update ループが 1 系統で済みます。つまり、「Touch を特別扱いしない」ことがこの節の中心です。 

### キーボードを先に接続する

まず `InputController.attach()` でキーボードを受けます。キー名は `normalizeKey()` を通って lower-case にそろうので、Touch 側でも同じ名前を使う前提で考えると、あとが楽になります。特に 1 文字ではないキーは、最初に比較名をそろえておくと、Touch ボタンとキーボードを同じ条件で扱いやすくなります。

### `installTouchControls()` で Touch を追加する

そのあと `installTouchControls()` で Touch ボタンを組み込みます。ホールド入力用のボタンは `press()` / `release()` に入り、アクション入力用のボタンは `pulseAction()` として 1 回分の押下に変換されます。サンプル側はキーボードと Touch を区別せず、`input.has()` や `input.wasActionPressed()` だけを見れば済みます。 

```js
import InputController from "./webg/InputController.js";

const input = new InputController(document);

input.attach({
  onKeyDown: (key) => {
    if (key === "escape") pauseGame();
  }
});

input.installTouchControls({
  touchDeviceOnly: false,
  autoSpread: true,
  groups: [
    {
      id: "move",
      buttons: [
        { key: "arrowleft", label: "←", kind: "hold", ariaLabel: "move left" },
        { key: "arrowright", label: "→", kind: "hold", ariaLabel: "move right" }
      ]
    },
    {
      id: "turn",
      buttons: [
        { key: "a", label: "A", kind: "hold", ariaLabel: "turn left" },
        { key: "d", label: "D", kind: "hold", ariaLabel: "turn right" }
      ]
    },
    {
      id: "action",
      buttons: [
        { key: "r", label: "R", kind: "action", ariaLabel: "reset" }
      ]
    }
  ],
  onAction: ({ key }) => {
    if (key === "r") resetPlayer();
  }
});

function update() {
  if (input.has("arrowleft")) moveLeft();
  if (input.has("arrowright")) moveRight();
  if (input.has("a")) rotateLeft();
  if (input.has("d")) rotateRight();
}
```

この形の利点は、処理の分岐が「キーボードか Touch か」ではなく、「どのキー状態か」だけになることです。新しいサンプルでは、まずこの経路を標準として考えるのが自然です。 

### アクションマップを使う場合

`InputController.registerActionMap()` を使っているサンプルでは、Touch ボタンもアクション名へ寄せるとさらに整理しやすくなります。`installTouchControls()` のアクションボタンは内部で `pulseAction()` を呼ぶため、`wasActionPressed("pause")` や `getAction("jump")` と自然につながります。これは `webg/unittest/input_controller` や `webg/samples/breakout` のように、キーボードと Touch を同じゲームプレイ操作へ束ねたいサンプルで特に有効です。 

### Canvas ジェスチャーでカメラを動かす

ここまでの説明は主に仮想ボタン入力ですが、モバイルでシーンを直接見回したい場合は、Touch ボタンだけでは足りません。カメラの回転、平行移動、ズームのような連続入力は、画面下の固定ボタンではなく `canvas` 上のポインターイベントで扱うほうが自然です。つまり、仮想ボタンと Canvas ジェスチャーは役割を分けるべきです。 

`tile_sim` では、この方針でモバイル向けカメラジェスチャーを組み込んでいます。マウスではドラッグで orbit、ホイールで zoom、クリックで tile inspection を行い、Touch では 1 本指ドラッグで orbit、2 本指ドラッグで camera target の平行移動、ピンチで zoom、タップで tile inspection を行います。仮想ボタンは引き続き移動や `GO` のような固定入力に使い、ジェスチャーは canvas 側で直接受けます。 

実装の考え方としては、まず `canvas.style.touchAction = "none"` を設定し、`pointerdown / pointermove / pointerup / pointercancel` で Touch ポインターを追跡します。1 本指と 2 本指を分けて扱い、1 本指では `dx / dy` を orbit の yaw / pitch へ反映し、2 本指では中心移動を pan、指間距離の変化を zoom に使います。タップとドラッグを分けたいときは、移動量の threshold を設け、threshold を超えなかった `pointerup` だけを選択処理へ回します。こうしておくと、移動系の Touch ボタンを画面下に置いたままでも、シーンの見回しは canvas 上を直接触って行えます。 

## `Touch` を直接使う場合

通常のサンプルでは `InputController` を通すほうが自然ですが、`Touch` を直接使う経路もあります。これは、内部テストや最小構成の確認に向いています。`webg/unittest/touch` はこの形を採っており、Touch 自体の挙動やレイアウト密度を見やすくしています。つまり「ホールド入力とアクション入力がどう通知されるか」を最短で確かめたいときに使う経路です。 

```js
import Touch from "./webg/Touch.js";

const keyState = new Set();
const press = (key) => keyState.add(String(key).toLowerCase());
const release = (key) => keyState.delete(String(key).toLowerCase());

const touch = new Touch(document, {
  touchDeviceOnly: false
});

touch.create({
  autoSpread: true,
  groups: [
    {
      id: "move",
      buttons: [
        { key: "arrowleft", label: "←", kind: "hold" },
        { key: "arrowright", label: "→", kind: "hold" }
      ]
    },
    {
      id: "action",
      buttons: [
        { key: "r", label: "R", kind: "action" }
      ]
    }
  ],
  onPress: ({ key }) => press(key),
  onRelease: ({ key }) => release(key),
  onAction: ({ key }) => {
    if (key === "r") resetPlayer();
  }
});
```

この形では、Touch の通知をそのまま追えます。その代わり、キーボードとの統合やアクションマップの管理は自分で持つ必要があります。したがって、新規サンプルの標準経路としてはやはり `InputController` を通すほうが安全です。アクションだけのボタンを簡単に並べたい場合は `createActionButtons()` も使えますが、ホールド入力を含むゲームプレイ操作では通常の `create()` のほうが自然です。 

## レイアウトと表示位置

`Touch` の root は、既定では画面下固定です。`left: 0`、`right: 0`、`bottom: 0` で配置され、safe-area-inset も考慮します。つまり、scene の上にカードを重ねる `UIPanel` と違い、`Touch` は画面端の固定 UI として読むべきです。ここを最初に受け入れると、ボタンサイズや配置モードの意味が見えやすくなります。 

ボタン数に応じてサイズも少しずつ変わります。5 個以下では `52px`、6 個から 8 個では `48px`、9 個以上では `44px` のコンパクト表示になります。さらに `applyLayoutMode()` は横幅とボタン数を見て、単一行、複数行、展開配置を切り替えます。`autoSpread: true` のときは、左右へ開いた配置も選ばれます。`webg/unittest/touch` は `?touchButtons=<数>` でボタン数を増やし、この挙動を確認するためのテストです。 

デバッグ用途として、root には `data-touch-count`、`data-touch-density`、`data-touch-layout`、`data-touch-spread` が付きます。`webg/unittest/touch` ではこれらの値も HUD に出し、現在どのレイアウトが選ばれているかを見られるようにしています。つまり、Touch のレイアウトは CSS の見た目だけでなく、内部状態としても追跡しやすくなっています。 

## よく使う option

`Touch` で特によく使う option は、それぞれ「どの環境でどう見せたいか」を決めるものです。`touchDeviceOnly` は `true` が既定で、coarse pointer 端末だけで表示したいときに向いています。デスクトップで常時確認したい場合は `false` にします。`force` はポインター判定に関わらず coarse pointer 扱いにしたいときの確認用です。`className` は root に独自クラスを足し、サンプルごとの CSS 調整をしやすくします。`onAnyPress` は、どのボタンでも押された瞬間を 1 か所で拾いたいときに使います。 

これらは見た目そのものを作る option というより、「どの端末でどう確認するか」「どのタイミングを拾いたいか」を決める option だと理解すると整理しやすくなります。Touch 入力の設計では、ボタン形状よりも「表示条件」と「入力経路」のほうが重要です。 

## 参照すべき sample と unittest

Touch の最小テストとしては `webg/unittest/touch` が最短です。ホールド入力とアクション入力の違い、`?touchButtons=<数>` による密度変化、レイアウト属性を確認するにはここが向いています。キーボードと Touch を `InputController` に統合する標準経路を確認したいなら `webg/unittest/input_controller` が自然です。アクションマップや pulse action まで含めて見られます。 

ゲーム内で Touch ボタンをゲームプレイ操作に使う代表例は `webg/samples/breakout` です。左右移動、開始、pause、restart をキーボードと同じ経路へ流しています。さらに、移動系の仮想ボタンと Canvas ジェスチャーを併用する例としては `webg/samples/tile_sim` がもっとも分かりやすくなります。TileMap ランタイム、`GO` / preset / reset の Touch ボタン、ドラッグ / ピンチによるカメラ操作を一緒に読めるため、本章の考え方がかなり具体化されます。 

## 使い分けとよくあるミス

キーボードと Touch を同じロジックで扱いたいなら、`InputController` を使います。Touch ボタンそのものの最小挙動を確認したいなら、`Touch` を直接使います。scene 上の操作パネルやメニューを組みたいなら `UIPanel`、会話やチュートリアルは `DialogueOverlay`、Touch ボタンの意味を画面上に補足したいなら `Message` や `WebgApp` の guide / status helpers を併用します。つまり、入力代替 UI と情報表示 UI を混ぜないことが、この章全体の結論です。 

よくあるミスも整理しておきます。`event.key` をそのまま比較してしまう場合は、`toLowerCase()` を通してから比較します。ホールド入力とアクション入力を同じ扱いにしてしまう場合は、移動はホールド、reset / confirm はアクションへ分けます。Touch と `UIPanel` を同じ用途で使ってしまう場合は、入力代替は `Touch`、scene の操作 UI は `UIPanel` に分けます。デスクトップで確認したいのに `touchDeviceOnly: true` のままにしてしまう場合は、確認時だけ `false` にします。Touch ボタンの意味が利用者に伝わらない場合は、`Message` か `WebgApp.createHelpPanel()` のような説明系 UI を併用します。 

要するに、この章で最も大事なのは、「仮想ボタンを出すこと」と「その入力をアプリの入力系へ通すこと」を別々に考えないことです。`Touch` は見た目のためだけの部品ではなく、入力の入口でもあります。ここが見えると、モバイル向けサンプルの構成がかなり整理しやすくなります。
