# 21 パーティクルと軽量エフェクト

最終更新：2026-04-08 JST

ポストプロセスが画面全体の見え方を整える工程だとすると、パーティクルは scene の中にある局所的な演出を足す工程です。火花、煙、拾得エフェクト、破片のような短命の演出は、モデルを毎回作り込むよりも billboard ベースの軽量エフェクトで表現したほうが扱いやすい場面が多くなります。`webg` では、この役割を `ParticleEmitter` が担います。

この章では、`webg/ParticleEmitter.js` と `webg/WebgApp.js` を土台に、パーティクルをどのように scene へ組み込み、どの値を明示して emit するかを整理します。まず背景と役割を確認し、そのあと `WebgApp.createParticleEmitter()` を使う最小例、preset 切り替え、独自ループで描く場合の考え方を current 実装に沿って見ます。

## 21.1 この章で押さえること

### 21.1.1 `ParticleEmitter` は billboard 群を時間で更新する部品

`ParticleEmitter` は 1 個の mesh を回すクラスではありません。多数の billboard particle を生成し、life、velocity、gravity、drag を毎フレーム更新しながら描画するヘルパーです。短命の演出をまとめて扱うのが役割です。

### 21.1.2 `preset` は texture と既定値の束であり、`emit()` の省略機能ではない

current 実装の `emit()` は、`position`、`positionSpread`、`velocity`、`velocitySpread`、`gravity`、`color`、`colorSpread` などの配列を明示的に受け取ります。preset は texture の見た目と defaults の束を持ちますが、`emit()` 呼び出し時に曖昧な補完をする API ではありません。必要なベクトルや色は使う側がはっきり渡す前提です。

### 21.1.3 標準の入口は `WebgApp.createParticleEmitter()`

`ParticleEmitter` を直接 `new` して `init(gpu)` することもできますが、通常は `WebgApp.createParticleEmitter()` を使うのが自然です。これで GPU 初期化と emitter の登録がまとめて行われます。

### 21.1.4 `WebgApp.start()` を使う場合、更新と描画は自動で流れる

`WebgApp` はフレームループの中で `updateParticleEmitters()` と `drawParticleEmitters()` を呼びます。そのため、`app.start()` を使う通常のサンプルでは、利用者が毎フレーム `emitter.update()` や `emitter.draw()` を個別に書かなくても動きます。

### 21.1.5 独自ループでは `draw(eyeNode, projectionMatrix)` を明示する

一方で、独自ループを組んで `WebgApp.start()` を使わないサンプルでは、`ParticleEmitter.draw(eyeNode, projectionMatrix)` を自分で呼ぶ必要があります。`samples/circular_breaker` の `particleEffects.js` は、この使い分けを読みやすくまとめた例です。

## 21.2 この章の流れ

最初に、パーティクルがなぜ軽量エフェクトとして便利なのかを整理します。次に `WebgApp.createParticleEmitter()` で emitter を作り、`emit()` に何を渡すかを current API で確認します。そのあとで preset の扱いと独自ループの場合を見て、最後に導入時の注意点をまとめます。

## 21.3 背景

ゲームやデモでは、当たり判定のヒット、アイテム取得、着地、破壊、煙、きらめきといった短時間の演出が頻繁に必要になります。こうした演出を毎回 mesh や animation で作ると、用途に対して重くなりがちです。そこで役立つのが、画面に正対する billboard を多数使い、短い寿命の中で色や大きさを変えながら見せるパーティクルです。

`webg` の `ParticleEmitter` は、この用途に合わせて設計されています。内部では particle slot を一定数持ち、`emit()` で空いている slot に spawn 情報を書き込み、`update()` で位置と寿命を進め、`draw()` で billboard へ流します。必要なら影用 billboard も同時に描けるため、地面に接する火花や煙も簡潔に表現できます。

## 21.4 `WebgApp` に emitter を登録する

通常のアプリでは、まず `WebgApp.createParticleEmitter()` を呼ぶところから始めます。このヘルパーは `ParticleEmitter` を生成し、GPU 初期化を行い、`WebgApp` の emitter 一覧へ登録します。

```js
import WebgApp from "./webg/WebgApp.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  clearColor: [0.08, 0.10, 0.15, 1.0]
});
await app.init();

const sparkEmitter = await app.createParticleEmitter({
  name: "sparkEmitter",
  maxParticles: 320,
  useShadow: true,
  groundY: -7.5,
  preset: "spark",
  seed: 31
});
```

この例で `maxParticles` は同時に生かせる particle 数、`useShadow` は影用 billboard を使うかどうか、`groundY` は影を置く高さ、`preset` は texture と既定値の束を選ぶ名前です。`seed` を固定すると、乱数を使う spread のばらつきが再現しやすくなります。

`WebgApp.start()` を使う通常の構成なら、ここで emitter を作っておけば、フレームループの中で `updateParticleEmitters()` と `drawParticleEmitters()` が自動で呼ばれます。利用者コードでは「いつ emit するか」に集中しやすくなります。

## 21.5 `emit()` では値を明示して出す

current 実装の `emit()` は、spawn に必要なベクトルや色を明示的に受け取ります。`ParticleEmitter` は内部で安易な補完を行わないため、どの方向へ飛ばし、どれくらい散らし、どの色で出すかを利用者コードがはっきり示せます。これは、演出の意図をサンプルや書籍コードから直接読めるようにするためにも重要です。

```js
const emitSparkBurst = (x, y, z) => {
  sparkEmitter.emit(24, {
    position: [x, y, z],
    positionSpread: [0.3, 0.3, 0.3],
    velocity: [0.0, 12.0, 0.0],
    velocitySpread: [8.0, 4.0, 8.0],
    gravity: [0.0, -18.0, 0.0],
    drag: 0.06,
    life: 0.9,
    lifeSpread: 0.2,
    size: 1.6,
    sizeSpread: 0.5,
    color: [1.0, 0.85, 0.40, 1.0],
    colorSpread: [0.0, 0.08, 0.10, 0.0],
    shadowAlpha: 0.55,
    shadowScale: 1.1,
    shadowY: -7.5
  });
};
```

ここで `positionSpread` は発生位置のばらつき、`velocitySpread` は飛び出し方向のばらつき、`gravity` は時間経過による加速度、`drag` は速度減衰、`life` は寿命秒数、`size` は billboard の大きさです。`shadowAlpha`、`shadowScale`、`shadowY` は影用 billboard の見え方を決めます。

この形で明示しておくと、火花は上へ強く飛ぶ、煙は速度を弱くして寿命を長めにする、拾得エフェクトは重力を弱めて上昇気味に見せる、といった違いをコード上ですぐ読み取れます。`unittest/particle_emitter` が「使い方が分かる test」になっているのは、この明示的な spawn 指定があるからです。

## 21.6 preset は見た目の方向性をまとめる

`ParticleEmitter` は `spark`、`smoke`、`debris`、`pickup` などの preset を持ちます。preset の役割は、particle texture の見た目や、演出の既定方向をまとめることです。利用中に切り替えたい場合は `setPreset()` を呼びます。

```js
sparkEmitter.setPreset("smoke", {
  texture: {
    width: 96,
    height: 96,
    style: "smoke",
    noiseScale: 5.8,
    noiseAmount: 0.68
  },
  defaults: {
    life: 1.4,
    lifeSpread: 0.3,
    size: 2.8,
    sizeSpread: 1.0
  }
});
```

`setPreset()` は必要に応じて procedural billboard texture も再生成します。ここでの defaults は「こういう演出に寄せる」という束であり、実際の `emit()` 呼び出しでは依然として明示的なパラメータを使う前提です。暗黙に不足分を補ってくれる API だと期待しないほうが、安全で読みやすいコードになります。

## 21.7 `WebgApp.start()` と独自ループの違い

通常のサンプルでは `WebgApp.start()` を使うため、利用者コードは emit の条件だけを書けば十分です。

```js
app.start({
  onUpdate: ({ deltaSec }) => {
    actor.update(deltaSec);
    if (actor.justHit) {
      emitSparkBurst(actor.hitX, actor.hitY, actor.hitZ);
    }
  }
});
```

この場合、particle の更新と描画は `WebgApp` 側のフレームループに統合されています。scene draw のあとに emitter draw が流れるため、一般的なサンプルではこの構成で足ります。

一方で、独自ループを組みたい場合は `draw()` を自分で呼びます。`samples/circular_breaker/particleEffects.js` は、この構成を小さなヘルパーにまとめています。

```js
const alive = sparkEmitter.draw(app.eye, app.projectionMatrix);
```

この書き方は、`WebgApp.start()` を使わずに独自ループを組むサンプルや、描画順を明示したいサンプルで向いています。反対に、通常の教材サンプルでは emitter の更新と描画を手でばらさず、`WebgApp.start()` へ任せたほうが流れを追いやすくなります。

## 21.8 軽量エフェクトとしての使いどころ

パーティクルが向いているのは、短時間で消える視覚効果です。具体的には、ヒット時の spark、着地の dust、壊れたときの debris、アイテム取得時の pickup glow などが代表です。これらは geometry よりも時間変化が主役であり、billboard のほうが軽くて調整しやすくなります。

逆に、長時間残る object、当たり判定を持つ object、視点から回り込んで形を見せたい object には向きません。その場合は particle ではなく `Shape` や model asset を使い、必要に応じて animation や material 側で演出するほうが自然です。

## 21.9 注意点

まず、`emit()` で渡す配列や数値を省略しないことが大切です。current 実装の `ParticleEmitter` は、不足したベクトルや色を自動補正する方針ではありません。`position`、`velocity`、`gravity`、`color` とそれぞれの spread を明示したほうが、意図が読みやすく、バグも見つけやすくなります。

次に、`maxParticles` を小さくしすぎると、長寿命の particle が残っている間に新しい burst を出せなくなります。逆に大きくしすぎると、不要に多くの slot を管理することになります。演出の同時発生数を見積もって決めるほうが扱いやすくなります。

影用 billboard を使う場合は、`groundY` や `shadowY` の高さが scene の床と合っているかも確認します。値がずれると、particle 本体は自然でも影だけが浮いて見えることがあります。`samples/circular_breaker` が `SHADOW_Y` を定数化しているのは、このずれを避けるためです。

`WebgApp.start()` を使う構成で、自分でも `emitter.draw()` を呼んでしまうと二重描画になります。通常のループに乗せるのか、独自ループで自分で draw するのかを最初に決めて、どちらか一方へ寄せるほうが安全です。

## 21.10 関連サンプル

最初に読むなら `unittest/particle_emitter` が最も分かりやすくなります。fake renderer による自動確認と、実画面での visual sample が同じファイルにまとまっているため、`preset -> emit -> update -> draw -> clear` の流れを追いやすいからです。ゲームへの組み込み例としては `samples/circular_breaker` がよく、ヒット時の spark を gameplay code から切り離して扱う構成を確認できます。
