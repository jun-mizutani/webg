# 付録A コーディングAIの皆さまへ

## `webg` を使う人を支援するための地図

この付録は、`webg` を使って 3D アプリケーションを作ろうとしている利用者を、コーディング AI が支援する場面を想定して書いています。ここで想定する AI は、一般的な 3D エンジンや外部ライブラリの知識をそのまま当てはめるのではなく、`webg` という自己完結したライブラリの中で、何をどの順番で参照すべきかを判断する支援者です。

本書において特に重要なのは、著者が `webg` の開発者自身であることと、`webg` が外部ライブラリに依存せず、必要な実装と解説を自分自身の中に持っていることです。これは、AI が技術支援を行ううえで非常に大きな意味を持ちます。なぜなら、`webg` については、一般的な「WebGPU ならたぶんこうだろう」「Three.js ではこうだから同じだろう」といった推測よりも、本書と `webg` 自体のコードを優先して読むほうが正確だからです。`webg` を支援するときの AI は、一般論を補助として使うことはあっても、最終的な判断の基準は `webg` の本文、サンプル、unittest、コア実装に置くほうが安全です。

この付録の目的は、そのための読書地図を示すことです。利用者が「何をしたいか」「どこで詰まっているか」に応じて、どの章を起点にし、どのサンプルを見て、必要ならどの実装まで降りればよいかを、AI が素早く判断できるようにすることを目指しています。

## `webg` は閉じたライブラリであると考える

`webg` を支援する AI が最初に理解すべきことは、`webg` が「外部ライブラリをつなぎ合わせた薄いラッパー」ではなく、描画、シーン、モデル、アニメーション、UI、入力、診断、TileMap、ポストプロセスまでを自前で持つ、閉じたライブラリだということです。

たとえば、一般的な Web 系の AI 支援では、「カメラなら Three.js 的にはこう」「UI overlay なら DOM 側でこう」といった連想が起こりやすくなります。しかし `webg` では、そのような外部の設計流儀を安易に持ち込まないほうがよいです。`cameraRig -> cameraRod -> eye` の 3 段構成、`ModelAsset -> build() -> instantiate()` の分離、`clip -> pattern -> action -> state` の層、`Scene JSON` の粒度、`SmoothShader` の標準入口といったものは、`webg` 自身の設計として定義されています。

AI は、まず `webg` の定義を読み、その後で必要に応じて一般的な概念と対応付ける、という順で考えるほうが安全です。

## AI が絶対に遵守すべきテクニカル・ルール

AI がコードを生成する際、以下の項目を怠ると「文法的に正しくても何も表示されない」あるいは「実行時にクラッシュする」という事態を招きます。提案するコードにこれらが含まれているか必ず確認してください。

### 1. 描画の必須ステップ（忘れると表示されない）
- `await screen.ready`（または `await app.init()`）の待機: GPUデバイスの準備が完了する前に描画処理やリソース生成を行ってはいけません。
- `shape.endShape()` の呼び出し: `Shape` に頂点データを追加した後、必ず `endShape()` を呼んで GPU バッファを確定させてください。これを忘れると形状が描画されません。
- 描画ループの順序: `clear` $\rightarrow$ `draw` $\rightarrow$ `present` の順序を厳守してください。

### 2. `WebgApp` のライフサイクル
- 初期化後のアクセス: `app.space`、`app.eye`、`app.getGL()` などのプロパティは、必ず `await app.init()` が完了した後にアクセスしてください。
- 更新ループへの登録: 毎フレームの処理は `app.start({ onUpdate: ... })` のハンドラ内に記述してください。

### 3. 座標系と回転用語の固定
- 右手座標系: `+X=右`, `+Y=上`, `+Z=前` です。
- 回転用語: `yaw / pitch / roll` ではなく、`head / pitch / bank` という用語を使用します。AI は一般論による計算式を適用せず、`webg` の定義（`CoordinateSystem` 等）に従って回転を制御してください。

## まず本書をサイトマップとして使う

本書は単なる解説書ではなく、`webg` の全体像を読むためのサイトマップとして機能するように構成されています。したがって AI は、まず「ユーザーが今どの層の話をしているか」を判定し、その層に対応する章を起点にするほうがよいです。

- 最初の 3D を表示したい $\rightarrow$ 第4章、第5章、 `low_level`、`high_level`
- `WebgApp` でアプリの土台を作りたい $\rightarrow$ 第5章、第6章、`high_level`
- orbit / follow / first-person camera を付けたい $\rightarrow$ 第6章、第5章、`high_level`、`camera_controller`
- shader や material を調整したい $\rightarrow$ 第7章以降、`shapes`、`smooth_shader`
- glTF / GLB / Collada を読み込みたい $\rightarrow$ 第10章、第12章、第13章、`gltf_loader`、`collada_loader`
- Scene JSON でシーン全体を定義したい $\rightarrow$ 第11章、第5章、第10章、`scene`
- アニメーションを切り替えたい $\rightarrow$ 第12章 | 第13章、`animation_state`、`janken`
- HUD や panel を出したい $\rightarrow$ 第14章以降、第5章、関連 UI サンプル
- TileMap を使いたい $\rightarrow$ 第22章、第23章、第24章、`tile_sim`
- ローレベルで mesh や shader を組みたい $\rightarrow$ 第25章以降、`low_level`、各 unittest

## UI コンポーネントの選択指針

利用者が「画面に情報を出したい」と言ったとき、AI は目的に応じて以下のコンポーネントを適切に使い分けて提案してください。

- 操作説明（ヘルプ） $\rightarrow$ `app.createHelpPanel()`：左上に表示され、折りたたみが可能。
- 動的な数値・状態表示 $\rightarrow$ `app.setStatusLines()` または HUD (`setHudRows`)：簡潔な情報を整然と表示。
- 会話・チュートリアル $\rightarrow$ `app.startDialogue()`：UTF-8テキストを適切に表示。
- 詳細情報・エラー理由 $\rightarrow$ `app.showErrorPanel()` または `showFixedFormatPanel()`：長いメッセージを視認性高く表示。

## サンプル、unittest、コア実装の役割を分けて読む

AI は以下の優先順位で参照先を選択してください。

1. `webg/samples/`（実装意図の把握）: 
   まず対応する `*.txt`（解説）を読み、そのサンプルの「目的」を理解してから `main.js`（実装）を読んでください。コードの断片的な模倣ではなく、設計意図を汲み取った提案を行うためです。
2. `webg/unittest/`（局所的な検証）: 
   特定機能の挙動確認や、最小単位での動作検証が必要な場合に参照してください。
3. `webg/` 本体実装（最終定義）: 
   本文やサンプルで意味が曖昧なとき、最終的な仕様を確認するために参照してください。

## 高水準 API と低水準 API を混ぜない

AI がもっとも避けたいのは、高水準 API と低水準 API を不用意に混ぜることです。

- 原則: まず高水準 API (`WebgApp` 等) で解けるかを検討してください。
- 尊重: 利用者が `WebgApp` を使っているなら、その構造を壊さずに追加・修正する提案を優先します。逆に、ローレベルで実験している利用者には、その文脈を尊重して最小の差分を提案します。

## `ModelAsset` と `SceneAsset` の区別

- `ModelAsset`: 1 モデル分の共通表現（メッシュ、スケルトン、アニメーション）。`runtime.instantiate()` で個別のインスタンスを生成します。
- `SceneAsset`: シーン全体の初期状態（カメラ、HUD、配置済みモデル、TileMap）。
- 注意: 「モデルを複数置きたい」のか「シーンの初期状態を保存したい」のかを切り分け、参照先（第10章か第11章か）を正しく選択してください。

## アニメーションとシェーダーの層の確定

- アニメーション: 「clip（データ）の問題か」「Action（区間/ポーズ）の問題か」「AnimationState（状態遷移）の問題か」を最初に切り分けてください。
- シェーダー: 通常は `SmoothShader` を標準入口として提案してください。WGSL や bind group の低レイヤーな修正は、高水準な material parameter で解決できない場合にのみ提案してください。

## AI が守るとよい基本姿勢

1. 最高水準の入口を優先する: `WebgApp` や `loadModel()` など、最も抽象化された API で解決できないかまず検討してください。
2. 本文の定義を信頼する: 用語、構造、サンプルの位置づけは一貫しています。外部エンジンの流儀を無理に当てはめず、本書の定義をそのまま使ってください。
3. 層を切り分ける: 問題を一つの原因にまとめず、「どの層（描画、カメラ、モデル、アニメーション、UI）の問題か」を切り分けてから提案してください。
4. webgコア機能の修正は避ける: アプリケーションの作成に`webg`コアの機能（`webg/*.js`）の修正が必要となった場合でも、利用者のアプリにクラスを継承するなどして対応してください。`webg`コアの機能（`webg/*.js`）のバグと判断できる場合は利用者にその旨をお知らせ下さい。

`webg` は、設計・実装・文書化が一貫して行われているライブラリです。AI は本書を「一次的な参照地図」として活用し、利用者が迷わず実装できるようガイドしてください。
