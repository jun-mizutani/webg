# 28 スキニング

最終更新：2026-04-10 JST

この章は、Blender などで作成した armature 付きの glb / glTF を `loadModel()` で読み込む高レベルな使い方の背景にある仕組みを、「スキニングとは何をしているのか」という観点から低水準の教材コードで理解するための章です。対象は `webg/SmoothShader.js`、`webg/Skeleton.js`、`webg/Shape.js`、`samples/skinning`、`unittest/skinning_basic` と、それらを簡略化した `book/examples/28_01.html`、`book/examples/28_02.html` です。

## 28.1 この章で押さえること

### 28.1.1 普段の利用では glb import が入口だが、仕組みは低水準で説明できる

通常のアプリ開発では、骨とウェイトを自分で手入力するより、Blender などで作成した glb を読み込むほうが自然です。ただし、見た目が崩れたときに何が壊れているかを判断するには、低水準の仕組みを知っておく必要があります。

### 28.1.2 スキニングの中心は「1 頂点を複数の bone 座標系で動かして混ぜる」こと

1 つの頂点は、1 本の bone だけで動くとは限りません。境界付近の頂点は、複数の bone の変形結果を重み付きで混ぜた位置へ移動します。

### 28.1.3 `Skeleton` は bone 階層と matrix palette を作り、`SmoothShader` は GPU 側でブレンドする

CPU 側では `Skeleton.updateMatrixPalette()` が各 bone の変形行列を並べ、GPU 側では `SmoothShader` の頂点シェーダーが bone index と weight を使ってブレンドします。

### 28.1.4 低水準教材では「円柱を 2 本の bone で曲げる」と理解しやすい

いきなり複雑な glb を読むより、円柱の側面メッシュを自前で作り、下半分は bone0、上半分は bone1 に強く影響されるようにすると、スキニングの意味がかなり明確になります。

## 28.2 この章の流れ

最初に高レベル経路と低レベル経路の役割分担を整理し、そのあとで 1 頂点を 2 本の bone で変形する最小式を説明します。続いて、低水準で円柱を作り、`Skeleton` と `SmoothShader` へつなぎます。最後に、Blender から glb をインポートするときも本質的には同じことが起きていると確認します。

## 28.3 高レベル経路と低レベル経路の役割分担

普段のアプリ開発では、次のような高レベル経路を使うことが多くなります。

```js id="p7r9sd"
const runtime = await app.loadModel("./character.glb");
runtime.instantiate(app.space);
```

この経路では、joint 階層、inverse bind matrix、頂点ごとの weight、animation clip は、すでに glb 側に入っています。読者は `Action` や `AnimationState` を組み合わせて再生を進めれば十分です。

ただし、次のような場面では高レベル経路だけでは判断しにくくなります。

1. どの bone がどの頂点を引っ張っているか分からない
2. 曲がり目が不自然な原因が、weight なのか bone なのか分からない
3. `SmoothShader` の skinning 経路が内部で何をしているか把握したい
4. importer が出した skeleton 情報を疑う前に、最小原理を自力で確認したい

そのため、本章では「高レベル利用の代わりに低レベルだけを使う」のではなく、「高レベル利用の背景を確認するために、低レベルで最小構成を一度作ってみる」という位置づけで説明します。

## 28.4 スキニングを 1 頂点だけで考える

まずは、円柱全体や glb 全体ではなく、1 つの頂点だけを考えます。

レスト姿勢の頂点を `p_rest` とします。bone0 と bone1 それぞれについて、skinned mesh のローカル座標系で見た現在姿勢の bone 行列を `L0`、`L1`、レスト姿勢から作られた bone offset inverse を `M0^-1`、`M1^-1` とすると、各 bone で変形した頂点は次のように書けます。

```text id="0v8o0g"
p0_local = L0 * M0^-1 * p_rest
p1_local = L1 * M1^-1 * p_rest
```

ここで weight を `w0`、`w1` とし、`w0 + w1 = 1` になるようにしておくと、最終頂点は次の重み付き平均になります。

```text id="6m01z1"
p_final_local = p0_local * w0 + p1_local * w1
```

ここで重要なのは、この段階では scene 全体の world 座標ではなく、skinned mesh 自身のローカル座標系で考えていることです。objectNode 自体の移動や回転がある場合は、その共通変換があとからメッシュ全体へ一括で掛かります。

これは「2 種類の座標系で 1 つの頂点を変形移動して、それらの平均座標を最終的な頂点とする」という説明を、`webg` の実装に対応する形へ言い換えたものです。

`webg` では、この `L * M^-1` を bone ごとに並べた配列が matrix palette であり、`SmoothShader` はそれを頂点シェーダーでブレンドします。

### 28.4.1 CPU 側で 1 頂点を計算するとこうなる

次のコードは、`Skeleton.updateMatrixPalette()` のあとで `bone.worldMatrix` を current bone 行列として読み、1 頂点だけを CPU 側でブレンドする最小例です。`28_01` はこの helper をそのまま実行して marker を置く page になっています。

```js id="n7l2ri"
function buildSkinMatrix(bone) {
  const skin = bone.worldMatrix.clone();
  skin.mul(bone.getBofMatrix());
  return skin;
}

function transformVertexByBone(restPosition, bone) {
  const skin = buildSkinMatrix(bone);
  return skin.mulVector(restPosition);
}

function blendVertex(restPosition, influences) {
  const out = [0.0, 0.0, 0.0];
  for (let i = 0; i < influences.length; i++) {
    const moved = transformVertexByBone(restPosition, influences[i].bone);
    out[0] += moved[0] * influences[i].weight;
    out[1] += moved[1] * influences[i].weight;
    out[2] += moved[2] * influences[i].weight;
  }
  return out;
}
```

この helper 自体は描画に必須ではありません。実際の描画では GPU 側で同じことが行われます。ただし、教材としては「GPU がやっていることを CPU 側でも再現できる」と確認できるのが重要です。実装上は `Skeleton.updateMatrixPalette()` が `bone.worldMatrix` を skinned mesh ローカル座標系の current pose へ更新し、`SmoothShader` が同じ情報から matrix palette を使ってブレンドします。

実際の page では、次の順で呼ぶと本文の式とそのまま対応づけられます。

```js id="r4z0v5"
skeleton.updateMatrixPalette();

const rootOnlyLocal = transformVertexByBone(restPosition, rootBone);
const childOnlyLocal = transformVertexByBone(restPosition, childBone);
const blendedLocal = blendVertex(restPosition, [
  { bone: rootBone, weight: w0 },
  { bone: childBone, weight: w1 }
]);
```

この段階で得られるのは object local 座標です。画面上へ marker を置いて確認したいときは、最後に objectNode の world 行列を 1 回だけ掛けます。

```js id="myfyul"
const rootOnlyWorld = objectNode.getWorldMatrix().mulVector(rootOnlyLocal);
const childOnlyWorld = objectNode.getWorldMatrix().mulVector(childOnlyLocal);
const blendedWorld = objectNode.getWorldMatrix().mulVector(blendedLocal);
```

## 28.5 低水準で 2-bone 円柱を作る

仕組みが分かったら、次は頂点群へ広げます。題材は「下から上へ細かく分割した円柱側面」です。

1. `Shape` で円柱側面の頂点を並べる
2. `Skeleton` に root bone と child bone を作る
3. 各頂点に `addVertexWeight()` で 2 本分の weight を入れる
4. `shape.endShape()` で GPU 用バッファへ変換する
5. `SmoothShader` で `has_bone: 1` を有効にして描画する

次のコードは、その最小教材版です。

```js id="ydgqsh"
import Shape from "./webg/Shape.js";
import Skeleton from "./webg/Skeleton.js";

function createTwoBoneCylinder(gpu, options = {}) {
  const height = Number(options.height ?? 8.0);
  const radius = Number(options.radius ?? 0.45);
  const rings = Math.max(2, Math.floor(options.rings ?? 14));
  const segments = Math.max(3, Math.floor(options.segments ?? 24));

  const shape = new Shape(gpu);
  shape.setAutoCalcNormals(true);

  const skeleton = new Skeleton();
  shape.setSkeleton(skeleton);

  const rootBone = skeleton.addBone(null, "rootBone");
  const childBone = skeleton.addBone(rootBone, "childBone");

  rootBone.setRestPosition(0.0, 0.0, 0.0);
  childBone.setRestPosition(0.0, height * 0.5, 0.0);

  skeleton.bindRestPose();
  skeleton.setBoneOrder(["rootBone", "childBone"]);

  for (let ring = 0; ring <= rings; ring++) {
    const v = ring / rings;
    const y = height * v;
    for (let segment = 0; segment < segments; segment++) {
      const u = segment / segments;
      const angle = u * Math.PI * 2.0;
      const x = Math.cos(angle) * radius;
      const z = -Math.sin(angle) * radius;

      const vIndex = shape.addVertexUV(x, y, z, u, v) - 1;

      // 下ほど rootBone、上ほど childBone の influence を強くする
      shape.addVertexWeight(vIndex, 0, 1.0 - v);
      shape.addVertexWeight(vIndex, 1, v);
    }
  }

  for (let ring = 0; ring < rings; ring++) {
    const row0 = ring * segments;
    const row1 = (ring + 1) * segments;
    for (let segment = 0; segment < segments; segment++) {
      const next = (segment + 1) % segments;
      const a = row0 + segment;
      const b = row0 + next;
      const c = row1 + segment;
      const d = row1 + next;
      shape.addTriangle(a, c, b);
      shape.addTriangle(b, c, d);
    }
  }

  shape.endShape();
  return { shape, skeleton, rootBone, childBone };
}
```

### 28.5.1 このコードで重要な点

#### `Skeleton` は bone の親子関係だけでは終わらない

`setRestPosition()` でレスト姿勢を置き、`bindRestPose()` で bone offset inverse を再計算し、`setBoneOrder()` で shader が読む palette 順を確定します。ここがそろわないと、頂点へ weight を入れても意図どおりに曲がりません。

#### `addVertexWeight()` は「頂点がどの bone にどれだけ引っ張られるか」を入れる

この例では 2 本だけを使っていますが、`Shape.endShape()` は最終的に 4 本分までの bone index / weight を `vertexBuffer1` へ詰めます。`SmoothShader` の skinning 経路はその slot を前提にしています。

#### 円柱側面だけに絞ると、スキニングの観察がしやすい

今回は上下面を閉じていません。これはサボっているのではなく、「側面の曲がり」と「weight 境界の見え方」に集中するためです。

## 28.6 `SmoothShader` は GPU 側で何をしているか

CPU 側で `Skeleton.updateMatrixPalette()` が呼ばれると、bone ごとの `currentBoneMatrixLocal * inverseRestGlobal` が 12 float 単位で並びます。

`SmoothShader` の頂点シェーダーは、各頂点の `index` と `weight` を読み、4 本までの bone 行列を重み付きで合成して、最終的な skinning 行列を作ります。その行列を頂点位置と法線へ掛けてから、通常の static mesh と同じように view / projection へ流します。

重要なのは、「bone ごとに頂点を個別に描き直している」のではなく、「1 頂点について 4 本までの influence を 1 回の頂点シェーダー実行でブレンドしている」ことです。

そのため、切り分けるときは次の順で考えると迷いにくくなります。

1. `Shape` に bone index / weight が入っているか
2. `Skeleton` の rest pose と bone order が正しいか
3. `Skeleton.updateMatrixPalette()` が current pose を反映しているか
4. `shape.setMaterial("smooth-shader", { has_bone: 1 })` が有効か
5. `weight_debug` を使うと influence 分布が意図どおりか

## 28.7 実行例は 2 本に分けて検証する

この章では、検証対象を混ぜないために example を 2 本へ分けます。

### 28.7.1 `book/examples/28_01.html`

`28_01` は bone と marker だけを表示し、代表頂点 1 個について次の 4 点を比較するための例です。

1. rest 位置
2. `bone0` だけで変形した位置
3. `bone1` だけで変形した位置
4. weight 付き平均を取った最終位置

この page では円柱 メッシュ を出さず、数式どおりに marker が動くかを最優先で確認します。見やすさのため、代表点は円柱軸から少し離した probe point を使いますが、検証している原理自体は同じです。visible marker 自体は、前節の CPU helper をそのまま実行して得た座標を表示しています。

```js id="7i3pdq"
skeleton.updateMatrixPalette();

const rootOnlyLocal = transformVertexByBone(restPosition, rootBone);
const childOnlyLocal = transformVertexByBone(restPosition, childBone);
const blendedLocal = blendVertex(restPosition, [
  { bone: rootBone, weight: w0 },
  { bone: childBone, weight: w1 }
]);

const rootOnlyWorld = objectNode.getWorldMatrix().mulVector(rootOnlyLocal);
const childOnlyWorld = objectNode.getWorldMatrix().mulVector(childOnlyLocal);
const blendedWorld = objectNode.getWorldMatrix().mulVector(blendedLocal);
```

同時に `28_01` の内部では、`M0^-1 * p_rest` と `M1^-1 * p_rest` をそれぞれ bone node にアタッチした参照点も別に持っています。status 行に出している `verify` は、その参照点の world 座標と CPU helper の結果との差です。つまり `28_01` は「本文の helper が正しいか」を見せる page であり、参照点はその正しさを確かめるための内部チェックとして使っています。

### 28.7.2 `book/examples/28_02.html`

`28_02` は円柱 メッシュ の見え方だけを確認する例です。marker は出さず、`SmoothShader` による曲がり方と、`weight_debug` を切り替えたときの分布だけを見ます。

つまり、`28_01` は「1 頂点の原理検証」、`28_02` は「メッシュ全体の見え方検証」という役割分担です。

## 28.8 Blender / glb の高レベル経路へ戻す

ここまでの低水準例を見たあとで glb の高レベル経路へ戻ると、Blender で行っている作業の意味が読みやすくなります。

Blender 側でやっているのは、概ね次の内容です。

1. bone 階層を作る
2. 各 bone のレスト姿勢を決める
3. 頂点ごとにどの bone がどれだけ効くかを weight paint する
4. animation clip としてボーンごとの時系列姿勢（キーフレーム）として持つ

glb を import すると、`webg` はこれらを `ModelAsset`、`Skeleton`、`Animation` として取り込みます。したがって、高レベル経路は「低水準の仕組みを使わない」のではなく、「低水準で手で書いていた部分を importer が肩代わりしてくれる」と考えるのが実態に近くなります。

## 28.9 よくあるつまずき

### 28.9.1 `has_bone: 1` を入れたのに曲がらない

bone index / weight が shape へ入っていないか、skeleton の palette 順と vertex 側 index が対応していない可能性があります。

### 28.9.2 bone は動いているのにメッシュだけレスト姿勢のまま

`has_bone: 1` を有効にした `SmoothShader` ではなく、static mesh 向け設定のまま描いていると、そのままでは変形しません。skinned mesh では `SmoothShader` に対して `has_bone: 1` が必要です。

### 28.9.3 曲がるが境界が折れすぎる

weight を極端に切り替わるようにしすぎている可能性があります。まず `weight_debug` で分布を見て、境界付近が 0 / 1 だけになっていないかを確認します。

### 28.9.4 importer を疑う前に低水準の最小例を見る

複雑な glb では、問題が animation、skeleton 名、material、texture、node scale のどこにあるか分かりにくくなります。最小の 2-bone 円柱で仕組みを再確認すると、どの層を疑うべきかを切り分けやすくなります。
