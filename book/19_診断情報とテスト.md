# 19 診断情報とテスト

最終更新：2026-04-09 JST

この章では、`WebgApp`、`DebugConfig`、`DebugDock`、`Diagnostics`、`DebugProbe` を通して、診断情報の使い方を整理します。`webg` の診断情報は、アプリの現在状態を画面で読み、そのままクリップボード経由で共有するための仕組みです。

いまの `webg` では、描画された画面の状態を AI に伝えたいとき、まず `DebugDock` の `Copy Summary` または `Copy JSON` を使います。すると、その瞬間のカメラ位置、視野角、シーン規模、警告、サンプル固有の状態がクリップボードへコピーされます。利用者はそれをそのまま貼り付けるだけで、AI に「見えているはずの状態」を数値やテキストとして渡せます。つまり、診断情報は単なる補助表示ではなく、アプリの現在状態を共有するための正式な出口です。

## この章で押さえること

### A. `DebugDock` は現在状態の表示とコピーのための場所

`DebugDock` には、`Current State` の表示欄と、`Copy Summary`、`Copy JSON` の 2 つのボタンだけがあります。ここから現在状態を読み、そのままコピーできます。

### B. もっとも重要な操作は `F9` のあとに `M`

`WebgApp` の既定設定では、デバッグ用のキー入力は prefix 方式です。厳密には「`F9` を押したあとに `M`」で debug / release を切り替えます。同じ仕組みで、「`F9` を押したあとに `C`」で summary、「`F9` を押したあとに `V`」で JSON をコピーできます。

### C. `Copy Summary` は人と AI が最初に読む要約

summary は、`Overview`、`Latest Issue`、`Key Stats`、`Warnings`、`Recent Details` の順で並ぶテキストです。まずはこれを読むのが標準です。

### D. `Copy JSON` は構造付きの詳細情報

JSON には summary と同じ基礎情報に加え、`context.webgAuto` 配下へ camera、shader、material の一覧が入ります。AI に詳しい解析を頼むときはこちらが向いています。

### E. 自動統計に加えて、そのアプリならではの情報を足す

カメラ位置や `triangleCount` のような共通情報は `WebgApp` が自動で集めます。そこへ `mode`、`focusDistance`、`clipCount`、`score` のような、そのアプリで意味のある値を追加すると、要約が読みやすくなります。

## 補助ファイルの役割を先に整理する

診断情報まわりの実装は、1 つの巨大な仕組みではなく、役割の違う 4 つの部品を組み合わせています。ここを先に分けて理解しておくと、`WebgApp` を使う場合も、独自 UI を組む場合も、どこを見ればよいか判断しやすくなります。

### `DebugConfig.js`

`DebugConfig.js` は、debug / release の切り替えと、その切り替えに連動する診断系の可否をまとめて管理する部品です。背景としては、diagnostics、probe、console 出力のような debug 用機能を、sample ごとに個別実装せず同じ基準で切り替えたいという事情があります。

`DebugConfig` は、`mode` を `debug` または `release` として持ち、その mode に応じて `enableConsole`、`enableDiagnostics`、`enableProbe` のような debug 向け機能をまとめて切り替えます。

使い方は 2 通りあります。通常の sample では `WebgApp` が内部で `DebugConfig.setMode()` を呼ぶので、利用者は `F9` のあとに `M` で mode を切り替えるだけで済みます。一方、`samples/sound` や `samples/circular_breaker` のように `WebgApp` を使わない経路では、`DebugConfig.setMode()`、`DebugConfig.isDebug()`、`DebugConfig.isEnabled("enableProbe")` のように直接参照して、debug 表示や保存操作の可否を分けます。

### `DebugDock.js`

`DebugDock.js` は、診断 report を画面で読み、必要ならその場でコピーするための表示部品です。背景としては、console を開かなくても現在状態を見られ、そのまま AI や他の利用者へ貼り付けられる場所を desktop 側に用意したい、という狙いがあります。

このファイル自身は report を計算しません。`Current State` に summary text を表示し、`Copy Summary` と `Copy JSON` のボタンを並べる、という UI の責務に絞られています。つまり、診断内容を組み立てるのは `Diagnostics.js` や `WebgApp` であり、`DebugDock` はそれを見せるための view だと考えると整理しやすくなります。

使い方は、ふつうは `WebgApp` 経由です。`WebgApp` が constructor 内で `new DebugDock(...)` を組み立て、`updateDebugDock()` から summary text を流し込みます。利用者が直接 `DebugDock` を new する場面は多くありません。表示条件は `DebugDock.isActive()` にまとまっていて、debug 用 UI が有効であること、十分な幅があること、細いポインター環境であること、のような desktop 寄りの条件を満たしたときだけ現れます。

### `Diagnostics.js`

`Diagnostics.js` は、現在状態を report object にまとめ、その report を summary / text / JSON へ整形し、必要ならコピーや保存へ流す中心部品です。背景としては、dock に出す文字列、clipboard に渡す文字列、JSON として保存する内容を別々に作ると、表示と共有内容がずれやすくなるためです。

そのため `Diagnostics.js` の意味は、「診断情報の正本を作る部品」と考えると分かりやすくなります。`createSuccessReport()` や `createErrorReport()` で report を作り、`addDetail()`、`addWarning()`、`mergeStats()` で中身を足し、最後に `toSummaryText()`、`toText()`、`toJSON()`、`copySummary()`、`copyJSON()` で用途ごとの形へ変換します。`DebugDock` が表示している `Current State` も、最終的にはこの report を summary 化したものです。

使い方は、`WebgApp` を使う場合と使わない場合で少し違います。`WebgApp` を使う sample では `setDiagnosticsStage()`、`mergeDiagnosticsStats()`、`addDiagnosticsDetail()` のような wrapper を通すだけで report を組み立てられます。一方、`samples/sound` のような独自 UI では `Diagnostics.createSuccessReport()` を直接呼び、独自の stats や details を足して `Diagnostics.copyJSON()` や `Diagnostics.downloadText()` へ渡します。

### `DebugProbe.js`

`DebugProbe.js` は、「連続的に取り続ける log」ではなく、「欲しい瞬間を 1 回だけ採取する」ための one-shot helper です。背景としては、モード切り替え直後や入力直後のように、毎フレーム変わる状態の中から特定タイミングの 1 枚を切り出したい、という場面が多いためです。

このファイルの意味は、採取要求の待ち行列です。`request()` で「何フレーム後に、どの関数で採取するか」を登録し、毎フレーム `update(frameCount)` を呼んで、条件を満たした 1 件だけを発火します。採取結果は `lastResult` に残るので、あとで「最後に何を採取したか」も確認できます。常時出力を垂れ流す部品ではなく、必要な瞬間だけ report を固定する部品です。

使い方は、`WebgApp` では `configureDiagnosticsCapture()` と `captureDiagnosticsSummary()` / `captureDiagnosticsReportJSON()` の組み合わせが基本です。内部では `DebugProbe.request()` と `updateDebugProbe()` が働き、指定フレーム後に report を採取してコピーします。`samples/sound` や `samples/circular_breaker` では `new DebugProbe()` を直接作り、`collect` で report を返す関数を登録して、独自の copy / save 操作へつないでいます。

## 1. `DebugDock` は何をするか

`DebugDock` は、右側に出る HTML の dock です。`webg/DebugDock.js` を読むと、ここにある操作は次の 2 つだけです。

1. `Copy Summary`
2. `Copy JSON`

その下に `Current State` が表示されます。ここには `Diagnostics.toSummaryText()` が返す summary テキストがそのまま表示されます。つまり、画面で見えている `Current State` と、`Copy Summary` でコピーされる内容は、同じ current report を基にしています。

この構成にしている理由は、画面で読んだ内容と、AI に貼り付けた内容がずれにくくなるためです。別々の経路で別々の文字列を作るのではなく、現在の report を 1 つ組み立て、その report を summary と JSON に整形して使い分けています。

## 2. `DebugDock` を表示する方法

`WebgApp` の既定の `debugTools.mode` は `debug` です。ただし、`DebugDock` は常に見えるわけではありません。`DebugDock.isActive()` の条件を見ると、次の条件を満たしたときだけ表示されます。

1. debug mode であること
2. ビューポート幅が既定で 960px 以上であること
3. 細いポインター、つまり通常のマウス操作を前提にした環境であること

そのため、デスクトップでは見えていても、スマートフォンや狭いブラウザー幅では表示されないことがあります。これは不具合ではなく、`DebugDock` がデスクトップ向けに設計されているためです。

もっとも重要な操作は、debug / release の切り替えです。現在の `WebgApp` では prefix 方式なので、次の順で操作します。

1. `F9` を押す
2. その直後に `M` を押す

これで debug / release が切り替わります。文書ではしばしば「`F9 + M`」と簡略に書けますが、実際の入力処理は同時押しではなく「`F9` のあとに `M`」です。

同じ仕組みで、次の操作も使えます。

1. `F9` のあとに `C`
   summary をコピーします
2. `F9` のあとに `V`
   JSON をコピーします
3. `F9` のあとに `Esc`
   待機状態をキャンセルします

release では `enableDiagnostics` と `enableProbe` が無効になり、debug 用の dock や補助表示も隠れます。そのため、コピーや `DebugDock` を使いたいときは、まず `F9` のあとに `M` で debug へ戻すのが基本です。

## 3. クリップボード共有が中心である

`webg` では、診断情報の中心はクリップボード共有です。`Copy Summary` と `Copy JSON` は、どちらも `navigator.clipboard.writeText()` を使ってテキストをクリップボードへ書き込みます。コピーした内容は、そのまま AI との会話欄へ貼り付けられます。

この点で、いまの診断情報は「ブラウザーの開発者向け画面のコンソールを開いて読む」ことを前提にしていません。もちろん、必要ならブラウザーの「ウェブ開発ツール」や「デベロッパーツール」を使えます。多くのブラウザーでは、Windows なら `Ctrl` と `Shift` と `I` を同時に押すと開けます。その中にある `Console` パネルへログを出す使い方もあります。しかし、`webg` の標準の共有経路は console ではなく、`Copy Summary` と `Copy JSON` です。

したがって、まず dock からコピーする流れを基準にすると実際の使い方と一致します。

## 4. `Copy Summary` で何が得られるか

summary は `Diagnostics.toSummaryText()` で作られます。構成は次の 5 つです。

1. `[Overview]`
2. `[Latest Issue]`
3. `[Key Stats]`
4. `[Warnings]`
5. `[Recent Details]`

### 4.1 `Overview`

ここには、最低限次の項目が入ります。

1. `source`
   どのファイルやアセットを見ているか
2. `system`
   どの系統のサンプル、またはどのアプリか
3. `stage`
   現在の段階。`ready`、`build`、`runtime`、`runtime-probe` など
4. `ok`
   その report が成功扱いか失敗扱いか
5. `timestamp`
   採取時刻

AI に状況を渡すときは、まずこの `Overview` を見ると、「何についての report なのか」がすぐ分かります。

### 4.2 `Latest Issue`

ここには、最新の error があれば error、なければ最新の warning が入ります。何もなければ `(none)` です。

利用者が最初に見るべきなのは、実はここです。`Latest Issue` に重要な warning があれば、細かい数値を見る前に「いま何が怪しいか」の当たりを付けられます。

### 4.3 `Key Stats`

ここには、診断で最初に見たい統計値が優先順で並びます。順序は `Diagnostics.getSummaryStatKeys()` で定義されています。主な項目の意味は次のとおりです。

1. `eyeX`, `eyeY`, `eyeZ`
   カメラのワールド座標です。カメラが極端に遠くへ飛んでいないかを見ます
2. `eyeYaw`, `eyePitch`, `eyeBank`
   カメラの向きです。モデルがあるのに別方向を見ていないかを見ます
3. `cameraTargetX`, `cameraTargetY`, `cameraTargetZ`
   カメラが向こうとしている target です
4. `eyeDistance`
   eye と target の距離です。ズームしすぎ、近すぎ、遠すぎを見ます
5. `fovX`, `fovY`
   現在の視野角です。極端な広角や狭角になっていないかを見ます
6. `appShaderClass`
   アプリ全体で使っている shader クラス名です
7. `vertexCount`, `triangleCount`
   現在のシーンで参照している geometry の規模です
8. `boneCount`
   skeleton を持つ構成かどうかを見るための骨数です
9. `nodeCount`
   `Space` にあるノード数です
10. `shapeCount`
    シーンに置かれている shape instance の数です
11. `meshCount`
    共有 resource 単位で数えた geometry 数です
12. `visibleShapeCount`, `hiddenShapeCount`
    表示中と非表示の shape 数です
13. `skinnedShapeCount`
    skeleton を持つ shape 数です
14. `canvasWidth`, `canvasHeight`
    実際の描画バッファのサイズです
15. `displayWidth`, `displayHeight`
    CSS 上の表示サイズです
16. `frameCount`
    進んだフレーム数です
17. `envOk`
    環境チェックを含めた現在の report が正常扱いかどうかです
18. `uptimeSec`
    起動からの経過秒です

このほか、`entryCount`、`clipCount`、`runtimeAnimations`、`primitiveCount`、`modelCount` のようなサンプル固有の値が追加される場合があります。これらは利用者が `mergeDiagnosticsStats()` で足した値です。

### 4.4 `Warnings`

warning 配列の末尾側から最大 4 件が表示されます。たとえば `checkEnvironment()` が返した注意点や、利用者が `addDiagnosticsWarning()` で入れた内容がここへ出ます。

### 4.5 `Recent Details`

detail 配列の末尾側から最大 8 件が表示されます。ここは「そのアプリで何を見たいか」を短い行で追加する場所です。

たとえば次のような値を入れると、読者にとって意味のある summary になります。

1. `mode=follow`
2. `focusDistance=26.0`
3. `bgmPreset=boss`
4. `weightMode=hard`
5. `clip=walk`

## 5. `Copy JSON` で何が増えるか

`Copy JSON` は summary と同じ current report を基にしていますが、`detailLevel: "json"` で組み立て直されるため、追加の構造情報が入ります。特に重要なのは `context.webgAuto` です。

ここには次の情報が入ります。

1. `context.webgAuto.camera.target`
   camera target の配列
2. `context.webgAuto.camera.eyePosition`
   eye の位置配列
3. `context.webgAuto.camera.eyeAttitude`
   eye の姿勢配列
4. `context.webgAuto.rendering.appShaderClass`
   アプリ全体の shader クラス
5. `context.webgAuto.rendering.shapeShaderClasses`
   現在シーンで使っている shader クラス一覧
6. `context.webgAuto.rendering.shapeShaderUsage`
   shader ごとの使用数
7. `context.webgAuto.rendering.materialIds`
   material ID 一覧
8. `context.webgAuto.rendering.materialUsage`
   material ごとの使用数

AI に「どの shader で描いているのか」「同じ material が何個使われているのか」まで渡したいときは、summary ではなく JSON を使います。

## 6. `Current State` とコピー結果の関係

`Current State` も `Copy Summary` も、同じ summary テキストを使います。違うのは更新タイミングです。

`Current State` は毎フレーム再計算されるのではなく、`WebgApp` 側で 1 秒のキャッシュを持っています。これは dock の表示コストを下げるためです。一方で `Copy Summary` と `Copy JSON` は、ボタンを押した時点で `forceRefresh: true` で report を作り直します。

したがって、通常は `Current State` を読み、正確にその瞬間の値を渡したいときだけボタンを押してコピーする、と考えると分かりやすくなります。

## 7. 最小構成

もっとも基本的な書き方は次のとおりです。

```js
const app = new WebgApp({
  document,
  debugTools: {
    mode: "debug",
    system: "scene",
    source: "./scene.json"
  }
});

await app.init();
app.setDiagnosticsStage("runtime");
app.configureDebugKeyInput();
```

この時点で、少なくとも次のことができるようになります。

1. desktop 条件を満たせば `DebugDock` が表示される
2. `Copy Summary` と `Copy JSON` が使える
3. `F9` のあとに `M`、`C`、`V` が使える
4. 環境の基本情報と自動統計が summary / JSON へ入る

## 8. どんな情報が自動で入るか

`WebgApp` は、利用者が何も追加しなくても、現在の report に次のような情報を自動で足します。

### 8.1 起動直後の環境情報

`app.init()` の最後では `checkEnvironment({ stage: "ready" })` が呼ばれます。ここで次のような値が report に入ります。

1. `webgpuSupported`
2. `hasScreen`
3. `hasShader`
4. `hasProjection`
5. `hasSpace`
6. `hasEye`
7. `hasInput`
8. `hasMessage`

これにより、利用者が最低限の起動確認を毎回手書きしなくても済みます。

### 8.2 現在のシーン規模

`collectCurrentSceneStats()` により、次の統計が自動で入ります。

1. `nodeCount`
2. `shapeCount`
3. `meshCount`
4. `vertexCount`
5. `triangleCount`
6. `boneCount`
7. `visibleShapeCount`
8. `hiddenShapeCount`
9. `skinnedShapeCount`

特に `meshCount`、`vertexCount`、`triangleCount` は共有 resource 単位で数えます。そのため、`Shape.createInstance()` で instance を増やしても、geometry の再利用ができていれば数がむやみに膨らみません。

### 8.3 カメラと描画条件

次の値も自動で入ります。

1. `eyeX`, `eyeY`, `eyeZ`
2. `eyeYaw`, `eyePitch`, `eyeBank`
3. `cameraTargetX`, `cameraTargetY`, `cameraTargetZ`
4. `eyeDistance`
5. `fovX`, `fovY`
6. `canvasWidth`, `canvasHeight`
7. `displayWidth`, `displayHeight`
8. `frameCount`
9. `uptimeSec`
10. `appShaderClass`

このため、読者向けのサンプルでカメラ診断用の専用テキストを毎回別に作る必要は、以前よりかなり減っています。

## 9. 利用者が追加するコード

自動で入る情報だけでは、そのアプリの意味は十分に伝わらないことがあります。そこで利用者は、アプリ固有の情報を追加します。

### 9.1 `stage` を切り替える

まず、いま何の段階かを report に残します。

```js
app.setDiagnosticsStage("fetch");
sceneAsset = await SceneAsset.load("./scene.json");

app.setDiagnosticsStage("build");
sceneRuntime = await app.loadScene(sceneAsset.getData());

app.setDiagnosticsStage("runtime");
```

これで AI や他の利用者が report を読んだときに、どの段階の情報かを見分けやすくなります。

### 9.2 `stats` を追加する

サンプル固有の数値は `mergeDiagnosticsStats()` で追加します。

```js
app.mergeDiagnosticsStats({
  mode: state.mode,
  focusDistance: dof.focusDistance.toFixed(1),
  focusRange: dof.focusRange.toFixed(1),
  blurScale: dof.getBlurScale().toFixed(2),
  clipCount,
  score
});
```

ここに入れるべきなのは、「そのアプリの利用者が最初に知りたい値」です。たとえば次のように考えると整理しやすくなります。

1. 視点制御サンプルなら `mode`
2. 被写界深度サンプルなら `focusDistance`
3. ローダーサンプルなら `clipCount`
4. ゲームサンプルなら `score`

### 9.3 `details` を追加する

短い補足は `addDiagnosticsDetail()` で追加します。

```js
app.addDiagnosticsDetail(`clip=${selectedClipName}`);
app.addDiagnosticsDetail(`targetPos=(${x.toFixed(2)}, ${y.toFixed(2)}, ${z.toFixed(2)})`);
app.addDiagnosticsDetail(`bgmPreset=${selectedBgmPreset}`);
```

details は `Recent Details` に出るので、短く読める情報に向いています。長い説明文より、`key=value` の形にそろえるほうが読みやすくなります。

### 9.4 `warnings` を追加する

注意を促したい内容は `addDiagnosticsWarning()` で追加します。

```js
if (envReport.warnings?.length) {
  app.addDiagnosticsWarning(envReport.warnings[0]);
}
```

warning は `Latest Issue` と `Warnings` の両方で目に入りやすくなるので、単なる補足より優先して読ませたい内容に向いています。

### 9.5 `checkEnvironment()` を併用する

`checkEnvironment()` は、projection、shape、weight、message などの基本整合を確認するための report を返します。現在のサンプルでもよく使われています。

```js
const envReport = app.checkEnvironment({
  stage: "runtime-check",
  shapes: [activeShape]
});

app.mergeDiagnosticsStats({
  envOk: envReport.ok ? "yes" : "no",
  envWarning: envReport.warnings?.[0] ?? "-"
});
```

この形にしておくと、利用者は summary を見ただけで「そのシーンが基本条件を満たしているか」を判断できます。

## 10. one-shot probe を使う

連続更新中の状態ではなく、「切り替えた直後の 1 回だけ」を固定したいことがあります。そのときは `configureDiagnosticsCapture()` と `createProbeReport()` を使います。

```js
function makeProbeReport(frameCount) {
  const report = app.createProbeReport("runtime-probe");
  Diagnostics.addDetail(report, `mode=${state.mode}`);
  Diagnostics.addDetail(report, `targetPaused=${state.targetPaused ? "yes" : "no"}`);
  Diagnostics.mergeStats(report, {
    frameCount,
    targetAngleDeg: state.targetAngleDeg.toFixed(2)
  });
  return report;
}

app.configureDiagnosticsCapture({
  labelPrefix: "camera_controller",
  collect: () => makeProbeReport(app.screen.getFrameCount())
});
```

これを登録しておくと、`captureDiagnosticsSummary()` や `captureDiagnosticsReportJSON()` を使って、1 フレーム後などの状態を採取できます。

```js
app.captureDiagnosticsSummary({ afterFrames: 1 });
app.captureDiagnosticsReportJSON({ afterFrames: 1 });
```

probe は、モード切り替え直後、アニメーション再開直後、入力直後など、一瞬の状態を確定させたいときに向いています。

## 11. 起動失敗を report に残す

起動に失敗した場合は、画面へエラーパネルを出すだけでなく、diagnostics report もエラー版へ差し替えておくと、あとで共有しやすくなります。

```js
document.addEventListener("DOMContentLoaded", () => {
  start().catch((error) => {
    app?.setDiagnosticsReport?.(Diagnostics.createErrorReport(error, {
      system: "camera_controller",
      source: "samples/camera_controller/main.js",
      stage: app?.getDiagnosticsReport?.()?.stage ?? "start"
    }));
    app?.showErrorPanel?.(error, {
      title: "camera_controller failed",
      id: "start-error"
    });
  });
});
```

この形にしておけば、利用者は失敗時でも `Copy Summary` や `Copy JSON` で状況をそのまま渡せます。

## 12. `WebgApp` を使わない場合

通常は `WebgApp` を使うほうが自然です。ただし、`samples/sound` のように独自 UI を組み、`Diagnostics.js` を直接使う方法もあります。

```js
const report = Diagnostics.createSuccessReport({
  system: "sound",
  source: "samples/sound/main.js",
  stage: "runtime"
});

Diagnostics.addDetail(report, `soundEffect=${selectedSoundEffect}`);
Diagnostics.mergeStats(report, {
  masterVol: ui.masterVol.value,
  seVol: ui.seVol.value,
  bgmVol: ui.bgmVol.value
});
```

この経路は、`WebgApp` が持つ自動統計は使いません。その代わり、独自 UI の状態をそのまま report へまとめられます。音のように、3D シーン統計より調整値の記録が重要な場合はこちらが向いています。

## 13. どのサンプルを参照するとよいか

利用者が実例を読みたいときは、次のサンプルが特に役立ちます。

1. `samples/scene`
   `Copy Summary` と `Copy JSON` を使う基本例です
2. `samples/camera_controller`
   カメラ関連の `mode` や target 状態を追加する例です
3. `samples/dof`
   `focusDistance` などのパラメータを summary へ載せる例です
4. `samples/model_shape`
   build 経路と validator 状態を report へ載せる例です
5. `samples/collada_loader`
   loader と runtime の接続状態を report に残す例です
6. `samples/sound`
   `WebgApp` を使わずに `Diagnostics.js` を直接使う例です

## 14. まとめ

現在の `webg` の診断情報は、canvas 上の文字表示を主役にした補助機能ではありません。主役は current report と、その report をクリップボードへ持ち出す `Copy Summary`、`Copy JSON` です。利用者は `DebugDock` を見て、必要ならボタンや `F9` のあとに `C` / `V` でコピーし、そのまま AI に渡せます。

もっとも重要な操作は、`F9` のあとに `M` です。これで debug / release を切り替え、`DebugDock` やコピー機能を使える状態へ戻せます。そのうえで、自動統計に対してアプリ固有の `stats`、`details`、`warnings` を足していくと、読者にも AI にも伝わる診断情報になります。
