# 3Dグラフィックスの基礎

この章は、`webg` で 3D を扱うときに必要になる基本的で重要な定義を整理した章です。

3D グラフィックスでは、座標系、回転、行列、クォータニオン、UV、ノーマルマップのうち、どれか一つだけを個別に理解していても、実際にコードを書く段階では知識がつながらず、迷いやすくなります。特に、「Yが上か、Zが上か」「Y 軸まわりの回転を yaw と呼ぶのか head と呼ぶのか」「行列は行優先なのか列優先なのか」「ベクトルを左に置くのか右に置くのか」「クォータニオンの回転は何番目の要素か」が曖昧なままだと、コードが読めても手が止まりやすくなります。この章では、その曖昧さを避けるために、`webg` が採用している定義を最初に説明します。

この章で最初に押さえたい基準は、次のとおりです。`webg` は右手系で、`+X=右`、`+Y=上`、`+Z=前` を基本軸とします。回転用語は `head / pitch / bank` を使い、一般的な 3D 用語では `head = yaw`、`pitch = pitch`、`bank = roll` に対応します。行列は列優先で保持し、ベクトルは列ベクトルとして右側へ置き、式は `M * v` の形で読みます。クォータニオンの内部順序は `[w, x, y, z]` です。glTF のように `[x, y, z, w]` を使うデータとは、この点で順序が異なります。また、UV は概念上 bottom-left 基準で考え、ノーマルマップは `RGB` をそのまま色として読むのではなく、`0..1` を `-1..1` に戻して `X / Y / Z` の方向ベクトルとして扱います。透視投影は `near`、`far`、縦方向の FOV、aspect 比で決まり、FOV が小さいほど望遠寄り、大きいほど広角寄りの見え方になります。

この章では、まず座標系の向きと回転の基準をそろえ、そのあとでテクスチャ座標とノーマルマップ、行列とベクトルの流儀、透視投影、クォータニオン、glTF / GLB との対応へと進みます。ここで基準をそろえておくと、`Node` の姿勢、カメラの orbit、GLB 読み込み、ノーマルマップ、スケールの話まで、同じ土台の上で読めるようになります。

## 座標系と回転の基準

`webg` の座標系は右手系です。右手の親指を `+X`、人差し指を `+Y`、中指を `+Z` と考えると、その向きがそのまま `webg` の基本軸になります。

![座標系と回転の基準](fig03_01_coordinate_rotation_basics.jpg)

*webg は右手系で +X=右、+Y=上、+Z=前 を基本軸とし、回転の順序は head / pitch / bank = Y / X / Z の対応で読むと整理しやすくなります。*

「画面の奥が前なのか、手前が前なのか」という点は誰でも混乱しやすい問題です。 誰からどんな状況で見て「前」なのかという文脈が無ければ答えは決まりません。 航空分野で一般的に使われる座標系は、NED座標系 (North-East-Down)といって、X軸は真北方向、Y軸は真東方向、Z軸は地球の重心方向（真下）を指す座標系です。X軸の示す方向を進行方向と考えます。このようにワールド座標系(地球)と航空機の進行方向を定義すると航空機にとっての「前」ははっきりします。パイロットの見ている方向（画面の奥）とは別の話です。

同じように `webg` の座標系は、**EUS座標系(East-Up-South)**と呼ぶことができます。X軸は真東方向、Y軸は真上方向、Z軸は真南を指す座標系です。Z座標の値が増える方向を前とすると、`+X` は右、`+Y` は上、`+Z` は前となります。さて、「画面の奥が前なのか、手前が前なのか」の答えは「カメラが向いている方向が前」とすると「画面の奥が前」と考えるのが自然ですが、「状況による」としか言えません。

回転の呼び方についても、最初に統一しておく必要があります。`webg` では、回転を `head / pitch / bank` で表します。一般的な 3D 用語に置き換えると、`head` は `yaw`、`pitch` はそのまま `pitch`、`bank` は `roll` です。 回転軸は `head=Y`、`pitch=X`、`bank=Z` となります。 ヘッドは左右、ピッチは上下、バンクは頭を傾ける動作になります。 どこかで物音がした場合、まず音の方向を向き（ヘッド）、上を見上げて（ピッチ）の順序と考えれば自然です。

| webg    | 一般的な呼び名 | 回転軸 |
| ------- | ------- | --- |
| `head`  | `yaw`   | Y 軸 |
| `pitch` | `pitch` | X 軸 |
| `bank`  | `roll`  | Z 軸 |


右ネジの法則: 回転軸に沿って右ネジを回したとき、ネジが進む方向が正の回転方向として定義されます。
視点による見え方の違い:
- 軸の正方向へ「向かって」見ると、正の回転は時計回りに見えます。
- 軸の正方向側から「原点を向いて」見ると、正の回転は反時計回りに見えます。
Bank(Roll)回転: 「前方向（画面奥）」をZ軸の負方向とした場合、その軸周りの正のRoll回転は原点から奥を見た視点で時計回りとなります。右手の親指を軸の正方向に向け、他の指を軽く曲げたときの指先の方向が回転角の正の向きです。

```js
node.setAttitude(90, 0, 0);  // head +90
```

`pitch` は X 軸まわりです。右から見て、正の `pitch` は前方向 `+Z` を下方向 `-Y` へ回します。たとえば `pitch = 90` にすると、前を向いていた軸は真下を向きます。

```js
node.setAttitude(0, 90, 0);  // pitch +90
```

`bank` は Z 軸まわりで、一般的には `roll` に相当します。前から見て、正の `bank` は右方向 `+X` を上方向 `+Y` へ回します。飛行機でいう「機体を傾ける」回転に近いものです。

```js
node.setAttitude(0, 0, 90);  // bank +90
```

`webg` の `Matrix.setByEuler(head, pitch, bank)` と `Quat.eulerToQuat(head, pitch, bank)` は、Y / X / Z の順で回転するオイラー角を扱います。ここで大事なのは、「引数の順番が `head, pitch, bank` であること」と、「それぞれがどの軸を表すかが固定されていること」です。3DCG の資料では `XYZ`、`ZYX`、`yaw-pitch-roll` などの表記が混在しやすいのですが、`webg` を読むときは、「`head(Y) -> pitch(X) -> bank(Z)` の名前で統一されている」と考えます。

## テクスチャ座標とノーマルマップの読み方

3D では、形そのものだけでなく、表面にどのように画像を貼るか、そして光に対してどのように凹凸を見せるかも重要です。ここで基準になるのが UV とノーマルマップです。

![UV とノーマルマップの読み方](fig03_02_uv_normalmap.jpg)

*UV は概念上 bottom-left 基準で読み、ノーマルマップは RGB をそのまま色としてではなく X / Y / Z の方向ベクトルとして扱います。*

`webg` のテクスチャ座標は、概念上は bottom-left 基準です。これは Blender の考え方に合わせた整理です。`U` は右へ行くほど増え、`V` は上へ行くほど増えます。したがって、左下が `(0, 0)`、右上が `(1, 1)` になります。この定義そのものは、`Shape` に手動で UV を入れるときも、`TileMap` が top face に明示的な UV を入れるときも同じです。

四角形に 1 枚の画像を貼るなら、概念上は次のように考えます。

```js
[
  { pos: [-1, -1, 0], uv: [0, 0] },
  { pos: [ 1, -1, 0], uv: [1, 0] },
  { pos: [ 1,  1, 0], uv: [1, 1] },
  { pos: [-1,  1, 0], uv: [0, 1] }
]
```

ただし、画像を GPU へアップロードしたときの上下の見え方は、概念上の UV と画像の格納向きで食い違うことがあります。その場合に直すべきなのは、基本的には画像入力側の `flipV` です。概念上の UV 自体を top-left 基準へ変えてしまうと、ジオメトリ側の意味と画像入力側の意味が混ざり、後でノーマルマップや明示的な UV を読むときに分かりにくくなります。これは TileMap の上面 UV を読むときにも同じです。UV は「ジオメトリがどのように貼られるか」を決める値であり、画像の上下補正は「テクスチャをどのように読み込むか」の側へ分けて考えるほうが整理しやすくなります。

ノーマルマップも、表面情報を読むうえで大切です。ノーマルマップは色そのものを表すテクスチャではなく、「そのピクセルの法線方向をどちらへ傾けるか」を持つテクスチャです。教材用の `book/examples/NormPhong.js` では、ノーマルマップを次の式で読みます。

```wgsl
let ntex = textureSampleLevel(uNormalTexture, uSampler, input.vTexCoord, 0.0).xyz
         * 2.0
         - vec3f(1.0, 1.0, 1.0);
```

ここでの意味は、`R -> X`、`G -> Y`、`B -> Z` です。ただし、そのまま `0..1` の色としては使いません。`0..1` を `-1..1` に戻してから、接空間の方向ベクトルとして使います。たとえば、`R = 0.5` は `X = 0`、`G = 0.5` は `Y = 0`、`B = 1.0` は `Z = 1` です。よくある「ほぼ青いノーマルマップ」は、「大きく傾けず、おおむね表面の正面を向いている」ことを意味します。

教材用の `book/examples/NormPhong.js` は、フラグメントシェーダー内で `dpdx / dpdy` と UV 微分から tangent を再構成し、`bitangent = cross(nnormal, tangent)` として右手系の TBN を作っています。つまり、ノーマルマップの `X / Y / Z` は接空間の `T / B / N` に対応し、最後にビュー側の法線へ変換してライティングに使われます。初学者向けには、ノーマルマップの色は「赤が左右方向、緑が上下方向、青が手前方向の傾き」と読むと十分ですが、ここでいう左右・上下はワールド軸ではなく、その面のローカルな接空間だという点を混同しないことが大切です。

## 行列とベクトルの流儀

`webg` の `Matrix` は 4×4 行列を使い、内部配列 `mat[16]` は列優先です。`set(row, column, val)` が `column * 4 + row` を使っているため、配列の並びは次の形になります。

```text
| m[0]  m[4]  m[8]  m[12] |
| m[1]  m[5]  m[9]  m[13] |
| m[2]  m[6]  m[10] m[14] |
| m[3]  m[7]  m[11] m[15] |
```

このとき、平行移動は最後の列に入り、`m[12] = tx`、`m[13] = ty`、`m[14] = tz` です。`Matrix.position()` もこの 3 要素へ書き込みます。左上の 3×3 部分は、現在のランタイムでは主に回転基底として使われます。`CoordinateSystem.setMatrix()` は `quat` から 3×3 の回転成分を作り、その後で平行移動成分を最後の列へ入れます。現在の `CoordinateSystem` / `Node` は `position + quat + uniform scale` から local matrix を再構築するため、標準フローでも uniform scale まではシーングラフ上で扱えます。ただし、non-uniform scale はまだ対象外です。

`webg` では、ベクトルを列ベクトルとして右側に置く考え方を採用しています。つまり、数式としては次のように読みます。

```text
v' = M * v
```

4 次元同次座標で書くと、次の形です。

```text
|x'|   | m00 m01 m02 tx |   |x|
|y'| = | m10 m11 m12 ty | * |y|
|z'|   | m20 m21 m22 tz |   |z|
|w'|   |  0   0   0   1 |   |1|
```

`Matrix.mulVector([x, y, z])` は、ちょうどこの考え方で `w=1` の点を掛けています。

```js
let x = m[0] * v[0] + m[4] * v[1] + m[8]  * v[2] + m[12];
let y = m[1] * v[0] + m[5] * v[1] + m[9]  * v[2] + m[13];
let z = m[2] * v[0] + m[6] * v[1] + m[10] * v[2] + m[14];
```

この形から分かるとおり、`webg` では行ベクトル `v * M` を標準にはしていません。数学として行ベクトル流儀を知っていても問題ありませんが、`webg` のコードを読むときは列ベクトル流儀に頭を切り替えたほうが混乱しにくくなります。行列合成も同じ前提で読みます。`Matrix.mul(mb)` は `this = this * mb`、`Matrix.lmul(mb)` は `this = mb * this` です。親子階層では `CoordinateSystem.setWorldMatrix()` が `world = parent * local` になるように作られているため、列ベクトル流儀では右側の変換から先に作用します。つまり、`parent * local * v` と書いたとき、最初に `local` が点へ掛かり、その結果に `parent` が掛かると読むと整理しやすくなります。

## 透視投影行列と視野角

3D のカメラを考えるときは、位置と姿勢だけでなく、「どれくらい広い範囲を写すか」も必要です。この役割を持つのが透視投影行列です。`webg` では `Matrix.makeProjectionMatrix(near, far, vfov, ratio)` がこれを作ります。 

![視野角と見え方の違い](fig03_03_projection_fov.jpg)

*FOV が小さいほど望遠寄りで写る範囲は狭くなり、FOV が大きいほど広角寄りで写る範囲は広くなります。*

```js
makeProjectionMatrix(near, far, vfov, ratio) {
  let h = 1.0 / Math.tan(vfov * 0.5 * Math.PI / 180);
  let w = h / ratio;
  let q = 1.0 / (near - far);

  this.makeUnit();
  this.mat[0] = w;
  this.mat[5] = h;
  this.mat[10] = far * q;
  this.mat[11] = -1.0;
  this.mat[14] = far * near * q;
  this.mat[15] = 0.0;
}
```

ここで重要なのは、`vfov` が縦方向の視野角だという点です。`h = 1 / tan(vfov / 2)` なので、視野角を小さくすると `h` は大きくなり、結果として画面へ写る範囲は狭くなります。これは写真でいう焦点距離を長くしたときの見え方に近く、望遠寄りの圧縮された印象になります。反対に、視野角を大きくすると写る範囲は広がり、広角寄りの強い遠近感になります。 

実務上は、次のように読むと分かりやすくなります。FOV が小さいと、画面に入る範囲は狭くなり、同じ位置から見ると対象は大きく見え、焦点距離が長いレンズの印象に近づきます。反対に、FOV が大きいと、画面に入る範囲は広くなり、同じ位置から見ると対象は小さく見え、焦点距離が短いレンズの印象に近づきます。 

縦 FOV だけでは、横方向にどこまで写すかは決まりません。そこで `ratio = width / height` を使い、`w = h / ratio` として横方向の拡大率を決めます。横長画面では同じ縦 FOV でも横へ広く写り、縦長画面では横方向の見え方が詰まります。`webg` の `Screen.getRecommendedFov(base)` は、この aspect 比の違いによって見え方が極端に変わりすぎないよう、標準 FOV を少し調整するヘルパーです。横長画面では `base` をそのまま使い、縦長に近づくほど縦 FOV を広げます。これにより、同じサンプルでも縦長端末で窮屈に見えにくくなります。 

`near` と `far` は、カメラから見てどこまでを描画対象にするかを決める裁断面です。`near` より手前、`far` より奥のものは投影されません。位置や姿勢が正しくても、clip 範囲の外なら画面には現れません。この 2 つは見える範囲だけでなく、深度バッファの精度にも関わります。特に `near` を極端に小さくし、`far` を極端に大きくすると、深度精度が荒れやすくなります。必要以上に広い範囲を取るより、scene の実寸に合わせた値にしたほうが扱いやすくなります。 

また、対象を大きく見せたいとき、カメラを近づける方法と FOV を狭める方法は同じではありません。カメラを近づけると遠近感そのものが変わり、手前と奥の差が強く出ます。一方で FOV を狭めると、同じ位置のまま写る範囲だけが狭まり、望遠寄りの見え方になります。この違いは、カメラ制御を考えるときにとても重要です。どこから見るかは `EyeRig` や `Node` の側で決まり、どう写るかは投影行列の側で決まります。 

## クォータニオンとは何か

クォータニオンは、3D の回転を表す 4 要素の数です。`webg` の `Quat` では、配列の順序を `[w, x, y, z]` で持ちます。回転ゼロは単位クォータニオン `[1, 0, 0, 0]` です。ここで `w` は実部、`x, y, z` は虚部と考えます。

```js
this.q = [1.0, 0.0, 0.0, 0.0];
```

クォータニオンを使う理由は、オイラー角だけで姿勢を持つよりも、連続回転の合成がしやすく、補間しやすく、ジンバルロックを避けやすいためです。`webg` の `CoordinateSystem` も、内部ではオイラー角をそのまま保持していません。`setAttitude()` や `rotate()` を呼ぶと、最終的には `Quat` へ変換して保持します。

`webg` では、クォータニオンは最終的に回転行列へ変換して使われます。`CoordinateSystem.setMatrix()` は、`quat` から 3×3 回転成分を `Matrix.setByQuat()` で作り、そこへ平行移動を最後の列へ入れる流れです。逆方向の変換もあり、`CoordinateSystem.setByMatrix(matrix)` では、行列をコピーしたあと `Quat.matrixToQuat(matrix)` でクォータニオンを復元し、`matrix.getPosition()` で位置を取り出します。つまり、ランタイムにおける姿勢は、「クォータニオンを保持し、必要なときに行列へ展開する」という流れになっています。

クォータニオンの単軸回転は、軸 `axis = (ax, ay, az)` と回転角 `theta` があるとき、概念的には次の形で書けます。 

```text
q = [ cos(theta / 2),
      ax * sin(theta / 2),
      ay * sin(theta / 2),
      az * sin(theta / 2) ]
```

`webg` の `setRotateX()`、`setRotateY()`、`setRotateZ()` は、これを X / Y / Z 軸それぞれに特化した形です。クォータニオンが回転を表すなら、長さ 1 の単位クォータニオンである必要があります。繰り返し計算すると丸め誤差で少しずつずれるため、`Quat.normalize()` で長さをそろえます。`Quat.slerp()` も最後に `normalize()` を呼びます。補間や合成の後に単位長へ戻すのは、回転表現を壊さないための基本です。 

数学としては、ベクトル `v` をクォータニオン `q` で回すとき、概念的には次の形で考えます。

```text
v' = q * v * q^-1
```

ここで `v` は純虚クォータニオンとして埋め込んだベクトルです。ただし、`webg` の通常フローでは毎回この式を直接計算するよりも、クォータニオンを回転行列へ変換して使う場面が多くなります。初心者向けには、「クォータニオンは回転そのものを持ち、行列はそれを座標変換として使いやすい形に展開したもの」と理解しておけば十分です。

## glTF / GLB と `webg` のクォータニオン・行列

glTF / GLB との対応では、クォータニオンの並び順の違いを注意する必要があります。webg は数学的な表記に合わせてクォータニオンを実部が先にくる [w, x, y, z] で保持します。これは誤りではありませんが、glTF の [x, y, z, w] とは順序が異なるため、入出力時には明示的な並べ替えが必要です。glTF のクォータニオンは `[x, y, z, w]` で渡ってくるのに対し、`webg` の `Quat` は `[w, x, y, z]` です。`SceneLoader` と `ModelBuilder` では、この並び替えをしてから行列へ変換しています。したがって、glTF の値をそのまま `Quat.q` へ入れると順序がずれるため、ローダー側で変換していることを理解しておくほうが安全です。

![クォータニオンと glTF の並び順](fig03_04_quat_and_gltf_order.jpg)

*webg のクォータニオンは [w, x, y, z]、glTF / GLB は [x, y, z, w] なので、読み込み時に並び順を変換してから runtime 姿勢へつなぎます。*

また、glTF の TRS はローダー側で読み込めます。現在のランタイムでは、`Node` / `CoordinateSystem` も uniform scale を持てます。そのため、glTF から来た回転・平行移動・一様スケールを、そのままシーングラフ上の local matrix として扱えます。一方で、non-uniform scale はまだ対象外なので、ここは現時点の前提として分けて考える必要があります。GLB 読み込みを追うときにこの基準を先に持っておくと、「読み込まれた transform がどの層で解釈されているか」がかなり見えやすくなります。

## まとめ

この章で最も大事なのは、3D の個別概念をばらばらに覚えないことです。`webg` では、座標系は右手系で `+X=右`、`+Y=上`、`+Z=前`、回転は `head / pitch / bank = Y / X / Z`、行列は列優先で `M * v`、クォータニオンは `[w, x, y, z]`、UV は概念上 bottom-left 基準、ノーマルマップは `RGB -> XYZ` の方向ベクトル、透視投影は `near`、`far`、縦 FOV、aspect で決まる、という前提で統一されています。ここを最初にそろえておくと、このあと出てくる `Node`、カメラ、GLB 読み込み、ノーマルマップ、TileMap、スキニングまで、同じ土台の上で読みやすくなります。

次章では、こうしてそろえた基礎定義を前提にして、`Screen`、`Space`、`Shape`、`Matrix`、`SmoothShader` をつないだ最初の描画フローを見ていきます。
