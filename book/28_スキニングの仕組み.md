# スキニングの仕組み

普段の利用では、armature 付きの glb を読み込んで animation を再生すれば十分です。しかし、見た目が崩れたときに何が壊れているかを判断するには、スキニングの原理そのものを知っておく必要があります。本章では、1 頂点を複数の bone でどう変形して混ぜるのかという最小原理から出発し、`Skeleton`、`SmoothShader`、2-bone 円柱の教材コードを通して skinning の仕組みを低水準で確認します。

ここで最初に押さえたいのは、普段の入口はあくまで glb import だという点です。通常のアプリ開発では、骨とウェイトを手で入力するより、Blender などで作成した glb を `loadModel()` で読み込むほうが自然です。ただし、その高レベル経路の背景で何が起きているかを知らないと、変形が不自然なときに原因を切り分けにくくなります。つまり、この章は「高レベル利用の代わりに低水準だけを使う」ための章ではなく、「高レベル利用の背景を理解するために、低水準で最小原理を一度確かめる」ための章です。

さらに重要なのは、スキニングの中心が「1 頂点を複数の bone 座標系で動かして混ぜること」にある点です。1 つの頂点は 1 本の bone だけで動くとは限りません。境界付近の頂点は、複数の bone の変形結果を重み付きで混ぜた位置へ移動します。`webg` では、CPU 側では `Skeleton.updateMatrixPalette()` が各 bone の変形行列を並べ、GPU 側では `SmoothShader` の頂点シェーダーが bone index と weight を使ってブレンドします。つまり、CPU は「どの bone が今どんな姿勢か」を palette として整え、GPU は「各頂点がその palette のどこをどれだけ使うか」を実行する役割分担になっています。

## 高レベル経路と低レベル経路

普段のアプリ開発では、次のような高レベル経路を使うことが多くなります。

```js
const runtime = await app.loadModel("./character.glb");
runtime.instantiate(app.space);
```

この経路では、joint 階層、inverse bind matrix、頂点ごとの weight、animation clip は、すでに glb 側へ入っています。読者は `Action` や `AnimationState` を組み合わせて再生を進めれば十分です。つまり、通常利用では「bone をどう作るか」よりも、「読み込まれた runtime をどう使うか」が主題になります。 

ただし、次のような場面では高レベル経路だけでは判断しにくくなります。どの bone がどの頂点を引っ張っているか分からないとき、曲がり目が不自然な原因が weight なのか bone なのか分からないとき、`SmoothShader` の skinning 経路が内部で何をしているか把握したいとき、importer が出した skeleton 情報を疑う前に最小原理を自力で確認したいときです。そのため本章では、「高レベル利用の代わりに低レベルだけを使う」のではなく、「高レベル利用の背景を確認するために、低レベルで最小構成を一度作ってみる」という位置づけで説明します。つまり、高レベル経路と低レベル経路は対立するものではなく、理解の層が違うだけです。

## 1 頂点だけでスキニングを考える

まずは、円柱全体や glb 全体ではなく、1 つの頂点だけを考えます。ここで原理を小さくしておくと、あとで多数頂点へ広げたときにも見通しがよくなります。つまり、この節が章全体の基礎になります。

レスト姿勢の頂点を `p_rest` とします。bone0 と bone1 それぞれについて、skinned mesh のローカル座標系で見た現在姿勢の bone 行列を `L0`、`L1`、レスト姿勢から作られた bone offset inverse を `M0^-1`、`M1^-1` とすると、各 bone で変形した頂点は次のように書けます。 

```text
p0_local = L0 * M0^-1 * p_rest
p1_local = L1 * M1^-1 * p_rest
```

ここで weight を `w0`、`w1` とし、`w0 + w1 = 1` になるようにしておくと、最終頂点は次の重み付き平均になります。 

```text
p_final_local = p0_local * w0 + p1_local * w1
```

ここで重要なのは、この段階では scene 全体の world 座標ではなく、skinned mesh 自身のローカル座標系で考えていることです。objectNode 自体の移動や回転がある場合は、その共通変換があとからメッシュ全体へ一括で掛かります。`webg` では、この `L * M^-1` を bone ごとに並べた配列が matrix palette であり、`SmoothShader` はそれを頂点シェーダーでブレンドします。つまり、まず 1 頂点で式を理解し、そのあとでそれが全頂点に対して GPU で同時に実行されると考えると整理しやすくなります。

### CPU 側で確かめる最小例

次のコードは、`Skeleton.updateMatrixPalette()` のあとで `bone.worldMatrix` を current bone 行列として読み、1 頂点だけを CPU 側でブレンドする最小例です。`28_01` はこの helper をそのまま実行して marker を置く page になっています。

```js
function buildSkinMatrix(bone) {
  const skin = bone.worldMatrix.clone();
  skin.mul(bone.getBofMatrix());
  return skin;
}

function transformVertexByBone(restPosition, bone) {
  const skin = buildSkinMatrix(bone);
  return skin.mulVector(restPosition);
}

function blendVertex(restPosition, influences) {
  const out = [0.0, 0.0, 0.0];
  for (let i = 0; i < influences.length; i++) {
    const moved = transformVertexByBone(restPosition, influences[i].bone);
    out[0] += moved[0] * influences[i].weight;
    out[1] += moved[1] * influences[i].weight;
    out[2] += moved[2] * influences[i].weight;
  }
  return out;
}
```

この helper 自体は描画に必須ではありません。実際の描画では GPU 側で同じことが行われます。ただし、教材としては「GPU がやっていることを CPU 側でも再現できる」と確認できるのが重要です。実装上は `Skeleton.updateMatrixPalette()` が `bone.worldMatrix` を skinned mesh ローカル座標系の current pose へ更新し、`SmoothShader` が同じ情報から matrix palette を使ってブレンドします。つまり、この helper は GPU 側の処理を人間が追いやすい形へいったん取り出したものです。

実際の page では、次の順で呼ぶと本文の式とそのまま対応づけられます。

```js
skeleton.updateMatrixPalette();

const rootOnlyLocal = transformVertexByBone(restPosition, rootBone);
const childOnlyLocal = transformVertexByBone(restPosition, childBone);
const blendedLocal = blendVertex(restPosition, [
  { bone: rootBone, weight: w0 },
  { bone: childBone, weight: w1 }
]);
```

この段階で得られるのは object local 座標です。画面上へ marker を置いて確認したいときは、最後に objectNode の world 行列を 1 回だけ掛けます。

```js
const rootOnlyWorld = objectNode.getWorldMatrix().mulVector(rootOnlyLocal);
const childOnlyWorld = objectNode.getWorldMatrix().mulVector(childOnlyLocal);
const blendedWorld = objectNode.getWorldMatrix().mulVector(blendedLocal);
```

## 2-bone 円柱を低水準で作る

仕組みが分かったら、次は頂点群へ広げます。題材は「下から上へ細かく分割した円柱側面」です。この教材では、下ほど `rootBone`、上ほど `childBone` の influence を強くすることで、weight の意味を視覚的に追いやすくします。つまり、「どの bone がどのあたりを引っ張るか」が分かりやすい最小教材として円柱側面を選んでいます。

1. `Shape` で円柱側面の頂点を並べる
2. `Skeleton` に root bone と child bone を作る
3. 各頂点に `addVertexWeight()` で 2 本分の weight を入れる
4. `shape.endShape()` で GPU 用バッファへ変換する
5. `SmoothShader` で `has_bone: 1` を有効にして描画する 

次のコードは、その最小教材版です。 

```js
import Shape from "./webg/Shape.js";
import Skeleton from "./webg/Skeleton.js";

function createTwoBoneCylinder(gpu, options = {}) {
  const height = Number(options.height ?? 8.0);
  const radius = Number(options.radius ?? 0.45);
  const rings = Math.max(2, Math.floor(options.rings ?? 14));
  const segments = Math.max(3, Math.floor(options.segments ?? 24));

  const shape = new Shape(gpu);
  shape.setAutoCalcNormals(true);

  const skeleton = new Skeleton();
  shape.setSkeleton(skeleton);

  const rootBone = skeleton.addBone(null, "rootBone");
  const childBone = skeleton.addBone(rootBone, "childBone");

  rootBone.setRestPosition(0.0, 0.0, 0.0);
  childBone.setRestPosition(0.0, height * 0.5, 0.0);

  skeleton.bindRestPose();
  skeleton.setBoneOrder(["rootBone", "childBone"]);

  for (let ring = 0; ring <= rings; ring++) {
    const v = ring / rings;
    const y = height * v;
    for (let segment = 0; segment < segments; segment++) {
      const u = segment / segments;
      const angle = u * Math.PI * 2.0;
      const x = Math.cos(angle) * radius;
      const z = -Math.sin(angle) * radius;

      const vIndex = shape.addVertexUV(x, y, z, u, v) - 1;

      // 下ほど rootBone、上ほど childBone の influence を強くする
      shape.addVertexWeight(vIndex, 0, 1.0 - v);
      shape.addVertexWeight(vIndex, 1, v);
    }
  }

  for (let ring = 0; ring < rings; ring++) {
    const row0 = ring * segments;
    const row1 = (ring + 1) * segments;
    for (let segment = 0; segment < segments; segment++) {
      const next = (segment + 1) % segments;
      const a = row0 + segment;
      const b = row0 + next;
      const c = row1 + segment;
      const d = row1 + next;
      shape.addTriangle(a, c, b);
      shape.addTriangle(b, c, d);
    }
  }

  shape.endShape();
  return { shape, skeleton, rootBone, childBone };
}
```

このコードで重要なのは 3 つあります。`Skeleton` は bone の親子関係だけでは終わらず、`setRestPosition()` でレスト姿勢を置き、`bindRestPose()` で bone offset inverse を再計算し、`setBoneOrder()` で shader が読む palette 順を確定すること。`addVertexWeight()` は「頂点がどの bone にどれだけ引っ張られるか」を入れること。円柱側面だけに絞ることで、スキニングの観察がしやすくなることです。とくに `Shape.endShape()` は、スキニングメッシュの場合、bone index と weight を別の頂点属性スロットへ詰めます。`SmoothShader` の skinning 経路はこの構成を前提にしています。

## GPU 側では何が起きているか

CPU 側で `Skeleton.updateMatrixPalette()` が呼ばれると、bone ごとの `currentBoneMatrixLocal * inverseRestGlobal` が 12 float 単位で並びます。ここで CPU 側の役割は終わりではなく、むしろ GPU へ渡すための準備が整った状態だと考えると分かりやすくなります。

`SmoothShader` の頂点シェーダーは、各頂点の `index` と `weight` を読み、4 本までの bone 行列を重み付きで合成して、最終的な skinning 行列を作ります。その行列を頂点位置と法線へ掛けてから、通常の static mesh と同じように view / projection へ流します。つまり、28.2 節で 1 頂点に対して手で書いた式が、ここでは多数の頂点に対して GPU で並列実行されていると考えれば十分です。

重要なのは、「bone ごとに頂点を個別に描き直している」のではなく、「1 頂点について 4 本までの influence を 1 回の頂点シェーダー実行でブレンドしている」ことです。そのため、切り分けるときは次の順で考えると迷いにくくなります。`Shape` に bone index / weight が入っているか。`Skeleton` の rest pose と bone order が正しいか。`Skeleton.updateMatrixPalette()` が current pose を反映しているか。`shape.setMaterial("smooth-shader", { has_bone: 1 })` が有効か。`weight_debug` を使うと influence 分布が意図どおりか。つまり、問題を shader だけ、あるいは importer だけのせいにせず、「頂点属性」「skeleton」「palette」「shader 設定」の順に切り分けると理解しやすくなります。

## 検証用 example を 2 本に分ける

この章では、検証対象を混ぜないために example を 2 本へ分けます。ここでの考え方は、「数式が正しいか」と「メッシュ全体の見え方が正しいか」を同じページで一度に見ようとしないことです。つまり、役割を分けた 2 本立てにすることで、どこを確認しているのかが明確になります。

### `28_01`: 1 頂点の原理を確認する

`28_01` は bone と marker だけを表示し、代表頂点 1 個について、rest 位置、`bone0` だけで変形した位置、`bone1` だけで変形した位置、weight 付き平均を取った最終位置の 4 点を比較するための例です。この page では円柱メッシュを出さず、数式どおりに marker が動くかを最優先で確認します。見やすさのため、代表点は円柱軸から少し離した probe point を使いますが、検証している原理自体は同じです。visible marker 自体は、前節の CPU helper をそのまま実行して得た座標を表示しています。つまり、`28_01` は「本文の式がそのまま画面上で確かめられる」ことを主眼にした数式検証用ページです。 

### `28_02`: メッシュ全体の見え方を確認する

`28_02` は円柱メッシュの見え方だけを確認する例です。marker は出さず、`SmoothShader` による曲がり方と、`weight_debug` を切り替えたときの分布だけを見ます。つまり、こちらは「数式の正しさ」ではなく「GPU による skinning 結果の見え方」を確認するためのページです。`28_01` は「1 頂点の原理検証」、`28_02` は「メッシュ全体の見え方検証」という役割分担です。この分離によって、数式と描画結果のどちらに問題があるのかを切り分けやすくなります。 

## Blender / glb の高レベル経路へ戻す

ここまでの低水準例を見たあとで glb の高レベル経路へ戻ると、Blender で行っている作業の意味が読みやすくなります。つまり、高レベル経路は別の仕組みではなく、今まで手でやっていたことを importer が肩代わりしてくれているだけだと理解できます。

Blender 側でやっているのは、概ね次の内容です。bone 階層を作ること。各 bone のレスト姿勢を決めること。頂点ごとにどの bone がどれだけ効くかを weight paint すること。animation clip としてボーンごとの時系列姿勢を持つこと。glb を import すると、`webg` はこれらを `ModelAsset`、`Skeleton`、`Animation` として取り込みます。したがって、高レベル経路は「低水準の仕組みを使わない」のではなく、「低水準で手で書いていた部分を importer が肩代わりしてくれる」と考えるのが実態に近くなります。つまり、この章の低水準教材は、高レベル import の意味を読み解くための下敷きです。 

## よくあるつまずき

`has_bone: 1` を入れたのに曲がらない場合は、bone index / weight が shape へ入っていないか、skeleton の palette 順と vertex 側 index が対応していない可能性があります。つまり、最初に疑うべきは shader よりも「頂点属性」と「bone 順」の整合です。

bone は動いているのにメッシュだけレスト姿勢のままなら、skinning 対応 shader ではなく static mesh 向け設定のまま描いている可能性があります。skinned mesh では `BonePhong`、`BoneNormPhong`、または skinning 対応の `SmoothShader` 経路を使い、`has_bone: 1` が有効である必要があります。

曲がるが境界が折れすぎる場合は、weight を極端に切り替えすぎている可能性があります。まず `weight_debug` で分布を見て、境界付近が 0 / 1 だけになっていないかを確認します。つまり、見え方の問題を bone の回転だけで説明しようとせず、weight 分布も同時に見るのが大切です。 

複雑な glb で問題が起きたときは、animation、skeleton 名、material、texture、node scale のどこに原因があるか分かりにくくなります。そんなときは importer を疑う前に、この 2-bone 円柱の最小例へ戻ると、どの層を疑うべきか切り分けやすくなります。つまり、この章の教材は importer の代わりではなく、切り分けの基準として役立ちます。

## まとめ

この章で最も大事なのは、スキニングを「bone がメッシュを動かす魔法の仕組み」として見ないことです。本質は、1 頂点を複数の bone 座標系で変形し、その結果を重み付きで混ぜることにあります。CPU 側では `Skeleton` がそのための matrix palette を作り、GPU 側では `SmoothShader` が各頂点ごとに blend を行います。つまり、低水準で見た 1 頂点の式と、高レベルで使う glb import は本質的に同じ仕組みの別の見え方です。

また、この章では 2-bone 円柱という最小教材を通して、「レスト姿勢」「inverse bind」「weight」「palette 順」「shader 設定」という切り分け軸をはっきりさせています。複雑な glb がうまく動かないときでも、まずこの最小原理へ戻れると、どの層に問題があるのかを判断しやすくなります。つまり、この章はスキニングを手で実装するためだけでなく、高レベル import の結果を正しく読むための基礎章として位置づけると理解しやすくなります。
