# タイルマップの使い方

TileMap を build したあと、本当に大事になるのはランタイム側の運用です。どの cell にいるかをどう持つか、node をどこへ置くか、表示窓をどう追従させるか、クリック選択や path 探索をどうアプリへつなぐかといった処理は、構築後にはじめて現れます。本章では、build 後の TileMap をアプリ側からどう使うかを、処理フローに沿って見ていきます。 

ここで最初に押さえたいのは、TileMap ランタイムの API が意図的に少数へ絞られていることです。`getTile()`、`canMove()`、`findPath()`、`followCell()`、`pickCell()`、`placeNodeOnCell()` を押さえると、現在の TileMap サンプルと unittest の大半を追えます。また、TileMap を使うときは論理位置と見た目位置を分けて持つほうが安全です。cell 上の論理位置はアプリ側の state として持ち、見た目のワールド座標や tween は `getNodePositionOnCell()` や `placeNodeOnCell()` で解決するほうが混乱しにくくなります。さらに重要なのは、表示窓の更新と表示更新は別手順だということです。`followCell()` や `setDisplayArea()` で矩形を変えたあと、`refreshTileColors()` で現在の見え方を更新する、という 2 段を忘れないことが大切です。

## runtime API の全体像

TileMap の public API は多くありません。まずは次のものを押さえると、ほとんどのサンプルを読めます。ここで重要なのは、これらの API が「盤面データを読んで判定するもの」と「見た目の world 座標へ変換するもの」に分かれている点です。つまり、TileMap は論理盤面の基準を返し、アプリ側はそれを使って見た目や入力を組み立てます。 

| API                                         | 役割                               | まず使う場面                     |
| ------------------------------------------- | -------------------------------- | -------------------------- |
| `getTile(x, y)`                             | cell を読む                         | 現在位置や goal の取得             |
| `getHeight(cell)`                           | height だけを読む                     | HUD や移動条件の表示               |
| `getWorldPosition(cell)` / `toWorld(cell)`  | cell center のワールド座標を得る           | camera target や effect の配置 |
| `getNodePositionOnCell(cell, lift, radius)` | node を置く位置を返す                    | 移動アニメーションの目標位置             |
| `placeNodeOnCell(node, cell, lift, radius)` | node をその場へ置く                     | 初期配置や teleport             |
| `canMove(from, to)`                         | 隣接 cell 間の移動可否を返す                | arrow 移動の判定                |
| `findPath(start, goal)`                     | path を返す                         | ルート表示や AI の下地              |
| `setDisplayArea(rect)`                      | 可視範囲を更新する                        | スクロールを手で制御するとき             |
| `followCell(cell, options)`                 | cell を基準に可視範囲を追従させる              | player 追従                  |
| `pickCell(origin, dir)`                     | ray から hit cell を返す              | クリック選択                     |
| `refreshTileColors(selected)`               | 選択色と `displayArea` に応じた表示状態を更新する | 選択変更後とスクロール後               |

この一覧だけを見ると API が少なく感じられますが、そのぶん責務がはっきりしています。TileMap のランタイムは、盤面を大きく変形するのではなく、「今どの cell を見ているか」「そこへ進めるか」「そこをどう見せるか」を返す helper として使うほうが current 実装に合っています。 

## 論理位置と見た目位置を分ける

TileMap を使うときに最も大事なのは、論理位置と見た目位置を混ぜないことです。player や marker の位置をワールド座標へ直接書くより、「今いる cell」を持っておくほうが扱いやすくなります。つまり、論理位置は TileMap の cell で持ち、見た目の node 座標はその都度 TileMap に計算させる、という分担にすると混乱しにくくなります。

```js
let playerCell = tileMap.getTile(1, 1);
tileMap.placeNodeOnCell(playerNode, playerCell, 0.08, 0.50);
```

`placeNodeOnCell()` は即時に位置を反映します。動きを滑らかに見せたいときは、目標位置だけ TileMap から受け取り、node 側のアニメーションを使います。つまり TileMap 自体は「移動中」の時間経過を持たず、論理位置と world 座標をつなぐだけです。見た目の時間変化は node 側の責務になります。 

```js
const nextCell = tileMap.getTile(playerCell.col + 1, playerCell.row);
if (!tileMap.canMove(playerCell, nextCell)) {
  return;
}

const targetPosition = tileMap.getNodePositionOnCell(nextCell, 0.08, 0.50);
playerNode.animatePosition(targetPosition, {
  durationMs: 250
});
playerCell = nextCell;
```

ここで大事なのは、`playerCell` の更新と node のアニメーションを別に持つことです。TileMap は「どの cell にいるか」を管理し、node は「どう見せるか」を管理すると考えると整理しやすくなります。 

## 盤面上にオブジェクトを置く

TileMap 上へ object を置くときは、`getNodePositionOnCell()` と `placeNodeOnCell()` を中心に考えると分かりやすくなります。前者は world 座標だけを返し、後者は node へ即時適用します。初期配置や teleport なら `placeNodeOnCell()`、移動先だけ知りたいなら `getNodePositionOnCell()` という使い分けです。 

特に、effect や camera target を cell 上へ置くときは `getWorldPosition(cell)` / `toWorld(cell)` も役立ちます。これらは node の足元補正や半径補正を含まず、cell center の world 座標だけを返します。つまり、TileMap は盤面上の「配置の基準点」を複数の粒度で返してくれる、と考えると使い分けやすくなります。 

## 移動 animation を組み込む

### `TileMap` と `Node.animatePosition()` の役割分担

TileMap を使うアプリでは、「入力を受けたフレームで次の cell を決める」「見た目は 200ms から 300ms くらいかけて次の cell へ移る」という構成が頻出します。ここで迷いやすいのは、TileMap 自体が時間経過を持たないことです。`canMove()`、`getNodePositionOnCell()`、`followCell()` はすべて即時の計算であり、移動アニメーションを進めるのは `Node.animatePosition()` と `Space.update(deltaMs)` の役目です。つまり、TileMap ランタイムは「次にどこへ行けるか」は返しますが、「何ミリ秒かけて見せるか」は返しません。ここを最初に分けておくと、TileMap と animation の責務を混同しにくくなります。

`WebgApp.start()` を使う場合は、毎フレームの順序がすでに決まっています。`onUpdate(ctx)` が呼ばれたあとに、`WebgApp` 側で `space.update(deltaMs)` が自動で呼ばれます。つまりサンプル側では、`onUpdate()` の中で移動要求を受け、必要なら `animatePosition()` を開始するだけで十分です。逆に、`WebgApp` を使わず自分で loop を組む場合は、各フレームで `space.update(deltaMs)` を必ず呼ばないと tween が進みません。通常経路では `WebgApp` が時間進行を引き受け、例外的な独自ループでは自分で進行を管理することになります。 

`webg/samples/tile_sim` のような TileMap サンプルで、盤面処理の流れだけに絞ると、次のようになります。ここで見てほしいのは、TileMap が論理位置と world 目標座標を返し、アニメーションそのものは node 側で進めていることです。盤面処理と見た目の動きを分けて保つ構成になっています。

```js
let playerCell = tileMap.getTile(3, 3);
let moveTween = null;

tileMap.placeNodeOnCell(playerNode, playerCell, 0.08, 0.50);
tileMap.refreshTileColors();

function tryMoveBy(dx, dy) {
  if (moveTween && !moveTween.isFinished()) {
    return false;
  }

  const nextCell = tileMap.getTile(playerCell.col + dx, playerCell.row + dy);
  if (!tileMap.canMove(playerCell, nextCell)) {
    return false;
  }

  const targetPosition = tileMap.getNodePositionOnCell(nextCell, 0.08, 0.50);
  tileMap.followCell(nextCell, {
    width: 8,
    height: 6,
    followCol: 2,
    followRow: 2
  });
  tileMap.refreshTileColors();

  moveTween = playerNode.animatePosition(targetPosition, {
    durationMs: 250,
    easing: "outCubic",
    onComplete: () => {
      moveTween = null;
      playerCell = nextCell;
      tileMap.refreshTileColors();
    }
  });
  return true;
}

app.start({
  onUpdate: ({ deltaSec }) => {
    if (app.wasActionPressed("move_up")) {
      tryMoveBy(0, -1);
    }
    if (app.wasActionPressed("move_down")) {
      tryMoveBy(0, 1);
    }
    if (app.wasActionPressed("move_left")) {
      tryMoveBy(-1, 0);
    }
    if (app.wasActionPressed("move_right")) {
      tryMoveBy(1, 0);
    }
  }
});
```

この例では `deltaSec` を直接使っていませんが、問題ありません。理由は、`animatePosition()` が開始されたあと、その進行は `WebgApp` が内部で呼ぶ `space.update(deltaMs)` に任せているためです。`onUpdate()` の役割は「この frame に新しい移動を始めるか」を決める入口として使うことです。 

### player と `displayArea` の時間を分ける

TileMap のアプリでは、player 本体と表示範囲を同じ速度で動かすと、盤面全体が同時に滑って見えて落ち着かないことがあります。そのため、移動演出と `displayArea` の追従は別の時間軸として持つほうが見え方を整えやすくなります。これは仕様上の必須条件ではありませんが、利用者視点ではかなり有効です。つまり、「論理位置の更新」「player の移動演出」「表示窓の追従」は同じ出来事から始まっても、同じ時間で動かす必要はありません。 

実装上は、player の移動 tween と、camera target や camera pivot の tween を別オブジェクトとして管理します。TileMap は `followCell()` で「次にどの範囲を見せるべきか」を返し、camera 自体の補間はサンプル側が持つ形にしておくと、後から速度だけを調整しやすくなります。ここでも TileMap は「盤面上のどこを見るか」を返し、時間のかかる見せ方はアプリ側が持つ、という分担が保たれています。 

## `displayArea` をスクロールさせる

広いマップで盤面の一部だけを見せるときは、`followCell()` を使うと現在の cell を基準に可視範囲を追従させられます。ここで大事なのは、`displayArea` の更新と見た目の更新は同じ処理ではないことです。矩形を変えるだけでは node の表示状態は変わらないため、`refreshTileColors()` を続けて呼ぶ必要があります。つまり、この節の核心は「矩形を変える手順」と「見た目を更新する手順」を 2 段で理解することです。 

```js
const displayArea = tileMap.followCell(playerCell, {
  width: 8,
  height: 6,
  followCol: 2,
  followRow: 2
});

tileMap.refreshTileColors(selectedHit);
```

`followCol` と `followRow` は、「追従対象 cell を表示窓の左上から何 cell 目に置くか」を表します。たとえば `followCol: 2` なら、player は常に窓の左から 3 列目付近へ寄るように見えます。広い盤面で player を完全中央に固定するより、少し前方や手前へ寄せたほうが読みやすい場面では、この値が効きます。つまり `followCell()` は単なる「中心寄せ」ではなく、「窓の中のどこに対象を置くか」を決める helper だと考えると整理しやすくなります。 

ここで忘れやすいのが `refreshTileColors()` です。`setDisplayArea()` や `followCell()` を呼んだだけでは node の表示状態は更新されません。可視範囲を変えた直後に `refreshTileColors()` を呼ぶと、現在の窓に合わせて表示対象と選択色がそろいます。つまり、「表示窓を変えたのに画面が追従しない」という問題の多くは、この 2 段目を忘れていることにあります。

## クリックで cell を選ぶ

TileMap の `pickCell(origin, dir)` は、ray と cell box の交差から最も近い cell を返します。戻り値には `cell`、`point`、`hitFace`、`hitHeight` が入ります。ここで大事なのは、TileMap 自体が「選択状態」を持つのではなく、選択された hit をアプリ側が保持し、その見た目の更新だけを `refreshTileColors()` に委ねることです。つまり、選択の意味はアプリ側、色更新は TileMap helper 側、という分担になります。

```js
const hit = tileMap.pickCell(ray.origin, ray.dir);
if (!hit) {
  return;
}

if (hit.hitFace === "top") {
  selectedHit = hit;
} else {
  selectedHit = hit;
}
tileMap.refreshTileColors(selectedHit);
```

`hitFace` は `top`、`bottom`、`wall-x+`、`wall-x-`、`wall-y+`、`wall-y-` のどれかです。top を選択として使い、wall は別の編集操作に使う、という分け方がしやすくなっています。`webg/samples/tile_sim` と `webg/unittest/tilemap` は、ここをクリック処理の中心にしています。つまり、選択そのものはアプリ側の state として持ちつつ、TileMap は「どの cell に当たったか」と「その状態に応じてどう色を付けるか」を助ける部品として使うのが自然です。

## 移動判定と path 探索

`canMove(from, to)` は「隣接 cell 間で高さ差が 1 以内なら進める」という基本ルールです。さらに `terrain === "wall"` の cell は blocked として扱われます。ここで重要なのは、手動移動と path 探索が同じ規則を使うことです。つまり、見た目の段差、操作入力、AI のルート決定が同じ盤面ルールへそろうようになっています。この一貫性があるため、TileMap を盤面の唯一の基準として扱いやすくなります。 

```js
const nextCell = tileMap.getTile(playerCell.col, playerCell.row - 1);
if (tileMap.canMove(playerCell, nextCell)) {
  playerCell = nextCell;
}
```

最短経路が必要なときは `findPath(start, goal)` を使います。

```js
const path = tileMap.findPath(playerCell, goalCell);
if (path) {
  console.log(path.map((cell) => `(${cell.col},${cell.row})`).join(" -> "));
}
```

この path も `canMove()` と同じルールを使います。つまり、見た目の段差、手動移動、AI のルートが同じ基準にそろいます。ここが TileMap の運用上かなり重要で、移動判定と経路探索が別ルールになると、プレイヤーは進めるのに AI は進めない、といったずれが出やすくなります。 

## 座標の読み方を混ぜない

TileMap を扱うときに混乱しやすいのは、配列の並び、方向名、入力名が別物だという点です。つまり「データの並び順」「盤面上の列と行」「キーボード入力としての上方向」を同じものと考えないほうが安全です。この区別を最初に持っておくと、camera-relative movement の節もかなり読みやすくなります。 

| 項目                | 意味       | どう読むか             |
| ----------------- | -------- | ----------------- |
| `tiles[y][x]`     | data の並び | 左上から右へ、次に下へ読む     |
| `x` / `col`       | 列        | 東西方向の index       |
| `y` / `row`       | 行        | 南北方向の index       |
| `ArrowUp`         | input 名  | row の増減とは同義ではない   |
| `north` / `south` | 方向名      | 配列 index そのものではない |

特に orbit camera を使うサンプルでは、`ArrowUp` を押したからといって常に `row - 1` になるわけではありません。画面上の上方向へ動くために camera の向きに応じて grid 移動を変換する場合は、`resolveCameraRelativeGridMove(yawDeg, key)` を使います。ここで大事なのは、この helper が TileMap の内部座標を変えるのではなく、「入力の見え方」を camera に合わせて変換しているだけだという点です。TileMap の論理盤面自体はそのまま保たれています。

```js
import { resolveCameraRelativeGridMove } from "./webg/TileMap.js";

const relativeMove = resolveCameraRelativeGridMove(orbitYaw, "arrowup");
const nextCell = tileMap.getTile(
  playerCell.col + relativeMove.dx,
  playerCell.row + relativeMove.dy
);
```

このヘルパーは TileMap の内部座標を変えるものではなく、「入力の見え方」を camera に合わせるヘルパーです。`webg/samples/tile_sim` と `webg/unittest/tilemap` がこの使い方の実例です。つまり、座標系を変換しているのではなく、入力解釈を変えているだけだと理解すると整理しやすくなります。 

## `edgeCutMask` と見た目の整理

TileMap の見た目は、高さのある box をそのまま最後まで texture 付きで描く形ではありません。構築の途中で、各 cell は周囲 4 方向との高低差から `edgeCutMask` を持ちます。east / west / north / south のどの辺を切るかを 4 bit で表し、renderer はその bit を見て上面の辺を内側へ寄せます。ここで runtime 利用者として重要なのは、「見た目は edge を削って整理しているが、論理 box は full height のまま残る」という点です。つまり、描画と判定で役割が分かれています。 

同時に、描画用 shape 自体は top から `visibleHeightSteps` 分だけを薄い四角錐台の cap として描き、その下側は texture を使わない foundation で埋めます。これにより、背の高い cell でも縦方向の texture 情報が増えすぎず、斜め視点で背景が抜けにくい形を保ちながら、上面まわりの形を読み取りやすくします。ここで大事なのは、見た目を薄くしても TileMap の論理 box は full height のまま残ることです。`pickCell()` や `canMove()` は cell の高さ情報をそのまま使い、描画だけを整理しています。つまり、`edgeCutMask` はまず見た目のための値であり、利用者が最初に追うべきなのは mask の算術そのものより「描画と判定が別責務だ」という点です。

削れ量そのものは、実装上は `TILE_SURFACE_INSET_RATIO = 0.4` を使って決めています。ただし、この `0.4` は cell 全幅を `1.0` とした比率ではありません。実際には `width = cell.box.maxx - cell.box.minx`、`halfWidth = width * 0.5`、`halfDepth = depth * 0.5` を先に求め、そのうえで `inset = Math.min(halfWidth, halfDepth) * 0.4` としているため、「cell の中心から辺までの半幅を `1.0` としたときの `0.4`」です。正方形 cell なら、セル全幅に対しては 20% 分だけ辺を内側へ寄せる意味になります。これは見た目の説明として押さえておくと便利ですが、通常の運用では mask 自体を手で作る必要はありません。 

ここでの `width` や `depth` の単位は、TileMap 専用の別単位ではなくワールド座標の長さです。`cellSize` と `cellGap` から決まる cell の実寸そのものなので、既定値の `cellSize = 4.0`、`cellGap = 0.18` なら、cell の実幅は `4.0 - 0.18 = 3.82` ワールド unit、半幅は `1.91` ワールド unit になります。この場合の削れ量は `1.91 * 0.4 = 0.764` ワールド unit です。以前の説明で「削れ 0.2」と書いていたのは、cell 全幅に対して 20% という意味で、実装の `0.4` はその半幅基準の値に対応しています。runtime 利用者としては、「削れ量は world 座標系の長さで決まり、論理盤面とは別の描画表現だ」と読むと整理しやすくなります。 

UV の考え方も、この cap と foundation の役割分担に合わせています。cap の上面だけが、cell 外形を `0..1` に正規化した top projection を使います。つまり、削れが `0.2` なら、上面の端は `U=0.2` や `U=0.8` をそのまま使い、上面だけを再び `0..1` へ引き延ばしません。こうしておくと、上面の切り込み幅を texture 上でもそのまま読めます。一方で側面は、面ごとに独立した単純な UV を使います。ここでは「どの面を見ているか」が分かりやすいことを優先し、上からの投影を共有しません。つまり、見た目の簡潔さを優先した貼り方になっています。runtime 利用者として重要なのは、「TileMap の見た目は通常の shape より少し専用化されている」という理解です。 

foundation 側は top material の texture / normal map を流用しつつ、側面 UV だけを面正面基準の矩形へ切り替えています。つまり、まず左下 `(0,0)`、右上 `(1,1)` の面 UV を置き、そのあと foundation の高さに応じて上端 `V` を補正します。色だけ少し暗くしているのは、上面の主役感を残したまま土台と認識しやすくするためです。ここも runtime 利用者としては、「cap が主役で、foundation は補助」と読むと整理しやすくなります。 

また、TileMap 実装を読むときに見落としやすいのが、`Shape` 既定の texture mapping mode です。`Shape` は既定では sphere 用の mode `0` を持っており、このまま明示的に指定した UV を入れると triangle 追加時の seam 補正が働き、手動で指定した UV の `0..1` 境界まで書き換えてしまいます。TileMap では top face と側面の UV を自前で決めているため、shape 作成時に mode `1` を明示し、旧来の seam 補正を止めてから頂点を積んでいます。さらに、TileMap の論理的な UV は `webg` 全体の前提どおり Blender と同じ bottom-left 基準です。ここは top-left 基準へ変えるのではなく、画像テクスチャを確認用に貼るときに必要なら `flipV` を画像入力側で入れます。論理的な UV と画像の上下補正を混ぜると、削れた上面の `topuv` 計算まで壊れやすくなるため、役割を分けて考えるほうが安全です。通常ここを直接書き換える必要はありませんが、「なぜ TileMap の見た目が普通の cube と少し違うか」を理解する助けになります。 

さらに、面の表裏はヘルパー間で暗黙反転させず、`addPrismSlice()` や smooth 用の shared-vertex cap 定義時点で外側 CCW にそろえるのが現在の前提です。たとえば north face は `[topNE, bottomNE, bottomNW, topNW]`、top face は `[topNW, topSW, topSE, topNE]` のように、実際にその面を外から見たときに CCW になる順で書いてあります。ここを前提にしておくと、法線方向、`frontFace: "ccw"`、foundation 側面 UV の「左下 `(0,0)` / 右上 `(1,1)`」の対応を同じ読み方で追えます。これも主に実装読解の補助ですが、TileMap の見た目と判定が別層で構成されていることを理解する助けになります。 

重要なのは、利用者が通常この値を直接書かなくてよいことです。TileMap は `assignEdgeCutMasks()` で自動計算し、`applySurfaceShapes()` で見た目へ反映します。利用者側で必要になるのは、次の 2 つくらいです。`getEdgeCutMask(cell)` で今の cell がどの削れ方か知りたいとき、`describeEdgeCutMask(mask)` で HUD やデバッグテキストに短い表記を出したいときです。形状そのものを確認したいときは、まず `height` と `terrain` の定義、`edgeCutMask` の説明、実際の表示結果を順に見比べるほうが整理しやすくなります。通常のゲームやサンプルでは、mask を手で持つより TileMap に計算を任せるほうが安全です。 

## よくあるつまずき

`tiles` を一部だけ書いて build に失敗する場合、TileMap は盤面の欠けを補完しません。`width` と `height` を決めたら、その範囲を全部埋める必要があります。また、`tiles` と `generator` を同時に書くことはできません。どちらか一方だけを使います。固定マップと自動生成を混ぜたくなったら、最終的な `tiles` を自分で作ってから TileMap へ渡すほうが安全です。 

`displayArea` を動かしたのに表示範囲が変わらない場合は、`setDisplayArea()` や `followCell()` のあとで `refreshTileColors()` を呼び忘れていることが多くなります。矩形変更と見た目更新は別処理です。また `canMove()` を呼んだのに object が動かない場合は、`canMove()` が `true` / `false` を返すだけであり、node の位置更新は `placeNodeOnCell()` か `animatePosition()` を使って別に行う必要があります。つまり、TileMap の判定と見た目の更新は別責務です。 

日本語の説明を TileMap に直接持たせたくなることもありますが、TileMap は盤面データと判定の部品です。チュートリアル文や会話文は `DialogueOverlay` や `UIPanel` に分けるほうが自然です。表示系の使い分けは第14章や第15章の整理とつながります。 

glb の自キャラを載せたのにアニメーションが進まない場合は、`animatePosition()` の完了だけ見ていて `AnimationState.update()` を毎フレーム呼んでいない可能性があります。あるいは `WebgApp.start()` を使っていない場合に `space.update(deltaMs)` を呼んでいない可能性があります。位置 tween と骨アニメーションは、どちらもフレームループに乗せて継続進行させる必要があります。TileMap 上のキャラクターは「盤面上に置く処理」と「毎フレーム進める処理」の両方が必要です。

## 例を通して確かめる

`webg/samples/tile_sim` は TileMap とアニメーション付きキャラクターを組み合わせた例です。`WebgApp`、`Scene JSON`、`terrainMaterials`、camera-relative movement、`followCell()`、`pickCell()`、戦術盤面上での actor 配置と更新の分担をまとめて確かめられます。実運用を見るなら最初の参照先として最も分かりやすく、TileMap を「盤面基盤」としてどう使うかが見えやすい sample です。

`webg/unittest/tilemap` は TileMap の API を小さく追いたいときに役立つ例です。`canMove()`、`findPath()`、`pickCell()`、`displayArea`、node 配置の関係を読みやすく確かめられます。最初に API の動きだけを確認したいなら、こちらから読むほうが整理しやすくなります。 

`webg/samples/animation_state` は 1 つの glb アセットと `Action` / `AnimationState` をどうつなぐかを見る例です。TileMap と直接は結びついていませんが、「アニメーションの開始地点」と「毎フレームの更新地点」を分ける考え方はそのまま使えます。TileMap 上の actor に骨アニメーションを載せるときの考え方の補助として見ると役立ちます。`webg/samples/janken` は、同じ glb を 2 体配置し、それぞれ独立したステートマシンを持たせる例です。TileMap 上で複数のアニメーション付きキャラクターを置きたいときに見比べやすくなります。

この章で最も大事なのは、TileMap ランタイムを「論理位置を返す部品」として扱い、node や animation はその結果を受けて見た目を作る、と分けて考えることです。`getTile()`、`canMove()`、`findPath()`、`followCell()`、`pickCell()`、`placeNodeOnCell()` を基準にしておくと、TileMap を使うサンプルの処理フローはかなり読みやすくなります。次の第24章では、その考え方を `tile_sim` という実例の中でさらに具体化していきます。
