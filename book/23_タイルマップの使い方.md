# タイルマップの使い方

TileMap をビルドした後に極めて重要となるのが、ランタイムでの運用です。「どのセルにキャラクターが存在するかをどう管理するか」「ノードをどこに配置するか」「表示ウィンドウをどう追従させるか」、そして「クリックによる選択やパス探索の結果をどうアプリケーションに連携させるか」といった処理は、構築が終わった後の実装フェーズで初めて現れる課題です。本章では、ビルド後の TileMap をアプリケーション側からどのように利用するかを、処理フローに沿って解説します。

まず押さえておきたいのは、TileMap ランタイムの API が意図的に少数に絞られている点です。`getTile()`、`canMove()`、`findPath()`、`followCell()`、`pickCell()`、`placeNodeOnCell()` の主要な API を理解すれば、現在の TileMap サンプルやユニットテストの大部分を把握できます。

また、TileMap を利用する際は「論理的な位置」と「視覚的な位置」を分けて管理することが推奨されます。セル上の論理位置はアプリケーション側の状態（State）として保持し、見た目のワールド座標や Tween アニメーションについては `getNodePositionOnCell()` や `placeNodeOnCell()` を用いて解決することで、実装の混乱を防ぐことができます。さらに重要な点として、表示範囲の更新と描画の更新は別手順であることに注意してください。`followCell()` や `setDisplayArea()` で矩形範囲を変更した後、`refreshTileColors()` を呼び出して現在の見え方を更新するという「2段階の手順」を忘れないことが大切です。

## ランタイム API の全体像

TileMap のパブリック API はシンプルに設計されています。これらの API は大きく分けて「盤面データを参照して判定を行うもの」と「視覚的なワールド座標へ変換するもの」の2つの役割に分類されます。TileMap は論理盤面の基準値を返し、アプリケーション側がそれを用いて視覚表現や入力処理を組み立てるという役割分担になっています。

| API | 役割 |
| :--- | :--- | :--- |
| `getTile(x, y)` | セル情報を取得する |
| `getHeight(cell)` | 高さ情報のみを取得する |
| `getWorldPosition(cell)` / `toWorld(cell)` | セル中心のワールド座標を取得する |
| `getNodePositionOnCell(cell, lift, radius)` | ノードを配置すべき位置を返す |
| `placeNodeOnCell(node, cell, lift, radius)` | ノードを指定位置へ即座に配置する |
| `canMove(from, to)` | 隣接セル間の移動可否を判定する |
| `findPath(start, goal)` | パス（経路）を返す |
| `setDisplayArea(rect)` | 可視範囲を更新する |
| `followCell(cell, options)` | 指定セルを基準に可視範囲を追従させる |
| `pickCell(origin, dir)` | レイキャストからヒットしたセルを返す |
| `refreshTileColors(selected)` | 選択色と可視範囲に応じた表示状態を更新する |

API の数は少ないですが、その分役割が明確です。TileMap ランタイムは盤面を動的に変形させるものではなく、「今どのセルを参照しているか」「そこへ移動可能か」「それをどう見せるか」を回答するヘルパーとして利用するのが、現在の実装に最も適した運用方法です。

## 論理的な位置と視覚的な位置の分離

![論理位置と見た目位置の分離図](fig23_01_tile_logic_visual.jpg)

*TileMap では、論理位置と見た目位置を分けて考えることで、盤面処理とアニメーション補間を無理なく両立できます。*

TileMap を運用する上で最も重要な原則は、論理位置と視覚的な位置を混同しないことです。プレイヤーやマーカーの位置をワールド座標で直接管理するのではなく、「現在どのセルにいるか」という論理位置を保持する方が制御しやすくなります。論理位置は TileMap のセルで管理し、見た目のノード座標はその都度 TileMap に計算させるという分担を徹底しましょう。

```js
let playerCell = tileMap.getTile(1, 1);
tileMap.placeNodeOnCell(playerNode, playerCell, 0.08, 0.50);
```

`placeNodeOnCell()` は即座に位置を反映させます。一方で、動きを滑らかに見せたい場合は、目標位置のみを TileMap から取得し、ノード側のアニメーション機能を利用します。TileMap 自体は「移動中」という時間経過の状態を持たず、あくまで論理位置とワールド座標を紐付ける役割に徹しています。見た目の時間的変化はノード側の責務となります。

```js
const nextCell = tileMap.getTile(playerCell.col + 1, playerCell.row);
if (!tileMap.canMove(playerCell, nextCell)) {
  return;
}

const targetPosition = tileMap.getNodePositionOnCell(nextCell, 0.08, 0.50);
playerNode.animatePosition(targetPosition, {
  durationMs: 250
});
playerCell = nextCell;
```

このように、`playerCell` の更新（論理的な移動）とノードのアニメーション（視覚的な移動）を切り離して管理することで、コードの整理が容易になります。

## オブジェクトの配置と移動アニメーション

盤面上にオブジェクトを配置する際は、`getNodePositionOnCell()` と `placeNodeOnCell()` を使い分けます。前者はワールド座標のみを返し、後者はノードへ位置を即座に適用します。初期配置やテレポートには `placeNodeOnCell()` を使い、移動先を計算してアニメーションさせたい場合は `getNodePositionOnCell()` を利用してください。

また、エフェクトやカメラターゲットを配置する場合は `getWorldPosition(cell)` または `toWorld(cell)` が有効です。これらはノードの足元補正や半径補正を含まない、純粋なセル中心のワールド座標を返します。

### TileMap と Node.animatePosition() の役割分担

TileMap を利用したアプリケーションでは、「入力を受けたフレームで次のセルを決定し、その後 200ms〜300ms かけて視覚的に移動させる」という構成が一般的です。ここで注意すべきは、TileMap 自体は時間経過を管理していない点です。`canMove()` や `getNodePositionOnCell()`、`followCell()` はすべて即時計算であり、アニメーションを進行させるのは `Node.animatePosition()` と `Space.update(deltaMs)` の役割です。

`WebgApp.start()` を使用している場合、`onUpdate(ctx)` の後に `WebgApp` 内部で `space.update(deltaMs)` が自動的に呼ばれます。そのため、サンプルコードでは `onUpdate()` 内で移動要求を受け取り、`animatePosition()` を開始させるだけで十分です。もし `WebgApp` を使わずに独自のループを構築する場合は、毎フレーム必ず `space.update(deltaMs)` を呼び出さないと Tween アニメーションが進まないため注意してください。

`webg/samples/tile_sim` では、この役割分担に基づいた実装が行われています。

```js
let playerCell = tileMap.getTile(3, 3);
let moveTween = null;

tileMap.placeNodeOnCell(playerNode, playerCell, 0.08, 0.50);
tileMap.refreshTileColors();

function tryMoveBy(dx, dy) {
  if (moveTween && !moveTween.isFinished()) {
    return false;
  }

  const nextCell = tileMap.getTile(playerCell.col + dx, playerCell.row + dy);
  if (!tileMap.canMove(playerCell, nextCell)) {
    return false;
  }

  const targetPosition = tileMap.getNodePositionOnCell(nextCell, 0.08, 0.50);
  tileMap.followCell(nextCell, {
    width: 8,
    height: 6,
    followCol: 2,
    followRow: 2
  });
  tileMap.refreshTileColors();

  moveTween = playerNode.animatePosition(targetPosition, {
    durationMs: 250,
    easing: "outCubic",
    onComplete: () => {
      moveTween = null;
      playerCell = nextCell;
      tileMap.refreshTileColors();
    }
  });
  return true;
}

app.start({
  onUpdate: ({ deltaSec }) => {
    if (app.wasActionPressed("move_up")) {
      tryMoveBy(0, -1);
    }
    if (app.wasActionPressed("move_down")) {
      tryMoveBy(0, 1);
    }
    if (app.wasActionPressed("move_left")) {
      tryMoveBy(-1, 0);
    }
    if (app.wasActionPressed("move_right")) {
      tryMoveBy(1, 0);
    }
  }
});
```

この例では `deltaSec` を直接使用していませんが、`animatePosition()` の進行は内部的な `space.update(deltaMs)` に委ねられているため問題ありません。`onUpdate()` の役割は、あくまで「このフレームで新しい移動を開始するか」を判定するエントリーポイントとして機能しています。

### プレイヤーと表示範囲（displayArea）の時間軸の分離

プレイヤー本体と表示範囲（`displayArea`）を全く同じ速度で動かすと、盤面全体が同時に滑るような挙動になり、視覚的に不安定な印象を与えることがあります。そのため、移動演出と `displayArea` の追従を別の時間軸で管理することを推奨します。

実装上は、プレイヤーの移動 Tween と、カメラターゲットやピボットの Tween を別オブジェクトとして管理します。TileMap の `followCell()` で「次にどの範囲を表示すべきか」という目標値を決定し、実際の補間速度はアプリケーション側で調整することで、より心地よい視覚体験を実現できます。

## 表示範囲（displayArea）のスクロール制御

広大なマップの一部を表示する場合、`followCell()` を用いて特定のセルを基準に可視範囲を追従させます。ここで重要なのは、先述した通り「矩形範囲の更新」と「見た目の更新」を分けて考えることです。

```js
const displayArea = tileMap.followCell(playerCell, {
  width: 8,
  height: 6,
  followCol: 2,
  followRow: 2
});

tileMap.refreshTileColors(selectedHit);
```

`followCol` と `followRow` は、追従対象のセルを表示ウィンドウの左上から数えて何セル目に配置するかを指定します。例えば `followCol: 2` と設定すれば、プレイヤーは常にウィンドウの左から3列目付近に位置することになります。プレイヤーを完全な中央に固定せず、あえて前方や手前に寄せることで、進行方向の視認性を高めることが可能です。

`setDisplayArea()` や `followCell()` を呼び出しただけではノードの表示状態は更新されません。必ず直後に `refreshTileColors()` を呼び出し、現在の可視範囲に合わせて表示対象と選択色を更新してください。「表示範囲を変更したのに画面が追従しない」という問題の多くは、この更新処理の漏れが原因です。

## セルの選択処理

`pickCell(origin, dir)` は、レイ（Ray）とセルボックスの交差判定を行い、最も近いセルを返します。戻り値には `cell`、`point`、`hitFace`、`hitHeight` が含まれます。

ここで注意したいのは、TileMap 自体は「どのセルが選択されているか」という状態を保持しない点です。選択状態（Hit 情報）はアプリケーション側で保持し、その状態に基づいた見た目の更新のみを `refreshTileColors()` に委ねる構成になります。

```js
const hit = tileMap.pickCell(ray.origin, ray.dir);
if (!hit) {
  return;
}

// hitFace に応じて処理を分けることが可能
if (hit.hitFace === "top") {
  selectedHit = hit;
} else {
  selectedHit = hit;
}
tileMap.refreshTileColors(selectedHit);
```

`hitFace` には `top`、`bottom`、`wall-x+`、`wall-x-`、`wall-y+`、`wall-y-` のいずれかが格納されます。例えば「上面（top）をタップしたときは選択し、側面（wall）をタップしたときは編集操作を行う」といった使い分けが可能です。

## 移動判定とパス探索

`canMove(from, to)` は、「隣接セル間で高さ差が 1 以内であれば移動可能」という基本ルールに基づいています。また、`terrain === "wall"` と定義されたセルは進入不可（Blocked）として扱われます。

重要なのは、手動での移動判定と AI によるパス探索が全く同じルールを使用している点です。これにより、視覚的な段差、プレイヤーの操作、AI のルート決定がすべて同一の盤面ルールに統合され、一貫性のある挙動が保証されます。

```js
const nextCell = tileMap.getTile(playerCell.col, playerCell.row - 1);
if (tileMap.canMove(playerCell, nextCell)) {
  playerCell = nextCell;
}
```

最短経路が必要な場合は `findPath(start, goal)` を使用します。

```js
const path = tileMap.findPath(playerCell, goalCell);
if (path) {
  console.log(path.map((cell) => `(${cell.col},${cell.row})`).join(" -> "));
}
```

移動判定と経路探索で異なるルールを適用してしまうと、「プレイヤーは進めるのに AI は進めない」といった不整合が生じやすくなります。TileMap を盤面の唯一の基準として扱うことが、安定した運用への近道です。

## 座標系の解釈と混同の防止

TileMap を扱う際、配列のインデックス、方向名、入力キーの名称が混在するため、これらを明確に区別して考える必要があります。「データの並び順」「盤面上の列と行」「キーボード入力の上方向」を同一視せず、以下のように整理してください。

| 項目 | 意味 | 解釈方法 |
| :--- | :--- | :--- |
| `tiles[y][x]` | データの格納順 | 左上から右へ、次に下へ向かって走査 |
| `x` / `col` | 列 (Column) | 東西方向のインデックス |
| `y` / `row` | 行 (Row) | 南北方向のインデックス |
| `ArrowUp` | 入力名称 | 必ずしも `row` の減算を意味しない |
| `north` / `south` | 方向名称 | 配列インデックスと直接的に同一ではない |

特に Orbit Camera を使用する場合、`ArrowUp` キーが押されたからといって常に `row - 1` へ移動するとは限りません。カメラの向きに応じてグリッド移動を変換する必要がある場合は、`resolveCameraRelativeGridMove(yawDeg, key)` を利用します。このヘルパー関数は TileMap の内部座標を書き換えるものではなく、あくまで「入力の解釈」をカメラの向きに合わせて変換するものです。

```js
import { resolveCameraRelativeGridMove } from "./webg/TileMap.js";

const relativeMove = resolveCameraRelativeGridMove(orbitYaw, "arrowup");
const nextCell = tileMap.getTile(
  playerCell.col + relativeMove.dx,
  playerCell.row + relativeMove.dy
);
```

## 視覚表現の最適化と `edgeCutMask`

TileMap の視覚表現は、単純なボックスを並べてテクスチャを貼る手法とは異なります。構築プロセスにおいて、各セルは周囲 4 方向との高低差に基づいた `edgeCutMask` を保持します。これは East / West / North / South のどの辺を削るかを 4 ビットで表したもので、レンダラーはこのマスクを参照して上面の辺を内側に寄せます。

ここで重要なのは、「見た目上はエッジを削って整理しているが、論理的なボックスはフルサイズ（Full Height）のまま保持されている」という点です。つまり、描画処理と当たり判定処理で役割が分かれています。

描画面では、上部から `visibleHeightSteps` 分だけを薄い四角錐台のキャップ（Cap）として描き、その下部をテクスチャを使用しないファンデーション（Foundation）で埋めています。これにより、高さのあるセルであってもテクスチャ情報が過剰に増えるのを防ぎ、斜め視点での背景の抜けを防ぎつつ、上面の形状を明確に提示できます。

内部的な削れ量は `TILE_SURFACE_INSET_RATIO = 0.4` という比率で決定されています。これはセル全幅に対する比率ではなく、「中心から辺までの半幅」に対する比率です。正方形のセルの場合、全幅に対して 20% 分だけ辺を内側に寄せる計算になります。

また、UV 座標の設計もこの構造に合わせて最適化されています。キャップの上面のみが、セル外形を `0..1` に正規化したトッププロジェクション（Top Projection）を使用します。これにより、削れが 0.2 であれば、テクスチャ上の `U=0.2` や `U=0.8` の位置がそのまま端となり、テクスチャ上の切り込み幅を直感的に管理できます。一方、側面は面ごとに独立した単純な UV を使用し、視認性を優先しています。

さらに、実装上の詳細として、`Shape` の初期テクスチャマッピングモードを `1` に設定し、従来のシーム補正を無効化しています。これにより、自前で定義した UV 座標が書き換えられるのを防いでいます。UV の基準は `webg` 全体の仕様に従い、Blender と同様の Bottom-Left 基準です。画像テクスチャの上下が反転している場合は、UV 計算を修正するのではなく、画像入力側で `flipV` を適用して調整してください。

面の表裏については、`addPrismSlice()` や共有頂点キャップの定義時点で、外側に向かって反時計回り（CCW）になるよう統一されています。これにより、法線方向や `frontFace: "ccw"` の設定、側面 UV の対応関係を一貫して追跡することが可能です。

利用者がこれらの値を直接操作する必要はほとんどありません。TileMap は `assignEdgeCutMasks()` で自動計算し、`applySurfaceShapes()` で見た目に反映させます。利用者が関与するのは、`getEdgeCutMask(cell)` で現在の削れ方を確認する場合や、`describeEdgeCutMask(mask)` でデバッグテキストを表示する場合のみです。

## よくあるつまずき

マップデータの不備
`tiles` 配列の一部のみを定義してビルドした場合、TileMap は欠損部分を自動的に補完しません。`width` と `height` で指定した範囲をすべて埋める必要があります。また、`tiles`（固定マップ）と `generator`（自動生成）を同時に指定することはできません。両方を組み合わせたい場合は、最終的な `tiles` 配列を自身で構築してから TileMap に渡してください。

表示の更新漏れ
`displayArea` を変更したのに画面が変わらない場合、`setDisplayArea()` や `followCell()` の後に `refreshTileColors()` を呼び忘れている可能性が高いです。矩形の変更と見た目の更新は別処理であることを意識してください。

判定と移動の混同
`canMove()` は移動可能かどうかの真偽値（true/false）を返すだけであり、ノードを実際に移動させるわけではありません。ノードの位置を更新するには、別途 `placeNodeOnCell()` や `animatePosition()` を呼び出す必要があります。

UI とデータの分離
セルに直接日本語の説明文を持たせたい場合がありますが、TileMap はあくまで盤面データと判定の部品です。チュートリアルや会話文などのテキスト情報は、`DialogueOverlay` や `UIPanel` といった UI レイヤーで管理するのが適切です。

アニメーションの停止
GLB モデルを配置したのにアニメーションが進まない場合は、`animatePosition()` の完了だけを監視し、`AnimationState.update()` を毎フレーム呼んでいない可能性があります。また、`WebgApp.start()` を使用していない環境で `space.update(deltaMs)` の呼び出しを忘れているケースも考えられます。位置の Tween と骨格アニメーションは、どちらもフレームループ内で継続的に更新させる必要があります。

## 例を通して確かめる

`webg/samples/tile_sim` は、TileMap とアニメーション付きキャラクターを組み合わせた総合的な例です。`WebgApp`、`Scene JSON`、`terrainMaterials`、カメラ相対移動、`followCell()`、`pickCell()`、そしてアクターの配置と更新の分担について、実運用に近い形で確認できます。TileMap を「盤面基盤」としてどう活用すべきかを学ぶための最適なサンプルです。

`webg/unittest/tilemap` は、個別の API 挙動を詳細に確認したい場合に有用です。`canMove()` や `findPath()`、`pickCell()` などの単体動作を簡潔に追うことができます。

`webg/samples/animation_state` は、GLB アセットと `Action` / `AnimationState` の連携を示す例です。TileMap と直接的な関係はありませんが、「アニメーションの開始」と「毎フレームの更新」を分ける考え方は共通しています。TileMap 上のキャラクターに骨格アニメーションを実装する際の参考になります。また、`webg/samples/janken` では、複数の独立したステートマシンを持つキャラクターの配置例を確認でき、TileMap 上で複数のユニットを制御する際のヒントになります。

本章で最も重要なのは、TileMap ランタイムを「論理的な位置を算出する部品」として扱い、ノードやアニメーションは「その結果を受けて視覚化する」という役割分担を明確にすることです。`getTile()`、`canMove()`、`findPath()`、`followCell()`、`pickCell()`、`placeNodeOnCell()` の 6 つの API を軸に据えることで、TileMap を利用した処理フローをスムーズに理解できるはずです。次の第 24 章では、これらの考え方を `tile_sim` という実例の中でさらに具体的に掘り下げていきます。
