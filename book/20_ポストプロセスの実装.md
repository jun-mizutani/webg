# ポストプロセスの実装

Bloomや被写界深度（Depth of Field）のようなエフェクトは、単にシェーダーを追加するだけで実現できる機能ではありません。一度シーンをオフスクリーン（画面外）へ描き出し、その結果を後段のフルスクリーンパスで加工してから、最終的にキャンバスへ戻すという工程が必要になります。本章では、この描画フローの再構築を軸に、`BloomPass`、`DofPass`、`VignettePass` の導入手順を解説します。

対象となるファイルは、`webg/BloomPass.js`、`webg/DofPass.js`、`webg/VignettePass.js`、`webg/RenderTarget.js`、`webg/FullscreenPass.js`、`webg/SeparableBlurPass.js`、および各サンプルの `webg/samples/bloom/main.js`、`webg/samples/dof/main.js`、`webg/unittest/vignette/main.js` です。

ここで重要なのは、ポストプロセスは3Dシーン描画の内部処理ではなく、描画完了後に適用する「別工程」であるという点です。`BloomPass`、`DofPass`、`VignettePass` はいずれも、「3Dシーンを描き終えた後のテクスチャ」を加工するコンポーネントとして機能します。したがって、導入時にまず変更すべきは `SmoothShader` のような個別マテリアル設定ではなく、`WebgApp.start()` における描画フローです。具体的には、`autoDrawScene: false` に設定し、`onBeforeDraw()` でオフスクリーンへ3Dシーンを描画し、`onAfterDraw3d()` でポストプロセスを適用してキャンバスへ出力する構成にすることで、処理の流れを整理できます。

さらに、`BloomPass` と `DofPass` は共通して、「`beginScene()` $\rightarrow$ `app.space.draw(app.eye)` $\rightarrow$ `render()`」という3つのステップで構成されます。この一連の流れを1フレームの中で崩さないことが、正しくエフェクトを適用させるための鍵となります。また、Bloomはカラー情報のみで動作しますが、被写界深度（DOF）は後段で深度情報を読み取ります。そのため、`DofPass` の内部では `RenderTarget({ sampleDepth: true })` が使用されています。最後に、HUDやデバッグオーバーレイを重ねる場合は、ポストプロセス処理の後に `app.screen.clearDepthBuffer()` を呼び出す構成が必要です。これはエフェクト自体の機能ではなく、描画フローを適切に制御するための処理です。

## ポストプロセスの概念と描画フロー

ポストプロセスとは、シーン内の個々のオブジェクトのシェーダーを書き換える手法ではありません。まず3Dシーンをオフスクリーンターゲットへ描き出し、その結果得られたカラーテクスチャや深度テクスチャを、後段のフルスクリーンパスで読み取って加工します。つまり、シーン全体の描画を完了させた後で、「画面全体に対して1枚の後処理をかける」というアプローチです。

このフローを理解せずに導入すると、「`BloomPass` を実装したのに画面に変化がない」「DOFを導入したが深度情報が読み取れない」「Vignetteを適用したことでHUDの描画順序が崩れた」といった問題に直面しやすくなります。そのため、各パスの詳細な説明に入る前に、まずは共通の描画フローを確立させます。

![ポストプロセス描画フロー図](fig20_01_postprocess_flow.jpg)

*ポストプロセスでは、シーンを一度オフスクリーンへ描き出し、その結果に後処理を適用した後でHUDを重ねます。*

### `autoDrawScene: false` による制御

ポストプロセスを導入する際、最初に行うべきは描画フローの変更です。`WebgApp` の標準的な描画設定のままでは、シーンが直接キャンバスへ出力されるため、その間にBloomやDOFといった処理を挟むことができません。そのため、ポストプロセスを利用するサンプルでは `autoDrawScene: false` に設定しています。

### 基本的な実装サイクル

最小構成のコードは以下の通りです。

```js
import WebgApp from "./webg/WebgApp.js";
import FullscreenPass from "./webg/FullscreenPass.js";
import BloomPass from "./webg/BloomPass.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  autoDrawScene: false
});
await app.init();

const bloom = new BloomPass(app.getGL(), {
  width: app.screen.getWidth(),
  height: app.screen.getHeight()
});
await bloom.ready;

const debugPass = new FullscreenPass(app.getGL(), {
  targetFormat: app.getGL().format
});
await debugPass.init();

app.start({
  onUpdate: ({ screen }) => {
    bloom.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    bloom.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    bloom.render(app.screen, {
      clearColor: app.clearColor
    });

    app.screen.clearDepthBuffer();
  }
});
```

このコードにおいて重要なのは、`bloom.beginScene()` がシーン描画の入口となり、`bloom.render()` がポストプロセス処理の入口となる点です。3Dシーンを描画する責任は、引き続き `app.space.draw(app.eye)` が担っており、`BloomPass` はその前後を繋ぐ役割を果たしています。`DofPass` においても考え方は同様であり、異なるのは後段で参照するデータ（深度など）や、保持する中間ターゲットの構成のみです。

### HUDの重ね合わせ

ポストプロセス適用後も `Message` やデバッグオーバーレイを正しく表示させたい場合は、最終パスの後に深度バッファをクリアし直す必要があります。`webg/samples/bloom` や `webg/samples/dof` では、ポストプロセスの後に `app.screen.clearDepthBuffer()` を呼び出しています。これはエフェクトの一部ではなく、HUDを自然に重ねるためのフロー制御です。

## BloomPass の実装と活用

### 役割と仕組み

`BloomPass` は、シーン内の輝度が高い部分のみを抽出し、それをぼかして元のシーンに加算することで、光がにじみ出すような視覚効果を演出するヘルパーです。`webg/samples/bloom` は、この処理フローを学習するためのサンプルとして構成されており、「カラー情報を読み取って光をにじませる」という、ポストプロセスの基本を理解するのに適しています。

内部では主に `sceneTarget`、`extractTarget`、`extractHeatTarget`、そして `SeparableBlurPass` が管理するぼかし用ターゲットを使用します。利用者は `new BloomPass(gpu, options)` でインスタンス化し、`await bloom.ready` で準備完了を待機した後、`beginScene`、`render`、`resizeToScreen` といったAPIを通じて制御します。また、しきい値（threshold）やぼかし強度（blur）などのパラメータをセッター経由で調整可能です。

### 最小構成の実装

```js
import BloomPass from "./webg/BloomPass.js";

const bloom = new BloomPass(app.getGL(), {
  width: app.screen.getWidth(),
  height: app.screen.getHeight(),
  threshold: 0.68,
  extractIntensity: 1.0,
  softKnee: 0.35,
  bloomStrength: 1.15,
  exposure: 1.0,
  toneMapMode: 0,
  blurScale: 1.0,
  blurIterations: 2,
  blurRadius: 1.0
});
await bloom.ready;

app.start({
  onUpdate: ({ screen }) => {
    bloom.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    bloom.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    bloom.render(app.screen, {
      clearColor: app.clearColor
    });
    app.screen.clearDepthBuffer();
  }
});
```

この実装例で注目すべき点は、`BloomPass` を導入しても3Dシーンの描画主体は変わらないということです。変更されるのは、シーンを一度 `sceneTarget` へ描き出し、その後に「明るい部分の抽出」と「合成」という別工程を挟む点だけです。

### デバッグビューによる調整

中間結果を可視化するには `FullscreenPass` を利用します。`BloomPass` は中間ターゲットへアクセスするためのゲッターを提供しているため、それを `debugPass.draw()` に渡すことで、内部状態で何が起きているかを確認できます。ポストプロセスは中間テクスチャを可視化できることで理解度が飛躍的に向上するため、あらかじめデバッグ経路を構築しておくことを推奨します。

```js
import FullscreenPass from "./webg/FullscreenPass.js";

const debugPass = new FullscreenPass(app.getGL(), {
  targetFormat: app.getGL().format
});
await debugPass.init();

let bloomView = "composite";

app.start({
  onUpdate: ({ screen }) => {
    bloom.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    bloom.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    bloom.render(app.screen, {
      clearColor: app.clearColor
    });

    if (bloomView !== "composite") {
      const debugSource = bloomView === "scene"
        ? bloom.getSceneTarget()
        : bloomView === "extract"
          ? bloom.getExtractTarget()
          : bloomView === "extractHeat"
            ? bloom.getExtractHeatTarget()
            : bloomView === "blurA"
              ? bloom.getBlurTargetA()
              : bloomView === "blurTargetB";

      app.screen.beginPass({
        clearColor: app.clearColor,
        colorLoadOp: "clear",
        depthView: null
      });
      debugPass.draw(debugSource);
    }

    app.screen.clearDepthBuffer();
  }
});
```

特に `extract` と `extractHeat` を比較することで、しきい値や `softKnee` がどのように効いているかを視覚的に把握でき、効率的にパラメータを調整することが可能です。

## DofPass の実装と活用

### 役割と仕組み

`DofPass` は、シーンのカラー情報と深度情報の両方を参照し、合焦面（フォーカス面）だけを鮮明に残し、それ以外をぼかすエフェクトです。`webg/samples/dof` では、フォーカスガイドとデバッグビューを同時に表示し、`focusDistance`（合焦距離）と `focusRange`（合焦範囲）の意味を視覚的に理解できるように構成されています。Bloomとの最大の違いは、後段で深度情報を利用する点にあります。

内部では `sceneTarget`、`depthDebugTarget`、`focusDebugTarget`、および `SeparableBlurPass` のぼかしターゲットを保持しています。主要なAPIは `BloomPass` と同様ですが、`projectionNear` や `projectionFar` といった投影行列に関する設定が必要になります。

### 最小構成の実装

```js
import DofPass from "./webg/DofPass.js";

const dof = new DofPass(app.getGL(), {
  width: app.screen.getWidth(),
  height: app.screen.getHeight(),
  projectionNear: app.projectionNear,
  projectionFar: app.projectionFar,
  focusDistance: 34.0,
  focusRange: 6.0,
  maxBlurMix: 1.0,
  blurScale: 0.5,
  blurIterations: 2,
  blurRadius: 2.4
});
await dof.ready;

app.start({
  onUpdate: ({ screen }) => {
    dof.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    dof.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    dof.render(app.screen, {
      clearColor: app.clearColor
    });
    app.screen.clearDepthBuffer();
  }
});
```

ここでも `beginScene()` $\rightarrow$ `app.space.draw(app.eye)` $\rightarrow$ `render()` のサイクルを維持しています。ただし、`DofPass` では `sceneTarget` がカラー情報に加えて、サンプリング可能な深度情報を保持している必要があります。ここはDOF導入において最も見落としやすい重要なポイントです。

### 深度デバッグとフォーカスマスク

`DofPass` は、内部的に深度（depth）や合焦範囲（focusMask）をカラーターゲットへ出力するヘルパーを備えています。DOFの調整は、深度条件と合焦範囲が視覚的に分からないと困難であるため、Bloom以上にデバッグビューの活用が不可欠です。

```js
import FullscreenPass from "./webg/FullscreenPass.js";

const debugPass = new FullscreenPass(app.getGL(), {
  targetFormat: app.getGL().format
});
await debugPass.init();

let dofView = "composite";

app.start({
  onUpdate: ({ screen }) => {
    dof.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    dof.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    dof.render(app.screen, {
      clearColor: app.clearColor
    });

    if (dofView !== "composite") {
      const debugSource = dofView === "scene"
        ? dof.getSceneTarget()
        : dofView === "depth"
          ? dof.getDepthDebugTarget()
          : dofView === "focusMask"
            ? dof.getFocusDebugTarget()
            : dofView === "blurA"
              ? dof.getBlurTargetA()
              : dofView === "blurTargetB";

      app.screen.beginPass({
        clearColor: app.clearColor,
        colorLoadOp: "clear",
        depthView: null
      });
      debugPass.draw(debugSource);
    }

    app.screen.clearDepthBuffer();
  }
});
```

`depth` と `focusMask` を切り替えて確認することで、`focusDistance` が投影設定（near/far）と整合しているか、どの領域がシャープに維持されているかを即座に判断できます。

## VignettePass の実装と活用

### 役割と仕組み

`VignettePass` は、最終的な画面出力に対して適用する最も軽量なエフェクトです。BloomやDOFのように複雑な中間ターゲットを必要とせず、「1枚のカラーテクスチャを読み込み、周辺部の輝度を減衰させる」というシンプルな処理を行います。`webg/unittest/vignette` では、この最短構成を確認できるテストが実装されています。ポストプロセスの基本的な流れを最小構成で検証したい場合に最適です。

### 最小構成の実装

`VignettePass` 自体はオフスクリーンターゲットを保持しないため、ソースとなるターゲットを別途用意する必要があります。

```js
import FullscreenPass from "./webg/FullscreenPass.js";
import VignettePass from "./webg/VignettePass.js";

const sceneTarget = app.screen.createRenderTarget({
  label: "my-scene",
  format: app.getGL().format,
  hasDepth: true
});
await sceneTarget.ready;

const rawScenePass = new FullscreenPass(app.getGL());
await rawScenePass.init();

const vignettePass = new VignettePass(app.getGL(), {
  centerX: 0.5,
  centerY: 0.5,
  radius: 0.74,
  softness: 0.42,
  strength: 0.92,
  tint: [0.0, 0.0, 0.0, 1.0]
});
await vignettePass.init();

app.start({
  onUpdate: ({ screen }) => {
    sceneTarget.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    app.screen.clear(sceneTarget);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    vignettePass.render(app.screen, {
      source: sceneTarget,
      clearColor: app.clearColor
    });
    app.screen.clearDepthBuffer();
  }
});
```

元のシーンと比較したい場合は、`vignettePass.render()` の代わりに `rawScenePass.draw(sceneTarget)` を呼び出すことで確認できます。Vignetteは処理が非常に単純であるため、「ソーステクスチャを別途用意して渡す」という基本さえ押さえれば容易に導入可能です。

## ポストプロセスを支える共通部品

各エフェクトの内部で利用されている共通コンポーネントについて解説します。

- `RenderTarget`: オフスクリーン描画の先となるターゲットです。ポストプロセスにおいて「最初にどこへ描画するか」を制御する土台となります。
- `FullscreenPass`: 1枚のテクスチャを現在のパスへ出力するための最小限のヘルパーです。デバッグビューの表示や、エフェクト適用前のシーンとの比較表示に多用されます。いわば「中間ターゲットを可視化するためのツール」です。
- `SeparableBlurPass`: 横方向と縦方向のぼかし処理を交互に繰り返す（ピンポン処理）ことで、効率的にガウスぼかしなどを実現する共有ヘルパーです。`BloomPass` と `DofPass` の内部で利用されています。

これらの部品を理解することで、各エフェクトがどのような共通基盤の上に構築され、どこに固有の処理（抽出や深度判定など）を持っているのかが明確になります。

## 導入パターンと使い分け

初めてポストプロセスを導入する場合、`BloomPass` または `VignettePass` から始めることを推奨します。これらは「オフスクリーン $\rightarrow$ フルスクリーン $\rightarrow$ キャンバス」という基本フローを追いやすく、深度情報の参照という複雑な工程を含まないためです。まずはカラー情報のみを扱う後処理に慣れ、その後にDOFのような深度依存のパスへ進むのが効率的です。

`DofPass` は、投影行列の `near` / `far` やカメラ設定と密接に結びついているため、カメラの視野角や距離感が確定していない段階では調整が困難です。シーンの基本カメラ設定が落ち着いた段階で導入してください。また、どのパスにおいても、中間テクスチャを可視化できるようにしておくことで、パラメータ調整の時間を大幅に短縮できます。特にDOFでは、`depth` と `focusMask` を確認せずに調整を行うと、原因の切り分けが困難になります。

## トラブルシューティング

画面に変化が見られない場合
`autoDrawScene: false` に設定されていない可能性があります。その場合、シーンが直接キャンバスへ描画されるため、ポストプロセス処理がスキップされます。`beginScene()` と `render()` が正しく描画フローに組み込まれているか確認してください。

リサイズ後に解像度がずれる場合
`onUpdate()` 内で `bloom.resizeToScreen(screen)` や `dof.resizeToScreen(screen)` を呼び出していない可能性があります。オフスクリーンターゲットは、キャンバスのサイズ変更に合わせて適切にリサイズする必要があります。

被写界深度（DOF）が正しく効かない場合
投影設定の `near` / `far` がカメラの設定と不整合を起こしているか、デバッグビューで深度を確認せずにパラメータを調整している可能性があります。まずは `depth` と `focusMask` を表示し、意図した領域が合焦しているかを確認してください。

HUDが表示されなくなった場合
ポストプロセスの最終パスの後に `app.screen.clearDepthBuffer()` を呼び出していない可能性があります。これにより、ポストプロセスで書き込まれた深度情報が残り、後から描画するHUDが深度テストで棄却されてしまいます。

VignettePass が動作しない場合
`VignettePass` は内部でソーステクスチャを生成しません。シーンをあらかじめ別の `RenderTarget` へ描画し、そのテクスチャを `render({ source })` に渡す必要があります。実装しただけで動作するのではなく、適切にソースを指定することが必須です。
