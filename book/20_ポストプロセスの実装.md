# ポストプロセスの実装

Bloom や被写界深度のようなエフェクトは、シェーダーを少し足せば済む機能ではありません。いったんシーンをオフスクリーンへ描き、その結果を後段のフルスクリーンパスで加工してから、最後にキャンバスへ戻す必要があります。本章では、この描画フローの組み替えを軸に、`BloomPass`、`DofPass`、`VignettePass` の導入手順を整理します。対象となるのは `webg/BloomPass.js`、`webg/DofPass.js`、`webg/VignettePass.js`、`webg/RenderTarget.js`、`webg/FullscreenPass.js`、`webg/SeparableBlurPass.js`、`webg/samples/bloom/main.js`、`webg/samples/dof/main.js`、`webg/unittest/vignette/main.js` です。 

ここで重要なのは、ポストプロセスが 3D シーン描画の内部ではなく、その後段に挟む別工程だという点です。`BloomPass` も `DofPass` も `VignettePass` も、「3D シーンを描き終えたあとのテクスチャ」を加工する部品です。したがって導入するときに最初に変えるのは `Phong` や `NormPhong` ではなく、`WebgApp.start()` の描画フローです。`autoDrawScene: false` にして、`onBeforeDraw()` でオフスクリーンへ 3D シーンを描き、`onAfterDraw3d()` でポストプロセスをキャンバスへ戻す形にすると整理しやすくなります。 

さらに重要なのは、`BloomPass` と `DofPass` がどちらも「`beginScene()` → `app.space.draw(app.eye)` → `render()`」の 3 段で使うことです。この 3 段を 1 フレームの中で崩さないことが最も大切です。加えて、Bloom はカラーだけを読めば動作しますが、被写界深度は深度も後段で読みます。そのため `DofPass` の内部では `RenderTarget({ sampleDepth: true })` が使われます。最後に、HUD やデバッグオーバーレイを重ねるなら、ポストプロセス後に `app.screen.clearDepthBuffer()` を呼ぶ構成が必要です。これはエフェクトそのものではなく、描画フローを崩さないための処理です。 

## ポストプロセスとは何か

ポストプロセスは、シーン内の各 object のシェーダーを書き換える機能ではありません。まず 3D シーンをオフスクリーンターゲットへ描き、その結果のカラーテクスチャや深度テクスチャを、後段のフルスクリーンパスで読み取って加工します。つまり、scene を描いたあとに「画面全体へ 1 枚の後処理をかける」と考えると分かりやすくなります。 

この流れを理解せずに導入すると、`BloomPass` を作ったのに画面が変わらない、DOF を実装したのに深度が読めない、Vignette を呼んだのに HUD の順番が崩れる、といった問題が起きやすくなります。したがって、この章では各 pass の説明より先に、まず描画フローそのものをそろえることを優先します。 

## 共通の描画フロー

### `autoDrawScene: false` にする

ポストプロセスを入れるときに最初に変えるのは、シェーダーではなく描画フローです。`WebgApp` の通常描画をそのまま使うと、シーンは直接キャンバスへ出るため、あとから Bloom や DOF を挟めません。そのため、ポストプロセス処理を使うサンプルでは `autoDrawScene: false` にしています。 

### `beginScene()` → `space.draw()` → `render()`

最小構成は次の形です。

```js
import WebgApp from "./webg/WebgApp.js";
import FullscreenPass from "./webg/FullscreenPass.js";
import BloomPass from "./webg/BloomPass.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  autoDrawScene: false
});
await app.init();

const bloom = new BloomPass(app.getGL(), {
  width: app.screen.getWidth(),
  height: app.screen.getHeight()
});
await bloom.ready;

const debugPass = new FullscreenPass(app.getGL(), {
  targetFormat: app.getGL().format
});
await debugPass.init();

app.start({
  onUpdate: ({ screen }) => {
    bloom.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    bloom.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    bloom.render(app.screen, {
      clearColor: app.clearColor
    });

    app.screen.clearDepthBuffer();
  }
});
```

このコードで重要なのは、`bloom.beginScene()` がシーン側の入口であり、`bloom.render()` がポストプロセス側の入口だということです。3D シーンを描画する責任は、あくまで `app.space.draw(app.eye)` にあります。BloomPass はその前後をつなぐ部品です。DofPass でも考え方は同じで、違うのは後段で何を読むか、どの中間ターゲットを持つかだけです。 

### HUD を最後に重ねる

ポストプロセス後も `Message` やデバッグオーバーレイを重ねたい場合は、最終パスのあとで深度をクリアし直す構成が必要です。`webg/samples/bloom` と `webg/samples/dof` は `app.screen.clearDepthBuffer()` をポストプロセスのあとに呼んでいます。これはエフェクトの一部ではなく、最後に HUD を自然に重ねるためのフロー制御です。 

## `BloomPass`

### 役割

`BloomPass` は、シーンの中で明るい部分だけを抜き出し、それをぼかして、最後に元のシーンへ足し戻すヘルパーです。`webg/samples/bloom` は、この処理フローをそのまま見せる教材用サンプルになっています。ここでは「カラーだけを読んで後段で光をにじませる」という、ポストプロセスの最初の 1 本として理解しやすい構成になっています。 

内部では主に `sceneTarget`、`extractTarget`、`extractHeatTarget`、そして `SeparableBlurPass` が持つぼかしターゲットを使います。利用者が触る API は、`new BloomPass(gpu, options)`、`await bloom.ready`、`bloom.beginScene(screen, clearColor)`、`bloom.render(screen, options)`、`bloom.resizeToScreen(screen)`、そして threshold、softKnee、blur まわりの setter 群です。 

### 最小構成

```js
import BloomPass from "./webg/BloomPass.js";

const bloom = new BloomPass(app.getGL(), {
  width: app.screen.getWidth(),
  height: app.screen.getHeight(),
  threshold: 0.68,
  extractIntensity: 1.0,
  softKnee: 0.35,
  bloomStrength: 1.15,
  exposure: 1.0,
  toneMapMode: 0,
  blurScale: 1.0,
  blurIterations: 2,
  blurRadius: 1.0
});
await bloom.ready;

app.start({
  onUpdate: ({ screen }) => {
    bloom.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    bloom.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    bloom.render(app.screen, {
      clearColor: app.clearColor
    });
    app.screen.clearDepthBuffer();
  }
});
```

この例で見るべきなのは、BloomPass を導入しても 3D シーンの描画主体は変わらないという点です。変わるのは、シーンをいったん `sceneTarget` へ描き、その後に明るい部分の抽出と合成を別工程で行うことです。ポストプロセスを「シーンのあとに 1 枚の画面全体処理を挟むもの」と理解すると、流れが見失いにくくなります。 

### デバッグビュー

中間結果を見たいときは `FullscreenPass` を使います。`BloomPass` は中間ターゲットへアクセスする getter を持っているので、それをそのまま `debugPass.draw()` へ渡せます。ポストプロセスは中間テクスチャを見られるだけで理解しやすさがかなり上がるため、最初からこの経路を用意しておくと調整が速くなります。 

```js
import FullscreenPass from "./webg/FullscreenPass.js";

const debugPass = new FullscreenPass(app.getGL(), {
  targetFormat: app.getGL().format
});
await debugPass.init();

let bloomView = "composite";

app.start({
  onUpdate: ({ screen }) => {
    bloom.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    bloom.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    bloom.render(app.screen, {
      clearColor: app.clearColor
    });

    if (bloomView !== "composite") {
      const debugSource = bloomView === "scene"
        ? bloom.getSceneTarget()
        : bloomView === "extract"
          ? bloom.getExtractTarget()
          : bloomView === "extractHeat"
            ? bloom.getExtractHeatTarget()
            : bloomView === "blurA"
              ? bloom.getBlurTargetA()
              : bloom.getBlurTargetB();

      app.screen.beginPass({
        clearColor: app.clearColor,
        colorLoadOp: "clear",
        depthView: null
      });
      debugPass.draw(debugSource);
    }

    app.screen.clearDepthBuffer();
  }
});
```

Bloom の調整では、`extract` と `extractHeat` を見比べるだけでも、しきい値や `softKnee` の効き方をかなり理解しやすくなります。 

## `DofPass`

### 役割

`DofPass` は、シーンカラーと深度の両方を見ながら、合焦面だけをシャープに残し、それ以外を blur へ寄せるヘルパーです。`webg/samples/dof` は focus guide とデバッグビューを同時に出し、`focusDistance` と `focusRange` の意味が見えるようにしています。Bloom と違って後段で深度を読むため、「カラーだけを読む pass」ではない点が最大の違いです。 

内部では `sceneTarget`、`depthDebugTarget`、`focusDebugTarget`、`SeparableBlurPass` のぼかしターゲットを持ちます。利用者がまず覚えるべき API は、`new DofPass(gpu, options)`、`await dof.ready`、`dof.beginScene(screen, clearColor)`、`dof.render(screen, options)`、`dof.resizeToScreen(screen)`、そして focus / blur 関連の setter 群です。 

### 最小構成

```js
import DofPass from "./webg/DofPass.js";

const dof = new DofPass(app.getGL(), {
  width: app.screen.getWidth(),
  height: app.screen.getHeight(),
  projectionNear: app.projectionNear,
  projectionFar: app.projectionFar,
  focusDistance: 34.0,
  focusRange: 6.0,
  maxBlurMix: 1.0,
  blurScale: 0.5,
  blurIterations: 2,
  blurRadius: 2.4
});
await dof.ready;

app.start({
  onUpdate: ({ screen }) => {
    dof.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    dof.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    dof.render(app.screen, {
      clearColor: app.clearColor
    });
    app.screen.clearDepthBuffer();
  }
});
```

この例でも、`beginScene()` → `app.space.draw(app.eye)` → `render()` の 3 段を保っています。ただし DofPass では、`sceneTarget` がカラーに加えてサンプリング可能な深度も持っていなければならない点が違います。ここが DOF 導入で最も外しやすい条件です。 

### 深度デバッグと focus mask

`DofPass` は `depth` と `focusMask` をカラーターゲットへ出すヘルパーも内部で回しているので、デバッグ表示用の入力をそのまま画面へ出せます。DOF は深度条件と合焦範囲が視覚的に分からないと調整しづらいため、Bloom 以上にデバッグビューが重要です。 

```js
import FullscreenPass from "./webg/FullscreenPass.js";

const debugPass = new FullscreenPass(app.getGL(), {
  targetFormat: app.getGL().format
});
await debugPass.init();

let dofView = "composite";

app.start({
  onUpdate: ({ screen }) => {
    dof.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    dof.beginScene(app.screen, app.clearColor);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    dof.render(app.screen, {
      clearColor: app.clearColor
    });

    if (dofView !== "composite") {
      const debugSource = dofView === "scene"
        ? dof.getSceneTarget()
        : dofView === "depth"
          ? dof.getDepthDebugTarget()
          : dofView === "focusMask"
            ? dof.getFocusDebugTarget()
            : dofView === "blurA"
              ? dof.getBlurTargetA()
              : dof.getBlurTargetB();

      app.screen.beginPass({
        clearColor: app.clearColor,
        colorLoadOp: "clear",
        depthView: null
      });
      debugPass.draw(debugSource);
    }

    app.screen.clearDepthBuffer();
  }
});
```

`depth` と `focusMask` を切り替えて見られるようにしておくと、`focusDistance` の見え方が projection の `near` / `far` とずれていないか、どこがシャープ領域として残っているかをすぐ確認できます。 

## `VignettePass`

### 役割

`VignettePass` は最終色に対する最小のエフェクトです。Bloom や DOF のように複数段のターゲットを持たず、「1 枚のカラーテクスチャを読み、周辺だけを減衰させる」ことに集中しています。`webg/unittest/vignette` は、その最短の構成を確認するテストです。ポストプロセスの流れそのものを最小構成で確認したいときに向いています。 

### 最小構成

`VignettePass` 自体はオフスクリーンターゲットを持たないので、source 側は別に用意します。

```js
import FullscreenPass from "./webg/FullscreenPass.js";
import VignettePass from "./webg/VignettePass.js";

const sceneTarget = app.screen.createRenderTarget({
  label: "my-scene",
  format: app.getGL().format,
  hasDepth: true
});
await sceneTarget.ready;

const rawScenePass = new FullscreenPass(app.getGL());
await rawScenePass.init();

const vignettePass = new VignettePass(app.getGL(), {
  centerX: 0.5,
  centerY: 0.5,
  radius: 0.74,
  softness: 0.42,
  strength: 0.92,
  tint: [0.0, 0.0, 0.0, 1.0]
});
await vignettePass.init();

app.start({
  onUpdate: ({ screen }) => {
    sceneTarget.resizeToScreen(screen);
  },
  onBeforeDraw: () => {
    app.screen.clear(sceneTarget);
    app.space.draw(app.eye);
  },
  onAfterDraw3d: () => {
    vignettePass.render(app.screen, {
      source: sceneTarget,
      clearColor: app.clearColor
    });
    app.screen.clearDepthBuffer();
  }
});
```

元のシーンと見比べたいだけなら、`vignettePass.render()` の代わりに `rawScenePass.draw(sceneTarget)` を呼べば比較できます。Vignette は後段処理としては非常に単純なので、「source テクスチャを別に用意する」という基本だけを押さえれば理解しやすくなります。 

## 共通部品: `RenderTarget` / `FullscreenPass` / `SeparableBlurPass`

`RenderTarget` はオフスクリーン描画先そのものです。ポストプロセスでは「最初にどこへ描くか」を切り替えるための土台になります。`FullscreenPass` は、1 枚のテクスチャを現在のパスへ出す最小ヘルパーで、デバッグビュー、コピーパス、元のシーンの比較表示でよく使います。`SeparableBlurPass` は横ぼかしと縦ぼかしをピンポンで回す共有ヘルパーで、`BloomPass` と `DofPass` はこれを内部で使っています。 

これらは主役のエフェクトではありませんが、理解と調整のためには非常に重要です。`FullscreenPass` を「中間ターゲットを画面で見るための表示器」、`SeparableBlurPass` を「ぼかし処理の共通土台」と考えると、各エフェクトが何を差分として持っているかが見えやすくなります。 

## 導入パターンと使い分け

最初の 1 本として導入するなら、`BloomPass` か `VignettePass` が安全です。どちらも「オフスクリーン → フルスクリーン → キャンバス」の形を追いやすく、深度を後段で読むところまでは要求しません。まずカラーだけを使う後段処理に慣れてから、DOF のような深度依存の pass へ進むほうが理解しやすくなります。 

`DofPass` は projection の `near` / `far` や camera 設定と強く結びつくため、カメラの視野角や距離感がまだ固まっていない段階では調整しづらくなります。シーンの基本カメラが落ち着いてから入れるほうが扱いやすくなります。また、どの pass でも中間テクスチャを見られるようにしておくと、パラメータ調整がかなり速くなります。特に DOF では、`depth` と `focusMask` を見ずに調整を進めると原因の切り分けが難しくなります。 

## よくあるつまずき

パスを作ったのに画面が変わらない場合は、`autoDrawScene: false` にしておらず、シーンが直接キャンバスへ描画されている可能性があります。`beginScene()` と `render()` を描画フローへ組み込んでください。ポストプロセスは pass を生成するだけではなく、描画の流れ自体を切り替えて初めて動きます。 

resize 後に解像度がずれる場合は、`onUpdate()` で `bloom.resizeToScreen(screen)` や `dof.resizeToScreen(screen)` を呼んでいないことが多くなります。オフスクリーンターゲットは画面サイズへ追従させる必要があります。 

被写界深度が効かない場合は、projection の `near` / `far` がカメラと合っていないか、あるいはデバッグビューで深度を確認せずにパラメータを動かしている可能性があります。まず `depth` と `focusMask` を見るほうが安全です。 

HUD が出なくなった場合は、ポストプロセス処理の最終パスのあとで `app.screen.clearDepthBuffer()` を呼んでいない可能性があります。これは「エフェクトの一部」ではなく、「HUD を最後に重ねるためのフロー制御」だと考えると理解しやすくなります。 

`VignettePass` は source テクスチャを自分で作りません。シーンをどこかの `RenderTarget` へ描画し、そのテクスチャを `render({ source })` へ渡す必要があります。作っただけで動くと考えないほうが安全です。 

## 関連 sample / test

Bloom の標準サンプルは `webg/samples/bloom` です。パラメータ行、デバッグドック、`extractHeat`、blur quality 切替まで含めて確認できます。DOF の標準サンプルは `webg/samples/dof` で、focus guide、depth debug、focus mask、blur target の違いを確認できます。最小のポストプロセス処理テストとしては `webg/unittest/vignette` が向いています。`RenderTarget`、`FullscreenPass`、`VignettePass` の関係を最短で追えます。第15章の HUD / overlay と合わせて読むと、「後段処理のあとに何を重ねるか」という役割分担もさらに整理しやすくなります。
