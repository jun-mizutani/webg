# 22 タイルマップとは

最終更新：2026-04-09 JST

この章では、webg の `TileMap` を「どう定義し、どう構築し、ランタイムでどう使うか」という流れで見ていきます。

背景として、TileMap は単なる 2D 配列ではありません。各 cell は 3D 空間上の厚みを持つ床として扱われ、見た目、移動判定、選択、path 探索、表示範囲のスクロールを同じデータから扱います。ここが曖昧なままだと、見た目だけ tile を作って移動ルールが別になったり、クリック判定だけが独自処理になったりして、仕様がぶれやすくなります。この章では、そのぶれを避けるために、TileMap を使うときの流れを最初にそろえます。

## この章で押さえること

### A. TileMap は「見た目の床」ではなく、盤面の基準となる唯一のデータ

TileMap の `height` と `terrain` は、描画用の飾りではありません。`canMove()`、`pickCell()`、`findPath()`、`followCell()` も同じ cell データを見ます。つまり、TileMap を使うときは「先に cell データを決め、あとから node や camera を合わせる」と考えるほうが安全です。

### B. `tiles` を手で書く場合は、盤面を欠けなく全部埋める

`width` と `height` を決めたら、`tiles` を使う経路では、その範囲の全 cell を 1 回ずつ定義する必要があります。欠けている cell や重複した cell があると `build()` はエラーになります。手作りのマップを使うときは、ここを最初に確認すると原因を追いやすくなります。

### C. `displayArea` は盤面そのものではなく、現在見せる窓

TileMap では、盤面全体の定義と「今どこを見せるか」を分けます。`width` / `height` と `tiles` / `generator` が盤面そのもので、`displayArea` はその上に重ねる可視範囲です。広い地形で camera を追従させたいときは、この `displayArea` を `followCell()` で動かします。

### D. `canMove()` は判定だけを返し、実際の移動や演出は別に行う

`canMove(from, to)` は「その cell へ進めるか」を返すヘルパーです。node の位置更新や hop 演出までは行いません。見た目を動かすときは `getNodePositionOnCell()` や `placeNodeOnCell()` を使い、必要なら `Node.animatePosition()` を重ねます。

### E. `edgeCutMask` は通常は読むだけでよい

TileMap は構築の途中で周囲 4 方向の高さ差から `edgeCutMask` を自動で計算します。利用者が最初からこの値を手で作る必要はありません。見た目の削れ方を確認したいときに `getEdgeCutMask()` や `describeEdgeCutMask()` を読む、という理解で十分です。

## この章の流れ

TileMap を使うときは、次の順番で考えると迷いにくくなります。

1. `scene.tileMap.width` と `scene.tileMap.height` を決めます
2. 盤面の作り方を `tiles` か `generator` のどちらか 1 つに決めます
3. 必要なら `displayArea` を付けて、広いマップのどこを見せるかを決めます
4. texture や normal map を使うなら、構築前に `terrainMaterials` を定義します
5. `WebgApp.loadScene(scene)` か `TileMap.fromScene(...).build()` でランタイムを作ります
6. ランタイムでは `getTile()`、`canMove()`、`placeNodeOnCell()`、`followCell()`、`pickCell()` を使って移動と選択を組み立てます
7. `displayArea` や選択状態を変えたら、`refreshTileColors()` で現在の見え方を更新します

迷ったときの参照先は次の 4 つです。

1. TileMap を使った代表サンプルは `webg/samples/tile_sim`
2. API の最小確認は `webg/unittest/tilemap`
3. アニメーション付きキャラクターと TileMap を合わせた見方も `webg/samples/tile_sim` が中心になります

## 1. TileMap は何をまとめる部品か

TileMap は、3D 空間上の高さ付き盤面を 1 つのランタイムとして扱うヘルパーです。役割は大きく 5 つあります。

1. `tiles` または `generator` から full grid の cell を組み立てること
2. `terrainMaterials` を使って terrain ごとの見た目を分けること
3. `pickCell()`、`canMove()`、`findPath()` で盤面の判定をまとめること
4. `displayArea` と `followCell()` で可視範囲のスクロールを扱うこと
5. `placeNodeOnCell()` で「cell の上に object を置く」基準位置を返すこと

ここで大事なのは、TileMap が path 探索専用でも、描画専用でもないことです。見た目、入力、判定を同じ cell データへ寄せるための基盤として使います。

## 2. まず用意する Scene JSON

TileMap は `scene.tileMap` として定義します。最初に覚えるべき項目は、`width`、`height`、`tiles` または `generator`、そして必要なら `displayArea` です。

### 2.1 手作り map を使う最小例

この形は puzzle や固定ステージに向いています。

```js
const scene = {
  tileMap: {
    width: 4,
    height: 3,
    displayArea: {
      x: 0,
      y: 0,
      width: 4,
      height: 3
    },
    tiles: [
      { x: 0, y: 0, height: 0, terrain: "floor" },
      { x: 1, y: 0, height: 0, terrain: "floor" },
      { x: 2, y: 0, height: 1, terrain: "slope" },
      { x: 3, y: 0, height: 2, terrain: "peak" },

      { x: 0, y: 1, height: 0, terrain: "floor" },
      { x: 1, y: 1, height: 1, terrain: "slope" },
      { x: 2, y: 1, height: 1, terrain: "slope" },
      { x: 3, y: 1, height: 2, terrain: "peak" },

      { x: 0, y: 2, height: 0, terrain: "floor" },
      { x: 1, y: 2, height: 0, terrain: "floor" },
      { x: 2, y: 2, height: 1, terrain: "slope" },
      { x: 3, y: 2, height: 1, terrain: "wall" }
    ]
  }
};
```

この例で重要なのは、`width=4`、`height=3` に対して 12 cell を全部書いていることです。TileMap は「書かれていない cell を自動で floor 扱いする」仕様ではありません。手作業で抜けが出やすいので、最初は小さい盤面から始めるほうが安全です。

### 2.2 自動生成を使う最小例

この形は terrain の見え方確認や広いマップの試作に向いています。

```js
const scene = {
  tileMap: {
    width: 20,
    height: 15,
    displayArea: {
      x: 0,
      y: 0,
      width: 10,
      height: 8
    },
    generator: {
      type: "noise",
      seed: 42,
      heightMin: 0,
      heightMax: 5
    }
  }
};
```

`generator` を使うと、`width * height` 分の cell は自動で埋まります。`terrain` は height から自動で `floor` / `slope` / `peak` へ分類されます。最初に形の確認だけをしたいときは、この経路のほうが早く始められます。

### 2.3 `displayArea` の意味

`displayArea` は、盤面全体ではなく「現在見せる窓」です。`x` / `y` / `width` / `height` で書く形がいちばん分かりやすく、内部では `minCol` / `maxCol` / `minRow` / `maxRow` を持つ矩形へ正規化されます。

現在の TileMap 実装では、描画と pick はこの窓そのものではなく、その周囲に少し余白を足した範囲を候補にします。

既定では `displayAreaPadding = 2` として扱い、画面端で tile が急に消えないようにしています。

また、cell の見た目は full height の柱を最後まで texture 付きで描くのではなく、上面側だけを薄い cap として描きます。

既定では `visibleHeightSteps = 1` なので、各 cell は「最上面から 1 step 分だけ」を texture 付き cap として描画し、その下側は必要なぶんだけ texture を使わない foundation で埋めます。

移動判定や pick に使う full height box とは役割を分けています。

```js
displayArea: {
  x: 3,
  y: 2,
  width: 8,
  height: 6
}
```

この例なら、列 3 から 10、行 2 から 7 までを見せます。盤面より大きい矩形を書いても、TileMap 側で盤面サイズへ収まるように制限されます。必要なら `displayAreaPadding` を scene 定義側で上書きして、描画余白の広さを調整できます。

### 2.4 TileMap の constructor で設定する値

`scene.tileMap` には `width`、`height`、`tiles`、`generator` だけでなく、TileMap ランタイムの既定値も含められます。ここは単なる飾りではなく、cell のワールド座標、見た目の厚み、表示窓の追従位置に直接効きます。コードを読むときは、「盤面の平面寸法を決める値」「高さ座標を決める値」「描画上の cap の見せ方を決める値」「表示窓の follow を決める値」に分けて追うと理解しやすくなります。

#### `cellSize`

1 cell を grid 上で何 unit 間隔で並べるかを決めます。`createCellSpec()` では `col * cellSize`、`row * cellSize` を基準にして cell の `minx` / `maxx` / `minz` / `maxz` を組み立てるため、盤面全体の広さや cell 中心の位置計算にそのまま入ります。`build()` の最後に計算する `bounds.centerX`、`bounds.centerZ` もこの値を使います。つまり、`cellSize` は「見た目の 1 マス幅」であると同時に、「ワールド上で隣の cell がどれだけ離れるか」を決める値です。

#### `cellGap`

隣り合う cell のあいだに残す隙間です。`createCellSpec()` では `cellGap * 0.5` を左右に配り、cell の box を内側へ少し縮めます。これにより、cell 同士が完全に接した一枚板ではなく、盤面として読みやすい区切りが出ます。`tileSize = cellSize - cellGap` は cell の実幅そのものなので、cap shape や foundation shape の平面寸法を読むときも、この値の感覚を基準にすると追いやすくなります。

#### `floorY`

すべての cell に共通する底面の Y 座標です。`createCellSpec()` では `box.miny` に入り、cell の中心 Y や foundation の高さ計算の基準にもなります。つまり `floorY` を下げると、盤面全体がワールドの下側へ沈み、`baseTopY` や `heightStepY` が同じでも柱の全高は大きくなります。TileMap の高さ表現を読むときは、「底がどこにあるか」を決める値として見ると分かりやすくなります。

#### `baseTopY`

`height = 0` の cell でも、上面をどの高さへ置くかを決める基準値です。`createCellSpec()` では `topY = baseTopY + height * heightStepY` を使うため、`baseTopY` は「地表の 0 段目」をどこに置くかの基準になります。たとえば `floorY` を固定したまま `baseTopY` を上げると、すべての cell の上面が一様に持ち上がり、height 差はそのままでも見た目の柱は厚くなります。

#### `heightStepY`

cell の `height` が 1 増えるごとに、上面 `topY` を何 unit 上げるかを決める値です。`height` は論理的な段数で、実際のワールド高さは `heightStepY` を掛けて作ります。したがって `heightStepY` を大きくすると、同じ `height = 3` でもより背の高い柱に見えます。TileMap の移動判定や `edgeCutMask` は論理 height を見ますが、描画上の高さ差や node を置く位置はこの値を通してワールド座標へ変換されます。

#### `visibleHeightSteps`

これは論理高さそのものではなく、描画上で上面側を何 step 分だけ texture 付きの cap として見せるかを決める値です。`resolveVisibleCapHeight()` では `heightStepY * visibleHeightSteps` を cap の厚みとして計算し、残りの高さは `resolveFoundationHeight()` により texture を使わない foundation 側へ回します。そのため `visibleHeightSteps = 1` のときは、各 cell の最上面から 1 段分だけが texture 付きの主役として見え、背の高い cell でも側面の情報量が増えすぎません。ここは「盤面データを変える値」ではなく、「同じ盤面をどう見せるかを変える値」です。

#### `surfaceShading`

cap 部分の法線を `flat` のまま保つか、`smooth` で共有するかを決める値です。既定値は `flat` で、top face と各 side face が独立した頂点を持ち、cell の edge が立った見え方になります。`smooth` を指定すると、cell 上部の cap だけを shared vertex で組み立て直し、top と斜面の法線が corner ごとに平均化されます。これにより、tile の gap や foundation は保ったまま、上面まわりだけを少し滑らかに見せられます。現在の smooth cap はさらに、上端へ横方向比率 0.05、縦方向もほぼ同程度の bevel を入れて中央の平らな面を少し残すため、bevel 帯そのものも視認しやすくなっています。

ここで大事なのは、`smooth` がマップ全体を 1 枚の地形 sheet に変えるわけではないことです。`cellGap` による隙間と、cell ごとの独立した foundation はそのまま残ります。つまり、「盤面の区切りは残しつつ、各 cell の上部だけを共有法線で試す」ための option です。まずは `flat` を比較基準にして通常の見え方を確認し、そのあとで `smooth` を試すと差が追いやすくなります。

```js
tileMap: {
  width: 8,
  height: 8,
  visibleHeightSteps: 1,
  surfaceShading: "smooth",
  tiles
}
```

#### `displayFollowCol`

`followCell()` を使って表示窓を player などへ追従させるとき、対象 cell を表示矩形の左から何列目に置くかを決める既定値です。`followCell()` は `minCol = cell.col - followCol` を基準に表示矩形を組み立てるため、この値が 0 なら対象 cell は左端寄り、1 なら左から 2 列目、2 ならさらに中央寄りに置かれます。camera そのものの角度を決める値ではなく、「スクロール後に対象が窓のどこへ来るか」を決める値と考えると読みやすくなります。

#### `displayFollowRow`

`displayFollowCol` の行方向版です。`followCell()` では `minRow = cell.row - followRow` を使うため、対象 cell を表示矩形の上から何行目に置くかがこの値で決まります。見下ろし型のパズルや斜め見下ろし camera では、player を完全中央に置くより少し手前寄りに保つほうが先読みしやすい場合があるため、この値で表示窓の追従位置を調整します。

#### `displayAreaPadding`

実際の `displayArea` の外側へ、何 cell 分余白を足して描画候補と pick 候補を集めるかを決める値です。`getDisplayAreaCells()` では `minCol - padding`、`maxCol + padding` のように範囲を少し広げて cell を集めるため、表示窓ぎりぎりで tile が急に消えたり、画面端で pick が不安定になったりするのを抑えられます。つまり `displayArea` が「今見せたい窓」だとすると、`displayAreaPadding` は「その窓の外側にどれだけ緩衝帯を持つか」を決める値です。

これらの既定値をまとめて持てることで、TileMap は「盤面データ」と「その盤面をワールド上でどう見せるか」を同じ scene 定義の中で扱えます。コードを読むときは、`cellSize` / `cellGap` が XZ 平面、`floorY` / `baseTopY` / `heightStepY` が Y 座標、`visibleHeightSteps` が描画表現、`displayFollowCol` / `displayFollowRow` / `displayAreaPadding` が表示窓の挙動に属している、と分けて追うと混乱しにくくなります。

## 3. texture や normal map を使うときの `terrainMaterials`

terrain ごとの見た目を変えたいときは、`scene.tileMap.terrainMaterials` を構築前に定義します。ここは `TileMap.build()` の中で読まれるため、構築後に差し替えるより、Scene JSON を作る段階でそろえるほうが自然です。

```js
scene.tileMap.terrainMaterials = {
  default: {
    materialId: "norm-phong",
    color: [0.70, 0.74, 0.84, 1.0],
    texture: groundTexture,
    normal_texture: groundNormalTexture,
    use_texture: 1,
    use_normal_map: 1
  },
  floor: {
    color: [0.30, 0.54, 0.96, 1.0]
  },
  slope: {
    color: [0.36, 0.70, 0.42, 1.0]
  },
  peak: {
    color: [0.92, 0.76, 0.42, 1.0]
  },
  wall: {
    color: [0.52, 0.30, 0.30, 1.0]
  }
};
```

`default` が土台になり、`floor` や `peak` の個別設定があれば上書きされます。normal map を効かせる場合は `NormPhong` のような対応シェーダーを選ぶ必要があります。`webg/samples/tile_sim` は terrain ごとの見え方をサンプル全体の中で確認しやすい参照先です。

ここで注意したいのは、TileMap の surface UV は通常の `Shape` の自動 UV とは少し考え方が違うことです。TileMap では、`cap` の上面だけが「cell の外形を上から見た投影座標」を使い、外形の左端を `U=0`、右端を `U=1`、上端を `V=0`、下端を `V=1` として扱います。たとえば east / west を 0.2 ずつ削った cell なら、上面の U 範囲は `0.2 .. 0.8` になります。ここを上面だけ `0 .. 1` に再正規化しないのが重要です。

一方で側面は、各面ごとに独立した単純な UV を使います。上面と同じ投影をそのまま共有するのではなく、面単位で `topuv` と `bottomuv` を結んだ分かりやすい貼り方を保つことで、削れた斜面と垂直面の両方が同じ規則で読めるようにしています。そのため、TileMap の見た目を調整したいときは、追加の UV 数値を増やすよりも、`edgeCutMask` 後の形状と texture 画像そのものの見え方を意識するほうが自然です。foundation の側面だけは別扱いで、面を正面から見たときに左下が `(0,0)`、右上が `(1,1)` になる矩形 UV を先に置き、そのあと高さ分だけ上端 `V` を補正する手順を使います。`surfaceShading: "smooth"` を使う場合も、外周 shoulder 側には従来の `topuv`、中央の平らな top 側にはそれを少し内側へ寄せた UV、bottom 側には `bottomuv` を持たせるため、貼り方の規則自体は大きく変わりません。

実装上の注意として、TileMap は `addVertexPosUV()` で明示的に指定した UV を各頂点へ直接入れています。このとき `Shape` の texture mapping mode を既定の `0` のままにすると、球面 UV 用の seam 補正が `addTriangle()` 側で走り、`0..1` の手動で指定した UV まで改変されます。TileMap ではこれを避けるため、shape 作成時に texture mapping mode を `1` に切り替えて、旧来の sphere seam 補正が入らないようにしています。ここを忘れると、上面へ 1 枚画像を貼るだけのはずが、edge 側を引き延ばしたような見え方になります。

もう 1 つ大事なのは、TileMap の論理的な UV 自体は `webg` 全体の規約に合わせて Blender と同じく「下が 0」であることです。つまり、上面の `nw / ne / sw / se` を考えるときも、`v` は south 側が `0`、north 側が `1` です。一方で WebGPU へ画像を取り込むと、画像の上下の見え方とはそのまま一致しない場合があります。そのため、`num256.png` のような確認用画像を使うときは、論理的な UV をひっくり返すのではなく、画像入力側で `flipV` の補正を入れて向きを合わせるのが正しい整理です。

さらに、面の表裏も TileMap 実装を読むときの重要な前提です。webg のシェーダー既定値は `frontFace: "ccw"` なので、外側から見て反時計回り（CCW）に並んだ頂点が表面になります。現在の `TileMap.js` では `addPrismSlice()` が north / east / west / south / top / bottom の各面を「外側から見て CCW」の順で `addQuadFace()` へ渡し、`addQuadFace()` はその順をそのまま `addPlane()` に渡します。以前のように helper 側で逆転させる前提ではなくなったため、面の点順を読むだけで法線の向きを追えるようになっています。

## 4. build の標準フロー

新規アプリでは、`WebgApp` と `loadScene()` を使う経路がいちばんそろえやすくなります。

```js
import WebgApp from "./webg/WebgApp.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  camera: {
    target: [8.0, 0.0, 8.0],
    distance: 18.0,
    yaw: 0.0,
    pitch: -26.0,
    bank: 0.0
  }
});

await app.init({
  shader: "norm-phong"
});

const runtime = await app.loadScene(scene);
const tileMap = runtime.tileMap;
```

この形の利点は、Screen、Shader、Space、camera を別々に組み立てなくてよいことです。tileMap だけでなく primitives や models も同じ `scene` に含められるので、サンプルを大きくしていくときにも処理フローを変えずに済みます。

TileMap 単体を直接構築したいときは、`TileMap.fromScene()` を使います。

```js
import TileMap from "./webg/TileMap.js";

const tileMap = TileMap.fromScene(scene, app.space, app.getGL());
tileMap.build();
```

この経路は unittest や小さい確認用スクリプトに向いています。複数 entry を含む scene 全体を扱うなら、`app.loadScene(scene)` のほうが自然です。
