# インストールと実行環境

この章は、`webg` を手元のマシンに配置し、サンプルや unittest を実際に起動するまでの手順を整理した実行ガイドです。第1章がプロジェクト全体の入口であり、第4章以降が最初のアプリの組み立て方につながるとすると、この章は「まず何をどこへ置き、どう起動すればよいか」をそろえるための章です。 

`webg` リポジトリをそのまま手元へ置き、ローカルサーバー経由でサンプルを開く流れを標準的な実行手順とします。最初にここを明確にしておかないと、描画ロジックの問題と実行方法の問題が混ざりやすくなります。たとえば、実装自体は正しいのにファイルをブラウザで直接開いたためにアセット読み込みだけが失敗すると、「WebGPU のコードが壊れているのか」「ブラウザが対応していないのか」「パスが違うのか」を同時に疑うことになり、切り分けが難しくなります。 

この章で最初に押さえておきたいのは、`webg` では「パッケージをインストールする」というより、「リポジトリをそのまま使う」という形が基本だという点です。また、`webg/samples/index.html` や `webg/unittest/index.html` のような一覧ページはブラウザでファイルを直接開けることがありますが、実際に描画を行うサンプルや unittest は、ES Modules、`fetch()`、アセット読み込み、`localhost` のセキュアコンテキストが必要な条件を含むため、ローカルサーバーで開くほうが確実です。さらに、ローカルサーバーで起動できたとしても、WebGPU を使えるかどうかはブラウザ側の条件にも依存します。したがって、最初の段階では「サーバーの問題」と「ブラウザの WebGPU 対応」を分けて考えることが重要です。

## なぜローカルサーバーを使うのか

`webg` のサンプルは、単なる HTML 1 枚ではありません。`main.js`、`webg/*.js`、`webg/samples/*/*.txt`、JSON、テクスチャ、glTF / GLB、Collada など、複数のファイルを経由して構成されています。このときブラウザは、「どのファイルをどのオリジンから読んでいるか」をセキュリティ上の重要な条件として扱います。`file://` で直接開くと、オリジンの扱いが HTTP 配信時と異なり、モジュールの import や `fetch()` がブラウザごとに不安定になりやすくなります。

そのため、`webg` ではローカルサーバーを 1 本立てて、リポジトリのルートディレクトリを配信し、`localhost` 経由でサンプルを開く方法を標準として採用するのが自然です。すでに LAN 内に Web サーバーがある環境では、そのサーバーに `webg` を配置し、配信された URL から同じ構成を開いても構いません。

この章の役割は、次の判断を早く行えるようにすることです。何をインストールすればよいか、リポジトリをどこから実行すればよいか、`file://` で直接開けるものとローカルサーバーが必要なものは何か、LAN 内の既存 Web サーバーに置いた場合はどのように開けばよいか、Mac / Windows / Linux でどのコマンドを使えばサーバーを立てられるか、最初にどの URL を開くと確認しやすいか、という点です。

また、この章は次のような場面で使います。初めて `webg` を手元で起動するとき、サンプル一覧は見えるのに個別ページが動かないとき、`file://` と `http://localhost` の違いを整理したいとき、OS ごとの簡単なローカルサーバー起動方法を知りたいとき、LAN 内の既存 Web サーバーへ置いて実行できるかを判断したいとき、他者や AI に「まずどう実行するか」を共有したいときです。

最初の段階で特に混乱しやすいのは、`webg/samples/index.html` が開けたので個別サンプルもそのまま `file://` で動くと思ってしまうこと、ブラウザのコンソールに CORS、module、fetch のエラーが出ても描画コードの問題だと考えてしまうこと、リポジトリのルートディレクトリではなく `webg/samples/scene/` のような途中階層をサーバーのルートディレクトリにしてしまうこと、LAN 内の Web サーバーへ置いたが `navigator.gpu` が出ない理由をサーバー側の問題だけだと思ってしまうこと、WebGPU 非対応のブラウザとローカルサーバー未使用の問題を混同すること、アセットを読むサンプルと自己完結に見えるサンプルを同じ条件で考えてしまうことです。 

## まずファイルを手元へ置く

`webg` は、リポジトリをそのまま手元に置く運用が基本です。方法は、`git clone` しても、フォルダごとコピーしても構いません。 

### `git clone` する場合

```bash
git clone https://github.com/jun-mizutani/webg.git
cd webg
```

履歴を残したまま更新したいときは、この方法が自然です。 

### フォルダごとコピーする場合
https://github.com/jun-mizutani/webg/archive/refs/heads/main.zip から zip ファイルをダウンロードして、webg フォルダごと作業場所へ置いても構いません。最初は展開した webg フォルダ中に例えば my_webg フォルダを作成して、その中にプロジェクトフォルダを作成すると、サンプルと同じインポートの書き方で webg のコア機能を試すことができます。 インポートパスさえ変更すればどこにおいても問題ありません。

```text
webg/
  book/
  samples/
  unittest/
  webg/
  my_webg/
```

`webg/samples/` や `webg/` だけを切り出すと相対パスが崩れるため、リポジトリのルートディレクトリを保ったまま開いてください。

## 直接開けるものと、直接開かないほうがよいもの

### フォルダから直接開きやすいもの

次のような静的なページは、`file://` で直接開いて読めることがあります。これらは主に一覧や説明を表示するページであり、アセット読み込みやモジュール実行に強く依存しません。 

* `webg/samples/index.html`
* `webg/unittest/index.html`
* 文書ファイル自体

### フォルダからブラウザで直接開かないほうがよいもの

次のような実行系ページは、ローカルサーバーで開くものとして考えてください。理由は、これらのページが `<script type="module">` による ES Modules、`fetch()` によるテクスチャ、JSON、glTF / GLB、Collada の読み込み、相対パスで参照する補助ファイル群、`navigator.gpu` を使う WebGPU 初期化に依存しているためです。 

* 最小コードの実行ページ
* `webg/samples/*/index.html` の大半
* `webg/unittest/*/index.html` の大半
* `webg/book/examples/*.html` の大半

リポジトリ内では、たとえば次の層が `fetch()` を使っています。したがって、ページ自体は開けても、テクスチャやモデルを読み込んだ瞬間に失敗することがあります。 

* `webg/Texture.js`
* `webg/ModelAsset.js`
* `webg/SceneAsset.js`
* `webg/Gltf.js`
* `webg/GltfShape.js`

### なぜローカルファイルをブラウザで直接開けないのか

ブラウザはローカルファイルを自由に読めるようにすると、他のファイルを勝手に読むページも作れてしまいます。そのため、`file://` 上のページには HTTP 配信時と同じ自由度を与えません。ここで関係するのは、主に **Same-Origin Policy (SOP)** と **CORS (Cross-Origin Resource Sharing)** です。SOP は、ブラウザがセキュリティを確保するための基本ルールで、ページがアクセスできるリソースを「同一オリジン」と「他オリジン」に区別します。同一オリジンとは、プロトコル、ホスト、ポートの 3 要素がすべて一致する場合です。いずれかが異なれば、別オリジンとして扱われます。たとえば、`http://example.com` で配信されたページは、`http://other.com` からのリソースや、`file://` で開かれたローカルファイルに制限なくアクセスできません。 

`file://` プロトコルで開かれたページは、HTTP 配信時とは異なる扱いを受けます。多くのブラウザでは、`file://` からのリクエストは特殊なセキュリティ領域として扱われ、CORS の仕組みが正しく機能しない、あるいは厳格に制限されます。CORS は、HTTP 通信においてサーバー側が許可を明示することで、異なるオリジンからのリクエストを認める仕組みです。しかし、`file://` 環境ではサーバーが存在しないため、この仕組みを利用できません。その結果、`fetch()` や ES Modules の読み込み時に、ブラウザが「同一オリジンではないリソースへのアクセスは許可しない」と判断し、エラーが発生します。 

これに対して、ローカルサーバーを使って `http://127.0.0.1:8000/` や `http://localhost:8000/` で開くと、相対パスのモジュール import や `fetch()` を通常の Web ページと同じ流れで扱えます。`http://localhost:8000/` で配信されたページは、同じオリジンのファイルを相対パスで読みやすくなります。一方で `file://` はオリジンの扱いが特殊であり、モジュールや `fetch()` がブラウザごとに制限されやすくなります。`webg` のサンプルは学習用であると同時に、複数ファイルを実際に読み込むアプリでもあるため、セキュリティ上の標準的な Web 実行条件にそろえておくほうが安全です。 

## LAN 内の Web サーバーへ置く場合

LAN 内に既存の Web サーバーがあるなら、`webg` のリポジトリ内容をその document root 配下へ配置し、配信された URL から実行できます。たとえば、次のような形です。

```text
http://192.168.1.20/webg/samples/index.html
http://devbox.local/webg/samples/index.html
```

この方法でも、モジュール import、`fetch()`、相対パスの解決は通常の Web ページと同じ流れで扱えます。ただし、WebGPU はブラウザによって secure context を要求します。`localhost` は例外的に扱われやすい一方で、LAN 内の IP アドレスやホスト名は、HTTP のままだと `navigator.gpu` が利用できないことがあります。

そのため、LAN 内の Web サーバーから `webg` を実行する場合は、まず URL 配信でモジュールとアセット読み込みが正常に動くことを確認し、そのうえで `navigator.gpu` が見えない場合は HTTP 配信そのものではなく secure context の条件を疑い、可能なら HTTPS で配信し、HTTPS が難しい場合は最初の動作確認だけでも `localhost` に戻して切り分ける、という順で考えるのが安全です。

## ローカルサーバーを起動する

この章では、リポジトリのルートディレクトリをカレントディレクトリにして起動する条件で書きます。

```bash
cd /path/to/webg
```

最も手軽なのは、Python 3 の簡易サーバーです。起動したら、ブラウザで `http://127.0.0.1:8000/samples/index.html` または `http://localhost:8000/samples/index.html` を開きます。

```bash
python3 -m http.server 8000
```

```text
http://127.0.0.1:8000/samples/index.html
http://localhost:8000/samples/index.html
```

### Mac での代表的な方法

#### 方法 A: Python 3

```bash
cd /path/to/webg
python3 -m http.server 8000
```

多くの環境で最初に試しやすい方法です。 

#### 方法 B: Node.js + `http-server`

```bash
cd /path/to/webg
npx http-server . -p 8000
```

Node 系の開発環境に慣れている場合はこちらでも扱いやすくなります。 

#### 方法 C: PHP built-in server

```bash
cd /path/to/webg
php -S 127.0.0.1:8000
```

PHP が入っている環境では、この方法も使えます。 

### Windows での代表的な方法

#### 方法 A: Python Launcher

```powershell
cd C:\path\to\webg
py -m http.server 8000
```

Windows では `py` がある環境なら、この方法が簡単です。 

#### 方法 B: Python 3

```powershell
cd C:\path\to\webg
python -m http.server 8000
```

`python` コマンドが通るなら、こちらでも構いません。 

#### 方法 C: Node.js + `http-server`

```powershell
cd C:\path\to\webg
npx http-server . -p 8000
```

Node.js の環境がある場合はこちらも使えます。 

### Linux での代表的な方法

#### 方法 A: Python 3

```bash
cd /path/to/webg
python3 -m http.server 8000
```

最も標準的な方法です。 

#### 方法 B: Node.js + `http-server`

```bash
cd /path/to/webg
npx http-server . -p 8000
```

Node 系の環境を使っているならこちらでも構いません。 

#### 方法 C: PHP built-in server

```bash
cd /path/to/webg
php -S 127.0.0.1:8000
```

PHP が入っている場合の代替手段です。 

他のアプリとポートが衝突するときは、ポートだけ変えてください。たとえば 9000 番なら次のようになります。 

```bash
python3 -m http.server 9000
```

```text
http://127.0.0.1:9000/samples/index.html
```

## 最初にどこを開くか

最初の確認では、いきなり複雑なサンプルへ行くより、まず一覧ページを開き、そのあと最小のサンプルへ進むほうが安全です。`webg/samples/index.html` は全体の入口として最も見やすく、そのあと `low_level`、`high_level`、`scene` を順に見ると、ローレベル、高水準、Scene JSON の 3 つの入り口を自然に確認できます。 

順番としては、次のように進めると迷いにくくなります。 

1. `samples/index.html` を開く
2. `low_level` で最小描画を確認する
3. `high_level` で `WebgApp` の最小構成を確認する
4. `scene` で Scene JSON の入口を確認する
5. その後に `sound`、`bloom`、`dof`、`tile_sim` など、目的に応じたサンプルへ進む

この流れにしておくと、問題が起きたときも「最小構成では動くのか」「高水準構成では動くのか」「アセット読み込みを含むと止まるのか」といった切り分けがしやすくなります。 

## 動かないときは何を確認するか

ローカルサーバーを立てても動かないときは、最初に「サーバーの問題」と「ブラウザの WebGPU 対応」を分けて考えます。まず URL で一覧ページが見えるか、個別サンプルの JS やアセット読み込みで 404 や CORS が出ていないかを確認します。そのうえで、`navigator.gpu` が利用できるかどうかを見ます。`webg/Screen.js` は `navigator.gpu` を確認して WebGPU を初期化するため、ここが無ければ描画は始まりません。つまり、ローカルサーバーの問題と WebGPU 非対応の問題は、最初に切り分けておく必要があります。

サンプル一覧は見えるのに個別ページが動かない場合は、相対パスの崩れ、アセット読み込み、`file://` 実行、途中階層をサーバールートにしたこと、ブラウザ側の WebGPU 条件などを順に疑うほうが安全です。最小の `low_level` まで戻して動くかどうかを確認すると、問題の層を絞りやすくなります。

## まとめ

この章で最も大事なのは、`webg` の実行は「パッケージを入れる」ことより、「リポジトリをそのまま置き、ローカルサーバー経由で開く」ことが基本だと最初に理解することです。 `file://` で静的な一覧が見えても、実行系ページは ES Modules、`fetch()`、アセット読み込み、WebGPU 初期化を含むため、標準手順としてはローカルサーバーを使うほうが安全です。

また、動かないときは「サーバーの問題」と「ブラウザ側の WebGPU 条件」を分けて考えることが大切です。この基準があると、第4章以降で最小描画や `WebgApp` を読み始めたときにも、実装の問題と実行条件の問題を混同しにくくなります。次章では、こうしてそろえた実行環境の上で、`webg` が採用している 3D の基礎定義、つまり座標系、回転、行列、クォータニオン、UV、ノーマルマップの基礎事項を整理していきます。
