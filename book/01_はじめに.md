# はじめに

`webg` は、JavaScript から WebGPU を使って 3D アプリを構築するためのライブラリです。PC やスマートフォンのブラウザ上で高速に動作する 3D アプリを作成することを目的としており、キー入力やタッチボタン、ピンチ操作、効果音や BGM の再生、GLB のアニメーションのインポート、パーティクル、被写界深度、フォグなど、3D 表現に必要となる機能を一通り備えています。外部ライブラリや各種のビルドツールを使う必要もなく、すべての必要なコードと詳細な解説は `webg` 自身に含まれています。 そのため、学習目的にも、AI を活用したアプリ開発にも適しています。また、WebGPU を使って 3D アプリを作るうえで必要になる座標計算や GPU 制御といったローレベルのプログラミングだけでなく、多くの機能が抽象化された `WebgApp` クラスによる高水準なプログラミングも行えます。

`webg` では、ローレベルと高水準の両方を扱えます。そのため、最初に必要なのはメソッド名を暗記することではなく、「どのような用途に向いたライブラリなのか」「どの文書から読むと理解しやすいのか」「サンプルと unittest をどう読み分ければよいのか」といった進み方の基準をつかむことです。

この章では、`webg` の全体像、向いている用途、プロジェクト内の配置、標準的な使い始め方、サンプルの見方、問題調査の考え方、そして現時点で未実装の範囲を順に整理します。細部のメソッド一覧やクラス単位の仕様は付録B「API一覧」に譲り、この章では「どこから入り、どう読み進めるか」を判断できるようにすることを目的にします。

## この本をどのように読むか

`webg` の大きな特徴は、3D の基礎からアプリ構成までを、１つのリポジトリの中で完結していることです。描画、シーングラフ、アセット、アニメーション、HUD、入力、診断情報、音声といった要素を、別々の外部資料を調査する必要なく順に追えるようになっています。これは、単に機能が多いということではなく、「3D アプリを構成する層がどこにあり、どの順番で読めばよいか」が見えやすい、という意味でもあります。

そのため、本書ではまず実行環境をそろえ、次に 3D の基礎と最小描画を押さえ、その後で `WebgApp`、アセット、アニメーション、TileMap、UI、診断情報、ポストプロセスへと進む流れを基準にしています。最初から細部の API 一覧を見るよりも、この順序を把握しておくほうが、学習の効率は高くなります。

また、`webg/samples/` は単なるデモの集まりではなく、教材でもあります。各サンプルは見栄えの確認だけでなく、「どのように作るか」を示すために構成されており、`main.js` と `*.txt` を合わせて読むことを前提にしています。一方で、`webg/unittest/` は公開サンプルとは役割が異なり、より局所的な確認や切り分けのための入口です。サンプルは統合例、unittest は局所検証という位置づけで読み分けると、問題の見方が整理しやすくなります。

`webg` を使い始めるときは、次の順で読むと迷いにくくなります。

1. 第2章「インストールと実行環境」で実行環境をそろえる
2. 第3章「3Dグラフィックスの基礎」で座標系と回転を押さえる
3. 第4章「WebGPUとwebgの最小描画」で最初の描画フローをつかむ
4. 第5章「WebgAppによるアプリ構成」で標準的なアプリ構成へ進む
5. `webg/samples/index.html` から `low_level`、`high_level`、`scene` を読む
6. 付録A「生成AIの皆さまへ」は生成AIを利用して開発する場合は最初に生成AIに読んでもらう
7. その後、必要な分野の章と付録へ進む

## `webg` はどのようなライブラリか

WebGPU を使った 3D 実装では、描画が出るところまで進めても、少し規模が大きくなるとシーングラフ、アセット、カメラ、入力、診断情報、音声といった処理が別に必要となります。特に AI と一緒に実装を進める場合は、「どこに何の役割があるか」「何を読めば今の挙動が分かるか」が曖昧だと、調査や修正の速度が大きく落ちます。

`webg` は、この問題を避けるために、3D アプリを構成する層をリポジトリ内で見分けやすくし、サンプルと文書を並べて読めるようにしています。ひとつの巨大なエンジンをブラックボックスとして使うのではなく、今のプロジェクトに必要な部分を順に読めるようにしていることが特徴です。

この章の役割も、まさにそこにあります。ここで知りたいのは API の詳細ではありません。`webg` がどのような 3D アプリに向いているのか、プロジェクトの中で何がどこに置かれているのか、どの文書から読み始めればよいのか、ローレベルと高水準のどちらで始めるべきか、代表的なサンプルはどれか、といった判断材料です。細部のメソッド一覧やクラス単位の仕様は付録「API一覧」に分け、この章は入口として位置づける文書にしています。

`webg` は、次のような場面に向いています。

* WebGPU を使った 3D アプリを、自分で構造を追いながら組みたいとき
* サンプルと実装を往復しながら学びたいとき
* 診断情報、HUD、タッチ入力、音声まで含めて、1 つのリポジトリの中で完結させたいとき
* AI に3Dアプリ開発を支援してもらうとき

また、`webg` を読むときは、最初にいくつかの前提をそろえておくと判断がぶれにくくなります。サンプルは単なるデモではなく教材でもあること、`Touch`、`Diagnostics`、`Message`、`UIPanel` などの補助機能もプロジェクトの一部として使うこと、3D の定義は第3章「3Dグラフィックスの基礎」にそろえること、キーボードと Touch を統一的に扱う場合は第16章「タッチ機能と入力」を参考にすることです。

## プロジェクトの中に何があるか

`webg` を読むときは、まずファイルの配置を役割ごとに見分けると分かりやすくなります。コアライブラリ本体は `webg/` にあり、`Screen`、`Space`、`Node`、`Shape`、`WebgApp`、`ModelBuilder`、`SceneLoader` などの実装がここにあります。

`webg/samples/` は、機能確認とデモを兼ねたサンプル群です。各サンプルには `main.js` と `*.txt` があり、`*.txt` が「何を示すサンプルか」を説明し、`main.js` がその実装を示します。`webg/samples/index.html` はその全体目次であり、`webg/samples/*/*.txt` の概要も転記しています。

`webg/unittest/` は、より小さく切り出したテストアプリの集まりです。機能の切り分けや検証に向いており、`webg/unittest/index.html` がその目次になります。

このように、`webg` はライブラリ本体、教材としてのサンプル、切り分け用の unittest を明確に分けています。第1章の段階でこの配置を頭に入れておくと、このあと章を進めるときにも迷いにくくなります。

## `webg` でできること

`webg` は、WebGPU 上で 3D アプリを組むための基礎を一通り備えています。描画基盤としては `Screen`、`Shader`、`Shape`、`Space`、`Node`、`RenderTarget` があり、アプリの入口として `WebgApp` があります。アセットとローダーには `ModelAsset`、`ModelLoader`、`ModelBuilder`、`SceneAsset`、`SceneLoader` があり、アニメーションには `Animation`、`Action`、`AnimationState`、`Skeleton` があります。カメラと入力には `EyeRig`、`Touch`、`InputController` があり、UI と診断情報には `Message`、`DialogueOverlay`、`UIPanel`、`FixedFormatPanel`、`Diagnostics`、`DebugProbe` が含まれます。音声には `AudioSynth` と `GameAudioSynth`、ポストプロセスには `FullscreenPass`、`BloomPass`、`DofPass`、`VignettePass` があります。

このように、`webg` は単に 3D を描画するだけでなく、3D アプリとして必要になる周辺機能まで視野に入れて設計されています。だからこそ、ローレベルの描画だけでなく、`WebgApp` を使った高水準な構成まで、ひとつのリポジトリで開発することができます。

## 標準的な使い始め方

`webg` の使い始め方には、大きく分けて 2 つの入口があります。ひとつは、ローレベルで最小構成を理解する方法です。`Screen`、`Space`、`Shape`、`Matrix`、`SmoothShader` を直接つなぐと、描画の流れを最短距離で確認できます。`webg/samples/low_level` は、この章の例をそのまま切り出したサンプルです。

```js
import Screen from "./webg/Screen.js";
import Space from "./webg/Space.js";
import Shape from "./webg/Shape.js";
import Primitive from "./webg/Primitive.js";
import Matrix from "./webg/Matrix.js";
import SmoothShader from "./webg/SmoothShader.js";

const screen = new Screen(document);
await screen.ready;
screen.setClearColor([0.1, 0.15, 0.1, 1.0]);

const shader = new SmoothShader(screen.getGL());
await shader.init();
Shape.prototype.shader = shader;

const projection = new Matrix();
const applyViewportLayout = () => {
  screen.resize(
    Math.max(1, Math.floor(window.innerWidth)),
    Math.max(1, Math.floor(window.innerHeight))
  );
  projection.makeProjectionMatrix(
    0.1,
    1000.0,
    screen.getRecommendedFov(55.0),
    screen.getAspect()
  );
  shader.setProjectionMatrix(projection);
};
applyViewportLayout();
window.addEventListener("resize", applyViewportLayout);
window.addEventListener("orientationchange", applyViewportLayout);
shader.setLightPosition([120.0, 180.0, 140.0, 1.0]);

const space = new Space();
const eye = space.addNode(null, "eye");
eye.setPosition(0.0, 0.0, 8.0);

const shape = new Shape(screen.getGL());
shape.applyPrimitiveAsset(Primitive.cube(2.0, shape.getPrimitiveOptions()));
shape.endShape();
shape.setMaterial("smooth-shader", {
  use_texture: 0,
  color: [1.0, 0.5, 0.3, 1.0]
});

const obj = space.addNode(null, "obj");
obj.addShape(shape);

function loop() {
  obj.rotateY(0.8);
  obj.rotateX(0.4);
  screen.clear();
  space.draw(eye);
  screen.present();
  requestAnimationFrame(loop);
}
loop();
```

この例で見たいのは、`await screen.ready`、projection の更新、`shape.endShape()`、`clear -> draw -> present` という基本順序です。コアの仕組みを理解したいときは、この最小構成から読むのが自然です。

もうひとつの入口は、`WebgApp` を使って標準的なアプリ構成に乗る方法です。サンプルや小規模アプリでは、こちらのほうが実用的です。`Screen`、`Space`、camera rig、入力、HUD、診断情報用の補助機能をまとめて初期化できるためです。`webg/samples/high_level` は、この構成の最小例です。

```js
import WebgApp from "./webg/WebgApp.js";
import Shape from "./webg/Shape.js";
import Primitive from "./webg/Primitive.js";
import EyeRig from "./webg/EyeRig.js";

const CAMERA_CONFIG = {
  target: [0.0, 0.0, 0.0],
  distance: 8.0,
  yaw: 0.0,
  pitch: 0.0,
  bank: 0.0
};

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  clearColor: [0.1, 0.15, 0.1, 1.0],
  camera: CAMERA_CONFIG
});

await app.init();

const shape = new Shape(app.getGL());
shape.applyPrimitiveAsset(Primitive.cube(2.0, shape.getPrimitiveOptions()));
shape.endShape();
shape.setMaterial("smooth-shader", {
  use_texture: 0,
  color: [1.0, 0.5, 0.3, 1.0]
});

const obj = app.space.addNode(null, "obj");
obj.addShape(shape);

const orbit = new EyeRig(app.cameraRig, app.cameraRod, app.eye, {
  document,
  element: app.screen.canvas,
  input: app.input,
  type: "orbit",
  orbit: {
    target: CAMERA_CONFIG.target,
    distance: CAMERA_CONFIG.distance,
    yaw: CAMERA_CONFIG.yaw,
    pitch: CAMERA_CONFIG.pitch,
    minDistance: 4.0,
    maxDistance: 18.0,
    wheelZoomStep: 1.0
  }
});
orbit.attachPointer();

app.setGuideLines([
  "Drag: orbit",
  "2-finger drag: pan",
  "Pinch / wheel: zoom",
  "Arrow keys: orbit",
  "[ / ]: zoom"
], {
  anchor: "bottom-left",
  x: 0,
  y: -2
});

app.setStatusLines([
  "high_level sample",
  "WebgApp + EyeRig"
], {
  anchor: "top-left",
  x: 0,
  y: 0
});

app.start({
  onUpdate: ({ deltaSec }) => {
    orbit.update(deltaSec);
    obj.rotateY(0.8);
    obj.rotateX(0.4);
  }
});
```

この構成では、カメラ操作、操作ガイド、ステータス表示まで、最初からアプリの中へ組み込めます。マウスだけでなく Touch もすぐ確認したい場合は、ローレベルの例よりも、こちらの入り方が向いています。

## サンプルはどのように読むか

サンプルを読むときは、`webg/samples/index.html` の一覧だけで終わらせず、対応する `*.txt` と `main.js` を合わせて見てください。`*.txt` は「何を示すサンプルか」を説明し、`main.js` はその実装です。この読み方を前提にすると、サンプルは単なる見本ではなく、教材として機能します。

最初の入口としては、次のサンプルが分かりやすくなります。

* `low_level`
  この章の最小描画例を、そのまま実行できるサンプル
* `high_level`
  `WebgApp + EyeRig` による標準的なアプリ構成の最小例
* `scene`
  Scene JSON、HUD、モデルの読み込み、入力をまとめて見る入口
* `shapes`
  primitive、texture、normal、wireframe の比較
* `gltf_loader`
  glTF / `ModelAsset` / static bake の見方
* `sound`
  `GameAudioSynth` と診断情報を合わせて確認する入口
* `bloom`
  offscreen scene と bloom 合成の入口
* `dof`
  被写界深度とデバッグ表示の入口

やや規模のあるアプリ例としては `circular_breaker`、TileMap 系の実践例としては `tile_sim` が役立ちます。

なお、`webg` では伝統的に `head / pitch / bank` という回転用語を使います。一般的な用語では `head = yaw`、`pitch = pitch`、`bank = roll` です。`EyeRig`、`CoordinateSystem`、`Animation` を読むときは、この対応を先に押さえておくと理解しやすくなります。座標系、回転軸、テクスチャ座標、行列、クォータニオンの前提は、第3章「3Dグラフィックスの基礎」にまとめています。

## 問題調査の進め方と、現時点での限界

3D アプリでは、描画結果だけでなく、「なぜその見え方になったか」を追えることが重要です。`webg` では、まず画面内の HUD やパネルで見える情報を出し、必要に応じて診断情報レポートと probe を残す流れを標準にしています。つまり、コンソールだけに頼るのではなく、「いま何が見えているか」をアプリ側でも共有しやすい設計を取っています。後の章で扱う `Diagnostics`、`DebugDock`、`FixedFormatPanel` は、この考え方の延長にあります。

一方で、現時点の `webg` にはまだ未実装の範囲もあります。PBR、shadow map、env map、高機能なエディターは未対応です。したがって、このライブラリを使うときは、「現在実装されている層を読みながら組み立てる」ことに価値があるかどうかで判断すると整理しやすくなります。ブラックボックスの巨大エンジンとして使うより、構造を追いながら必要な層を使っていくスタイルに向いています。

## まとめ

この章で最も大事なのは、`webg` を単なる API 集として見ないことです。`webg` は、WebGPU のローレベルな描画から `WebgApp` による高水準なアプリ構成までを、同じリポジトリの中で追えるようにしたライブラリです。だからこそ、最初に必要なのは API を暗記することではなく、「どこから読み始め、どの層を先に理解するか」を決めることです。

入口としては、第2章で実行環境をそろえ、第3章で座標系と回転を押さえ、第4章で最小描画を確認し、第5章で `WebgApp` の標準構成へ進む流れがもっとも自然です。そのうえで、samples は統合例、unittest は局所検証として読み分け、必要な分野だけ各章へ進むと、`webg` 全体がかなり追いやすくなります。

## 本書の歩き方

ここまでで、`webg` がどのようなライブラリであり、どのような用途に向いているのか、そしてプロジェクトの中に何があるのかを概観してきました。最後に、本書をどのような順番で読むと理解しやすいかを整理しておきます。本書は、最初から最後まで順に読んでも構いませんが、必要なところから読み進めても使えるように構成しています。`webg` は、外部ライブラリに依存せず、描画、シーン、モデル、アニメーション、UI、TileMap、ローレベル API までを自分自身の中に持っています。そのため本書もまた、単なる API 一覧ではなく、どの目的にはどの層を見ればよいかが分かるように章を並べています。

本書を読むときに最も大切なのは、いま自分が知りたいことが、どの部分の話なのかを先に決めることです。最初の 3D 表示を確認したいのか、アプリ全体の土台を作りたいのか、カメラを動かしたいのか、モデルやシーンを読み込みたいのか、アニメーションを切り替えたいのか、あるいはローレベルの `Shape` や `Shader` まで降りたいのかによって、起点にすべき章は変わります。`webg` には高水準 API と低水準 API の両方がありますが、まずは高水準の入口から入り、必要になったときだけ下位の章へ降りるほうが理解しやすくなります。

最初の入口としてもっとも自然なのは、第2章、第3章、第4章、第5章の流れです。第2章で実行環境をそろえ、第3章で座標系や回転の前提を押さえ、第4章で最小描画を確認し、第5章で `WebgApp` を使った標準的なアプリ構成へ進みます。この 4 章を通すと、`webg` を使って最初の 3D アプリを立ち上げるために必要な土台がそろいます。実際に何かを作り始めたい読者は、まずこの流れを一度通しておくと、その後の章で迷いにくくなります。

そのうえで、目的に応じて読み進めると、本書はさらに使いやすくなります。カメラ操作を整えたいなら第6章、見た目や陰影を調整したいなら第7章から第9章、glTF や Collada を読み込みたいなら第10章、シーン全体の初期状態を JSON で扱いたいなら第11章、アニメーションを切り替えたいなら第12章と第13章、UI 表示を整理したいなら第14章以降、TileMap を使いたいなら第22章以降が主な入口になります。ローレベル API や内部構造を直接追いたい場合は、第25章以降へ進むと、`Screen`、`Shader`、`Node`、`Matrix`、`Shape`、スキニングといったより根本の処理までたどれます。

サンプルの読み方にも、一定の順序があります。`webg/samples/` は単なる見本ではなく教材です。最初の表示なら `low_level` と `high_level`、モデル読み込みなら `gltf_loader`、シーン構成なら `scene`、アニメーション状態なら `animation_state` と `janken`、TileMap と actor の実例なら `tile_sim` が主な入口になります。まず本文でその機能の位置づけをつかみ、そのあと対応するサンプルを読み、必要ならコア実装へ降りる、という順で進めると理解しやすくなります。一方で `webg/unittest/` は、サンプルよりも局所的な確認や切り分けに向いています。何かがうまく動かないときや、ある機能だけを小さく確かめたいときには、unittest のほうが役立ちます。

また、本書では、似て見えても役割が異なるものを意図的に分けて説明しています。たとえば、`ModelAsset` は 1 モデル分の共通表現であり、Scene JSON はシーン全体の初期状態です。`build()` は共有リソースを含むランタイム化であり、`instantiate()` はそのランタイムから新しいシーンインスタンスを作る処理です。アニメーションも、`clip`、`action`、`state` を別の層として扱います。こうした区別を保ったまま読むと、サンプルの意味も、コア実装の意味も追いやすくなります。反対に、異なる層の概念を一つにまとめて理解しようとすると、後の章で混乱しやすくなります。

したがって、本書を読むときの基本は、まず目的に近い章へ入り、そこで紹介されているサンプルを見て、必要なら下位の章へ降りる、という順番です。最初からすべてを理解しようとする必要はありません。必要な入口を見つけ、そこで一度動かし、必要になったときに構造や内部へ戻る、という読み方で十分です。本書が、そのための地図として役立てば幸いです。
