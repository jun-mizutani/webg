# シェーダーの実装

この章は、`webg` のシェーダーを「利用者として使う」段階から一歩進めて、`Shader.js` と各派生クラスの実装をどのように追うかを整理する章です。対象は `webg/Shader.js`、`book/examples/Phong.js`、`book/examples/NormPhong.js`、`book/examples/BonePhong.js`、`book/examples/BoneNormPhong.js`、`webg/Background.js`、`webg/Font.js`、`webg/Wireframe.js`、`webg/FullscreenPass.js` です。第8章では WGSL の読み方を扱いましたが、この章で重心を置くのは CPU 側のクラス設計です。つまり本章では、「JavaScript 側で何を準備し、どのように GPU へ渡し、どのクラスがどこまでを引き受けているか」を見ることが中心になります。`Phong` 系 4 本は現行の標準入口ではなく、差分読解のために `book/examples` へ置いた教材用ファイルとして扱います。

この章で最初に押さえたいのは、`Shader.js` は単に WGSL の文字列を返すクラスではない、ということです。`webg` のシェーダークラスは、パイプライン作成、ユニフォームバッファの管理、bind group の生成、デフォルトテクスチャの用意、パラメータの反映、描画時の更新まで引き受けています。そのため、シェーダー実装を読むときは、「WGSL の関数」より先に「クラスが何を保持しているか」を見るほうが分かりやすくなります。さらに重要なのが、`Shape.setMaterial()` で登録したパラメータは最終的に各シェーダーの `doParameter()` へ流れ、そこで setter 群に振り分けられ、必要に応じて初期値へ戻される、という点です。ここが見えると、そのシェーダーが利用者へどのような入口を提供しているかがかなり明確になります。 

また、最初に比較するのに向いているのは `Phong` 系 4 本です。`Phong`、`NormPhong`、`BonePhong`、`BoneNormPhong` は、`webg` の 3D マテリアルを理解する中心にあります。静的メッシュ、ノーマルマップ、スキニング、その両方という差で見ると、実装差分を追いやすくなります。特に `Phong` は小さく基本的なシェーダーとして、教材的にも読みやすい入口です。第7章では通常の入口として `SmoothShader` を整理しましたが、本章では実装の読みやすさのために `Phong` 系を差分で追う構成を中心に据えます。

## まず `Shader.js` の共通構造を見る

`webg` では、多くのシェーダーが同じ流れを共有します。projection 行列や model-view 行列を受け取り、ユニフォームバッファを更新し、必要なテクスチャとサンプラーを bind group にまとめ、描画直前に GPU へ反映します。これを各シェーダーごとにばらばらに書くと、サンプル追加やデバッグオプション追加のたびに差分が増えます。`Shader.js` は、この共通部分を引き受ける基底クラスです。派生クラスは、その上に自分専用の WGSL、そのシェーダーが必要とするユニフォームレイアウト、初期パラメータ、bind group レイアウト、パラメータを反映する setter を追加します。 

`Shader.js` を読むときは、次の順番が分かりやすくなります。constructor で何を初期化しているか、`createResources()` で何を GPU 側に作るか、`getBindGroup()` で何を結び付けるか、`updateUniforms()` や setter が何をバッファへ書くか、`doParameter()` が派生クラスでどのように拡張されるか、という順です。この流れで見ると、各シェーダークラスが「どこまで共通で、どこから個別か」が整理しやすくなります。 

とくに重要なのは、`default` と `change` の 2 つのパラメータ表です。`default` はシェーダーの初期値、`change` は `Shape.shaderParameter()` で現在上書きされている値です。`doParameter(param)` は、指定のあるキーを setter へ流し、前回だけ上書きされていたキーを初期値へ戻します。この設計があるため、利用者は毎回すべてのパラメータを書き直さずに済みます。逆にここを理解しないと、「なぜ以前の値が残らないのか」「なぜ今回指定していない値が初期値へ戻るのか」が見えにくくなります。 

シェーダーを拡張するときは、新しいパラメータを追加したら、少なくとも初期値、setter、`doParameter()` 内での反映、必要なら `setDefaultParam()` 側の入口、そして文書とサンプルをそろえる必要があります。ここが抜けると、コードは存在しても利用者から見えない機能になりやすくなります。実装を足すときは、WGSL だけでなく CPU 側の入口と観察手段まで一緒に整える、というのが `webg` の文書構成と相性のよい進め方です。

## `Phong` 系 4 本を差分で読む

`Phong` 系 4 本は、順に比較していくと構造が見えやすくなります。基本形が `Phong`、そこへノーマルマップを加えたのが `NormPhong`、スキニングを加えたのが `BonePhong`、その両方を合わせたのが `BoneNormPhong` です。いきなり `BoneNormPhong` から読むより、`Phong` → `NormPhong` → `BonePhong` → `BoneNormPhong` の順で差分を追うほうが理解しやすくなります。 

### `Phong` をどう読むか

`Phong` は比較の基準です。最初に見るべきは、`Uniforms` が何を持つか、`doParameter()` がどの setter を呼ぶか、WGSL 側で `params0` / `params1` がどのように使われるか、の 3 点です。`Phong` のパラメータは、代表的には `params0 : vec4f` に `ambient, specular, power, emissive`、`params1 : vec4f` に `useTexture, backfaceDebug, ...` のようにまとめられています。JavaScript 側では `setAmbientLight() -> params0.x`、`setSpecular() -> params0.y`、`setSpecularPower() -> params0.z`、`setEmissive() -> params0.w`、`useTexture() -> params1.x`、`setBackfaceDebug() -> params1.y` という対応で読むと追いやすくなります。

`Phong` は、実用上よく使う基本シェーダーであり、同時に教材としても読みやすいクラスです。立方体、球、床、シーンの基本オブジェクトなど、まず最初に使うシェーダーの多くがこれです。ライティング、テクスチャ、fog、emissive、specular を一通り持ち、スキニングやノーマルマップはまだ要らない、という場面に向いています。その役割は、model-view 変換された `position` / `normal` を使って拡散反射と鏡面反射のライティングを計算し、必要であればベーステクスチャを掛け、最後に fog を混ぜて色を返すことです。`backface_debug` を有効にすると、裏面だけマゼンタで返すデバッグ表示もできます。 

利用者向けの最小形は、次のようになります。

```js
import Phong from "./book/examples/Phong.js";
import Shape from "./webg/Shape.js";

const shader = new Phong(gpu);
await shader.init();
Shape.prototype.shader = shader;

shader.setLightPosition([40.0, 90.0, 150.0, 1.0]);

const shape = new Shape(gpu);
shape.applyPrimitiveAsset(asset);
shape.endShape();

shape.setMaterial("phong", {
  color: [1.0, 0.9, 0.7, 1.0],
  ambient: 0.28,
  specular: 0.82,
  power: 34.0,
  use_texture: 1,
  texture
});
```

ここでの読み方は明快です。シーン全体で共有する光源はシェーダー側に置き、オブジェクトごとの質感は `shape.setMaterial("phong", {...})` で Shape 側へまとめて渡します。利用者がよく調整するのは `color`、`use_texture`、`texture`、`ambient`、`specular`、`power`、`emissive`、`backface_debug` です。開発者として追うときは、`setProjectionMatrix()`、`setModelViewMatrix()`、`setNormalMatrix()`、`setLightPosition()`、fog 系 setter もセットで見ておくと、どの値がシェーダー共通で、どの値が Shape ごとの差分かが整理しやすくなります。 

### `NormPhong` をどう読むか

`NormPhong` は、`Phong` を理解したあとに差分として読むのがもっとも分かりやすくなります。追加で見るべき点は、ノーマルマップ用テクスチャの binding が増えているか、`use_normal_map`、`normal_texture`、`normal_strength` を受ける setter があるか、フラグメントシェーダーでサンプリングした法線と元の法線をどのように混ぜているか、の 3 点です。WGSL 側でまず目に入るのは `@group(...) @binding(...) var uNormalTexture : texture_2d<f32>;` のようなノーマルマップ用テクスチャの binding です。その後、フラグメントシェーダーでは、ノーマルマップを読む、`dpdx` / `dpdy` から接線ベクトル / 従法線ベクトルを再構成する、TBN で接空間法線を戻す、`normal_strength` で混ぜる、という順で追うと、ノーマルマップの見え方がどこで決まるかが分かります。 

`NormPhong` は、`Phong` にノーマルマップを加えたマテリアルです。色だけでなく表面の凹凸感も見せたい場面で使います。役割としては、ベーステクスチャに加えてノーマルマップ用テクスチャを読み、フラグメントシェーダー内で再構成した TBN を使って法線を曲げることです。これにより、実際のポリゴン数を増やさずに細かい陰影差を出せます。利用者側では `use_normal_map`、`normal_texture`、`normal_strength` が `Phong` との差になります。ここで注意したいのは、ノーマルマップを ON にしても `normal_texture` を渡さなければ平坦な法線のデフォルトテクスチャが使われるため、「ノーマルマップを使っているのに何も起きない」ように見えることがある点です。これは不具合ではなくフォールバックであり、見え方を確実に変えたいなら `normal_texture` を明示的に渡します。 

### `BonePhong` をどう読むか

`BonePhong` で注目すべきなのは頂点シェーダーです。`Phong` と違って、頂点位置をそのまま使わず、ボーンパレットで変形してから後段へ渡します。代表的には `let i0 = i32(input.index.x) * 3; ...` のような式が現れ、`webg` では 1 本のボーンを `vec4 x 3` で持つため、4x4 行列そのままではなく、成分を詰め込んだ表現になっています。ここを理解すると、`BONE_VECTOR_COUNT` やオフセット設計が読みやすくなります。 

`BonePhong` を読む順番は、頂点入力に bone index / weight があるか、ユニフォームにボーンパレットがあるか、どこで 4 本までの影響を混ぜているか、変形後の `normal` をどう扱っているか、`weight_debug` の切り替えがどこにあるか、という流れが分かりやすくなります。`BonePhong` は、`Phong` にスキニングを加えたシェーダーです。役割は、頂点ごとの `boneIndex` と `weight` を読み、行列パレットから 4 本までのボーン行列を合成して頂点を変形することです。その後のライティング計算は `Phong` とほぼ同じです。`weight_debug` を ON にすると、重みを RGB で可視化するモードに切り替わります。 

利用者側では、`has_bone: 1` を渡すことと、Shape 側に bone index / weight が入っていることが前提になります。`Shape.endShape()` は、スキニングメッシュの場合、通常の頂点バッファとは別に bone index と weight 用のバッファを作り、`BonePhong` のパイプラインはこの 2 本構成を前提にしています。ここで重要なのは、`has_bone` を `1` にするだけでは曲がらないことです。skeleton が関連付いており、行列パレットが更新される状態で初めてスキニングが成立します。スキニングが不自然なときは、まず `weight_debug` で weight 分布を観察し、ボーンパレット以前に頂点属性が正しいかを切り分けるほうが安全です。 

### `BoneNormPhong` をどう読むか

`BoneNormPhong` は、`BonePhong` と `NormPhong` の差分を重ねて読むと分かりやすくなります。頂点シェーダー側は `BonePhong` に近く、フラグメントシェーダー側は `NormPhong` に近く、パラメータ群はスキニングとノーマルマップの両方を持ちます。そのため、いきなり `BoneNormPhong` から読むより、`Phong` → `NormPhong` → `BonePhong` → `BoneNormPhong` の順で比較するほうが構造が見えやすくなります。

`BoneNormPhong` は、「スキニングしながらノーマルマップも使う」マテリアルです。役割は、頂点変形は `BonePhong` と同じく行列パレットで行い、フラグメント側では `NormPhong` と同様にノーマルマップで法線を曲げることです。さらに `weight_debug` と `backface_debug` の両方を持ち、スキニングと陰影の両面を確認できます。ここでも注意したいのは、`weight_debug` を ON にすると重み色が優先されるため、ライティングやノーマルマップの見え方は確認できないことです。2 つを同時比較するのではなく、観察目的に応じて切り替えるほうが読みやすくなります。 

## 2D / オーバーレイ / ヘルパー系シェーダーをどう読むか

`Background`、`Font`、`BillboardShader`、`Wireframe`、`FullscreenPass` は、3D マテリアルと同じ読み方をすると少し混乱します。これらは「質感を決めるシェーダー」というより、「表示経路を作るシェーダー」として読むほうが理解しやすくなります。つまり、ライティングや材質係数よりも、どのような表示対象を、どのような経路で出すかに注目して読むクラスです。

`Background` は、画面全体や背景矩形をどのように出すかが関心の中心であり、テクスチャの向き補正も読みどころです。`Font` は、グリフ用フォントアトラスと色の扱いが中心で、`Text` や `Message` の内部実装の入口になります。`BillboardShader` は、カメラへ向く四角形をどのように作るかを見るクラスです。`Wireframe` は、メッシュのデバッグ表示をどのように描くかを見るもので、`FullscreenPass` は、レンダーターゲットのテクスチャを 1 枚そのまま出すパスをどのように作るか、という観点で読むと分かりやすくなります。

たとえば `Background` は、3D オブジェクトを使わずに画面後景へ画像や色を出すヘルパーです。`window` と `aspect` の扱い、画像の向きを合わせるための V 反転などを見ると、「なぜ通常の 3D オブジェクトとは別クラスになっているのか」が見えてきます。`Font` は 1 文字ごとの四角形を描くのではなく、同じ四角形とパイプラインを使い回し、dynamic offset で「何文字目か」「何色か」「どの位置か」を切り替える設計になっています。ここを理解すると、HUD やメッセージ表示のコスト感も読みやすくなります。 

## シェーダーを拡張するときに何をそろえるか

シェーダー実装を変更するときに重要なのは、WGSL だけを触って終わらせないことです。たとえば新しいパラメータを 1 つ追加する場合でも、WGSL の `Uniforms` にフィールドを増やしたら JavaScript 側の `uniformData` サイズとオフセットも増やす必要があります。binding 数を変えたら、レイアウトと `getBindGroup()` もそろえなければなりません。デバッグオプションを増やすなら、通常の初期値を壊さないよう constructor オプションか初期パラメータで扱うほうが安全です。変更が見える単体テストやサンプルも、できれば 1 本用意したほうがよい、というのがこの章の重要な実務的な結論です。

とくに `doParameter()` を忘れると、`Shape.setMaterial()` からパラメータが効かないシェーダーになります。これは見た目だけでは気付きにくく、あとで切り分けに時間が掛かりやすい箇所です。`webg` のシェーダーは、CPU 側のクラス設計、ユニフォーム更新、bind group、描画結果まで含めて一つの実装として読む必要があります。したがって、変更時も「文字列としての WGSL」ではなく、「入口から GPU までが一つながりになっているか」で確認するほうが安全です。

## 生成 AI がシェーダーを変更するときの注意

生成 AI がシェーダーを修正する際は、WGSL の `Uniforms` フィールドを増やしたら JavaScript 側の `uniformData` サイズとオフセットも増やすこと、binding 数を変えたらレイアウトと `getBindGroup()` もそろえること、デバッグオプションを増やしたら通常の初期値を壊さないよう constructor オプションか初期パラメータで扱うこと、`textureSample()` の位置は WGSL の非一様制御に関する制約を意識すること、変更が観察できる単体テストやサンプルを 1 本用意すること、を同時に見る必要があります。コードだけが正しくても、利用者や AI がどこを見れば差が分かるか分からない状態は避けたほうが安全です。変更後は、「どこを見れば差が分かるか」まで残すことが大切です。

## 読み進める順番と次に読む章

初めてシェーダー実装を追うなら、次の順が分かりやすくなります。まず第7章「シェーダーとマテリアル」で用途を確認し、第8章「WGSLの読み方」で WGSL の記法を押さえ、そのあと `Shader.js` を読んで共通構造をつかみ、`Phong` を読み、`NormPhong`、`BonePhong`、`BoneNormPhong` を差分で比較し、必要に応じて `Background`、`Font`、`FullscreenPass` を読む、という順です。この順番なら、サンプルを読むときにも「どこから見ればよいか」がぶれにくくなります。

## まとめ

この章で最も大事なのは、シェーダーを単なる WGSL 断片としてではなく、CPU 側のクラス設計、ユニフォーム更新、bind group、描画結果まで含めた一つの実装として読むことです。`Shader.js` は共通構造を引き受け、`doParameter()` は利用者向けの入口を整理し、`Phong` 系 4 本は差分として読むと構造が見えやすくなります。ここが見えてくると、シェーダーの変更やデバッグはかなり進めやすくなります。
