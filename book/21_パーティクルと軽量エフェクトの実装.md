# パーティクルと軽量エフェクトの実装

画面全体の見え方を変えるポストプロセスに対して、火花や煙のような短命の演出は scene の中へ局所的に差し込むほうが自然です。こうした表現を重いモデルや複雑な animation に頼らず軽く扱うために、`webg` では `ParticleEmitter` を使います。本章では、billboard ベースの particle をどのように emit し、通常の `WebgApp` フローへどう組み込むかを見ていきます。

ここで重要なのは、`ParticleEmitter` が 1 個の mesh を動かすクラスではないことです。多数の billboard particle を生成し、life、velocity、gravity、drag を毎フレーム更新しながら描画するヘルパーであり、短命の演出をまとめて扱うのが役割です。つまり本章の中心は、「どういう見た目の texture を使うか」よりも、「どんな burst を、どのタイミングで、どの程度の数だけ発生させるか」を整理することにあります。

さらに大事なのは、`preset` が emit の省略記法ではないことです。current 実装の `emit()` は、`position`、`positionSpread`、`velocity`、`velocitySpread`、`gravity`、`color`、`colorSpread` などを明示的に受け取ります。preset は texture の見た目と defaults の束ですが、不足した値を emitter 側が曖昧に補完する API ではありません。必要なベクトルや色は利用者がはっきり渡す前提です。これにより、「何がどの方向へ飛び、どのくらい散り、どの色で見えるか」がコードからそのまま読めるようになります。

## パーティクルを使う場面

ゲームやデモでは、当たり判定のヒット、アイテム取得、着地、破壊、煙、きらめきといった短時間の演出が頻繁に必要になります。こうした演出を毎回 mesh や animation で作ると、用途に対して重くなりがちです。そこで役立つのが、画面に正対する billboard を多数使い、短い寿命の中で色や大きさを変えながら見せるパーティクルです。局所的で短命な効果を、scene の流れを崩さず差し込めるのが強みです。

`webg` の `ParticleEmitter` は、この用途に合わせて設計されています。内部では particle slot を一定数持ち、`emit()` で空いている slot に spawn 情報を書き込み、`update()` で位置と寿命を進め、`draw()` で billboard へ流します。必要なら影用 billboard も同時に描けるため、地面に接する火花や煙も簡潔に表現できます。つまり、単に sprite を並べるのではなく、「短時間だけ存在する局所エフェクト」をまとめて運用するための部品として読むほうが current 実装に合っています。

## `ParticleEmitter` を `WebgApp` に登録する

通常のアプリでは、まず `WebgApp.createParticleEmitter()` を呼ぶところから始めます。このヘルパーは `ParticleEmitter` を生成し、GPU 初期化を行い、`WebgApp` の emitter 一覧へ登録します。つまり利用者コードは、パーティクル本体の初期化よりも、「何という emitter を、どの程度の容量で、どの preset で使うか」を先に決めればよくなります。

```js
import WebgApp from "./webg/WebgApp.js";

const app = new WebgApp({
  document,
  messageFontTexture: "./webg/font512.png",
  clearColor: [0.08, 0.10, 0.15, 1.0]
});
await app.init();

const sparkEmitter = await app.createParticleEmitter({
  name: "sparkEmitter",
  maxParticles: 320,
  useShadow: true,
  groundY: -7.5,
  preset: "spark",
  seed: 31
});
```

この例で `maxParticles` は同時に生かせる particle 数、`useShadow` は影用 billboard を使うかどうか、`groundY` は影を置く高さ、`preset` は texture と既定値の束を選ぶ名前です。`seed` を固定すると、乱数を使う spread のばらつきが再現しやすくなります。ここで `useShadow`、`groundY`、`shadowY` のような値は、局所エフェクトを地面へなじませるための補助として考えると分かりやすくなります。とくに spark や dust のような床との接触感が欲しい演出では、影用 billboard があるだけで scene への定着感がかなり変わります。 

`WebgApp.start()` を使う通常の構成なら、ここで emitter を作っておけば、フレームループの中で `updateParticleEmitters()` と `drawParticleEmitters()` が自動で呼ばれます。利用者コードでは「いつ emit するか」に集中しやすくなります。パーティクルの存在を scene の更新と描画へ統合する、という意味でも、この入口を標準として考えるのが自然です。

## `emit()` で burst を明示する

current 実装の `emit()` は、spawn に必要なベクトルや色を明示的に受け取ります。`ParticleEmitter` は内部で安易な補完を行わないため、どの方向へ飛ばし、どれくらい散らし、どの色で出すかを利用者コードがはっきり示せます。これは、演出の意図をサンプルや書籍コードから直接読めるようにするためにも重要です。つまり `emit()` は「preset を呼ぶ」よりも、「この瞬間の burst の条件を明示する」API として読むほうが実態に合っています。

```js
const emitSparkBurst = (x, y, z) => {
  sparkEmitter.emit(24, {
    position: [x, y, z],
    positionSpread: [0.3, 0.3, 0.3],
    velocity: [0.0, 12.0, 0.0],
    velocitySpread: [8.0, 4.0, 8.0],
    gravity: [0.0, -18.0, 0.0],
    drag: 0.06,
    life: 0.9,
    lifeSpread: 0.2,
    size: 1.6,
    sizeSpread: 0.5,
    color: [1.0, 0.85, 0.40, 1.0],
    colorSpread: [0.0, 0.08, 0.10, 0.0],
    shadowAlpha: 0.55,
    shadowScale: 1.1,
    shadowY: -7.5
  });
};
```

ここで `positionSpread` は発生位置のばらつき、`velocitySpread` は飛び出し方向のばらつき、`gravity` は時間経過による加速度、`drag` は速度減衰、`life` は寿命秒数、`size` は billboard の大きさです。`shadowAlpha`、`shadowScale`、`shadowY` は影用 billboard の見え方を決めます。この形で明示しておくと、火花は上へ強く飛ぶ、煙は速度を弱くして寿命を長めにする、拾得エフェクトは重力を弱めて上昇気味に見せる、といった違いをコード上ですぐ読み取れます。 

また、ここで大切なのは必要な値を省略しないことです。current 実装では、不足したベクトルや色を emitter 側が都合よく補ってくれる前提ではありません。`position`、`velocity`、`gravity`、`color` と、それぞれの spread を自分で明示するほうが、演出意図が保たれやすく、バグも見つけやすくなります。 

## preset の役割

`ParticleEmitter` は `spark`、`smoke`、`debris`、`pickup` などの preset を持ちます。preset の役割は、particle texture の見た目や、演出の既定方向をまとめることです。利用中に切り替えたい場合は `setPreset()` を呼びます。ただし、ここでも preset は「emit の代わり」ではなく、「この emitter をどんな見た目の系統に寄せるか」を決める束として理解するほうが自然です。

```js
sparkEmitter.setPreset("smoke", {
  texture: {
    width: 96,
    height: 96,
    style: "smoke",
    noiseScale: 5.8,
    noiseAmount: 0.68
  },
  defaults: {
    life: 1.4,
    lifeSpread: 0.3,
    size: 2.8,
    sizeSpread: 1.0
  }
});
```

`setPreset()` は必要に応じて procedural billboard texture も再生成します。ここでの defaults は「こういう演出に寄せる」という束であり、実際の `emit()` 呼び出しでは依然として明示的なパラメータを使う前提です。暗黙に不足分を補ってくれる API だと期待しないほうが、安全で読みやすいコードになります。つまり preset は texture と既定傾向の切り替えであり、spawn 条件の責務は `emit()` 側に残る、という分担です。 

## `WebgApp.start()` と独自ループの違い

通常のサンプルでは `WebgApp.start()` を使うため、利用者コードは emit の条件だけを書けば十分です。ここがもっとも標準的な使い方です。`WebgApp` はフレームループの中で `updateParticleEmitters()` と `drawParticleEmitters()` を呼ぶので、particle の更新と描画は scene の通常フローへ自然に統合されます。利用者は「何が起きたら emit するか」だけを追えばよく、particle の時間更新と描画順を別に考えなくて済みます。 

```js
app.start({
  onUpdate: ({ deltaSec }) => {
    actor.update(deltaSec);
    if (actor.justHit) {
      emitSparkBurst(actor.hitX, actor.hitY, actor.hitZ);
    }
  }
});
```

一方で、独自ループを組みたい場合は `draw()` を自分で呼びます。`samples/circular_breaker/particleEffects.js` は、この構成を小さなヘルパーにまとめています。これは通常経路ではなく、描画順やループ構造を自前で管理したいときの例外経路と考えると分かりやすくなります。

```js
const alive = sparkEmitter.draw(app.eye, app.projectionMatrix);
```

この書き方は、`WebgApp.start()` を使わずに独自ループを組むサンプルや、描画順を明示したいサンプルで向いています。反対に、通常の教材サンプルでは emitter の更新と描画を手でばらさず、`WebgApp.start()` へ任せたほうが流れを追いやすくなります。したがって、最初に「通常経路へ乗せるのか」「独自ループで draw するのか」を決めて、どちらか一方へ寄せるのが安全です。 

## 軽量エフェクトとしての使いどころ

パーティクルが向いているのは、短時間で消える視覚効果です。具体的には、ヒット時の spark、着地の dust、壊れたときの debris、アイテム取得時の pickup glow などが代表です。これらは geometry よりも時間変化が主役であり、billboard のほうが軽くて調整しやすくなります。scene 内で一瞬だけ発生して消えるものほど、`ParticleEmitter` の役割とよく合います。 

逆に、長時間残る object、当たり判定を持つ object、視点から回り込んで形を見せたい object には向きません。その場合は particle ではなく `Shape` や model asset を使い、必要に応じて animation や material 側で演出するほうが自然です。つまり、particle は「局所的で短命な効果」に強く、「存在そのものを見せる object」には向かないと考えると判断しやすくなります。 

## 注意点

まず、`emit()` で渡す配列や数値を省略しないことが大切です。current 実装の `ParticleEmitter` は、不足したベクトルや色を自動補正する方針ではありません。`position`、`velocity`、`gravity`、`color` とそれぞれの spread を明示したほうが、意図が読みやすく、バグも見つけやすくなります。ここを曖昧にすると、「なぜこの方向へ飛ぶのか」「なぜ色が変わったのか」がコードから読みにくくなります。 

次に、`maxParticles` を小さくしすぎると、長寿命の particle が残っている間に新しい burst を出せなくなります。逆に大きくしすぎると、不要に多くの slot を管理することになります。演出の同時発生数を見積もって決めるほうが扱いやすくなります。つまり `maxParticles` は「とりあえず大きくする値」ではなく、「何個の短命エフェクトが同時に走るか」を見積もって決めるべき値です。 

影用 billboard を使う場合は、`groundY` や `shadowY` の高さが scene の床と合っているかも確認します。値がずれると、particle 本体は自然でも影だけが浮いて見えることがあります。`samples/circular_breaker` が `SHADOW_Y` を定数化しているのは、このずれを避けるためです。軽量エフェクトは本体が小さいぶん、影の高さのずれが逆に目立ちやすいので、ここは早めにそろえておくほうが安全です。 

`WebgApp.start()` を使う構成で、自分でも `emitter.draw()` を呼んでしまうと二重描画になります。通常のループに乗せるのか、独自ループで自分で draw するのかを最初に決めて、どちらか一方へ寄せるほうが安全です。これも「通常経路」と「例外経路」を混ぜないことの一例として覚えておくと分かりやすくなります。 

## 関連 sample

最初に読むなら `unittest/particle_emitter` が最も分かりやすくなります。fake renderer による自動確認と、実画面での visual sample が同じファイルにまとまっているため、`preset -> emit -> update -> draw -> clear` の流れを追いやすいからです。まずはここで「particle がどう生成され、どう消えていくか」を確認すると、API の責務をつかみやすくなります。 

ゲームへの組み込み例としては `samples/circular_breaker` がよく、ヒット時の spark を gameplay code から切り離して扱う構成を確認できます。こちらは「emit そのものの確認」ではなく、「ゲームロジックのどの位置で particle を発生させるか」を読む sample として使うと分かりやすくなります。つまり、確認用は `unittest/particle_emitter`、組み込み例は `samples/circular_breaker` という 2 本立てで読むと整理しやすくなります。

第21章はこの形にしておくと、第20章のポストプロセスが画面全体の後段処理であり、パーティクルは scene 内の局所演出である、という役割差も読み取りやすくなります。
