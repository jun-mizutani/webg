# サウンドの設計

サウンド機能をアプリに追加する場合、単に効果音を1回鳴らせば十分というケースはほとんどありません。効果音（SE）とバックグラウンドミュージック（BGM）では目的が根本的に異なりますし、音量、ディレイ、リバーブ、そしてメロディの管理もそれぞれ独立して制御したいものです。

`webg` では、これらの違いを後から機能を追加して対応させるのではなく、設計段階から `AudioSynth` と `GameAudioSynth` に役割を分け、SEとBGMのバス（Bus）を分離した構成を採用しています。本章では、これら2つのクラスを中心に、`webg` のサウンド機能の使い方とどのような設計思想で構築されているかを解説します。

ここで重要なのは、`webg` のサウンド機能が単なる「音出しヘルパー」ではないという点です。Web Audio APIの制約である `AudioContext` の遅延初期化への対応、SEとBGMのバス分離、ディレイとリバーブの独立した調整、そしてBGMのための簡易シーケンサまでを一貫したフローとして管理しています。「何を鳴らすか」というコンテンツ側の視点と、「どう聞かせるか」という音響設計側の視点を明確に分けて考える設計になっています。

利用方法を簡単に言うと、通常は高レイヤーである `GameAudioSynth` を介して操作することになります。まず `const synth = new GameAudioSynth()` でインスタンスを生成し、ユーザーがボタンを押すなどの操作を行ったタイミングで `await synth.resume()` を呼び出してオーディオコンテキストを有効化します。その上で、`setMasterVolume()`、`setSeVolume()`、`setBgmVolume()` で全体の音量バランスを整え、SEであれば `playSe(name)`、BGMであれば `setMelody(name)` と `startBgm()` を使用して再生します。さらに、空間的な演出を行うディレイとリバーブは、バスごとに個別に調整可能です。つまり、「まずオーディオコンテキストを利用可能にし、次に『SEかBGMか』を選択し、最後に『音量を変えるのか、音色特性を変えるのか』を決定する」というのが標準的なワークフローとなります。

## サウンド設計の考え方

ゲームやインタラクティブなアプリケーションにおいて、単にオシレーターを鳴らすだけでは不十分な場合が多くあります。短く鋭い効果音と、心地よくループするBGMでは、求められる音響特性が異なるためです。また、ディレイやリバーブの量によって空間感が大きく変わるため、音を鳴らしながらリアルタイムで調整できる柔軟性が求められます。

`webg` ではこの要求に応えるため、機能を `AudioSynth`（低レイヤー）と `GameAudioSynth`（高レイヤー）に分離しています。低レイヤー側はオーディオルーティングと時間制御という基盤機能に集中し、高レイヤー側はゲーム向けのプリセット名で直感的に扱えるインターフェースを提供します。これにより、音響的な構造と実際の利用方法が分離され、コードの可読性と保守性が向上しています。

特に実用面で重要なのが、SEとBGMを最初から別バスとして扱う点です。ゲーム演出において「SEだけを強調したい」「BGMを控えめにして環境音を際立たせたい」といった状況は頻繁に発生するため、この分離は不可欠です。

さらに、リバーブ（残響）の設計においては、「量」と「特性」を明確に区別しています。`setSeReverb()` や `setBgmReverb()` はセンド/リターンの量（どれだけリバーブ成分を混ぜるか）を調整するAPIであり、`setSeReverbImpulse()` や `setBgmReverbImpulse()` はコンボリューションリバーブ（Convolver）に用いるインパルスレスポンス（IR）の特性、つまり「部屋（room）」「ホール（hall）」「プレート（plate）」といった空間の質や長さを変更するAPIです。量と特性を分けて制御することで、音響設計の意図を正確にUIやパラメータに反映させることができます。

## Web Audio API から見た `AudioSynth` の位置付け

ここまでの説明だけでも `AudioSynth` と `GameAudioSynth` の役割分担は追えますが、なぜそのようなクラス分割が必要になるのかを理解するには、さらに一段低い層である Web Audio API 側の考え方を見ておくと分かりやすくなります。`webg` のサウンド機能の正体は、ブラウザ標準の Web Audio API です。そしてこの API を理解するうえで最も重要なのが、音の流れを「オーディオグラフ」として組み立てるという発想です。

Web Audio API では、音の生成から最終出力までを、ノードと呼ばれる小さな部品の接続として表現します。これは、ギターのエフェクターをケーブルでつないだり、ミキサーで信号の流れを切り替えたりする物理的な機材構成にかなり近い考え方です。基本形は、「音源」から始まり、「加工」や「分岐」を経由して、「出力」へ届く一本の流れです。

例えば、オシレーターで音を作り、その音量を調整し、最後にリバーブを通してスピーカーへ送るなら、概念的には次のような接続になります。

`[OscillatorNode]` → `[GainNode]` → `[ConvolverNode]` → `[AudioDestinationNode]`

ここでの各ノードは次のような役割を持ちます。

- `OscillatorNode` のようなソースノードは、波形そのものを発生させる入口です
- `GainNode`、`BiquadFilterNode`、`ConvolverNode` のような加工ノードは、音量、音色、残響といった性質を変えます
- `AudioDestinationNode` は、最終的にスピーカーやヘッドホンへ送る出口です

この仕組み自体は明快ですが、実際に JavaScript で書き始めると、どのノードをいつ作り、どの順番で接続し、どのノードを SE 用に使い、どのノードを BGM 用に共有するか、といった管理が急に複雑になります。しかも、ゲームやアプリケーションでは「SE だけ音量を上げたい」「BGM だけディレイを変えたい」「SE と BGM で異なるリバーブを持たせたい」といった要求が自然に出てきます。これを毎回 Web Audio API の生のノード接続だけで書くと、アプリ側のコードは配線処理で埋まりやすくなります。

ここで必要になるのが「バス」という考え方です。バスとは、複数の音をいったんまとめて流すための経路です。SE 用の音を SE バスへ集め、BGM 用の音を BGM バスへ集めておけば、各音を個別に触らなくても、経路の途中にある `GainNode` や `ConvolverNode` を調整するだけで、系統全体の音量や空間特性をまとめて変えられます。`webg` が SE バスと BGM バスを分けているのは、この「経路ごとにまとめて制御したい」という実用上の要求に対応するためです。

`AudioSynth` が行っていることは、まさにこの複雑な配線作業の自動化です。`ensureContext()` は、まだ `AudioContext` が存在しないときにコンテキストを生成し、マスター、SE、BGM という主要な経路を用意します。続く `buildFxChain()` では、それぞれのバスへディレイ、フィードバック、フィルター、コンボリューションリバーブを接続し、乾いた音とエフェクト音の流れを組み立てます。つまり後の「内部構造」節で出てくるこれらのメソッドは、抽象的な初期化処理ではなく、Web Audio API のオーディオグラフを構築している作業そのものです。

この視点で見ると、`AudioSynth` は低レイヤーな「配線ボード」や「ミキシングコンソール」に相当します。その上にある `GameAudioSynth` は、その配線済み基盤を前提として、「minor_drive という BGM を選ぶ」「coin という SE を鳴らす」といった論理的な名前で扱える層です。したがって、本章の流れは「ブラウザ API が持つ物理的な音の流れ」から始まり、それを扱いやすい基盤へまとめた `AudioSynth` があり、その上でゲーム開発向けの名前付き操作を提供する `GameAudioSynth` がある、という三段構成として理解すると全体像をつかみやすくなります。

## `AudioSynth` と `GameAudioSynth` の役割分担

サウンド機能の役割は、大きく分けて次の3つの層で構成されています。

1.  `AudioSynth` (低レイヤー/基盤層)
    Web Audio APIのルーティング構築、SEの発音、BGMシーケンサ、エンベロープ制御、ディレイ・リバーブの処理、およびインパルスレスポンスの動的生成を担当します。「どうやって音を出すか」という物理的な実装を担う層です。
2.  `GameAudioSynth` (高レイヤー/インターフェース層)
    `AudioSynth` の上に構築され、メロディプリセットやSEカタログ、ゲーム向けの便利な発音ヘルパーを提供します。「どんな名前の音を鳴らすか」という論理的な管理を担う層です。
3.  `webg/samples/sound` (検証層)
    これらの機能を実際のUIを通じて調整し、確認するためのサンプル実装です。

このような階層構造を採用している理由は3つあります。第一に、単発のSEと連続的なBGMでは、バス構成もエンベロープの挙動も時間の扱いも異なるため、同一のAPIで無理に扱うのではなく、役割を分けることでシンプルに保つためです。第二に、ディレイやリバーブの「量」と「特性」を分けて管理することで、音響的な比較検証を容易にするためです。第三に、コアとなる基盤機能と、ゲーム固有のプリセットを分離することで、開発者がコアコードを汚さずにゲーム独自の音色設定を管理できるようにするためです。

## 標準的な利用フロー

### オーディオコンテキストの有効化

サウンド実装で最も注意すべき点は、音声の再生は必ず「ユーザーの操作後」に行わなければならないということです。多くのモダンブラウザでは、ユーザーが操作していない状態で突然音が鳴ることを防ぐため、音声再生を制限するオートプレイポリシーが導入されています。

そのため、`AudioSynth` はコンストラクタで即座に `AudioContext` を生成しません。ユーザーの操作をきっかけに `resume()` が呼ばれたとき初めて `ensureContext()` が動作し、`AudioContext` の生成とともにマスターバス、SEバス、BGMバス、およびディレイ・リバーブのルーティングが構築されます。`webg/samples/sound` に「Audio Start」ボタンが用意されているのは、このブラウザ制約を適切に処理するためです。

### SEとBGMの使い分けと空間調整

音声機能が有効化した後は、SEとBGMを別々のバスとして制御します。特にBGMは単なるファイルの再生ではなく、`scheduleBgm()` による先読みスケジューリングで進行するため、BPMやメロディを変更すると楽曲全体の流れが動的に変化します。

空間演出を行うディレイとリバーブの調整では、「どれだけ効かせるか（量）」と「どのような空間にするか（特性）」を分けて考えます。`setSeReverb()` などの量調整APIと、`setSeReverbImpulse()` などの特性変更APIを使い分けることで、音量バランスを維持したまま空間の広がりだけを変えるといった精密な調整が可能になります。

## 使い方の基本例

### 効果音（SE）の再生

アプリケーションでは `GameAudioSynth` を使用するのが一般的です。最初の1音を鳴らすまでの最もシンプルな実装例を以下に示します。

```js
import GameAudioSynth from "./webg/GameAudioSynth.js";

const synth = new GameAudioSynth();

// ユーザー操作（クリックなど）をきっかけにオーディオを有効化して再生する
button.addEventListener("click", async () => {
  // AudioContextを有効化し、ルーティングを構築する
  await synth.resume();
  
  // 全体の音量を設定
  synth.setMasterVolume(0.25);
  
  // プリセット名でSEを再生
  synth.playSe("poyoon");
});
```

### 音量バランスの調整

マスターボリュームを基準とし、SEとBGMの相対的なバランスを個別に設定します。

```js
synth.setMasterVolume(0.25); // 全体の基準音量
synth.setSeVolume(0.90);    // SEの相対音量
synth.setBgmVolume(0.75);   // BGMの相対音量
```

### 空間エフェクト（ディレイ・リバーブ）の調整

ディレイとリバーブの「量」を調整します。これにより、音の奥行きや空間の広がりを制御します。

```js
// SEディレイ: 時間, フィードバック, ウェット量
synth.setSeDelay(0.11, 0.26, 0.22);
// BGMディレイ: 時間, フィードバック, ウェット量
synth.setBgmDelay(0.18, 0.22, 0.18);

// SEリバーブ: センド量, リターン量
synth.setSeReverb(0.32, 0.55);
// BGMリバーブ: センド量, リターン量
synth.setBgmReverb(0.28, 0.48);
```

### リバーブ特性の変更

インパルスレスポンス（IR）を変更して、空間の質（部屋の大きさや材質など）を切り替えます。

```js
// SEのリバーブ特性を「ルーム」に設定
synth.setSeReverbImpulse({
  kind: "room",
  durationSec: 2.2,
  decay: 1.6
});

// BGMのリバーブ特性を「ホール」に設定
synth.setBgmReverbImpulse({
  kind: "hall",
  durationSec: 4.0,
  decay: 1.9
});
```

### エンベロープの調整

音の立ち上がり（アタック）や減衰（ディケイ）を調整します。SEは輪郭をはっきりさせ、BGMは持続感を出すといった使い分けが可能です。

```js
// SE用のエンベローププリセットを適用
synth.setSeEnvelopePreset("soft", {
  attack: 0.005,
  decay: 0.05,
  sustain: 0.55,
  release: 0.08
});

// BGM用のエンベロープを個別に設定
synth.setBgmEnvelope({
  attack: 0.03,
  decay: 0.20,
  sustain: 0.60,
  release: 0.40
});
```

### BGMプリセットの切り替え

メロディ名で直接切り替える方法と、ヘルパー関数を用いてBPMやエンベロープをまとめて適用する方法があります。

```js
import { applyGameBgmPreset } from "../GameAudioPresets.js";

// メロディプリセットを直接切り替えて開始
synth.setMelody("minor_drive");
synth.startBgm();

// 場面に応じたプリセット（BPMや音色設定を含む）を一括適用
applyGameBgmPreset(synth, "boss");
```

## 独自BGMをアプリ側で追加する

`GameAudioSynth` は既存プリセットを切り替えるだけのクラスではありません。アプリケーション側から `registerMelody()` を呼び出せば、ライブラリ本体を書き換えずに独自のBGMを追加できます。既存プリセットは「命名済みのテンプレート」であり、実際の入口は `AudioSynth` に実装されているメロディ登録機構です。したがって、ゲームごとのステージ曲、ボス戦専用の短いループ、会話中だけ使う静かな反復なども、アプリ側のコードだけで定義できます。

`webg` のBGMはオーディオファイルの再生ではなく、16ステップを1小節とする簡易シーケンサで組み立てられます。各ステップは8分音符単位で進み、ベースとリードが別々に予約再生されます。このため、「どのステップで鳴らすか」「どの程度音価を伸ばすか」「どの程度休符を混ぜるか」を数値として持つと、少ないデータ量でもゲーム向けのループを作りやすくなります。

以下は、アプリ側で `story_field` という新しいBGMを追加して使う例です。

```js
import GameAudioSynth from "./webg/GameAudioSynth.js";

const synth = new GameAudioSynth();

async function startStoryFieldBgm() {
  await synth.resume();

  synth.registerMelody("story_field", {
    scale: [0, 2, 4, 5, 7, 9, 11],
    bassPattern: [0, -5, -3, -2, 0, -4, -2, -5],
    bassOctave: -12,
    leadDegrees: [
      0, 2, 4, 5, 7, 5, 4, 2,
      0, 2, 4, 7, 9, 7, 5, 4,
      0, 2, 4, 5, 7, 9, 7, 5,
      4, 5, 7, 9, 11, 9, 7, 5
    ],
    leadHoldSteps: [
      1.0, 1.0, 1.0, 1.6, 1.0, 1.0, 1.0, 2.0,
      1.0, 1.0, 1.0, 1.7, 1.0, 1.0, 1.0, 2.4,
      1.0, 1.0, 1.0, 1.6, 1.0, 1.0, 1.0, 2.0,
      1.0, 1.0, 1.0, 1.8, 1.0, 1.0, 1.0, 2.8
    ],
    leadOctave: 12,
    leadType: "triangle",
    leadDur: 0.11,
    leadGain: 0.060,
    bassType: "triangle",
    bassDur: 0.16,
    bassGain: 0.074,
    rhythmDropWeakProb: 0.28,
    rhythmDropTailProb: 0.52
  });

  synth.setBpm(116);
  synth.setRootHz(220.0);
  synth.setBgmEnvelope({
    attack: 0.03,
    decay: 0.18,
    sustain: 0.60,
    release: 0.36
  });
  synth.setMelody("story_field");
  synth.startBgm();
}
```

この例では、`leadDegrees` を32要素にして2小節以上のフレーズを持たせています。`scheduleBgmStep()` 自体は1小節ごとに 0 から 15 のステップ番号を繰り返しますが、`leadDegrees[step % length]` という形で参照されるため、配列を長くすれば小節をまたいだ旋律を作れます。同様に、`bassPattern` は4要素でも動作しますが、8要素へ伸ばすと2小節で低音進行に差を付けやすくなります。

## BGMパラメータの意味

独自BGMを作る際は、単に数値を並べるのではなく、「どのパラメータが何を決めているか」を理解しておくと調整が速くなります。`registerMelody()` へ渡す主な項目は次のような役割を持ちます。

- `scale`: 度数を半音へ変換する基準です。`[0, 2, 3, 5, 7, 8, 10]` ならマイナー系、`[0, 2, 4, 5, 7, 9, 11]` ならメジャー系に近い響きになります
- `bassPattern`: ベースの進行です。1要素が4ステップぶん使われるため、4要素で1小節、8要素で2小節ぶんの低音進行になります。負の値も使えます
- `bassOctave`: ベース全体を何半音ずらすかを指定します。`-12` なら1オクターブ下です
- `leadDegrees`: リード旋律を明示的に書く配列です。`null` を置くとそのステップは休符になります
- `leadGate` と `leadDegreeStep`: `leadDegrees` を書かず、一定規則で鳴らしたいときの簡易指定です。短いループを素早く作るには便利ですが、細かい旋律を作るなら `leadDegrees` の方が扱いやすくなります
- `leadHoldSteps`: 各ステップの音価倍率です。`1.0` は基本長、`2.0` は約2倍の長さとして扱われ、フレーズ終端や強拍を伸ばすのに使います
- `leadOctave`: リード全体の高さです。既定プリセットでは `12` が多く、ルート音より1オクターブ上で旋律を鳴らしています
- `leadType` と `bassType`: オシレーター波形です。音色の印象を大きく左右します
- `leadDur` と `bassDur`: 各音の基本長です。`leadHoldSteps` を使う場合も、ここが最低値になります
- `leadGain` と `bassGain`: 各パートの音量です。BGMバス全体の音量とは別に、旋律と低音のバランスを整えるために使います
- `rhythmDropWeakProb`: 弱拍を休符にする確率です。高くすると空きが増え、軽いグルーブになります
- `rhythmDropTailProb`: 小節末尾寄りのステップを休符にする確率です。高くすると息継ぎ感が強くなります

重要なのは、`registerMelody()` では `leadHoldSteps`、`rhythmDropWeakProb`、`rhythmDropTailProb` が必須であることです。`webg` はこの部分を自動補正しないため、メロディ定義が足りない場合はその場で例外になります。これは、不完全なデータのまま曖昧に再生されるより、定義ミスを早く見つけるためです。

## 波形の選び方と音色の調整

`webg` のサウンド生成は `OscillatorNode` を使っているため、音色の印象はまず波形選択で決まります。既存プリセットでも `sine`、`square`、`sawtooth`、`triangle` を使い分けていますが、これは単なる好みではなく、役割に応じた使い分けです。

- `sine`: 倍音が少なく、丸く柔らかい音になります。余韻確認用、静かなBGM、背景的なSEに向きます
- `triangle`: `sine` より少し芯がありつつ、まだ柔らかい音です。汎用的で、BGMリードや軽いUI音に使いやすい波形です
- `square`: 輪郭が強く、レトロゲームらしい主張が出やすい波形です。メロディを前に出したいときや、短い通知音に向きます
- `sawtooth`: 倍音が多く、強く荒い印象になります。警告音、ボス戦BGM、派手なSEに向いています

波形の変更は、BGMなら `leadType` や `bassType`、SEなら `playTone()` や `playGameTone()` の `options.type` で行います。例えば同じ旋律でも、`triangle` から `square` へ変えるだけで「背景のBGM」から「主役のBGM」へ聞こえ方が大きく変わります。逆に、派手すぎると感じるSEは `sawtooth` から `triangle` や `sine` へ落とすだけで収まりやすくなります。

## エンベロープと主要パラメータの見方

サウンドを「短く切る」「余韻を残す」といった性格は、波形だけでなくエンベロープと各種パラメータで決まります。`AudioSynth` では SE 用と BGM 用で別々にエンベロープを持っており、意味は共通しています。

- `attack`: 音量が 0 からピークへ達するまでの時間です。短いほど鋭く、長いほど柔らかく立ち上がります
- `decay`: ピークから持続音量まで落ちる時間です。短いと歯切れがよく、長いと胴鳴りが残ります
- `sustain`: 減衰後に維持する音量比です。`0.7` ならピークの70%程度を保ちます
- `release`: ノート終了後に 0 へ消えるまでの時間です。短いと切れ味が出て、長いと余韻が増えます
- `gain`: その音そのものの音量です。ミックス全体の調整とは分けて考えます
- `dur`: ノートを鳴らす基本時間です。SEでは単発の長さ、BGMでは1音の最低長として効きます
- `pan`: 左右定位です。`-1` が左、`1` が右です。SEに少しだけ振ると、画面上の動きに対応させやすくなります
- `detune`: 半音ではなくセント単位の微調整です。数値を少しだけずらすと、同じ音を重ねても硬すぎない響きになります
- `when`: 現在時刻からどれだけ後に鳴らすかではなく、「オーディオコンテキスト上の絶対時刻」を渡します。複数音を少しずつ遅らせて重ねるときに使います

ディレイとリバーブにも役割の違いがあります。`setSeDelay(timeSec, feedback, wet)` と `setBgmDelay(timeSec, feedback, wet)` は反復の速さと量を決め、`setSeReverb(send, returnGain)` と `setBgmReverb(send, returnGain)` は残響成分の送り量と戻り量を決めます。さらに `setSeReverbImpulse()` と `setBgmReverbImpulse()` は、room、hall、plate の空間特性自体を切り替えるAPIです。量と特性を分けて考えると、音量を大きくせずに広がりだけを変える、といった調整がしやすくなります。

## 独自の効果音をアプリ側で作る

独自SEを使いたい場合も、必ずしも `GameAudioSynth` 本体へ新しいプリセット名を追加する必要はありません。アプリ側に「自分のゲームだけで使う音」を関数として持たせ、`playTone()` または `playGameTone()` を組み合わせれば十分です。これは特に、UI用の短い通知音や、特定の1シーンだけで使う演出音で有効です。

`playTone()` は最も低い入口で、波形、音量、エンベロープ、定位をすべて個別に指定できます。`playGameTone()` はその上にあり、まず `soft` や `pluck` のようなSEエンベローププリセットを選び、その上で必要な項目だけ上書きするためのヘルパーです。アプリ側で短時間に何種類も試したいときは `playGameTone()` が扱いやすく、完全に独自のエンベロープを作りたいときは `playTone()` が向いています。

以下は、宝箱が開いたときの音をアプリ側で独自に作る例です。

```js
import GameAudioSynth from "./webg/GameAudioSynth.js";

const synth = new GameAudioSynth();

function playTreasureOpen() {
  const t0 = synth.ctx.currentTime;

  // 最初の明るい立ち上がり
  synth.playGameTone(523.25, 0.08, "pluck", {
    when: t0,
    type: "triangle",
    gain: 0.07,
    release: 0.06,
    pan: -0.08
  });

  // 少し遅れて和音感を足す
  synth.playGameTone(783.99, 0.12, "soft", {
    when: t0 + 0.040,
    type: "triangle",
    gain: 0.06,
    release: 0.10,
    pan: 0.06
  });

  // 高域のきらめき
  synth.playGameTone(1174.66, 0.16, "soft", {
    when: t0 + 0.090,
    type: "sine",
    gain: 0.04,
    release: 0.16,
    pan: 0.20
  });
}

button.addEventListener("click", async () => {
  await synth.resume();
  playTreasureOpen();
});
```

このように、1つのSEは1音だけでなく、数十ミリ秒ずらした複数の音の重ね合わせとして作ると、単純なビープ音になりにくくなります。既存の `GameAudioSynth.gameSe` に入っている `coin`、`levelup`、`tail_probe` なども同じ考え方で構成されています。短いアタック音で輪郭を作り、その後に少し長い音を重ねると、エンベロープやリバーブの違いも聞き取りやすくなります。

## 独自SEを名前付きで管理する方法

アプリケーションが大きくなると、関数を直接呼ぶだけでは管理しにくくなります。その場合は、アプリ側に SE カタログを持たせると整理しやすくなります。考え方は `GameAudioSynth.playSe(name)` と同じですが、名前解決の表をアプリ側へ置きます。

```js
const customSe = {
  treasure_open: () => playTreasureOpen(),
  portal_start: () => playPortalStart(),
  dialogue_tick: () => playDialogueTick()
};

function playAppSe(name) {
  const se = customSe[name];
  if (!se) {
    throw new Error(`Unknown app sound effect: ${name}`);
  }
  se();
}
```

この方式なら、ライブラリ本体を変更せずに、ゲームごとの固有SEを整理できます。`webg` 本体に追加すべきなのは「多くのサンプルや複数アプリで共通に再利用される音」に限り、個別作品専用のSEはアプリ側に置く方が保守しやすくなります。

### ゲームイベントに応じた通知音の管理

「クリア」「失敗」などのイベント音は、SE名との対応表をアプリケーション側で管理することで、ゲームロジックから直感的に呼び出すことができます。

```js
import { playGameEventSound } from "../GameAudioPresets.js";

// イベント名でサウンドを再生
playGameEventSound(synth, "clear");
playGameEventSound(synth, "fail");
```

## `AudioSynth` の内部構造

### バス構成とルーティング

`AudioSynth` は音響基盤としての役割を担います。具体的には、`ensureContext()` と `resume()` による `AudioContext` の遅延初期化から始まり、マスターバス $\rightarrow$ SE/BGMバス $\rightarrow$ 各エフェクトチェーンというルーティングを構築します。また、`playTone()` や `playSe()` による単発音の生成、メロディデータに基づいたBGMの先読みスケジューリング、インパルスレスポンスの手続き的生成までをすべて管理しています。リバーブの特性決定ロジックもこのクラスに集約されており、`AudioSynth` が音響的な「物理層」として機能しています。

### インパルスレスポンスの動的生成

`AudioSynth.createImpulseResponse()` は、外部の音声ファイルを読み込むのではなく、数学的な計算によってステレオの `AudioBuffer` を生成します。

1.  プロファイルの取得: `getImpulseProfile(kind)` により、「room」「hall」「plate」それぞれの特性（反射の密度や減衰率）を決定します。
2.  テールの書き込み: `writeImpulseTail()` で、時間の経過とともに高域が減衰するノイズ列を生成し、残響の尾（テール）を作ります。
3.  初期反射の追加: `addEarlyReflections()` で、左右のチャンネルに時間差を持たせた初期反射を加えます。これが空間の「広さ」や「方向感」を決定する主要な要素となります。
4.  正規化: `normalizeImpulse()` でピーク音量を揃え、特性を切り替えても極端な音量差が出ないように調整します。

この手法により、外部アセットなしで多様な空間特性をシミュレートでき、特性の比較検証が容易になります。

### BGMシーケンサの仕組み

`webg` のBGMは、オーディオファイルの単純再生ではなく、スコア（楽譜）に基づいた予約再生方式を採用しています。`startBgm()` が処理を開始し、`scheduleBgm(lookAheadSec)` が少し先の未来までのノートを予約し、`scheduleBgmStep(step, when)` がベースとリードの音を組み立てます。

メロディプリセットは、単なる音階の並びではなく、スケール、ベースパターン、リードのゲート時間、音色タイプなどのパラメータセットとして定義されています。特にリードの音域を広く取り、ベースパターンを多ステップにすることで、単純な反復に聞こえない音楽的な展開を実現しています。

## `GameAudioSynth` の役割

`GameAudioSynth` は `AudioSynth` の高レイヤーラッパーとして、ゲーム開発者が使いやすいプリセット管理機能を提供します。具体的には、BGMメロディの登録、SEカタログの管理、ゲームイベント向けの音出しヘルパーなどの実装を担います。アプリケーション側からサウンドを制御する場合、原則としてこのクラスをエントリーポイントとして利用します。

## `webg/samples/sound` による検証

`webg/samples/sound` は、音響設計と確認を手早く繰り返すための専用ツールです。ここでは `WebgApp` を介さず、`GameAudioSynth` と `AudioSynth` の調整パラメータを直接操作できるインターフェイスになっています。

UIは「System（全体設定）」「Sound Effects（SE調整）」「Background Music（BGM調整）」の3つのブロックで構成されており、音量、ディレイ、リバーブ、エンベロープをリアルタイムに変更しながら聞き比べることができます。

特に `tail_probe` という検証用SEは、リバーブとエンベロープの差を明確に聞き分けるために設計されています。冒頭の短いアタック音で立ち上がりを、その後の持続音でディケイやリバーブのテールを追跡できるため、音響調整の基準音として活用してください。

## サウンド機能の拡張とカスタマイズ

サウンド機能を拡張したり、内部構造を変更したりする場合は、以下の関連箇所をセットで確認することをおすすめします。

- ルーティングの変更: `ensureContext()` および `buildFxChain()` を確認してください。
- リバーブ特性の変更: `getImpulseProfile()`、`writeImpulseTail()`、`addEarlyReflections()` のロジックを確認してください。
- BGMの挙動変更: `startBgm()`、`scheduleBgm()`、`scheduleBgmStep()` のスケジューリング処理を確認してください。
- プリセットの追加: `GameAudioSynth` および `samples/GameAudioPresets.js` に定義を追加し、それを利用するUI側を修正してください。
- サンプルのUI修正: `webg/samples/sound/main.js` および関連するドキュメント（`sound.txt` 等）を更新してください。

特にインパルスレスポンスをカスタマイズする際は、「エフェクトの量（センド量）」と「空間の特性（IR）」の制御を混同させないように設計することで、直感的に調整する環境を維持できます。
