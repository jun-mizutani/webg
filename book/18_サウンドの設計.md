# サウンドの設計

サウンド機能は、単に効果音を 1 回鳴らせれば十分というものではありません。SE と BGM では扱い方が異なり、音量、ディレイ、リバーブ、メロディの管理も別々に考えたくなります。`webg` では、この違いを後付けで吸収するのではなく、最初から `AudioSynth` と `GameAudioSynth` に役割を分け、SE / BGM のバスも分離した構成で扱います。本章では、`AudioSynth` と `GameAudioSynth` を中心に、`webg` のサウンド機能がどのような設計で組まれているかを見ていきます。 

ここで重要なのは、`webg` のサウンドが「効果音を 1 回鳴らすヘルパー」ではないことです。`AudioContext` の遅延初期化、SE と BGM のバス分離、ディレイとリバーブの独立した調整、BGM の簡易シーケンサを 1 つの流れとして扱います。音の調整を追いやすくするために、「何を鳴らすか」と「どう聞かせるか」を分けて考える設計になっています。 

最初に結論だけを言うと、通常の利用では `GameAudioSynth` から入るのが自然です。まず `const synth = new GameAudioSynth()` を作り、利用者がボタンを押すなどの操作を行ったあとで `await synth.resume()` を呼びます。そのうえで `setMasterVolume()`、`setSeVolume()`、`setBgmVolume()` で音量バランスを整え、SE なら `playSe(name)`、BGM なら `setMelody(name)` と `startBgm()` を使います。さらに、ディレイとリバーブはバスごとに分けて調整します。つまり、まず `AudioContext` を利用可能にし、そのあと「SE か BGM か」「量を変えたいのか、特性を変えたいのか」を分けて触るのが標準です。 

## `webg` のサウンドをどう分けて考えるか

ゲームやサンプルのサウンドでは、単にオシレーターを鳴らせば十分ということは多くありません。短い効果音とループする BGM では求める性格が異なりますし、ディレイやリバーブの量と空間感も別々に調整したくなります。さらに、同じ素材でもエンベロープやリバーブ設定で印象が大きく変わるため、音を鳴らしながらその場で調整できることが重要になります。 

`webg` のサウンド機能は、そのために `AudioSynth` と `GameAudioSynth` を分けています。基盤側は配線と時間制御に集中し、上位層はゲーム向けのプリセット名で扱えるようにすることで、音の構造と実際の使い方を分け、可読性を高めています。加えて、SE と BGM は最初から別バスとして扱われます。ゲームでは、SE だけを強調したい場面、BGM だけを控えめにしたい場面があるため、この分離は実用上かなり重要です。 

さらに大事なのは、リバーブの「量」と「性格」を別に考えることです。`setSeReverb()` / `setBgmReverb()` はセンド / リターンの量を変える API であり、`setSeReverbImpulse()` / `setBgmReverbImpulse()` は Convolver のインパルスレスポンス、つまり `room` / `hall` / `plate` の性格と長さを変える API です。量と性格を分けて考えると、音設計の説明と UI の操作が一致しやすくなります。 

## `AudioSynth` と `GameAudioSynth`

サウンド関連の役割は、次の 3 つの層に分けると分かりやすくなります。`AudioSynth` は Web Audio API の配線、SE 発音、BGM シーケンサ、エンベロープ、ディレイ、リバーブ、インパルスレスポンス生成を担当します。`GameAudioSynth` は、その上にメロディプリセット、SE カタログ、ゲーム向け発音ヘルパーを載せる上位層です。さらに `webg/samples/sound` は、これらを実際の UI を通して確認するためのサンプルです。ここで重要なのは、`AudioSynth` が「どう鳴らすか」の層であり、`GameAudioSynth` が「どんな名前で何を鳴らすか」の層だという点です。 

この構成を取る理由は 3 つあります。まず、SE と BGM を同じ API で無理に扱わないためです。単発の SE と連続する BGM では、バスもエンベロープも時間の扱いも異なります。次に、ディレイとリバーブの量と性格を分けて説明しやすくするためです。音量調整だけでなく、空間の特性そのものも扱えるようにしておくと、サンプルの比較がしやすくなります。さらに、ゲーム側の実装では「コアが用意する基盤」と「そのゲーム固有のプリセット」を分けたほうが、役割の切り分けが分かりやすくなります。共通部分は `GameAudioSynth` に寄せ、場面ごとの BGM プリセットや通知音の対応はサンプル側で持つと、あとから調整範囲を追いやすくなります。 

## 標準フロー

### 最初に `resume()` する

サウンドで最初に押さえるべきなのは、音の開始は利用者が画面を操作した後に行う、ということです。`AudioSynth` はコンストラクタで `AudioContext` を生成しません。ブラウザーには、利用者が何も操作していない状態で突然音が鳴るのを防ぐため、音声再生の開始を制限する仕組みがあります。`resume()` が呼ばれて初めて `ensureContext()` が `AudioContext` を生成し、マスター、SE バス、BGM バス、ディレイ、リバーブの経路を組みます。`webg/samples/sound` に `Audio Start` ボタンがあるのは、この制約をそのまま表に出しているためです。 

### SE と BGM を分けて扱う

サウンドを使うときに先に意識しておきたいのは、SE と BGM は別バスであり、音量や FX 設定もバスごとに変える前提だという点です。`resume()` 前は `AudioContext` が作成されていないので、まず利用者の操作をきっかけに音声機能を有効化する必要があります。さらに、BGM は単発再生ではなく `scheduleBgm()` の先読みで進行するため、BPM やメロディを変えると流れ全体が変わります。`webg/samples/sound` は `WebgApp` を使わず、音の UI を直接組んでいるサンプルなので、他のサンプルと入口が少し異なる点も意識しておくほうが安全です。 

### ディレイとリバーブを調整する

ディレイとリバーブの調整では、「どれだけ効かせるか」と「どんな空間にするか」を分けて考えます。`setSeReverb()` / `setBgmReverb()` は量、`setSeReverbImpulse()` / `setBgmReverbImpulse()` は IR の性格変更です。この分離があるため、音量バランスと空間特性を別の軸として扱えます。サンプルで比較聴取するときも、この分け方を意識すると違いを追いやすくなります。 

## 使い方の基本例

### 最初の SE を鳴らす

通常の利用におけるアプリやサンプルでは、`GameAudioSynth` を直接使うのが適切です。最初の 1 音を出すまでの流れとしては、次の形がもっとも素直です。 

```js
import GameAudioSynth from "./webg/GameAudioSynth.js";

const synth = new GameAudioSynth();

button.addEventListener("click", async () => {
  await synth.resume();
  synth.setMasterVolume(0.25);
  synth.playSe("poyoon");
});
```

最小例だけ抜き出すなら、次のようにも書けます。ここで重要なのは、`resume()` が利用者の操作の後に呼ばれることです。コンストラクタだけでは、まだ音は出力されません。 

```js
const synth = new GameAudioSynth();

await synth.resume();
synth.playSe("poyoon");
```

### SE と BGM の音量を分ける

SE と BGM のバランスは、最初にバスで分けておくほうが扱いやすくなります。マスターは全体の基準、SE / BGM の音量は相対バランスという役割で見ると整理しやすくなります。 

```js
synth.setMasterVolume(0.25);
synth.setSeVolume(0.90);
synth.setBgmVolume(0.75);
```

### ディレイとリバーブの量を調整する

ディレイとリバーブの「量」を変える API です。`setSeDelay()` / `setBgmDelay()` はディレイ時間、フィードバック、ウェット量を変えます。`setSeReverb()` / `setBgmReverb()` はセンド / リターンの量を変えます。ここは「空間の量」の調整です。 

```js
synth.setSeDelay(0.11, 0.26, 0.22);
synth.setBgmDelay(0.18, 0.22, 0.18);

synth.setSeReverb(0.32, 0.55);
synth.setBgmReverb(0.28, 0.48);
```

### リバーブの性格を変える

同じリバーブ量でも、IR の性格を変えると聞こえ方は大きく変わります。ここで変わるのはセンド / リターンの量ではなく、ConvolverNode に入るインパルスレスポンスそのものです。UI 上で言えば、`Reverb Kind`、`Reverb Length`、`Reverb Decay` がこの層に対応します。 

```js
synth.setSeReverbImpulse({
  kind: "room",
  durationSec: 2.2,
  decay: 1.6
});

synth.setBgmReverbImpulse({
  kind: "hall",
  durationSec: 4.0,
  decay: 1.9
});
```

### エンベロープを調整する

SE は単発音の輪郭が重要で、BGM は持続感が重要なので、エンベロープを別々に持っているほうが自然です。 

```js
synth.setSeEnvelopePreset("soft", {
  attack: 0.005,
  decay: 0.05,
  sustain: 0.55,
  release: 0.08
});

synth.setBgmEnvelope({
  attack: 0.03,
  decay: 0.20,
  sustain: 0.60,
  release: 0.40
});
```

### メロディプリセットと BGM プリセットを切り替える

BGM を melody 名とサンプル側のプリセットで扱う例です。`setMelody()` はメロディプリセットを直接切り替える API です。場面向けのまとまりを使いたい場合は、`samples/GameAudioPresets.js` のようなヘルパーで melody、BPM、root、BGM envelope をまとめて適用すると、コア側の担当を増やさずに済みます。

```js
import { applyGameBgmPreset } from "../GameAudioPresets.js";

synth.setMelody("minor_drive");
synth.startBgm();

applyGameBgmPreset(synth, "boss");
```

### 通知音をサンプル側でまとめる

ゲーム側で「開始」「停止」「クリア」などの通知音をまとめたいときは、SE 名の対応表をサンプル側に置くほうが追跡しやすくなります。ここでは「どの波形で何 Hz を鳴らすか」ではなく、「どのゲーム上の出来事を鳴らすか」で選べます。通知音の命名や割り当てはゲームごとに変わりやすいので、コアではなくサンプル / アプリ側に置くのが自然です。 

```js
import { playGameEventSound } from "../GameAudioPresets.js";

playGameEventSound(synth, "clear");
playGameEventSound(synth, "fail");
```

## `AudioSynth` の内部

### バス構成

`AudioSynth` は音の基盤を担当します。現在の実装では、`ensureContext()` と `resume()` による `AudioContext` の遅延初期化、マスター / SE バス / BGM バスの構築、SE / BGM それぞれのディレイ / リバーブチェーンの構築、`playTone()` と簡易 `playSe()` による単発音の発音、メロディを使った BGM の先読みスケジューリング、エンベローププリセットの保持、インパルスレスポンスの手続き生成までを引き受けています。IR 生成も別クラスではなく `AudioSynth.js` の中に入っているため、現在の `webg` ではリバーブの特性もこのクラスの役割として読むほうが実装に合っています。 

### インパルスレスポンスの作り方

`AudioSynth.createImpulseResponse(ctx, durationSec, decay, options)` は、録音済みのインパルスファイルを読むのではなく、手続き的にステレオ `AudioBuffer` を生成します。流れとしては、まず `getImpulseProfile(kind)` で `room` / `hall` / `plate` ごとのプロファイルを取り、左右 2 チャネルの `AudioBuffer` を作り、`writeImpulseTail()` で時間が進むほどスムージングを強めた減衰ノイズ列を書き込み、`addEarlyReflections()` で数本の初期反射を左右別タイミングで加算し、最後に `normalizeImpulse()` でピークをそろえます。この構成にしている理由は、外部アセットなしで `room` / `hall` / `plate` の性格差を比較できるようにするためです。 

`getImpulseProfile(kind)` は、`room`、`hall`、`plate` ごとに `alphaStart`、`alphaEnd`、`reflectionBurstMs`、`earlyReflections` を返します。`room` は短く近い反射、`hall` は広く長いテール、`plate` は明るく密な印象を意識したプロファイルです。ここで決めているのは単なる長さではなく「反射の性格」です。`writeImpulseTail()` は、単純なホワイトノイズ減衰ではなく、`alphaStart` から `alphaEnd` へ変化するスムージングを入れることで、テール後半の高域が少しずつ落ちやすいノイズ列を作ります。`addEarlyReflections()` は、左右チャンネルに時間差を持たせた初期反射を加え、`room` / `hall` / `plate` の聞こえ方の違いを作る主要な要素になっています。最後の `normalizeImpulse()` は、初期反射で上がりやすいピークをそろえ、`kind` を切り替えても極端な音量差になりにくくします。 

現行実装の既定値では、SE リバーブインパルスは `kind = "hall"`、`durationSec = 3.2`、`decay = 1.8`、BGM リバーブインパルスは `kind = "hall"`、`durationSec = 4.0`、`decay = 1.9` です。SE より BGM を少し長くしているのは、フレーズの尾を聞き取りやすくするためです。 

### BGM シーケンサの仕組み

`AudioSynth` の BGM は、単に 1 曲のオーディオファイルを再生するのではなく、`scheduleBgm()` と `scheduleBgmStep()` で少し先の音を予約しながら進行します。基本の流れは、`startBgm()` がスケジューリング処理を開始し、`scheduleBgm(lookAheadSec)` が少し先までのノートを予約し、`scheduleBgmStep(step, when)` がベースとリードを組み立て、最後に `stopBgm(fadeSec)` がフェードアウトして止める、というものです。この設計にしている理由は、BGM を「再生ファイル」ではなく「時系列で進むスコア」として扱うためです。 

メロディプリセットは、単なる音の並びではなく、小さなスコア断片として読むと分かりやすくなります。`scale`、`bassPattern`、`leadDegrees` または `leadGate`、`leadHoldSteps`、`leadDur`、`bassDur`、`leadType`、`bassType`、`leadGain`、`bassGain` などがまとまっており、特に `leadDegrees` を長く取り、`bassPattern` を 8 ステップ以上にすると、同じ 1 小節の反復に聞こえにくくなります。現在の `GameAudioSynth` は、この方針に沿って `minor_drive`、`bright_arp`、`hero_run`、`night_drive` などを定義しています。 

## `GameAudioSynth` の役割

`GameAudioSynth` は `AudioSynth` の上位層として、ゲーム向けのプリセットを格納しています。現在の実装では、主に `installMelodyPresets()` による BGM メロディプリセット登録、`installSePresets()` による SE カタログと実体の登録、`playGameTone()` によるゲーム向け Tone ヘルパー、`playSe()` / `playGameSe()` による SE 名の厳密解決、`getSoundEffectList()` / `getSoundEffectInfo()` による catalog と profile 情報の取得を担当します。通常の利用では、まずこちらから入るのが適切です。 

## `webg/samples/sound` の見方

`webg/samples/sound` は、音設計と確認を同じ画面で繰り返すサンプルです。ここでは `WebgApp` を使わず、`GameAudioSynth` と `AudioSynth` の調整 UI を直接組み合わせています。画面は大きく `System`、`Sound Effects`、`Background Music` の 3 つのブロックに分かれます。`System` は `Audio Start` と `Master Vol`、`Sound Effects` は SE の選択、連続比較、SE エンベロープ、SE ディレイ / リバーブ、`Background Music` は BGM の開始 / 停止、BPM、メロディ、BGM エンベロープ、BGM ディレイ / リバーブを扱います。このサンプルを使うときは、まず `Audio Start`、次に `Play SE` や `Audition All`、最後に `BGM Start` という順で見ると追跡しやすくなります。 

`tail_probe` は、リバーブとエンベロープの差を聞き分けやすくする確認用 SE です。冒頭の短い輪郭音でアタックを、その後の少し長い胴鳴りでディケイ / サステイン / リリースとリバーブテールを追跡しやすくしています。違いが分かりにくいときは、まずこれを選ぶと分かりやすくなります。さらに、このサンプルは 3D 表示や WebGPU diagnostics を持たないため、現在は音調整に必要な UI だけへ絞っています。重視しているのは、SE / BGM の素材選択、音量、ディレイ、リバーブ、エンベロープを短い操作で往復しながら聞き比べられることです。サンプル側の独自 BGM プリセットヘルパーを置かず、コアに含まれる `GameAudioSynth` と `AudioSynth` の確認に絞っている点も、この簡潔さに含まれます。 

## 変更時の注意点

サウンド周りを変更するときは、1 か所だけを見るとずれが生じやすいので、少なくとも次の組を一緒に確認するほうが安全です。バス構成を変えるなら `ensureContext()` と `buildFxChain()`、リバーブの特性を変えるなら `getImpulseProfile()`、`writeImpulseTail()`、`addEarlyReflections()`、BGM の流れを変えるなら `startBgm()`、`scheduleBgm()`、`scheduleBgmStep()`、サンプル / アプリ側のプリセットヘルパーを増やすなら `GameAudioSynth`、`samples/GameAudioPresets.js`、そのヘルパーを使うサンプル / アプリの UI と文書、sound サンプルの UI や説明を変えるなら `webg/samples/sound/main.js`、`webg/samples/sound/sound.txt`、`webg/samples/index.html` をまとめて見る、という形です。特に、インパルスレスポンスの仕組みを変える場合は、「量を変える API」と「特性を変える API」を混ぜないほうが安全です。
